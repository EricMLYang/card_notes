### AI 編程工具的上下文管理
https://www.youtube.com/watch?v=rmvDxxNubIg
一個很棒的演講，講的是 AI 編程工具的上下文管理。講者的核心觀點：不要把思考外包給 LLM。
為什麼這很重要？
講者提出「Dumb Zone」的概念。當上下文視窗填滿約 40%，AI 回應品質就開始下降。塞太多資訊，就會產生「AI Slop」。
解決方案是持續壓縮和審查：
- 研究系統運作，找出相關檔案
- 規劃具體步驟，包含程式碼片段
- 實作時保持上下文精簡
實際工作流程：
講者稱之為「刻意壓縮」。用 Claude Code 時，把進度壓縮成 markdown 檔。研究結果放進 research.md，實作步驟放進 plan.md，包含程式碼和行號。這樣新的上下文視窗能立即接手，不用重新理解整個程式庫。
關鍵洞察：子代理不是用來模仿團隊角色的。它們是用來控制上下文。讓子代理去探索程式庫，完成後回傳精簡摘要給父代理。父代理保持輕量專注。
用 Claude Code 時，你要自己審查計劃檔、驗證內容、決定何時壓縮。好的計劃要能讓你預測會有什麼改動。

### nthropic 有專門寫一篇文章來談
l.threads.com/?u=https%3A%2F%2Fwww.anthropic.com%2Fengineering%2Feffective-harnesses-for-long-running-agents&e=AT2diXNiTWJ4bunbvrLv6eb2mBu2fdu_1dS-LTL9RrQRUdFtvHRI5NXl6MOwHqwAcmaOw1BYl8HbK9vP_o2tDebWXnpszef1eB27iQneUbMxNlW-a8efELrza3xOduVbOf7J2yprUNxV
簡單地說，AI 寫程式就跟工程師交接是一樣的。每個來交接的工程師都是白紙，所以上一個工程師必須留下足夠的 artifacts，下一個工程師才能無縫接上。在 AI coding 上如果要確保這件事，那就必須讓任務可以在一個 context window 裡面解決，否則半成品的工作，即使經過 compaction，是無法交接的。
我自己工作也是類似這個 pattern，我一定會把每一個任務拆解成一個 storypoint 就能完成。這樣基本上永遠不會超出 context windows，也就不需要壓縮。工作完成之後，就更新 ticket，commit，然後重開 session 進行下一個 ticket。
Overarchiing view 可以透過一個 README index file 來記錄，每次更新 ticket 的時候順便更新他就行了，推薦妳試試看
