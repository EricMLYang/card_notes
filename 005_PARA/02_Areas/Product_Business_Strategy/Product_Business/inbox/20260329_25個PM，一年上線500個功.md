25個PM，一年上線500個功能，估值超過100億美元的 Ramp 的 CPO 分享了全公司用 AI 工作流，從客戶研究到寫程式碼到上線流程全部有實際 demo。 Peter Yang 的 youtube頻道，來賓是 Ramp 的 CPO Geoff。Ramp 是一家成長極快的金融科技公司，去年用大約 25 個 PM 就出了超過 500 個功能，營收超過 10 億美元。這集的內容幾乎每個段落都有具體場景和 demo，我把重點整理出來。

這訪談裡面有提到很多實際上軟體科技公司的一些AI應用，包括8天研究變成8分鐘，自己企業的內建資料庫，講者Geoff 幾乎每個段落都有很具體的應用分享，從客戶研究 agent 到 PM 用 Claude Code 5 分鐘出一個 production feature。

對於人才分級這段也很有趣，是跟AI的導入熟練度相關，包含面試和績效管理的方向，還有未來PM的兩種面向，很值得聽聽。

━━━━━━━━━━━━━━━━━━━━
▍客戶研究：8 天的工作 8 分鐘跑完
Ramp 建了一個叫 Voice of the Customer 的 agent，跑在 Slack 裡面，長得就像一個你可以 DM 的同事。

這個 agent 可以存取 Gong (Gon io) 通話錄音、Salesforce 筆記、app 內問卷、客服工單、客戶 email，還有 Snowflake 資料庫。PM 直接用自然語言問它問題，比方說「我們的採購產品最近有什麼客戶回饋？」，它就會去翻 90 天的客服工單和聊天紀錄，整理出主要議題，附上原始資料的連結讓你點進去看。

Geoff 在 demo 裡展示的那次查詢，從發問到整理完結果大約 8 分鐘。他說同樣的事讓人來做，跑完整個量大概要 8 天。

整理完之後還可以繼續追問。比方說針對某個特定問題，叫它拉出客戶原話、找 LogRocket session、列出客戶 ID 讓你自己去看、甚至幫你草擬 email 約客戶開會。整個流程都在 Slack 裡完成。

━━━━━━━━━━━━━━━━━━━━
▍資料分析：讓所有人都能自己查數據
Ramp 之前建了一個叫 Ramp Research 的內部查詢工具。任何人可以用英文直接問資料問題，像「自動寄送 email 的開信率是多少？」，它會自動理解資料庫 schema、跑查詢、產出結果，還會自動生成解讀。

Geoff 提到一個有趣的觀察：當查資料變得更容易，問問題的人就變多了，公司整體就變得更 data-driven。業務用它找特定城市的客戶、客服用它查產品使用模式、行銷用它看 campaign 成效。

後來他們把這套工具升級了。現在是用 Claude Code 加上自建的 Skills（其中一個是 data analyst skill，完整理解他們的資料庫和分析方法論）。現在可以直接跟 Claude 說「幫我做一份採購產品的完整績效報告，找出用戶加入的主要原因、漏斗的卡點，然後幫我想 10 個成長方案」，Claude 會產出一份完整的 HTML 報告，裡面的數據全部是從他們自己的資料庫拉的。

━━━━━━━━━━━━━━━━━━━━
▍PM 的 Claude Code Skill：對話式的需求收斂
Ramp 的 PM 團隊用 Claude Code 建了一個 product shaping skill。這個 skill 可以存取他們 Notion 裡所有的 persona 研究、過去的調研紀錄、其他正在進行的專案資訊。

用法是對話式的。PM 跟 Claude 說想做什麼，Claude 會像一個資深同事一樣反問你：主要目標是什麼？這裡有幾個 trade-off 你怎麼選？有沒有考慮到跟另一個專案的交集？它有完整的上下文，知道公司其他團隊在做什麼，所以提出的問題是有脈絡的。

Geoff 說他們已經不太在意 spec 文件本身了。Spec 只是過程中的一步，重點是透過這個對話快速收斂到可以動手做的狀態。而且現在讀 spec 的是 AI，所以 spec 的寫法也跟過去不一樣了。

━━━━━━━━━━━━━━━━━━━━
▍5 分鐘做出一個 production feature
Geoff 現場 demo 了 Ramp 內部的 Inspect 工具（基於 Claude Code 建的）。他輸入了一段很簡單的需求：在帳單列表上面加四個指標，顯示逾期金額、即將到期的帳單（0-30 天和 30-90 天）、還有待付總額。

Inspect 拿到需求之後，自動理解了任務、拆解了計畫、讀取了 codebase，然後開始實作。因為它已經載入了 Ramp 的 design component library，所以它知道指標卡片長什麼樣、模組該怎麼排、互動該怎麼做。

5 分鐘之後，一個完整的 production feature 就出來了。前端後端都有（雖然這次主要是前端，因為 API endpoint 本來就存在）。Geoff 可以直接開 PR 提交給工程團隊。他說 Ramp 目前 50% 的程式碼是 AI 寫的，2024 年 12 月的時候是 30%，他預估很快會到 80%。

━━━━━━━━━━━━━━━━━━━━
▍從 Slack 回報到 PR 合併，全自動
Ramp 有很多 Slack 頻道專門收集產品回饋。Geoff 展示了一個 UX 回報頻道的例子：有人回報說某個地方的搜尋應該要支援大小寫不分（case insensitive）。

流程是這樣的：回報進來之後，Inspect 自動分析問題、找到 codebase 裡對應的位置、建立 PR。這個案例從回報到 PR 合併，整個過程非常快。

同樣的邏輯也用在客服工單上。客服收到用戶搞不懂某個功能的工單，Inspect 會自動跑一遍、理解問題出在哪、建議修改方案、甚至直接開好 PR，等 PM 或工程師來 review 就可以上線。

━━━━━━━━━━━━━━━━━━━━
▍上線流程也是自動化的
功能做完之後，上線的流程本身也靠 AI 處理了一大半。
Ramp 有一個 release bot，會自動做這些事：拉出這個功能的完整上下文、產生產品預覽、從 Snowflake 撈這個功能的影響數據、從 Slack 頻道整理所有相關討論的摘要，然後合成一份 release report。

除此之外，help center 文章自動產出、內部教育訓練文件（這功能是什麼、怎麼用、為什麼做）自動產出、Slack 公告也可以自動發。

他們的部署流程是分階段的：先內部 dogfooding，再推到 alpha（研究群組的客戶），再推到 beta（大約 10% 的客戶），追蹤數據沒問題之後才 GA。重大功能要 GA 之前有一個 review 流程，但這個流程本身也是自動化的。

━━━━━━━━━━━━━━━━━━━━
▍全公司的 AI 導入框架：L0 到 L3
Ramp 把員工的 AI 熟練度分成四個等級。
-L0 是偶爾用用 ChatGPT 的人。Geoff 直說，如果一個人停在 L0 不願意學，很可能不會留在公司，因為這代表缺乏自主學習的意願。

-L1 是有自己建過 custom GPTs、用過 Notion agent 或 Claude Code 做過一些事的人。

-L2 是能做出一個自動化某部分工作的 app、能對別人的程式碼提 commit 或回饋的人。

-L3 是系統層級的建造者，能設計和維護影響整個團隊的 AI 工作流。

Ramp 的做法是把 L3 的人當作內部推廣者，讓他們幫其他人往上爬。具體措施包括：開放所有工具的存取權，不設 token 預算限制。統一設定好 MCP、Skills、內部 skill 資料庫，讓任何人都可以直接拿來用。在 all-hands 會議上讓非工程師展示他們用 AI 做的東西（財務團隊自己做了一個 treasury 管理系統、法務團隊做合約審查、行銷團隊自動化網站建置）。有固定的 office hours 讓人來問問題和 onboarding。有專門的 AI evangelist 角色，他們的工作就是幫人設定好環境、帶他們走過第一次。

━━━━━━━━━━━━━━━━━━━━
▍面試和績效管理也改了

招聘方面，Ramp 現在要求所有新進員工都要有基本的 AI 工具操作能力。面試 PM 的時候，有一個專門的環節要求候選人現場 demo 一個自己做的產品，說明怎麼做的、用了什麼工具、怎麼運作的。

在績效追蹤方面，他們（用 vibe coding 的方式）做了一個內部工具，可以看到每個員工在各個 AI 工具上的 token 使用量，包括 Notion AI、ChatGPT、Claude Code、Claude Cowork，還有他們內部的工具。用這個來了解誰在積極使用、誰需要更多協助。

Geoff 對成本的看法蠻直接的：一個員工的薪水跟他消耗的 token 費用比起來，token 的花費根本不到薪水的雙位數百分比。如果一個 agent 能幫你多做 10 倍的工作，花兩倍薪水的 token 費用也完全合理。他認為現階段應該把這個當成投資，不該在意短期的花費效率。

━━━━━━━━━━━━━━━━━━━━
▍PM 角色正在分成兩個方向
Geoff 認為 PM 這個角色正在往兩個方向分化。
第一個方向是 builder。程式碼的成本接近零之後，PM 不再寫 spec 然後交給工程師。PM 要自己動手做出產品、快速迭代、直接測試。就像他 demo 的那個 5 分鐘出 feature 的場景。

第二個方向是商業策略。工程師和設計師現在也能做出很好的產品，但他們通常缺乏對商業環境、競爭定位、分銷策略、商業模式的完整理解。PM 應該專注在：我們做的東西怎麼在市場上贏？怎麼定價？怎麼做通路？怎麼創造企業價值？

他也談到主管角色的變化。他自己減少了很多會議和 1:1，把時間拿來實際操作 AI 工具和 vibe coding。他的邏輯是：如果主管自己不懂這些工具能做到什麼，就沒辦法有效地回饋團隊，也沒辦法判斷流程裡哪裡可以改進。

他舉了一個例子：他跟團隊講了很多次「CTA 按鈕要放在 fold 以上」，講了可能一百次，但現在這個原則已經寫進他們的 design 自動化流程裡了。在 Figma 原型產出之前，這些基本原則就已經被檢查過了。他說自己的工作就是不斷找出「我重複在講的事情」，然後把它變成系統裡的規則，這樣下次就不用再講了。

━━━━━━━━━━━━━━━━━━━━
▍規劃週期從一年縮到三個月
Ramp 現在只做三個月的規劃。Geoff 的理由是：三個月內能做完的事，已經等於過去三年的量了。

他認為規劃的重點有三個：對齊策略方向（專注哪些問題、不做哪些事、鎖定哪些客群）、讓團隊有基本的承諾和當責、給業務團隊一些即將上線的功能資訊讓他們好跟客戶溝通。第三點也自動化了。團隊在 Notion 裡更新 backlog 和計畫之後，系統會自動產出 one-pager 和投影片，套上品牌規範，讓業務團隊直接拿去用。

他也提到，LLM 在不同層級之間的資訊翻譯上非常好用。跟 CEO 和董事會報告的內容、跟總監同步的內容、跟團隊溝通的內容，語言和粒度完全不同。以前這些翻譯工作吃掉大量主管的時間，現在快很多。

━━━━━━━━━━━━━━━━━━━━
▍Geoff 對現階段的判斷
他說 Ramp 看到的這些做法，大部分是最近幾個月才建出來的。他不覺得他們已經把所有事情都想通了，但他很確定一件事：這個東西發展得很快，現在看到的工具和方法，90 天後可能就完全不一樣了。

他對 PM 和 builder 的建議是：現在要投入時間學這些工具，不要等公司替你安排。自己去用、去踩坑、去體會那個「啊，原來可以這樣」的時刻。一旦體會到了，就回不去了。
