打造最強 AI Agent 的團隊，都在做減法

Manus 重寫了五次架構，每次都在拿掉東西。Anthropic 設計 Claude Code 的 scaffold 時，就預期它會隨模型進步而縮減。Replit 把一個 agent 拆成三個，但每一個都更簡單。這跟多數人想的完全相反，大家直覺認為更強的 Agent 等於更複雜的系統，但研究數據打臉：輸入量增長後，模型效能下降 15% 到 47%。

Dex Horthy 把臨界值劃在模型輸入容量的 40%。超過就進入他所稱的「笨蛋區」，Agent 開始犯錯，而且那些錯看起來像是模型不夠聰明，其實是你餵太多垃圾進去。模型是引擎，Harness 是整台車，沒有人買引擎。

產業共識似乎正在逐漸成形。單一扁平迴圈勝過複雜編排，檔案系統當延伸記憶，保留錯誤訊息而非清除，基礎工具優於花俏整合。但工具過載策略還在分歧中，Manus 全部載入再用 logit 遮罩限制輸出，Cursor 按需延遲載入，方向相反卻各自有效。而 Google 押注超大 context window 讓模型自己處理一切，其他團隊則走主動過濾路線。實戰中，重 harness 似乎節節勝利中。

給正在做 Agent 的朋友：錢花在 harness 上而不是花在挑模型上，漸進式揭露是架構決策而不是功能選項，你的 harness 應該愈來愈瘦而不是愈來愈胖。過度工程是 Agent 開發的預設失敗模式，能克制住不加東西的團隊才走得遠。

# 拆解結果

## A. 主脈絡與個人映射
- 主張 Agent 變強靠「減法設計」而非堆疊功能。
- 關鍵不是模型分數，而是 harness 對上下文、工具、回饋迴圈的治理。
- 對系統建造者價值：把複雜度預算放在編排品質，不放在功能數量。

## B. 卡片（Zettel）

序號 1
- 標題：4-高性能 Agent 的預設策略是減法，不是加法
- 類型：Heuristic
- 概念（50–300 字）：成熟團隊會反覆刪除不必要流程、工具與狀態，讓主迴圈更短、更可預測。複雜編排常帶來上下文污染與故障連鎖，表面能力上升但穩定性下降。減法不是保守，而是為可控性買保險。
- 重要性（1 句）：直接提升可維運性與錯誤定位速度。
- 邊界/反例（1–2 句）：多任務協同場景仍需分層，但每層都應維持最小責任。
- 知識鉤子：可與 quality gate 的 agentic workflow safety 對接。

序號 2
- 標題：4-Context 使用量超過臨界值後，錯誤會呈非線性上升
- 類型：Warning
- 概念（50–300 字）：當上下文塞入過多資訊，模型注意力分散與噪音干擾會迅速上升，看似聰明其實更容易犯低級錯。問題常被誤判為模型弱，實際是上下文治理失敗。
- 重要性（1 句）：提醒團隊先做 context budget 管理再談換模型。
- 邊界/反例（1–2 句）：大視窗模型可延後臨界點，但無法消除噪音效應。
- 知識鉤子：可連到 context engineering 與記憶分層卡片。

## C. 連結建議（組裝藍圖）
- 內部組裝：減法設計 + context 臨界值，形成 Agent 架構審查清單。
- 外部對接：連到 harness is key 與產品化地圖中的實戰踩坑。
