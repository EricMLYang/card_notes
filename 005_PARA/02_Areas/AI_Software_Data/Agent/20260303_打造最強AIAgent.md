打造最強 AI Agent 的團隊，都在做減法

Manus 重寫了五次架構，每次都在拿掉東西。Anthropic 設計 Claude Code 的 scaffold 時，就預期它會隨模型進步而縮減。Replit 把一個 agent 拆成三個，但每一個都更簡單。這跟多數人想的完全相反，大家直覺認為更強的 Agent 等於更複雜的系統，但研究數據打臉：輸入量增長後，模型效能下降 15% 到 47%。

Dex Horthy 把臨界值劃在模型輸入容量的 40%。超過就進入他所稱的「笨蛋區」，Agent 開始犯錯，而且那些錯看起來像是模型不夠聰明，其實是你餵太多垃圾進去。模型是引擎，Harness 是整台車，沒有人買引擎。

產業共識似乎正在逐漸成形。單一扁平迴圈勝過複雜編排，檔案系統當延伸記憶，保留錯誤訊息而非清除，基礎工具優於花俏整合。但工具過載策略還在分歧中，Manus 全部載入再用 logit 遮罩限制輸出，Cursor 按需延遲載入，方向相反卻各自有效。而 Google 押注超大 context window 讓模型自己處理一切，其他團隊則走主動過濾路線。實戰中，重 harness 似乎節節勝利中。

給正在做 Agent 的朋友：錢花在 harness 上而不是花在挑模型上，漸進式揭露是架構決策而不是功能選項，你的 harness 應該愈來愈瘦而不是愈來愈胖。過度工程是 Agent 開發的預設失敗模式，能克制住不加東西的團隊才走得遠。

怎麼跟創業有點像？

# 拆解結果

## A. 主脈絡與個人映射
- 主張 Agent 變強靠「減法設計」而非堆疊功能。
- 關鍵不是模型分數，而是 harness 對上下文、工具、回饋迴圈的治理。
- 對系統建造者價值：把複雜度預算放在編排品質，不放在功能數量。

## B. 卡片（Zettel）

序號 1
- 標題：4-高性能 Agent 的預設策略是減法，不是加法
- 類型：Heuristic
- 概念（50–300 字）：成熟團隊會反覆刪除不必要流程、工具與狀態，讓主迴圈更短、更可預測。複雜編排常帶來上下文污染與故障連鎖，表面能力上升但穩定性下降。減法不是保守，而是為可控性買保險。
- 重要性（1 句）：直接提升可維運性與錯誤定位速度。
- 邊界/反例（1–2 句）：多任務協同場景仍需分層，但每層都應維持最小責任。
- 知識鉤子：可與 quality gate 的 agentic workflow safety 對接。

序號 2
- 標題：4-Context 使用量超過臨界值後，錯誤會呈非線性上升
- 類型：Warning
- 概念（50–300 字）：當上下文塞入過多資訊，模型注意力分散與噪音干擾會迅速上升，看似聰明其實更容易犯低級錯。問題常被誤判為模型弱，實際是上下文治理失敗。
- 重要性（1 句）：提醒團隊先做 context budget 管理再談換模型。
- 邊界/反例（1–2 句）：大視窗模型可延後臨界點，但無法消除噪音效應。
- 知識鉤子：可連到 context engineering 與記憶分層卡片。

## C. 連結建議（組裝藍圖）
- 內部組裝：減法設計 + context 臨界值，形成 Agent 架構審查清單。
- 外部對接：連到 harness is key 與產品化地圖中的實戰踩坑。

# BreadCards

> 註：本檔已有 `# 拆解結果` 舊版拆解（保留作為 history）。本 BreadCards 為 lite 階段重新展開，加入更多橋接候選卡。

## A. 主脈絡與個人映射
- 論證骨架：頂尖 Agent 團隊（Manus、Anthropic Claude Code、Replit）都在做減法——拿掉編排層、拿掉工具、拿掉狀態。「重 harness、輕 prompt」勝過「複雜編排 + 強 prompt」；context 用量超過容量 40% 進入「笨蛋區」，錯誤呈非線性上升。
- 挑戰的預設：「更強 Agent = 更複雜系統」「換更強模型就會變好」「多代理協作會比單代理強」。研究數據打臉：輸入量增長後效能下降 15–47%。
- 個人映射：對 D2D Architect / Vertical AI 主軸，這是「複雜度預算分配」的 first principle——把複雜度花在編排品質與 context 治理，而不是功能數量。也呼應「AI 產品該更有主見（Opinionated）」。

## B. 候選卡（Lite）

序號 1
- 候選標題：高性能 Agent 的預設策略是減法，不是加法
- 分級：Core
- 類型：Heuristic
- 核心內容：成熟 Agent 團隊反覆刪除流程、工具、狀態，讓主迴圈更短更可預測。複雜編排往往帶來 context 污染與故障連鎖，表面能力上升但穩定性下降。減法不是保守，而是為可控性買保險——你的 harness 應該越來越瘦，不是越來越胖。
- 保留理由：直接可作為 Agent 架構 review 的第一條判準。
- 待補強處：何時必須加層次（多任務協同）的反例與最小責任邊界。
- 初步知識鉤子：Manus 五次重寫、Claude Code scaffold 隨模型進步而縮減、Quality gate / agentic workflow safety。

序號 2
- 候選標題：Context 用量超過 40% 進入「笨蛋區」，錯誤呈非線性上升
- 分級：Core
- 類型：Warning
- 核心內容：Dex Horthy 把臨界值劃在 model 輸入容量 40%。超過後 Agent 開始犯錯，且錯誤看起來像「模型不夠聰明」，其實是上下文治理失敗。常見元兇：載一堆 MCP server、塞一堆 onboarding 文件、把所有歷史 trace 留著。先做 context budget，再談換模型。
- 保留理由：給「Agent 不準是模型問題還是 context 問題」一個量化判別點。
- 待補強處：大視窗模型可延後但無法消除噪音效應；不同任務型別的臨界值是否相同。
- 初步知識鉤子：Context Engineering、Intentional Compaction、On-Demand vs Static doc。

序號 3
- 候選標題：產業共識的清單 — 單一扁平迴圈、檔案系統當記憶、保留錯誤訊息、基礎工具勝花俏整合
- 分級：Support
- 類型：Pattern
- 核心內容：頂尖團隊正在收斂的共同設計：(1) 單一扁平迴圈勝過複雜多層編排；(2) 檔案系統當延伸記憶（不是 vector DB）；(3) 保留錯誤訊息而非清除（讓 Agent 知道走過的死路）；(4) 基礎工具（read/write/grep/bash）優於高階花俏整合。
- 保留理由：可作為 Vertical AI Agent 的 default architecture checklist。
- 待補強處：當必須跨多個 sandbox / 多租戶時，「檔案系統當記憶」的隔離問題。
- 初步知識鉤子：OpenCloud 文件式記憶、Anthropic Agent SDK 設計哲學、Repo-as-Worker。

序號 4
- 候選標題：尚未收斂的歧見 — 工具過載策略仍分歧（全載 + logit mask vs 按需延遲載入 vs 大 context window）
- 分級：Question
- 類型：Question
- 核心內容：Manus 全部工具載入再用 logit mask 限制輸出；Cursor 按需延遲載入；Google 押注超大 context window 讓模型自己處理。三條路方向不同卻各自有效。這是當前 Agent 工具治理尚未收斂的議題，值得追蹤——最終勝出的策略可能取決於 model 自己的 attention pattern 是否能學會 ignore noise。
- 保留理由：是「現在投資哪一條路」的高價值未決問題。
- 待補強處：是否會出現「按使用情境分流」的混合解。
- 初步知識鉤子：Tool Selection、Logit Bias、Attention Sink。

序號 5
- 候選標題：錢花在 harness 上而不是花在挑模型上 — 過度工程是 Agent 開發的預設失敗模式
- 分級：Core
- 類型：Heuristic
- 核心內容：作者結論：(1) 預算放 harness 不放模型；(2) 漸進式揭露是架構決策不是功能選項；(3) 你的 harness 應該越來越瘦。能克制不加東西的團隊才走得遠——這跟創業的「砍 feature roadmap」是同一條紀律。
- 保留理由：把 harness 主軸 + 創業思維 + 預算分配三件事打通。
- 待補強處：何時減法該停（避免砍到 core capability）。
- 初步知識鉤子：harness is key、Opinionated AI、創業 PM 的 feature 砍刀。
