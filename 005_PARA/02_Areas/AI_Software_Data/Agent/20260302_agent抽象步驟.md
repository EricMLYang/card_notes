## 對 Data Analysis Agent 的適用性重新評估

### ReAct 作為核心框架：適合，但要理解它的定位

原文對 ReAct 的定位很清楚：

> "ReAct decouples reasoning and planning... this framework interleaves both reasoning traces and task-specific actions to create an iterative process of thinking and taking actions."

ReAct 的 Thought-Action-Observation 循環對數據分析的價值在於：

| ReAct 元素 | 數據分析的對應 | 實際效果 |
|-----------|---------------|---------|
| **Thought** | 決定下一步查什麼、為什麼 | 讓分析有邏輯脈絡，不是亂查 |
| **Action** | 執行 SQL/Python | 實際取得數據 |
| **Observation** | 看到查詢結果或錯誤訊息 | 決定是否需要修正或繼續深入 |

**但原文也明確指出 ReAct 的侷限**：

> "ReAct Agents can theoretically go on for hundreds of cycles, but need to remember their previous steps and outcomes to effectively continue when context becomes an issue."

這對數據分析很關鍵——當分析進行到第 10 步，Agent 可能已經忘了第 1 步查到什麼。

---

### Reflexion 比單純 ReAct 更適合數據分析

原文對 Reflexion 的描述解決了我先前提到的「分析品質評估」問題：

> "Reflexion uses three entities: Actor LLM (executes actions), Evaluator LLM (assesses quality), Self-Reflection LLM (generates nuanced feedback)"

這個三角色架構對數據分析的意義：

```
[Actor LLM] → 寫 SQL、跑分析
     ↓
[Evaluator LLM] → 「這個分析結果合理嗎？」「回答了原始問題嗎？」
     ↓
[Self-Reflection LLM] → 「結果不合理是因為沒考慮 X 因素，下一步應該...」
     ↓
[Actor LLM] → 根據反思調整分析
```

**這比單純的 ReAct 多了一層「分析品質」的把關**，而不只是「程式有沒有報錯」。

---

### Plan-and-Solve vs Least-to-Most：選擇要看分析類型

原文對這兩者的區分很重要：

**Plan-and-Solve**（零樣本，先規劃再執行）：
> "The LLM is asked to devise a plan first and then solve the problem step-by-step"

**Least-to-Most**（逐步解決，前一步的答案餵進下一步）：
> "The solution of each subproblem is fed back into the LLM when attempting to solve the next problem"

對數據分析而言：

| 分析類型 | 更適合的方法 | 原因 |
|---------|-------------|------|
| **確認式**（算本月營收、比較 YoY） | Plan-and-Solve | 步驟可預測，先規劃效率高 |
| **探索式**（找異常、挖洞察） | Least-to-Most | 下一步取決於上一步發現什麼 |

我先前說「Plan-and-Solve 對探索性分析過度僵化」，原文的 Least-to-Most 其實就是解法。

## 原文沒覆蓋、但數據分析必須補的

### 1. 數據品質的 Observation 解讀

原文的 Observation 主要是：
> "A reasoning step about the result of the action"

但數據分析的 Observation 需要更結構化：

```python
# 不只是「看到結果」，而是要檢查：
observation = {
    "execution_status": "success",  # 語法層
    "row_count": 1547,              # 完整性層
    "null_percentage": {"revenue": 0.02, "date": 0.0},  # 品質層
    "sanity_checks": {              # 業務邏輯層
        "revenue_positive": True,
        "date_range_valid": True,
        "total_matches_detail": False  # ← 這裡有問題
    }
}
```

### 2. 業務知識的注入點

原文提到：
> "Analyze the existing codebase/data"

但沒有說明如何讓 Agent 知道：
- `status = 1` 代表什麼
- 公司定義的「活躍用戶」口徑
- 某些時間段的數據有已知問題

這需要在 **Task Decomposition 之前** 加一個「Context Loading」步驟。

### 3. Evaluator 的評估標準

原文的 Evaluator：
> "An LLM that assesses the quality of the current trajectory"

但沒有說 Evaluator 根據什麼標準評估。對數據分析，你需要明確定義：

```python
evaluation_criteria = {
    "answers_original_question": bool,  # 有沒有回答到問題
    "data_completeness": float,         # 數據是否完整
    "logical_consistency": bool,        # 邏輯是否一致
    "business_reasonableness": bool,    # 結論是否符合業務常識
    "actionability": float              # 結論是否可執行
}
```

## 修正後的設計建議

基於原文，你的 Data Analysis Agent 架構應該是：

```
[1. Context Loading]
    ↓
[2. Task Decomposition] ← 用 Least-to-Most 或 Plan-and-Solve（視分析類型）
    ↓
[3. ReAct Loop]
    ├─ Thought: 決定下一步分析
    ├─ Action: 執行 SQL/Python
    └─ Observation: 結構化的結果檢查（不只是看數字）
    ↓
[4. Reflexion Layer]
    ├─ Evaluator: 根據明確標準評估分析品質
    └─ Self-Reflector: 總結問題、建議下一步
    ↓
[5. 迭代或輸出]
```

# BreadCards

## A. 主脈絡與個人映射
- 論證骨架：Agent 規劃與反思可拆三層——Task Decomposition、Action Sequencing（核心 ReAct）、Reflection & Self-Improvement（Self-Refine → Reflexion → TTRL/R-Zero），三者實現方式分別對應 Prompting、SFT、RL。
- 挑戰的預設：「Plan-and-Solve 一定比 ReAct 強」「Reflection 等於更多 prompt」。指出 Plan-and-Solve 對探索式分析過度僵化，Least-to-Most 才是探索式分析的解；Reflexion 的三角色（Actor/Evaluator/Self-Reflector）才是可驗證的反思。
- 個人映射：對 Data Analysis Agent / DecisionOps 主軸來說，原文補完了「分析品質回饋迴路」這一塊，但缺了三件事——數據品質的結構化 observation、業務知識注入點、Evaluator 的明確評估標準。這三件事正是 Vertical Data Agent 的護城河。

## B. 候選卡（Lite）

序號 1
- 候選標題：Plan-and-Solve vs Least-to-Most — 確認式分析 vs 探索式分析的選型分流
- 分級：Core
- 類型：Heuristic
- 核心內容：Plan-and-Solve 先規劃再執行，適合步驟可預測的「確認式分析」（算 YoY、月營收）；Least-to-Most 把上一步答案餵下一步，適合「下一步取決於上一步發現」的「探索式分析」（找異常、挖洞察）。同一個 Data Agent 不該用同一種規劃策略，要按分析意圖切換。
- 保留理由：給 Data Agent 規劃層一個明確的二分判準，可直接寫進架構決策。
- 待補強處：混合任務（先確認再探索）的銜接機制。
- 初步知識鉤子：探索式 vs 確認式分析、ReAct 適用邊界、Decision Loop 設計。

序號 2
- 候選標題：Reflexion 三角色架構（Actor / Evaluator / Self-Reflector）— 比單純 ReAct 多一層「品質把關」
- 分級：Core
- 類型：Pattern
- 核心內容：Reflexion 把 Agent 拆成三個 LLM 角色：Actor 負責執行、Evaluator 評估品質、Self-Reflection 生成下一輪指引。對 Data Agent 的關鍵價值是把「程式有沒有報錯」升級為「分析結果合不合理、有沒有回答原問題」，這正是 ReAct 在多步分析中常 drift 的失敗點。
- 保留理由：是 Vertical Data Agent 的核心架構模板。
- 待補強處：三個角色用同一模型還是不同模型；Evaluator 是否需要 fine-tune。
- 初步知識鉤子：LLM-as-judge、Anthropic 三類 grader、控制迴路驗證層。

序號 3
- 候選標題：Data Agent 的 Observation 必須結構化，不能只看「結果」
- 分級：Core
- 類型：Warning
- 核心內容：通用 ReAct 的 Observation 是「對結果的推理步驟」，但數據分析必須拆成多層：execution_status（語法層）、row_count / null_percentage（完整性層）、sanity_checks（業務邏輯層）。如果 Observation 只是「我看到結果了」，下一步推理會建在不可靠的地基上。
- 保留理由：直指 Data Agent 最常見的失敗模式——「執行成功但答案是錯的」。
- 待補強處：哪些 sanity_check 是 domain-agnostic、哪些必須 vertical 客製。
- 初步知識鉤子：Data Quality Validation、Chad Sanderson Data Contracts、Trace as Outcome。

序號 4
- 候選標題：業務知識注入點 — Task Decomposition 之前需要 Context Loading 步驟
- 分級：Support
- 類型：Pattern
- 核心內容：通用 Agent 教材跳過了「Agent 怎麼知道 status=1 代表什麼、什麼是公司定義的活躍用戶、哪些時段的數據有已知問題」。Vertical Data Agent 必須在 Task Decomposition 前插入 Context Loading（Glossary / Metric Definition / Known Issue），否則所有後續推理都失準。
- 保留理由：是 Vertical AI 「語意層」護城河的具體實踐節點。
- 待補強處：Context Loading 該存在 vector DB、UC Function 還是 system prompt。
- 初步知識鉤子：Semantic Layer、Metrics Catalog、Domain-specific Embedding。

序號 5
- 候選標題：Evaluator 必須有明確標準 — 「評估品質」本身是可量化的多維 rubric
- 分級：Support
- 類型：Heuristic
- 核心內容：原文 Evaluator「assesses the quality of the current trajectory」是模糊建議。實作時必須拆成多維 rubric：answers_original_question / data_completeness / logical_consistency / business_reasonableness / actionability。沒有這份 rubric，Evaluator 會變成另一個會幻覺的 LLM。
- 保留理由：把抽象的「品質評估」轉成可實作的 rubric 模板。
- 待補強處：每個 rubric 維度的權重該動態還是靜態。
- 初步知識鉤子：Anthropic LLM rubric grader、Multi-dimensional eval、Goodhart's Law。
