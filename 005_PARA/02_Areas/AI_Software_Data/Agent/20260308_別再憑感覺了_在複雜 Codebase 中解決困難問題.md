## [詮釋]


---
📌 別再憑感覺了: 在複雜 Codebase 中解決困難問題

Dex Horthy (12-Factor Agents 的作者) 在 AI Engineer 的演講重點整理。他聚焦在一個很實際的痛點: AI 寫程式在 greenfield 很好用，但遇到複雜的 brownfield codebase 就會產生大量 slop (低品質程式碼)，怎麼辦?

他們三人團隊花了八週，找到了方法做到 2-3 倍的產出提升而且品質不打折。

——

🧠 Context 就是一切

LLM 是無狀態的，影響輸出品質的唯一因素就是 context window 裡的內容。需要針對四個維度優化:

▸ 正確性: 不能有錯誤資訊，這是最致命的
▸ 完整性: 不能缺少關鍵資訊
▸ 大小: 不能塞太多噪音
▸ 軌跡: 對話的方向感 — 如果一直罵 AI 做錯，它看到的模式就是「做錯→被罵」，下一步最可能就是再做錯。不如開乾淨的 context 重來

——

🤪 「笨蛋區」(The Dumb Zone)

Context window 用量超過大約 40% 時，模型表現就會明顯下降。如果裝了一堆 MCP server，光是工具定義就把 context 塞到笨蛋區了，做什麼都不會有好結果。

——

🗜 刻意壓縮 (Intentional Compaction)

不管對話進行得順不順利，都可以讓 agent 把目前的 context 壓縮成一個 markdown 檔案，review 後再開新 context 接著做。壓縮紀錄要精確記錄: 目前在處理什麼、相關檔案和行號、具體問題描述。

——

🔀 Sub-agent 的正確用法

Sub-agent 不是用來擬人化角色的 (frontend agent、backend agent 拜託別了)，而是用來「控制 context」— fork 出新的 context window 去做搜尋和閱讀，只把精簡結論回傳主 agent。

——

🔬 Research → Plan → Implement (RPI) 工作流

整個流程圍繞一件事: 持續保持 context window 在「聰明區」

▸ Research: 用 sub-agent 做垂直切片式的 codebase 探索，產出研究文件
▸ Plan: 列出精確步驟，包含具體檔名、行號、程式碼片段。具體到「全世界最笨的模型也不太可能搞砸」的程度
▸ Implement: 照著計畫執行，保持 context 最小化

實測結果: 七個小時對 30 萬行 Rust codebase 提交了 35,000 行程式碼，估算相當於一到兩週的人工工作量。

——

🧭 On-Demand 壓縮 vs 靜態文件

很多團隊在 repo 放一堆 onboarding 文件給 agent 讀，但離原始碼越遠，「謊言密度」越高，而且文件會過期。更好的做法是隨需壓縮: 每次開始任務時用 sub-agent 對 codebase 做探索，壓縮的是「真相」而不是可能過期的文件。

——

🧩 不是每個任務都需要完整 RPI

▸ 改個按鈕顏色? 直接跟 agent 講
▸ 小功能? 簡單 plan 就夠
▸ 跨多個 repo 的中型功能? 做一輪 research 再建 plan
▸ 最複雜的任務? 完整 RPI 流程，可能還要多輪迭代

——

💡 核心訊息: 不要外包思考

AI 不能取代思考，它只能放大你已經做過 (或沒做過) 的思考。一行錯誤的 plan 可能導致一百行錯誤的 code，一行錯誤的 research 會讓整個下游全部歪掉。

人類的精力應該集中在最高槓桿的地方: review research 和 plan 的品質，而不是去讀每一行生成的 code。

# BreadCards

## A. 主脈絡與個人映射
- 論證骨架：AI Coding 在 brownfield 失敗，根因不是模型不夠強，是 context window 失控。Dex Horthy 提出 Research → Plan → Implement (RPI) 工作流，核心目標只有一個：持續維持 context 在「聰明區」（< 40%）。Sub-agent 的正確用途是「fork context 隔離」，不是擬人化角色分工。
- 挑戰的預設：「sub-agent 應該按角色分（frontend agent / backend agent）」「prompt 寫越長越好」「文件越多 onboarding 越順」。這篇全部反過來。
- 個人映射：對 D2D Architect 與 Coding Agent 主軸——RPI 流程把「人類精力配置」明確化（Review research/plan 比 Review code 槓桿高得多），這正是把 Cedric Chin / Andy Grove 的高槓桿決策觀念落地到 AI Coding 工作流。

## B. 候選卡（Lite）

序號 1
- 候選標題：Sub-agent 的正確用途是控制 context，不是擬人化角色
- 分級：Core
- 類型：Pattern
- 核心內容：Sub-agent 不是「frontend agent / backend agent / QA agent」這種角色擬人化，而是 fork 出新 context window 去做搜尋或閱讀，只把精簡結論回傳主 agent。這樣主 agent 的 context 永遠保持在聰明區。把 sub-agent 當「context 分隔牆」，不當「組織分工」。
- 保留理由：直接糾正最常見的 multi-agent 設計錯誤；可作為 Agent 架構審查的硬規則。
- 待補強處：sub-agent 結論回傳的格式設計（避免回傳本身就把主 context 撐爆）。
- 初步知識鉤子：12-Factor Agents、Context Engineering、減法 Agent 設計。

序號 2
- 候選標題：Research → Plan → Implement (RPI) — 把 AI Coding 拆成三個 context-isolated 階段
- 分級：Core
- 類型：Pattern
- 核心內容：RPI 三階段：Research 用 sub-agent 做 vertical slice 探索並產出 research doc；Plan 列具體檔名、行號、code snippet，具體到「最笨模型也搞不砸」；Implement 照計畫執行並保持 context 最小。每階段開新 context，避免污染。實測：3 人團隊 7 小時對 30 萬行 Rust codebase 提交 35,000 行（相當 1–2 週人工）。
- 保留理由：是目前 brownfield AI Coding 最具體可複製的工作流，可直接寫進團隊 SOP。
- 待補強處：「不需要完整 RPI」的任務分流（按按鈕顏色、小功能、跨 repo 中型功能、最複雜任務）。
- 初步知識鉤子：Spec-Driven Development、Plan-and-Execute、Coding Agent 流程化。

序號 3
- 候選標題：Context 四維度 — 正確性、完整性、大小、軌跡
- 分級：Core
- 類型：Principle
- 核心內容：LLM 是無狀態的，輸出品質完全取決於 context window 內容。需從四維優化：(1) 正確性（不能有錯誤資訊）；(2) 完整性（不能缺關鍵）；(3) 大小（不能塞噪音）；(4) 軌跡（對話方向感——一直罵 AI 做錯，模型看到的模式是「做錯→被罵」，下一步最可能再做錯）。軌跡污染後最好開乾淨 context 重來。
- 保留理由：把 context 治理拆成可獨立檢查的四個槓桿。
- 待補強處：四維度的優先級與量化指標（例：完整性怎麼測？軌跡怎麼判定已污染？）。
- 初步知識鉤子：Context Engineering、Trajectory poisoning、Steering Hooks。

序號 4
- 候選標題：On-Demand 壓縮勝過靜態文件 — 離原始碼越遠，謊言密度越高
- 分級：Core
- 類型：Heuristic
- 核心內容：很多團隊在 repo 放一堆 onboarding 文件給 agent 讀，但離原始碼越遠，「謊言密度」越高，且文件會過期。更好的做法是隨需壓縮：每次任務開始用 sub-agent 探索 codebase，壓縮的是「真相」而不是可能過期的文件。靜態文件適合人類，動態壓縮適合 agent。
- 保留理由：直接打掉「先寫文件再讓 AI 讀」的常見直覺；可重塑 repo 文件策略。
- 待補強處：何時靜態文件仍有價值（架構決策記錄、API 規範等較穩定的內容）。
- 初步知識鉤子：Repo-as-Worker、CLAUDE.md / AGENTS.md 的真正用途、Documentation Drift。

序號 5
- 候選標題：Intentional Compaction — 主動壓縮 context 為 markdown，再開新 session 接續
- 分級：Support
- 類型：Heuristic
- 核心內容：不管對話順不順，都可以叫 agent 把目前 context 壓成 markdown 檔案——精確記錄：目前處理什麼、相關檔案行號、具體問題描述。Review 後再開新 context 接著做。把「壓縮」從 emergency 動作變成 routine。
- 保留理由：是 RPI 流程之外更輕量、可日常使用的 context 治理動作。
- 待補強處：壓縮 markdown 的 template 設計；何時該壓縮 vs 何時該砍掉重來。
- 初步知識鉤子：Checkpoint Pattern、Memory Compaction、OpenCloud 文件式記憶。

序號 6
- 候選標題：人類精力應集中在 Review research / plan 的品質，不是讀每行 code
- 分級：Core
- 類型：Heuristic
- 核心內容：「不要外包思考」——AI 不能取代思考，只能放大你已做過（或沒做過）的思考。一行錯的 plan 會導致一百行錯的 code，一行錯的 research 會讓整個下游全歪。所以最高槓桿的人類介入點不是 code review，是 research / plan 的審閱。這是 AI Coding 時代資深工程師的角色重定義。
- 保留理由：把「人類在 AI 工作流中的位置」說清楚，是寫給工程主管 / 資深工程師的核心訊息。
- 待補強處：什麼樣的 review checklist 適合 research / plan 階段。
- 初步知識鉤子：Andy Grove 高槓桿活動、Kent Beck 工程角色重估、AI Coding Risk Governance。

序號 7
- 候選標題：不是每個任務都需要完整 RPI — 任務複雜度分流
- 分級：Support
- 類型：Heuristic
- 核心內容：任務分流：(1) 改按鈕顏色：直接講；(2) 小功能：簡單 plan；(3) 跨 repo 中型功能：一輪 research + plan；(4) 最複雜任務：完整 RPI 多輪迭代。避免對所有任務都套最重流程；流程強度應與任務複雜度匹配。
- 保留理由：把 RPI 變成可彈性使用的工具箱，不是 dogma。
- 待補強處：「中型 / 最複雜」的判別 heuristic。
- 初步知識鉤子：Process strength matching、Anthropic 「最簡單能解決問題的方案」原則。
