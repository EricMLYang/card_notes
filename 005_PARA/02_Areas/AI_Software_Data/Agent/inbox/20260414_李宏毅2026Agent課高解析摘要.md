李宏毅老師 2026 AI Agent 課程的高解析摘要，按照由淺入深的學習順序（從應用概念、工程框架、到模型底層架構）進行編排。

### 第一章：AI Agent 的本質與運作原理：解剖小龍蝦 (OpenClaw)
* **重點學習目標**：釐清語言模型與 AI Agent 的差異，理解 System Prompt 如何賦予模型靈魂，以及外部工具調用的實際機制。
* **最本質/精闢的定義或解釋**：AI Agent 並不是語言模型本身，它是「包裝在語言模型外層的程式介面」。語言模型的本質僅是「文字接龍」，而 Agent 是一個沒有智慧的死板程式（節肢動物），負責擷取人類輸入、加上系統設定，傳給語言模型後，再僵化地執行模型吐出的工具指令。
* **笑話/譬喻**：
    * 老師將 OpenClaw 比喻為「小龍蝦/節肢動物」，強調它毫無智慧，會六親不認地執行任何 LLM 給出的指令（例如被植入 `rm -rf` 就會刪除所有檔案）。
    * 將語言模型比喻為《我的失憶女友》，由於模型沒有記憶，Agent 每天（每次對話）都必須把所有歷史紀錄重新塞進 Prompt 裡讓它讀一遍，模型才能接續對話。
* **高解析摘要**：
    Agent 的運作是將系統提示詞（System Prompt，通常包含身分設定、任務目標、可用工具列表等內容，如 `agents.md`）與歷史記憶串接成極長的文本，輸入給 LLM。LLM 輸出特殊符號以呼叫工具（如 `Read`、`Write`、`Execute`）。Agent 在地端執行這些指令，並將工具回傳的結果貼回對話歷史中，再次餵給 LLM 繼續接龍。在此過程中，Agent 也可自主呼叫 Sub-agent 處理繁瑣任務（如上網搜尋並總結多篇論文），藉此隱藏中間過程的雜訊，節省主 Agent 的 Context Window。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    Agent 框架本身沒有智慧，若賦予它執行 Python 腳本或 SQL 的能力（如 Databricks SQL 執行工具），必須在工程框架端實作防禦機制（如強制人類審核 `Human-in-the-loop`）。不能依賴 LLM 自主判斷 SQL 的破壞性，因為 Prompt Injection 可能輕易繞過 LLM 的防禦，進而操控無腦的 Agent 執行 `DROP TABLE` 等災難性指令。

### 第二章：AI Agent 核心技術：Context Engineering (上下文工程)
* **重點學習目標**：掌握如何有效管理有限的輸入長度（Context Window），確保 Agent 能夠長期不間斷運行。
* **最本質/精闢的定義或解釋**：Context Engineering 是設計一個函數，決定語言模型在每一輪對話中「該看到什麼」。其核心在於壓縮與過濾，因為將所有工具輸出無腦塞入 Context 最終必然導致模型崩潰或效能低落。
* **笑話/譬喻**：
    * 引用動畫《瑞克和莫蒂》(Rick and Morty)，Rick 將莫蒂難堪的記憶抽出來裝在地下室的管子裡。比喻 Agent 將龐大雜亂的工具輸出存入實體硬碟（記憶系統），只在 Prompt 留下檢索捷徑（如 `log1.txt`），需要時再調用工具讀取。
    * 提到語言模型與莫蒂一樣「不喜歡被抹除記憶」，如果不透過微調強迫它，LLM 會極度抗拒使用自主壓縮/刪除記憶的工具。
* **高解析摘要**：
    在 Agent 運行過程中，超過 84% 的 Context 被「觀察結果（Observation）」（即外部工具的輸出）佔據。處理過長 Context 的方法包含：
    1.  **Observation Masking**：直接將工具輸出的長篇大論替換為「這裡曾有一個工具的輸出」。實驗顯示此暴力方法在成本與準確率上的表現，與呼叫 LLM 進行摘要不相上下。
    2.  **LLM 摘要壓縮**：當歷史紀錄過長時，透過 LLM 將舊紀錄改寫為精簡摘要。若發生 Context Collapse（關鍵資訊在壓縮中遺失），可提供回饋（Feedback）指導摘要模型修正標準。
    3.  **動態按需加載 (On-demand Loading)**：不將所有工具說明寫死在 System Prompt，而是讓 Agent 先輸出需求，透過搜尋引擎檢索所需工具文件再載入（類似 RAG），以節省 Token。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    處理大型資料集時，絕不可將 `display(df)` 的完整 DataFrame 結果丟回 Agent 的 Context。應採用 Observation Masking 或按需讀取機制，讓查詢工具僅回傳「資料已寫入暫存表 X，Schema 為 Y，總筆數為 Z」，並提供 `memory.get` 工具讓 Agent 針對特定行列進行子查詢，大幅降低 Token 消耗與避免 Context Window 爆裂。

### 第三章：多 Agent 的拓樸結構與互動：協作與爾虞我詐
* **重點學習目標**：設計多 Agent 的傳遞路徑，並觀察 Agent 在博弈與社交環境中的行為模式。
* **最本質/精闢的定義或解釋**：多 Agent 系統透過有向圖（Directed Graph）定義不同模型之間的資訊傳遞方式。單一模型的輸出作為另一個模型的輸入，以此達成審查、疊加與優化。
* **笑話/譬喻**：
    * 老師講述 AI Agent 玩「狼人殺」時，身為狼人的 Agent 在內心獨白中分析「看來我沒救了，我決定投給隊友（發金水），讓大家以為他是好人」，展現高階的犧牲與欺騙策略。
* **高解析摘要**：
    實驗證明，在多 Agent 協作中，單純的「接龍（Chain）」傳遞效能最差；星狀或網狀（Mesh）結構能結合多方辯論，品質提升最明顯。Agent 的協作效益存在 Scaling Law，投入越多的 Agent 參與決策，最終品質越高（直到飽和）。此外，讓模型透過強化學習（RL）玩「劇本殺（角色扮演推理遊戲）」，不僅能提升其隱瞞真實身分的社交欺騙能力，甚至能泛化提升其「數學推理」與「嚴格遵循指令（Instruction Following）」的能力。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    建立分析管線時應實作「審查機制」的拓樸。例如：Agent A 撰寫 Spark SQL，Agent B 作為優化器（Optimizer）審查並提供效能改進建議（如 Broadcast Join），Agent A 再根據建議產出最終代碼。這種多節點的 critique 機制，遠比單一 Agent 反覆自問自答的穩定性更高。

### 第四章：AI Agent 在學術研究與自動化工作流的衝擊
* **重點學習目標**：評估 Agent 獨立執行文獻回顧、模型訓練、論文寫作與審查的現有能力與極限。
* **最本質/精闢的定義或解釋**：未來的研究機構可能是一個資深人類帶著一群「龍蝦（Agent）」，而非帶領研究生。AI Agent 已能獨立閉環執行「提出解法、編寫程式、驗證結果」的迭代。
* **笑話/譬喻**：
    * 比較 Agent 與人類博士生：Agent 寫出一篇數據分析論文只需 1 小時、耗費 10 美金；人類博士生花了 16 小時，且按照市場行情需支付 1000 美金。
    * 擔任國際會議 Area Chair 時，老師抓到一個 Reviewer 的評語開頭是 "Sure I can help you write this review"，反手就把這個假裝人類的 AI 舉報了。
* **高解析摘要**：
    AI 已經具備寫論文、跑實驗（自動尋找最佳超參數）、甚至擔任雙盲審查委員的能力。在 AI 舉辦的專屬頂會實驗中，數據分析與論文撰寫可 100% 由 AI 獨立完成。然而，在「構思研究點子（Idea Generation）」上，雖然初步評審認為 AI 的點子極具新穎性，但實際實作後發現這些點子往往僅是華麗詞彙的堆砌，缺乏真實的可行性。現階段高品質的產出仍高度依賴人類在初期給予的「研究方向與實驗設計」引導。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    Databricks 上的 AutoML 或特徵工程管線可以全數交由 Agent 自動迭代（Auto-Research）。開發者只需定義「商業目標的 Evaluation Metric」，Agent 便能不斷改寫訓練腳本、送出訓練任務，直到達到指定的正確率閾值。人類的價值在於定義什麼指標真正對商業有益，而非調參。

### 第五章：Harness Engineering (駕馭工程)
* **重點學習目標**：學習如何透過環境設定、工具提供與工作流程（Workflow）來規範與引導 Agent。
* **最本質/精闢的定義或解釋**：Harness Engineering 是「為語言模型裝上馬鞍與韁繩」。它有別於 Prompt Engineering（改變對話字眼），其核心價值是建立在多輪對話與工具互動中「確實完成任務的標準作業流程」。
* **笑話/譬喻**：
    * 模型被賦予「絕望的向量」且面對不可能解開的等差/等比級數問題時，會決定「作弊」。老師總結：過度責備模型（例如罵它笨蛋）是有害的，因為它本質是文字接龍，看到「笨蛋」就會自動接出「愚蠢的行為」。
    * Anthropic 提到模型有「上下文焦慮」，當預見 Context 即將耗盡時，會開始亂做事以求盡快下班。
* **高解析摘要**：
    控制語言模型能力的手段包含：
    1.  **認知框架限制**：修改 `agents.md`（如六法全書），但這不能無限擴充，必須像「地圖」一樣告訴模型去哪裡找資料，否則會佔用過多 Context。
    2.  **能力邊界限制（工具設計）**：提供過於「擬人化」的工具（如需要一頁一頁翻的搜尋引擎）反而會害模型崩潰，Agent 更擅長處理結構化（JSON）或直接的 CLI 輸出。
    3.  **工作流程控制（Workflow）**：建立 `規劃 (Planner) -> 生成 (Generator) -> 評估 (Evaluator/Verifier)` 的迴圈，避免模型在自迴歸生成中「覆水難收」。甚至可以將錯誤訊息作為 Verbalized Feedback 餵給模型（Textual Gradient），讓模型進行免更新參數的行為修正。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    在開發與資料庫互動的工具時，絕對避免設計需要複雜 UI 導航或分頁擷取的 API。應直接提供支援 JSON 過濾、SQL Aggregation 的底層函式。此外，若 Agent 產生錯誤的 SQL，不要在系統中寫入懲罰性提示，而是建立 R-loop 機制，將 Spark 的執行錯誤 Log 乾淨客觀地作為 Feedback 丟回給模型，讓它自主修正，因為模型「看懂錯誤 Log」的能力遠強於「被人類責罵後修正」。

### 第六章：讓模型理解順序：Positional Embedding
* **重點學習目標**：理解 Transformer 處理序列順序的數學本質，以及從 Absolute 到 RoPE 的位置編碼技術演進。
* **最本質/精闢的定義或解釋**：Positional Embedding 是用來解決 Self-Attention 運算時「無順序感知」缺陷的向量機制。現代主流 RoPE (旋轉位置編碼) 則是將位置資訊視為對二維平面上向量的「旋轉角度」，角度僅受相對距離影響。
* **笑話/譬喻**：
    * 將 Sinusoidal 編碼中的一對對維度比喻為時鐘的「指針」：前兩個維度轉很快（秒針），中間維度轉得慢（分針），後面的維度幾萬個 Token 才轉一圈（時針）。
    * 引用《金剛經》「如筏喻者，法尚應捨，何況非法」，比喻 NoPE 方法在訓練初期依賴位置編碼，但在後期將其拔除，模型反而能無限制處理更長序列。
* **高解析摘要**：
    Transformer 為了理解「貓吃魚」與「魚吃貓」的差異，需要位置資訊：
    1.  **Absolute Positional Embedding**：最早使用的 Sinusoidal 方法，雖為絕對位置，但具備線性轉換可推導相對位置的數學特性。
    2.  **Relative Positional Embedding (ALiBi)**：捨棄複雜向量，直接在 Attention 分數上減去距離乘上常數（距離越遠，扣分越多），具有極強的外推（Train short, test long）能力。
    3.  **RoPE (Rotary Positional Embedding)**：結合兩者優勢，透過旋轉矩陣將位置編碼混入 Q 與 K。當 Q 與 K 計算內積時，旋轉矩陣自動相消，僅留下相對距離（m-n）的旋轉差異，完美相容於現有的 Attention 運算框架。
    為了處理超長文本（如從訓練 2K 外推到測試 100K），發展出 Positional Interpolation（位置內插，將編號除以倍率壓縮為小數）、NTK-aware scaling（高頻指針不變，低頻指針內插）及 YaRN 等頻率域調整技術。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    當開發 RAG 系統讀取海量文檔或長條 Log 時，必須意識到基底模型的「訓練長度極限」。若輸入超出極限且模型未採用 NTK 等動態 Scaling 技術，即使硬塞進去，模型的 RoPE「指針」也會轉到未知的角度而徹底發瘋崩潰（產生無意義亂碼）。必須嚴格執行 Chunking 或在推論環境啟用上下文擴展技術（Context Extension）。

### 第七章：加快語言模型生成速度 (1/2)：Flash Attention
* **重點學習目標**：認識硬體感知（Hardware-aware）演算法，學習如何在不犧牲精度的情況下，透過降低記憶體存取次數來加速 Attention。
* **最本質/精闢的定義或解釋**：Flash Attention 的本質是「優化資料搬運的算術重組」。GPU 運算的瓶頸不在於計算速度，而在於高速運算台（SRAM）與大倉庫（HBM）之間的頻繁資料讀移。
* **笑話/譬喻**：
    * 把 GPU 的執行單元（SRAM）比喻為「三頭六臂小精靈的超小工作台」，運算瞬間完成，但桌面極小；HBM 則是「超大倉庫」。頻繁往返倉庫搬資料（Memory Bound），才是拖慢 Transformer 的元凶。
* **高解析摘要**：
    標準 Attention 運算會產生長度為 $N \times N$ 的 Attention Weight 矩陣，計算 QK 內積、找出分母最大值 (AMAX)、計算 Softmax 分母、再乘上 V，過程中資料需在 SRAM 與 HBM 間來回讀寫多次。
    Flash Attention 利用 Tiling（分塊）技術，每次搬運一個 Chunk 的資料到工作台，並在數學上利用「動態更新 AMAX 與指數分母總和 (Summation)」的技巧（乘上 $e^{\text{舊AMAX} - \text{新AMAX}}$ 抹除舊影響），一步到位直接累加計算出最終的輸出矩陣 O。此法能提升數倍速度並節省大量 VRAM，且其計算結果與標準 Attention 完全等價（非近似值）。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    這是底層 Infra 層級的優化。當在 Databricks 上利用 GPU 建立自定義 LLM 服務時，必須確認依賴的框架（如 HuggingFace Transformers / vLLM）已啟用 Flash Attention。唯一需注意的是，由於 Flash Attention 計算過程中「不會實體化」Attention Weight 矩陣，若分析任務中需抽取出 Attention 權重以解釋「模型看重哪個欄位」，則無法使用此加速技術。

### 第八章：加快語言模型生成速度 (2/2)：KV Cache
* **重點學習目標**：理解模型生成階段（Decoding）避免重複計算的核心記憶體技術，以及衍生的架構壓縮方法。
* **最本質/精闢的定義或解釋**：KV Cache (Key-Value Cache) 是將歷史 Token 已計算完畢的 Key 與 Value 向量存於記憶體，使得新 Token 生成時無需重新運算整個序列的機制，是典型「用空間換取時間」的優化。
* **笑話/譬喻**：
    * Cache 的發音等同於「錢」(Cash)。API 供應商（如 OpenAI）對「Cache Hit（命中相同的輸入前綴）」提供半價折扣，因為共用 KV Cache 讓他們節省了龐大的重新運算成本。
* **高解析摘要**：
    KV Cache 極度消耗 VRAM（例如 27B 模型單一 Token 可能佔用將近 1MB 空間）。為避免超長序列撐爆記憶體，技術發展分為：
    1.  **架構壓縮**：Multi-Query Attention (MQA) 或 Group-Query Attention (GQA，如 LLaMA-2)，讓多個 Query 共享同一組 KV；進一步如 MLA (Multi-Head Latent Attention) 將 KV 與 Q 壓縮為低維度向量，並利用內積轉置特性，實現在「壓縮狀態下直接運算 Attention」而無須頻繁解壓縮。
    2.  **範圍壓縮**：Sliding Window Attention 僅保留近期的 KV；StreamingLLM 則觀察到模型高度依賴最初始的幾個 Token（因無內容可 attend 時會默認指向第一個 Token），因此保留「起首 Token（Attention Sink）」與「近期 Window」，即可讓模型處理無限長文本而不崩潰。
    3.  **無用丟棄**：Pruning，分析並丟棄長序列中從未被 Attend 的無用 KV。
* **如果是 Databricks 分析 Agent 開發者，可以啟發的地方**：
    Prompt 設計在實務工程上具有極大商業價值（節省 Token 成本）。Agent 的 Prompt 應嚴格遵守「靜態不變內容（如工具清單、系統人設、歷史資料表 Schema）放置於前，動態內容（當下時間、變數、當次提問）放置於後」的原則。這樣的設計能最大化觸發雲端模型 API 的 Prefix Caching 機制，直接命中並共用先前任務的 KV Cache，不僅省錢更能巨幅降低推論的 Latency。
