# AI 不應該急著類比成「人」——從 Context 的成本結構談起
 
## 部門可以，人不行
 
企業導入 AI 時，我覺得我們不應該急著把 AI 類比成「人」的角色。
 
「部門」是合理的類比。部門是責任的集合，在任務的分配上，可以用部門作責任的歸屬。但是在任務的執行上，不應該由部門內的「人」來做類比。
 
## 人類的 Context 模型：集中、昂貴、不可分切
 
為什麼過去用「人」來執行任務？因為沒有選擇。
 
我們之所以需要責任的集合，是為了讓任務可以快速被分配。而人的「context」是大的——透過一些文件、透過一些訓練，讓他變成長期記憶，人就能不那麼有效率、但可以穩定處理很多不穩定的事情。
 
理論上context少意味著學的東西少，要處理的事情簡單，以流水線的角度context越低 （每個人只做一件事晴），最終的產出效果越大。

問題是，人類很難分切 context。人的單價太貴了，每多一個 context 就要多一個人力。所以我們會希望 context 集中——如 711 店員那樣，透過很多很多 SOP，集中給同一個人做。
 
> **背景補充：** 傳統企業的組織設計本質上是一種 context 管理策略。部門、職位、SOP 都在解決同一個問題：如何讓有限且昂貴的人力，承載盡可能多的任務 context。這也是為什麼企業傾向「一人多工」而非「一事一人」——context 切換的成本雖高，但比多養一個人便宜。
 
## Agent 的 Context 模型：輕量、可分切、成本低
 
但是在 AI 的世界，任務的分配剛好是反過來的。
 
Agent 不一樣，他的 context 是可以被輕量分切的，成本低很多。而且他分切之後，在大型模型的基礎知識支持下，可以達到 context 極輕——你可以一百個任務給一百個不同的 context 做，類比成一百個不同的人來做。
 
所以在最有效率的前提下，我認為是「專門的 context 處理專門的事情」，不確定的 context 靠 model 能力支撐，才是把 agent / model 組織化的關鍵，也更容易量化、管理所有的 job queue。
 
所以雖然類比成人類來思考 agent 是抽象思考層次比較輕鬆的東西，但一旦你往下開始做任務的分配、管理、監督，你馬上就會發現這個思路是混亂的。
 
## 結論
 
所以我們的任務跟以前一樣——找到最有價值的 path，做最適合的 context 去 manage 這些 path，進行產出。
 
差別只在於：過去 context 的載體是人，貴且不可分切；現在 context 的載體是 agent，便宜且可以任意切分。組織設計的邏輯該翻過來了。

# BreadCards

## A. 主脈絡與個人映射
- **論證骨架**：作者從「組織設計的本質是 context 管理」切入，指出過去部門/SOP/一人多工的存在，根本原因是人類 context 昂貴且不可分切。Agent 的 context 成本結構完全相反（輕量、可分切、便宜），因此最高效的 Agent 組織方式應該是「專門 context 處理專門事情」，而非沿用人類組織的多工集中模式。
- **挑戰的預設**：挑戰「把 Agent 類比成人來做組織管理」的主流做法——用「部門」類比責任歸屬可以，但用「人」類比任務執行單元會導致架構設計混亂。
- **個人映射**：直接命中 profile 中「AI Agent 工程化：架構設計」與「限制理論(TOC)實戰化：辨識瓶頸、避免局部最佳化」的交叉點。context 成本結構差異是一個底層原理，可以解釋為什麼 Agent workflow 不該照搬人類組織圖，也補充了 Repo-as-Worker 視角中「為什麼每個 skill/agent 應該有獨立且最小化的 context」的設計決策依據。

序號 1
- 候選標題：組織設計的本質是 context 管理策略
- 分級：Core
- 類型：Principle
- 核心內容：傳統企業的部門、職位、SOP 都在解決同一個問題：如何讓有限且昂貴的人力承載盡可能多的任務 context。企業傾向「一人多工」而非「一事一人」，是因為人類 context 不可分切——每多一個 context 就要多一個人力，context 切換成本雖高但比多養人便宜。這讓組織設計本質上成為一種 context 的成本最佳化問題。
- 保留理由：用 context 成本結構重新解釋組織設計，是非顯而易見的底層原理，可跨領域遷移到 Agent 架構設計。
- 待補強處：此觀點在組織理論中的邊界？是否所有組織設計都可歸結為 context 管理？
- 初步知識鉤子：限制理論（TOC）、交易成本理論、Agent 架構設計、Repo-as-Worker

序號 2
- 候選標題：人類 vs Agent 的 context 成本結構是反過來的
- 分級：Core
- 類型：Principle
- 核心內容：人類的 context 模型是集中、昂貴、不可分切的——透過文件與訓練轉為長期記憶，讓一人穩定處理多種不穩定任務。Agent 的 context 模型反過來：輕量、可分切、成本低，一百個任務可以給一百個獨立 context 處理，且靠大型模型的基礎知識彌補 context 不足。這個成本結構差異，決定了 Agent 的最佳組織方式不應沿用人類模式。
- 保留理由：這是整篇文章的核心機制洞見，直接支撐後續所有架構決策。
- 待補強處：Agent context 真的「無限可分切」嗎？當任務間有強耦合時，分切反而增加通訊成本？
- 初步知識鉤子：context window 管理、Agent 記憶設計、微服務 vs 單體架構的類比

序號 3
- 候選標題：Agent 組織化原則：專門 context 處理專門事情
- 分級：Core
- 類型：Heuristic
- 核心內容：基於 Agent context 可輕量分切的特性，最高效的做法是「專門的 context 處理專門的事情，不確定的部分靠 model 基礎能力支撐」。這樣做的好處是每個 Agent 的 context 極輕、職責極清晰，更容易量化、管理所有 job queue。反之，若沿用人類的多工集中模式（一個 Agent 塞很多 context），會浪費 Agent 可分切的結構性優勢。
- 保留理由：將原理轉化為可執行的設計判準，直接可用於 Agent workflow 架構決策。
- 待補強處：「不確定的靠 model 能力支撐」的邊界？model 基礎能力不足時怎麼辦？
- 初步知識鉤子：Skill 設計、single responsibility principle、job queue 管理、Harness 框架

序號 4
- 候選標題：不要用「人」類比 Agent 的任務執行——部門可以，人不行
- 分級：Support
- 類型：Warning
- 核心內容：企業導入 AI 時，用「部門」類比責任歸屬是合理的（部門是責任的集合，可用於任務分配）。但用「人」類比任務執行單元是危險的——因為人類的 context 模式（集中、不可分切）和 Agent 的 context 模式（輕量、可分切）完全相反。一旦用人的思路做任務分配和監督，架構馬上會變混亂。
- 保留理由：作為 Warning 有獨立實用價值，可在團隊討論 Agent 導入時直接引用。
- 初步知識鉤子：Agent 擬人化陷阱、組織設計模式
