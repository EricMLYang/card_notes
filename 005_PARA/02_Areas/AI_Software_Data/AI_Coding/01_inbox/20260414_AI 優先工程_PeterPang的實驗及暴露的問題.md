## [重點]
AI 優先工作流程出問題時，解法不是「更努力」或「寫更好的提示詞」，而是「缺少什麼能力？怎麼讓它對代理可讀、可執行？」
這改變了工程師思考工作的方式。你不再是在除錯程式碼，而是在除錯那個產出程式碼的環境。

CI/CD 管線分六階段：驗證 CI → 建置部署 Dev → 測試 Dev → 部署 Prod → 測試 Prod → 發布。沒有階段可以跳過，沒有手動覆寫。管線是確定性的，代理可以預測結果並對失敗進行推論。

每個 pull request 觸發三道平行的 AI 審查（Claude Opus 4.6）：程式碼品質、安全性、相依性掃描。這些不是建議，是閘門。一天部署八次，沒有人類審查者能每個 PR 都保持注意力。
技術債累積：多項獨立研究正指向同一個方向。GitClear 分析了 1.53 億行程式碼，發現大量使用 AI 工具且缺乏適當監督時，技術債指標顯著攀升。Carnegie Mellon 對 800 多個 GitHub repo 的研究記錄了程式碼品質加速退化。Apiiro 對 Fortune 50 企業的研究發現安全漏洞呈指數增長。
AI 生成的程式碼通常看起來乾淨：格式工整，單獨看邏輯正確。風險在累積。每次 AI 提交都製造細微不一致，這裡用了不同的身分驗證模式、那裡重複了工具函式、某個查詢能跑但沒遵循團隊的 ORM 慣例。個別來看微不足道，合在一起就像複利。
先搞清楚瓶頸在哪。如果 PM 流程比開發時間還長，那才是該先處理的，不是一味加 AI 程式碼工具。在擴大代理規模之前，先把測試 harness 建好。沒有快速驗證機制的快速 AI，只是在快速累積債務。

治理缺口：Fortune 2026 年 4 月的報告指出，91% 的組織已在用 AI 代理，但只有 10% 有明確的管理策略。AI 代理如今是實際操作者，它們分析資料、啟動工作流程、在企業內部採取行動。但現有的治理框架是為人類設計的。

風險
初階人才管線：如果 54% 的工程領導者計畫因 AI 而減少初階聘用，而初階人員正是未來架構師的養成管道，那這個產業正在製造一個 3 到 5 年後才會浮現的結構性缺口。Pang 指出最有價值的技能是架構判斷力，也就是批評 AI、找失敗模式、壓力測試假設的能力。但這些技能需要多年的實戰編碼、犯錯和正式環境事故才能培養。
你跳不過操作員階段直接當架構師。如果操作員角色日漸空洞化（主要工作只剩審查 AI 的 PR 和驗證分流工單），晉升所需的經驗深度可能永遠養不成。

AI 優先工程：Peter Pang 的實驗，以及它暴露的問題
『沒有結構的速度，就是包裝比較好看的 Vibe Coding，而 Harness是組織的新產品，其他都是那匹馬』
Peter Pang（CREAO 共同創辦人、前 Meta LLaMA 技術負責人）上週在 X 上發了一串貼文，兩天內超過 41 萬次瀏覽。核心主張：CREAO 99% 的正式環境程式碼由 AI 撰寫。一項功能從構想到上線，可以在一天內走完。三個月前，同樣的事需要六週。
這不是在 IDE 裡裝了 Copilot 就收工的故事。Pang 描述的是把軟體的規劃、開發、測試、部署和組織方式全部拆掉重來。底層的變化是真的、正在加速，而且會逼每個工程領導者做一些他們不想做的決定。
AI 輔助和 AI 優先的差別
多數公司還在「AI 輔助」階段。工程師開 Cursor，PM 用 ChatGPT 寫初稿，QA 試著讓 AI 生測試。流程沒變，效率提升一兩成，結構上什麼都沒動。
Pang 定義的「AI 優先」不一樣：前提是 AI 就是主要建造者，然後圍繞這個前提重新設計流程、架構和組織。不問「AI 怎麼幫工程師」，問「怎麼重組一切，讓 AI 負責建造、工程師負責方向和判斷」。
差別在於乘數和加法。把 AI 嫁接到現有的 sprints上，收益有限。但如果圍繞 AI 的執行速度重新設計整個迴圈，得到的是完全不同的營運模式。
CREAO 的數據：25 名員工、10 名工程師，14 天內平均每天 3 到 8 次正式環境部署。按舊模式，這兩週連一次正式發布都擠不出來。
Harness Engineering：OpenAI 給這件事取了個名字
2026 年 2 月，OpenAI 提出了 Harness Engineering 這個概念。他們的 Codex 團隊做了一個超過百萬行程式碼的正式環境應用，沒有一行出自人手。所有應用邏輯、測試、CI 設定、文件和可觀測性配置都是 Codex 代理寫的。產品有每日活躍的內部使用者和外部 alpha 測試者，經歷過上線、部署、故障、修復，全程代理驅動。
Harness 的比喻是刻意的。馬是 AI 模型：強大、快速，但不知道該往哪跑。Harness 是韁繩、鞍具、銜鐵，也就是約束條件、回饋迴圈、文件規範、linter、CI 閘門和架構邊界，讓代理的產出變得可靠。
Pang 在 OpenAI 正式命名之前就獨立得出了同樣的結論：AI 優先工作流程出問題時，解法不是「更努力」或「寫更好的提示詞」，而是「缺少什麼能力？怎麼讓它對代理可讀、可執行？」
這改變了工程師思考工作的方式。你不再是在除錯程式碼，而是在除錯那個產出程式碼的環境。
技術堆疊長什麼樣
CREAO 的實作值得細看，因為它把討論從理論拉回工程現實。
Pang 的第一步是把所有程式碼整合進單一 monorepo。理由不是開發者方便，而是代理可讀性。程式碼分散在多個 repo 時，AI 代理看不到全貌、無法推論跨服務影響、無法在本機跑整合測試。統一的程式碼庫對代理而言是可讀的；碎片化的程式碼庫等於不存在。
但這是一項重大架構承諾，多數現有團隊做不到一夜轉換。
CI/CD 管線分六階段：驗證 CI → 建置部署 Dev → 測試 Dev → 部署 Prod → 測試 Prod → 發布。沒有階段可以跳過，沒有手動覆寫。管線是確定性的，代理可以預測結果並對失敗進行推論。
每個 pull request 觸發三道平行的 AI 審查（Claude Opus 4.6）：程式碼品質、安全性、相依性掃描。這些不是建議，是閘門。一天部署八次，沒有人類審查者能每個 PR 都保持注意力。
最值得注意的是自我修復回饋迴圈。每天早上 UTC 9:00，自動化健康檢查啟動。Claude Sonnet 4.6 查詢 CloudWatch，分析所有服務的錯誤模式，產生健康摘要。一小時後，分流引擎把正式環境錯誤分群，依九個嚴重度維度評分，在 Linear 自動建立調查工單（附範例日誌、受影響使用者、建議調查方向）。工程師推送修復後，同一管線負責審查、測試、部署。部署完成，分流引擎再查 CloudWatch，錯誤消失就自動關閉工單。
每個工具負責一個階段，沒有哪個工具試圖包辦一切。
架構師和操作員
Pang 論述中最讓人不舒服的部分不是技術，是人。
他認為 AI 優先的世界裡會出現兩種工程師。架構師（每個團隊一到兩人）負責設計標準作業程序，教 AI 怎麼工作：建測試基礎設施、整合系統、分流系統，決定架構邊界，替代理定義什麼叫「好」。這個角色需要能在 AI 生成的方案裡找出漏洞。

有一件出乎意料的事：初階工程師適應得比資深的快。初階人員沒有十年舊習慣要拋棄，而且工具放大了他們的影響力，他們很興奮。反而是有深厚傳統功力的資深工程師遭遇最大困難，他們花兩個月完成的工作，AI 一小時就做完了。
一份 2026 年的分析也指出類似趨勢：隨著 AI 工具接手任務指派、進度追蹤和狀態回報，傳統工程管理者的角色正在被壓縮。「AI 協調者」這個新角色正在多家公司出現，由資深工程師負責驗證和整合代理產出，而非自己寫程式碼。

風險
初階人才管線：如果 54% 的工程領導者計畫因 AI 而減少初階聘用，而初階人員正是未來架構師的養成管道，那這個產業正在製造一個 3 到 5 年後才會浮現的結構性缺口。Pang 指出最有價值的技能是架構判斷力，也就是批評 AI、找失敗模式、壓力測試假設的能力。但這些技能需要多年的實戰編碼、犯錯和正式環境事故才能培養。

Pang 的三重審查管線就是為了攔截這些問題。但偵測能力的進步能不能跟上生成速度？當審查者和生成者共享相同盲點時怎麼辦？這些問題還沒有答案。
模型依賴：Pang 坦承整個轉型要歸功於最近兩個月的模型進步。「Opus 4.5 做不到 Opus 4.6 做到的事。」這既是 AI 優先工程最強的論據，也是最大的弱點。
工程流程建立在特定前沿模型的能力上，就對供應商產生了存亡等級的依賴。定價調整、API 政策變動、模型更新導致能力退化，甚至暫時斷線，都可能在整個營運中產生連鎖反應。高 token 消耗加上長上下文視窗，對前沿模型 API 持續跑代理，成本可能很快就撐不住。
治理缺口：Fortune 2026 年 4 月的報告指出，91% 的組織已在用 AI 代理，但只有 10% 有明確的管理策略。AI 代理如今是實際操作者，它們分析資料、啟動工作流程、在企業內部採取行動。但現有的治理框架是為人類設計的。
CREAO 的做法（確定性管線、自動回滾、帶緊急停止的功能旗標）對 25 人的新創來說是合理的治理。能不能擴展到有法規要求和合規義務的大型組織，還不知道。
一人公司？
Pang 的預測：「一人公司將會成為常態。如果一位架構師加上代理就能完成 100 人的工作量，許多公司根本不需要第二名員工。」
Sam Altman 說過類似的話。媒體最近也在報導遠距醫療供應商 MEDVi（據稱兩名員工、估值 18 億美元）。但 Forrester 的分析提醒，這個案例可能有誤導性，估值不等於一家有實質意義的企業。
比較務實的版本其實已經在發生：5 到 15 人的團隊，達成過去需要一兩百人的產出。Imbue 的 CEO Kanjun Qiu 認為，獨立 AI 公司最可能先在消費者或專業消費者產品領域成功，這類產品不需要大型市場推廣團隊。B2B 要獨立運作困難得多，因為信任和關係很難用代理替代。
真正的限制不在技術，在組織韌性。如果你的公司只有一個人搞得懂那些代理，那個人離開，你就完了。最小可行的 AI 優先團隊不是一人，是兩人。
如果你是工程師
你的價值正在從程式碼產出轉向決策品質。快速寫程式碼的能力每個月都在貶值，評估、批判和引導 AI 的能力在增值。產品感，你能不能光看一個 AI 生成的 UI 就判斷它有問題，正在變成核心工程技能。
Pang 對他 19 歲實習生的建議：訓練批判性思維。學習評估論點、找漏洞、質疑假設。學習辨認好的設計。這些技能會隨時間複利。
如果你是技術領導者
先搞清楚瓶頸在哪。如果 PM 流程比開發時間還長，那才是該先處理的，不是一味加 AI 程式碼工具。在擴大代理規模之前，先把測試 harness 建好。沒有快速驗證機制的快速 AI，只是在快速累積債務。
另外，刻意經營架構師人才管線。別全面砍初階職位。設計 AI 增強的入職培訓，即使團隊縮小，也維持人才培養。
如果你在評估這個領域
注意三件事。一是已持續 AI 優先運作超過六個月的團隊（目前多數案例涵蓋期間只有幾個月）。二是 AI 優先程式碼庫中技術債累積速率的具體數據。三是首起大規模 AI 生成程式碼的重大安全事件，那起事件要麼驗證 harness 方法有效，要麼觸發監管過度反應。
總結
Peter Pang 的貼文串捕捉到一個真實現象。他描述的方法，Harness Engineering、自我修復回饋迴圈、確定性 CI/CD 管線、三重 AI 審查，不是空談。OpenAI、Anthropic 和多個獨立團隊各自走到了相似的原則。
但速度收益是真的，風險也是真的：模型依賴、技術債的隱形累積、治理缺口，以及整個系統仰賴的人類專業管線正在慢慢斷裂。
能活下來的公司不是跑最快的，是 harness 最強的，約束條件、回饋迴圈和人類判斷框架，讓 AI 的力量往有用的方向走。沒有結構的速度，就是包裝比較好看的 Vibe Coding。
Harness 是產品。其他都是那匹馬。

## A. 主脈絡與個人映射
- 論證骨架：CREAO/Pang 的 AI-First 實驗（99% 正式環境 code 由 AI 寫、25 人/10 工程師/14 天每天 3-8 次部署）證明圍繞 AI 重組整個營運迴圈，而非把 AI 嫁接到既有 sprint。Harness 是核心：monorepo（為 agent 可讀性）、6 階段確定性 CI/CD、3 道平行 AI 審查（Opus 4.6）、每天 9:00 自我修復回饋迴圈。組織分裂為架構師（1-2 人/team）vs 操作員。風險：初階人才管線斷裂、技術債複利、模型依賴、治理缺口。

- 候選標題：Monorepo 的新理由：不是開發者方便，而是 agent 可讀性
- 分級：Core
- 類型：Pattern
- 核心內容：CREAO 第一步就是把所有 code 整合進單一 monorepo。理由不是 DX，而是 agent legibility——分散在多個 repo 時，agent 看不到全貌、無法推論跨服務影響、無法本機跑整合測試。「碎片化的 codebase 對 agent 等於不存在。」這是架構決策的全新評分維度。
- 保留理由：把長期架構辯論（mono vs poly repo）從「人類視角」拉到「agent 視角」，提供策略性新論點。
- 待補強處：對既有 polyrepo 組織，遷移成本與風險？是否有「partial monorepo」的中間方案？
- 初步知識鉤子：與 20260407 harnessability、20260401 skill graph 可讀性、ambient affordances

序號 3
- 候選標題：6 階段確定性 CI/CD：沒有跳過、沒有手動覆寫
- 分級：Core
- 類型：Pattern
- 核心內容：CREAO 的 6 階段管線：驗證 CI → 建置部署 Dev → 測試 Dev → 部署 Prod → 測試 Prod → 發布。沒有階段可跳過，沒有手動覆寫。確定性是讓 agent 能對失敗進行推論的前提——非確定性的環境讓 agent 無法學習。一天 8 次部署的基礎設施。
- 保留理由：把「CI/CD」從工程實踐拉到「agent 可推論性」的層次；提供具體階段參考。
- 待補強處：每階段的具體判準與失敗處理？緊急情境下真的不能手動覆寫嗎？
- 初步知識鉤子：與 20260404 完整循環、20260403 Validate 階段、與「人類隱性 harness 顯性化」

序號 4
- 候選標題：自我修復回饋迴圈——監控 → 分群 → 評分 → 自動建工單 → 修復後自動關閉
- 分級：Core
- 類型：Pattern
- 核心內容：CREAO 每天 UTC 9:00 啟動：Claude Sonnet 4.6 查 CloudWatch → 產生健康摘要 → 一小時後分流引擎按 9 維度評分 → 在 Linear 自動建工單（附範例日誌、受影響使用者、建議調查方向）→ 工程師推修復後同管線審查/測試/部署 → 部署完再查 CloudWatch、錯誤消失自動關工單。每個工具負責一個階段，沒有哪個工具想包辦一切。
- 保留理由：是「持續自我健康檢查」的最完整實作範例；可遷移到 data product / decision system 的 ops 設計。
- 待補強處：9 個嚴重度維度具體是什麼？小團隊版本如何裁減？
- 初步知識鉤子：與 20260412 OpenAI 30 分鐘審查、20260404 每週自動檢查、20260401 評估迴圈、Atomic Skill（每個工具負責一階段）

序號 5
- 候選標題：架構師 vs 操作員——AI 優先組織的兩極化結構
- 分級：Core
- 類型：Pattern
- 核心內容：架構師（每團隊 1-2 人）設計 SOP、教 AI 怎麼工作、決定架構邊界、定義「好」、能在 AI 方案裡找漏洞。操作員（其他人）在架構師設計的系統內工作，AI 派工單給人類，人負責調查/驗證/核准修復。組織從金字塔變成「兩極」結構。
- 保留理由：對「未來工程師角色」的清晰二分，可作為個人定位 / 招募 / 晉升設計參考；與 20260403 Junior Grit、20260327 Stripe「判斷什麼值得做」呼應。
- 待補強處：中間層（資深 IC、tech lead）會發生什麼？是否真的二分，還是 spectrum？
- 初步知識鉤子：與 capture profile「IC 到管理者重定義」「執行力到判斷力遷移」、Andy Grove

序號 6
- 候選標題：反直覺——初階適應比資深快，因為沒有 10 年舊習慣要拋棄
- 分級：Core
- 類型：Warning
- 核心內容：Pang 觀察初階工程師適應 AI 優先比資深更快——他們沒有 10 年舊習慣要拋棄，工具放大他們的影響力，他們很興奮。資深工程師反而困難最大：花 2 個月的工作 AI 一小時做完，認知衝擊巨大。同時「AI 協調者」新角色由資深承擔（驗證、整合、不寫 code）。
- 保留理由：顛覆「資深=經驗優勢」的預設，提供高槓桿 framing；對個人職涯與團隊配置都有實戰價值。
- 待補強處：哪些「資深經驗」仍是不可替代的？哪些是純 legacy？需要更精細的二分。
- 初步知識鉤子：與 capture profile「品味作為降噪器」（資深的真價值）、20260403 Junior Grit、Kent Beck「工程角色重估」

序號 7
- 候選標題：初階人才管線斷裂——3-5 年後的結構性缺口
- 分級：Core
- 類型：Warning
- 核心內容：54% 工程領導者計畫因 AI 減少初階聘用，但初階是未來架構師的養成管道。最有價值的「架構判斷力」需要多年實戰編碼、犯錯、production 事故才養得成；你跳不過操作員階段直接當架構師。如果操作員角色空洞化（只剩審 AI PR + 驗工單），晉升所需的經驗深度永遠養不成。產業正在製造 3-5 年後才浮現的人才斷層。
- 保留理由：高槓桿的長期警告，對組織決策、教育、個人職涯規劃都有戰略意義；與 20260403 Junior Grit 完整呼應。
- 待補強處：如何刻意設計「AI 增強的初階成長路徑」？哪些 production 事故經驗是不可替代的？
- 初步知識鉤子：與 capture profile「長期主義」「判斷力作為槓桿」、與 20260403 Junior Grit、與 20260327 Stripe「判斷什麼值得做」

序號 8
- 候選標題：技術債複利——AI 生成 code 看似乾淨但累積有毒
- 分級：Core
- 類型：Warning
- 核心內容：GitClear 分析 1.53 億行 code，缺乏監督的 AI 工具大量使用使技術債指標顯著攀升。CMU 對 800+ GitHub repo 記錄品質加速退化。Apiiro 對 Fortune 50 發現安全漏洞指數增長。AI code 通常格式工整、單看邏輯正確，風險在累積——不同身分驗證模式、重複工具函式、不遵守 ORM 慣例。個別微小，合在一起像複利。
- 保留理由：三項獨立研究是硬證據；對「AI 寫得很好」的樂觀派提供反例護欄。
- 待補強處：哪些技術債類型最具複利效應？是否有早期偵測指標？
- 初步知識鉤子：與 20260403 過度設計、20260404「文件比新功能重要」、與 capture profile「AI Coding 風險治理」

序號 9
- 候選標題：模型依賴是 AI 優先工程的存亡級弱點——「Opus 4.5 做不到 4.6 做到的事」
- 分級：Core
- 類型：Warning
- 核心內容：Pang 坦承轉型歸功於最近兩個月模型進步，「Opus 4.5 做不到 Opus 4.6 做到的事」。整個工程流程建立在特定前沿模型能力上 → 對供應商產生存亡級依賴。定價調整、API 政策變動、模型更新導致能力退化、暫時斷線都可能連鎖反應。高 token + 長 context + 持續跑代理，前沿模型 API 成本可能撐不住。
- 保留理由：對「all-in AI 優先」提供關鍵風險視角；呼應 capture profile「AI 成本結構與定價經濟學」「訂閱制統計賭局」。
- 待補強處：如何設計多模型 fallback？哪些 harness 元件可隔離模型差異？
- 初步知識鉤子：與 20260401「擦乾淨眼鏡 vs 換更聰明的腦袋」、與 capture profile compute cost 地板、vendor lock-in

序號 10
- 候選標題：治理缺口：91% 已用 AI agent，只 10% 有明確管理策略
- 分級：Support
- 類型：Pattern
- 核心內容：Fortune 2026/4 報告：91% 組織已在用 AI agent，但只 10% 有明確管理策略。AI agent 已是實際操作者（分析資料、啟動流程、企業內部採取行動），現有治理框架是為人類設計。治理缺口比技術缺口更大、補起來更慢。

- 初步知識鉤子：與 20260412 OpenAI misalignment monitoring、與 capture profile「shared responsibility」「上線責任」

序號 11
- 候選標題：「沒有結構的速度＝包裝比較好看的 Vibe Coding」——Harness 是產品本身
- 分級：Core
- 類型：Principle
- 核心內容：核心隱喻：馬是模型（強大但不知道往哪跑），Harness 是韁繩/鞍具/銜鐵（約束、回饋、文件、linter、CI、架構邊界）。能活下來的公司不是跑最快的，是 harness 最強的——讓 AI 力量往有用方向走的約束/回饋/人類判斷框架。沒有結構的速度只是 Vibe Coding 升級版。
- 保留理由：金句級 framing，與 20260401「Model 是原料、Skill 是工具、Harness 是讓工具湊在一起的系統」、20260407「harness 是持續工程實踐」共同形成「harness 三部曲」核心信念。
- 待補強處：「結構不夠 vs 過度結構」的判準？避免從 Vibe Coding 走到 over-harness。
- 初步知識鉤子：與 20260401、20260407 共構主軸；與 capture profile「sound vs noise」、Vibe Coding

序號 12
- 候選標題：一人公司可能是噱頭——最小可行 AI-First 團隊是「兩人」不是「一人」
- 分級：Core
- 類型：Heuristic
- 核心內容：Pang 預測「一人公司會成為常態」，但本文反論：MEDVi 兩員工 18 億美元案例可能誤導，估值不等於實質企業。比較務實的版本是 5-15 人達成過去 100-200 人的產出。Imbue CEO 認為獨立 AI 公司先在 consumer/prosumer 成功，B2B 困難（信任/關係難代理）。真正限制不在技術，在組織韌性——「公司只有一個人懂 agent，那人離開就完了」。最小可行是兩人。
- 保留理由：對「一人公司」浪潮的 reality check；提供具體規模 framing；組織韌性視角是常被忽略的維度。
- 待補強處：兩人團隊的角色配置（兩個架構師？一架構師一營運？）？知識交接機制？
- 初步知識鉤子：與 capture profile「solo consultant / fractional executive」「一人公司」迷思、bus factor
