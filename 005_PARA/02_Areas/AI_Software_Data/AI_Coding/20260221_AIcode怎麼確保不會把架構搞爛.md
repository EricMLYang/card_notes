AI 寫的 code 越來越多，怎麼確保不會把架構搞爛？

Pragmatic Engineer 的 Gergely Orosz 深度拆解了 OpenAI Codex 團隊的做法：Harness Engineering

核心概念：用機械式規則強制架構一致性，而不是靠 code review 或人的紀律

具體做法是定義嚴格的單向依賴鏈：
Types → Config → Repo → Service → Runtime → UI
每一層只能依賴上一層，違反就自動擋掉

這樣不管是人寫的還是 AI 寫的 code，都不可能破壞架構

另一個有趣的點：Codex 團隊分成 CLI 團隊和 Web 團隊雙軌並行，各自獨立迭代但共用底層協議

當 AI agent 每天產出上千行 code，架構守護不能靠人，要靠 harness

# BreadCards

## A. 主脈絡與個人映射
- 論證骨架：當 AI 產出 code 速度遠超過人類審查速度，傳統靠 code review 與工程師紀律的架構防線會失效；唯一能跟上的方式是把架構規則「機械化」內嵌進工具鏈，讓違規自動被擋。
- 作者挑戰的預設：架構一致性必須靠資深工程師守門。實際上靠人會在 AI 規模下崩潰，必須靠 harness。
- 個人映射：呼應 Harness Engineering 主軸（loop / verification / guardrails），把架構防線視為「可被 AI 24/7 觸發的自動約束」，補強 Repo-as-Worker 視角中「結構性護欄」這一塊。

## B. 候選卡（Lite）

序號 1
- 候選標題：架構規則機械化是 AI 規模下的唯一防線
- 分級：Core
- 類型：Principle
- 核心內容：當 AI 每天產出上千行 code，依靠 code review 或人類紀律無法即時擋住架構違規。Codex 團隊的做法是把單向依賴鏈（Types → Config → Repo → Service → Runtime → UI）寫進工具鏈強制檢查，違規直接擋下。這把「守架構」從人類判斷移到 harness 自動約束，讓人類只需設計規則，不需逐 PR 巡邏。
- 保留理由：明確把 AI 規模下的架構治理問題與機制化解法綁在一起，是高訊號的取捨原則。
- 待補強處：缺乏失效條件與規則設計反例。
- 初步知識鉤子：Harness Engineering、可觀測性護欄、AI 24/7 運作的最小控制集、Code Review 的邊界。

序號 2
- 候選標題：單向依賴鏈作為架構不被 AI 搞爛的具體機制
- 分級：Support
- 類型：Pattern
- 核心內容：Codex 把架構分層為 Types → Config → Repo → Service → Runtime → UI，並規定每一層只能依賴上一層，違反就被工具自動擋掉。這是一個可被 AI 與人類同時遵守的硬性結構，因為它不依賴任何「品味」或「資深判斷」，只要分層定義穩定，違規就是可機器檢查的事實。
- 保留理由：把「架構守護機械化」具體化為一個可遷移的層級設計範本。
- 待補強處：六層適用邊界、跨層 shortcut 例外處理機制未說明。
- 初步知識鉤子：Layered Architecture、Adapter Pattern、Repo-as-Worker、Domain 分層治理。

序號 3
- 候選標題：雙軌獨立迭代 + 共用底層協議的團隊拆分
- 分級：Support
- 類型：Pattern
- 核心內容：Codex 團隊分成 CLI 團隊與 Web 團隊雙軌並行，各自獨立迭代但共用底層協議。這種設計讓兩個介面可以各自演進，而共享的底層協議成為跨團隊一致性的單一真實來源，避免「為了共用元件而拖累發版速度」與「各自亂來造成系統分裂」兩種極端。
- 保留理由：可遷移到資料平台多前端／多客戶情境，是組織與技術耦合的取捨範本。
- 待補強處：底層協議如何 versioning、衝突如何裁決未交代。
- 初步知識鉤子：Platform Engineering、API-first、多租戶資料平台、團隊拓撲（Team Topologies）。
