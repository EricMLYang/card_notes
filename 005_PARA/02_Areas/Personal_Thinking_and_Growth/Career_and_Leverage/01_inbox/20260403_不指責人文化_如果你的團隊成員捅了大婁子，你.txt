如果你的團隊成員捅了大婁子，你會怎麼應對? 
󠀠󠀠
這週在科技圈討論度最高的話題，莫過於 Claude Code 的某個部署失誤，導致 source map 公開，進而讓原本閉源的程式碼變公開。
󠀠󠀠
這幾天社群有非常多解析 Claude Code 原始碼的內容，但相比之下，我們覺得更值得討論的是「當團隊中有人捅了大婁子，Claude Code 的團隊負責人如何應對?」。
󠀠󠀠
在社群中，很多人在討論搞砸該次部署的工程師，會不會被開除；但 Claude Code 創作者兼負責人 Boris Cherny 公開回應
󠀠󠀠
「出錯在所難免。身為一個團隊，最重要的是認清楚問題從來不在於某個人，而在於流程、文化或基礎建設。
󠀠󠀠
就這次的情況而言，有一個手動部署步驟本應被更好地自動化。我們的團隊已針對這部分做了一些改善，後續還有幾項優化正在進行中」。
󠀠󠀠
這種「不指責人的事故檢討」是我們最推崇的應對方式。
󠀠󠀠
之前分享過，Google 公開的 SRE 專書當中，談到事故檢討的章節，花很大的篇幅在講不指責人的事故檢討 (感興趣的讀者，該章節連結放在下方留言處)。
󠀠󠀠
要做好事故檢討這件事，需要先釐清事故檢討的本質。造成事故對個人與團隊肯定都會有傷害，然而沒有人是完美不會犯錯的，所以事故是在所難免。不過，當事故已經發生了，該專注的就是解決，以及確保未來可以做更好。
󠀠󠀠
事故檢討的核心重點是從失敗中學習，具體來說，首先是要去釐清如何確保同樣的問題，在未來不會再次發生? 假如真的無法完全避免問題，那有沒有方法能讓問題及早被發現? 以及當發現問題後，有沒有更快解決問題的方法?
󠀠󠀠
Boris 的這個回應中，體現了「不指責人 + 專注在改善系統讓同樣事故不會再發生」，是我們認為很值得學習的典範。
󠀠󠀠󠀠
假如你對如何打造更可靠的軟體這主題感興趣，我們在 E+ 成長計畫有更多深入的討論，包含做監控(montoring)、可觀測性 (observability)、輪班 (on-call)、事故檢討等等，此外也有軟實力的內容談如何應對各種工作中的突發狀況。
