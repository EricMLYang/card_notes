【 開發者（非 Infra）必備資安知識：4 個階段、9 週、21 個檢查點 】
——從「資安要注意」到「清楚知道自己負責哪一份」

▋ 起點

S3 bucket 公開設定錯誤、IAM 權限給太寬、Access Key 存在 GitHub——這三件事每年讓企業損失數十億美元。

駭客的水準不是重點，重點是：開發者以為這些事情「有人在管」。

某次上線前資安審查報告回來，有一半的問題出在應用層。不是 Infra 沒設好，是開發時沒有把這些東西列進去。

這篇整理的，就是「開發者那份」的清單。

---

▋ 責任共擔模型

雲端開發者都該讀一遍 AWS 的 **Shared Responsibility Model**。

**AWS 負責「雲端本身」的安全**（Security *of* the Cloud）——機房、硬體、虛擬化層。

**你負責「雲端上」的安全**（Security *in* the Cloud）——IAM、資料加密、應用層。

S3 公開、IAM 過寬、Access Key 進 Git，全是你的責任，AWS 管不到。

---

▋ 最小可行資安（MVS）

資安是個無底洞，學完 IAM 還有 VPC，學完 VPC 還有 WAF。

這份清單的出發點是 **MVS（Minimum Viable Security）**——讓你的系統不是最容易被找到漏洞的那個。

有邊界，有底線，其他的依資源排優先。

---

▋ 第一階段：身份管理（2 週）

幾乎所有大型入侵事件的共同起點：某個不該有的人，拿到了不該有的憑證。

- **Root 帳號鎖起來。** 加上 MFA 後封存不用。
- **最小權限原則。** 每次開權限前問：這個 Role / User 最少需要什麼？
- **服務間通訊用 Role，不用 Access Key。** Key 出現在 `.env`、Git history 就等於公開密碼。
- 機密資訊存 **AWS Secrets Manager**，應用啟動時動態讀取，密碼不落地。

---

▋ 第二階段：網路與資料保護（3 週）

VPC 設計直覺：越敏感的東西，離外網越遠。

```
Internet → Public Subnet（LB）→ Private Subnet（App）→ Data Subnet（DB，不對外）
```

Security Group 預設拒絕全部，只開必要 port。加密兩個基本要求：

- **Encryption at rest** — S3、RDS 儲存加密，一個開關的事
- **Encryption in transit** — 強制 HTTPS，設定 HSTS header

S3 的 **Block Public Access 要保持開啟**，手動關掉導致 bucket 公開的案例比想像中多。

---

▋ 第三階段：API 與應用層（3 週）

應用層是開發者最能控制、也是漏洞最密集的地方。

**OAuth 2.0 + JWT**：Access Token 短期（約 15 分鐘），Refresh Token 長期。Token 不要存 localStorage——XSS 一段腳本就能全部讀走。改用 HttpOnly Cookie。

**輸入驗證**是防 Injection 的基本防線，白名單原則：只接受預期格式，其餘拒絕。AI 生成的 code 在 happy path 沒問題，但 Input Validation 需要 reviewer 主動檢查。

API Gateway 前加 **AWS WAF**，開啟 Managed Rules 自動過濾 SQL Injection、XSS、DDoS。

---

▋ 第四階段：自動化守門（1 週）

設定一次，每次 PR 自動執行。CI/CD pipeline 加三件事：

- **Secret Scanning** — 密碼有沒有 commit 進去（GitHub 免費版）
- **Dependency Check** — Dependabot 或 Snyk，套件已知漏洞
- **SAST** — SonarCloud，程式碼安全漏洞模式

整合時間約半天。沒設定的狀態下，每次上線都是在賭。

---

▋ 21 個必做檢查點

下次上線前，對一遍這份清單。

**身份管理（6 個）**
- [ ] Root 帳號已關閉日常使用，已開啟 MFA
- [ ] 所有 IAM User 都已開啟 MFA
- [ ] 服務間通訊使用 IAM Role，而非 Access Key
- [ ] 用 IAM Access Analyzer 掃描過寬的權限（建議每季一次）
- [ ] 資料庫密碼、API Key 存在 Secrets Manager，不在程式碼或 .env 裡
- [ ] Git history 已確認沒有明文 secret（truffleHog 或 git-secrets）

**網路與資料（6 個）**
- [ ] 資料庫放在 Data Subnet，不對外暴露
- [ ] Security Group 只開放必要 port，其餘全部拒絕
- [ ] S3 Block Public Access 保持開啟
- [ ] S3 和 RDS 已啟用 Encryption at rest
- [ ] 所有外部通訊強制 HTTPS，已設定 HSTS header
- [ ] RDS 自動備份已開啟（保留 ≥ 7 天），重要 S3 bucket 已開啟 Versioning

**API 與應用層（6 個）**
- [ ] JWT 驗證流程已正確實作，Access Token 有過期機制
- [ ] 所有外部輸入有 Input Validation（SQL Injection、XSS 防護）
- [ ] API 有 Rate Limiting / Throttling 設定
- [ ] API Gateway 前已掛 WAF（開啟 AWS Managed Rules）
- [ ] 錯誤訊息不洩漏系統細節（stack trace 不往外傳）
- [ ] CloudTrail 和 GuardDuty 已開啟

**前端與 CI/CD（3 個）**
- [ ] 前端 Token 使用 HttpOnly Cookie 或 Memory，不用 localStorage
- [ ] CSP Header 已設定，外部腳本有明確白名單
- [ ] CI/CD pipeline 有 Secret Scanning + Dependency Vulnerability Check

---

▋ 收尾

資安問題很少是「被高水平駭客入侵」，更多是「某個開發者以為有人在管，但其實沒有」。

把自己那份做好，不需要成為資安專家。21 個檢查點，從下次上線前開始對。
