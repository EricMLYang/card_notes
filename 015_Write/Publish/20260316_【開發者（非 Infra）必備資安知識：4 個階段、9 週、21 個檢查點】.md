【 開發者（非 Infra）必備資安知識：4 個階段、9 週、21 個檢查點 】
——從「資安要注意」到「清楚知道自己負責哪一份」

▋ 起點

S3 bucket 公開設定錯誤、IAM 權限給太寬、Access Key 存在 GitHub。

這三件事每年讓企業損失數十億美元。

駭客的水準不是重點，重點是：開發者以為這些事情「有人在管」。

外部產品開發的過程裡，「資安要注意」這句話我聽過很多次。

但沒有人告訴我具體是什麼。

要學到哪個程度？哪些是開發者的責任、哪些是 Infra 的、哪些是 AWS 的？

直到某次上線前資安審查報告回來，有一半的問題出在應用層。

不是 Infra 沒設好。是我們開發的時候沒有把這些東西列進去。

這篇整理的，就是「開發者那份」的清單。

---

▋ AWS 教給我最重要的一課

AWS 有一份文件，雲端開發者都應該讀一遍：**Shared Responsibility Model（責任共擔模型）**。

**AWS 負責「雲端本身」的安全**（Security *of* the Cloud）。

實體機房、硬體、虛擬化層、底層網路——這些 AWS 扛。

**你負責「雲端上」的安全**（Security *in* the Cloud）。

IAM 設定、資料加密、作業系統更新、應用層的所有東西——這些是你的。

很多資安事件的起點，是開發者以為上了雲就安全了。

S3 bucket 設為公開、IAM 權限給太寬、Access Key 出現在 Git——

這些全部都是 Security *in* the Cloud 的問題。責任在開發者這邊，AWS 管不到。

這張責任地圖，是學資安之前最重要的一個認知校正。

---

▋ 最小可行資安（MVS）

資安是個無底洞。

學完 IAM 還有 VPC，學完 VPC 還有 WAF，學完 WAF 還有滲透測試。

你可以無限往下挖，但挖到最後會發現大多數知識跟你日常開發的關係越來越遠。

這份清單的出發點是 **MVS（Minimum Viable Security）**。

一個實際的目標：讓你的系統在安全性上，不是最容易被找到漏洞的那個。

Code Review 的邏輯也是這樣——你不需要把每一行都審到完美，

但任何會進正式環境、影響資料安全的變更，不應該跳過系統層的審查。

有邊界，有底線，其他的依資源排優先。

4 個階段、9 週、21 個檢查點，這是我目前認為「夠用」的版本。

---

▋ 第一階段：身份管理（2 週）

幾乎所有大型入侵事件，追溯到最後都有一個共同起點：

某個不該有的人，拿到了不該有的憑證。

IAM 的實務核心只有三件事：

**Root 帳號鎖起來。** 只用來開帳號，加上 MFA 之後封存不用。

**最小權限原則。** 需要什麼開什麼，不多給一分。每次開權限前問一個問題：這個 Role / User 最少需要什麼？

**服務間通訊用 Role，不用 Access Key。** Access Key 是最容易洩漏的東西——`.env` 檔案、`config.json`、Git history，只要出現在這些地方就等於公開密碼。

機密資訊的正確存放方式是 **AWS Secrets Manager**：

應用程式啟動時動態讀取，密碼不落地、不在 code 裡流動。

這兩件事做好，已經比大多數沒特別注意的系統安全一個量級。

---

▋ 第二階段：網路與資料保護（3 週）

VPC 的設計只有一個直覺：越敏感的東西，離外網越遠。

```
Internet
  ↓
Public Subnet（Load Balancer）
  ↓
Private Subnet（應用程式伺服器）
  ↓
Data Subnet（資料庫，完全不對外）
```

Security Group 是每個 instance 的防火牆。

預設拒絕全部，只開放必要的 port。對外服務通常只需要 443。

資料加密有兩個基本要求，設定起來成本幾乎是零：

**Encryption at rest** — S3 和 RDS 的資料儲存時加密，在服務設定裡一個開關的事。

**Encryption in transit** — 強制 HTTPS，設定 HSTS header，不讓明文在網路上傳輸。

S3 有一個地方要特別注意：**Block Public Access 要保持開啟。**

這個開關被手動關掉、導致整個 bucket 公開可讀的案例，比想像中多很多。

---

▋ 第三階段：API 與應用層（3 週）

應用層是開發者最能控制的地方，也是漏洞最密集的地方。

身份驗證用 **OAuth 2.0 + JWT** 是現代 API 的標配。

Access Token 短期（15 分鐘左右），Refresh Token 長期，有明確過期機制。

Token 存放的位置很重要——**不要存在 localStorage**。

XSS 攻擊只需要一段注入的腳本，就可以把 localStorage 的內容全部讀走。

改用 HttpOnly Cookie，讓 JavaScript 碰不到 token。

輸入驗證是防止 Injection 攻擊的基本防線：

```python
# 有問題的寫法：直接拼接 SQL
query = f"SELECT * FROM users WHERE name = '{user_input}'"

# 正確做法：參數化查詢
query = "SELECT * FROM users WHERE name = %s"
cursor.execute(query, (user_input,))
```

白名單原則：只接受預期格式，其餘一律拒絕。長度、字元集、型別——都要驗。

這個部分也是 AI 輔助開發時最容易出問題的地方。

AI 生成的 code 在 happy path 上通常沒問題，但 Input Validation 和邊界條件的處理，需要 reviewer 主動去問那個問題。

API Gateway 前面加上 **AWS WAF**，開啟 AWS Managed Rules——

SQL Injection、XSS、常見 DDoS 模式，自動過濾，不需要自己從頭寫規則。

---

▋ 第四階段：自動化守門（1 週）

設定一次，每次 PR 自動執行。問題在進 main branch 之前就被攔截。

前端幾個常踩的點先確認：

- `v-html` / `dangerouslySetInnerHTML` 直接插入未驗證的 HTML → XSS 直接開門
- Token 存在 localStorage → XSS 一攻就全部洩漏
- 沒有設定 CSP（Content-Security-Policy）Header → 外部腳本可以任意執行

CI/CD pipeline 加三件事：

```
Secret Scanning — 有沒有把密碼 commit 進去（GitHub 有免費版）
Dependency Check — Dependabot 或 Snyk，套件有沒有已知漏洞
SAST — SonarCloud，程式碼有沒有常見安全漏洞模式
```

這三個工具，GitHub 上幾乎都有免費方案，整合時間大約半天。

沒有設定的狀態下，每次上線都是在賭。

---

▋ 21 個必做檢查點

下次上線前，對一遍這份清單。

**身份管理（6 個）**
- [ ] Root 帳號已關閉日常使用，已開啟 MFA
- [ ] 所有 IAM User 都已開啟 MFA
- [ ] 服務間通訊使用 IAM Role，而非 Access Key
- [ ] 用 IAM Access Analyzer 掃描過寬的權限（建議每季一次）
- [ ] 資料庫密碼、API Key 存在 Secrets Manager，不在程式碼或 .env 裡
- [ ] Git history 已確認沒有明文 secret（可用 truffleHog 或 git-secrets 掃描）

**網路與資料（6 個）**
- [ ] 資料庫放在 Data Subnet，不對外暴露
- [ ] Security Group 只開放必要 port，其餘全部拒絕
- [ ] S3 Block Public Access 保持開啟
- [ ] S3 和 RDS 已啟用 Encryption at rest
- [ ] 所有外部通訊強制 HTTPS，已設定 HSTS header
- [ ] RDS 自動備份已開啟（保留 ≥ 7 天），重要 S3 bucket 已開啟 Versioning

**API 與應用層（6 個）**
- [ ] JWT 驗證流程已正確實作，Access Token 有過期機制
- [ ] 所有外部輸入有 Input Validation（SQL Injection、XSS 防護）
- [ ] API 有 Rate Limiting / Throttling 設定
- [ ] API Gateway 前已掛 WAF（開啟 AWS Managed Rules）
- [ ] 錯誤訊息不洩漏系統細節（stack trace 不往外傳）
- [ ] CloudTrail 和 GuardDuty 已開啟

**前端與 CI/CD（3 個）**
- [ ] 前端 Token 使用 HttpOnly Cookie 或 Memory，不用 localStorage
- [ ] CSP Header 已設定，外部腳本有明確白名單
- [ ] CI/CD pipeline 有 Secret Scanning + Dependency Vulnerability Check

▋ 收尾

整理這份清單的過程中，我想到一個觀察：

資安問題很少是「被高水平駭客入侵」，

更多是「某個開發者以為有人在管，但其實沒有」。

這份清單的設計初衷，就是把那個模糊地帶說清楚——

Infra 有 Infra 的責任範圍，AWS 有 AWS 的，開發者有開發者的。

把自己那份做好，不需要成為資安專家，也不需要一次學完所有東西。

21 個檢查點，從下次上線前開始對。
