【 以前串接因果結構快把人燒乾，現在有人幫你把執行環境想好了 】
——從 Databricks 五環能力模型，重新理解「數據如何變決策」
ㅤ
ㅤ
▋ 以前做決策輔助研究的時候，我遇到一個很挫敗的現實。
ㅤ
理論上很美的東西，實作上串不起來。
ㅤ
因果發現需要特徵工程。
情境模擬需要多個變數同時校準。
滾動更新需要即時的資料傳導鏈。
ㅤ
每一塊單獨做都還行。
但要把它們接成一個能動的決策系統？
ㅤ
沒有 Agent 可以協助。
沒有語義層讓不同系統說同一種語言。
沒有上下文服務自動打包即時資訊。
ㅤ
全靠人工手動串接。
ㅤ
那段經歷讓我很確定一件事。
數據與決策之間的斷層，問題出在架構，不在技術本身。
ㅤ
所以當我看到 Databricks 提出的五環能力模型（Composable Canvas）時，第一反應是——「這就是我以前缺的那套執行環境。」
ㅤ
這篇文章想把五環模型拆開，對接到過去做決策系統時的實戰痛點。
ㅤ
ㅤ
▋ 為什麼「有數據」還是做不了決策
ㅤ
這件事有三個結構性障礙。每一個都跟演算法無關。
ㅤ
**整合技術債的指數成長。**
ㅤ
傳統架構是「應用程式中心」模式。每引進一個新 SaaS 工具，為了讓它拿到決策需要的數據，得跟現有的 n 個系統做點對點整合。
ㅤ
整合成本是 O(n²) 的。
ㅤ
實務上，IT 團隊 80% 的精力消耗在維護脆弱的 Data Pipeline 和同步延遲上。
真正在優化決策邏輯的時間少得可憐。
ㅤ
**語義的破碎化。**
ㅤ
決策需要完整的上下文。
但客戶行為、企業營運、內容資產被鎖在不同系統裡。
ㅤ
一個系統說的「客戶」，跟另一個系統說的「客戶」，可能根本是不同的東西。
「轉化」也是。
ㅤ
語義不一致的情況下，自動化決策只會更快地傳遞錯誤的判斷。
ㅤ
**靜態預測跟不上動態現實。**
ㅤ
傳統分析多是事後回溯或定點預測。
但商業場景裡的決策是連續的、滾動的。
ㅤ
一個關鍵指標更新了——比方 GDP 修正、終端價格劇變——如果系統沒辦法自動觸發傳導鏈修正，那個預測在產出的瞬間就過期了。
ㅤ
以前做決策輔助框架的時候，最痛的就是這裡。
想做滾動更新，但資訊到推論的傳導鏈全部是斷的。
外部條件一變，就得重新手動跑一輪。
ㅤ
ㅤ
▋ 五環模型：終於有人把這條路想清楚了
ㅤ
Databricks 這份白皮書，本質上在宣告「應用程式堆疊時代」的終結。
ㅤ
取而代之的是以「資料情報平台」為核心的可組合畫布。
ㅤ
讓我覺得有意思的地方，是它把企業數據資產重新拆成五種可治理的資料：
ㅤ
客戶資料（Customer）——行為與軌跡。
企業資料（Company）——庫存、財務等內部上下文。
內容資料（Content）——創意與非結構化資訊，AI Agent 推論的燃料。
代碼資料（Code）——模型權重、Prompt 與 Agent 技能。
控制資料（Control）——語義定義與治理護欄，確保 AI 行為可控。
ㅤ
特別想提「控制資料」這一類。
ㅤ
過去很少有人把「規則」和「標準」當成需要治理的數據資產。
但在 Agent 時代，你對品質的定義、對風險的容忍度、對決策邊界的判斷——這些如果沒有被編碼成可管理的數據，Agent 等於在裸奔。
ㅤ
ㅤ
▋ 五環怎麼運作
ㅤ
五環能力模型把「數據→決策→行動」拆成五層同心環。
我覺得這是白皮書裡最有建設性的部分。
ㅤ
**Ring 1：Data Core。**
統一存放五類資料。不再是每個 App 自己存自己的。
ㅤ
**Ring 2：Semantic Layer。**
標準化指標與邏輯，讓所有應用程式與 Agent 說同一種語言。
ㅤ
這就是前面說的語義破碎化的解法。
當所有人對「客戶」「轉化」「營收」的定義一致，AI 的推論才有共同基礎。
ㅤ
**Ring 3：CaaS（Context-as-a-Service）。**
動態決策的關鍵。把零散數據打包成即時上下文，供 Agent 隨時取用。
ㅤ
過去我最渴望的「資訊一更新，推論就跟著滾動」，在這一環有了工程上的解法。
ㅤ
**Ring 4：Decisioning。**
獨立的決策引擎，負責在多重目標下做效用計算與權衡。
ㅤ
這一環最被低估，也最重要。
它把決策邏輯從特定 App 中抽離，變成獨立服務。
ㅤ
**Ring 5：Apps & Agents。**
執行端。消費前四環提供的數據、語義、上下文與決策結果，轉化為行動。
ㅤ
ㅤ
▋ 以前做不到的事，現在怎麼接上的
ㅤ
回頭看以前做的決策輔助研究，核心邏輯其實沒有錯。
當時缺的是執行環境。
ㅤ
**產生方案——以前靠手動建模，現在靠語義驅動。**
ㅤ
做因果發現和情境模擬，以前得針對每個問題做繁瑣的特徵工程。
沒有人幫你管理「這個指標到底代表什麼」「這兩個變數之間有沒有因果關係」。
ㅤ
有了語義層（Ring 2），數據有了業務意義。
Agent 可以直接在語義層上推論，因果分析可以大規模自動化。
ㅤ
以前這件事完全不可想像。
ㅤ
**權衡方案——以前靠人腦扛，現在有獨立的決策層。**
ㅤ
過去做的研究核心是「不確定性與風險權衡」。
用價值函數和偏好建模，在多個方案中找最適的那一個。
但邏輯綁死在特定應用場景裡，換個部門就得重來。
ㅤ
Ring 4 把決策邏輯下沉為獨立服務。
企業可以跨部門、跨渠道套用一致的風險評估標準。
ㅤ
生成式 AI 把「產生方案」的成本打到極低之後，真正的商業壁壘已經轉移到「判斷」上。
判斷需要的是穩定的決策框架，不是更花俏的 Prompt。
ㅤ
**滾動更新——以前是奢望，現在 CaaS 接上了。**
ㅤ
以前最理想的願景：外部資訊一更新，預測就跟著滾動修正。
但沒有任何系統能自動做到。
ㅤ
Ring 3 提供了工程上的解法。
市場訊號進入 Data Core，CaaS 自動打包成最新上下文提供給 Agent。
Agent 隨即觸發推論，判斷結果是否需要修正。
ㅤ
人工介入不再是必要條件。
數據與決策之間的齒輪，真正嚙合了。
ㅤ
ㅤ
▋ 最深的體悟
ㅤ
看完這份白皮書，最觸動我的不是某個技術細節。
ㅤ
以前追求「精確預測」，常常陷在模型的世界裡。
覺得模型準一點、數據多一點、演算法新一點，決策品質就會提升。
ㅤ
但五環模型揭示的是更根本的東西：重點在 Harness——讓模型真正落地運作的那套執行環境。
ㅤ
在畫布上，數據是流動資產（Ring 1），邏輯是標準化語義（Ring 2），Agent 根據即時上下文（Ring 3）持續進行決策（Ring 4）並執行（Ring 5）。
ㅤ
Harness 包含的是迴圈、工具、上下文管理、持久化、驗證、限制條件。
兩個團隊用同一個模型，做出來的 Agent 體驗可以像完全不同的物種。差別就在 Harness 的設計。
ㅤ
再往深處想，Harness 其實承載的是品味。
你給 Agent 什麼工具、強制什麼順序、用什麼標準衡量成功、怎麼從失敗恢復——這些選擇反映的是你的價值觀。
ㅤ
在所有人都能複製貼上 Agent 的時代，價值觀才是真正的差異化。
ㅤ
ㅤ
▋ 這對做數據的人意味著什麼
ㅤ
架構現代化的本質，是把人的品味與判斷編碼進 Control Data 和 Decisioning Layer。
ㅤ
做數據的人的角色正在慢慢轉變。
從「維護好水管」，到「設計一套讓 Agent 穩定運行的機制」。
確保外部世界變動時，系統能依據你設定的價值觀，做出邏輯一致的決策。
ㅤ
對我來說，這是對過去那些願景的遲來實踐。
ㅤ
以前苦思良久的「數據如何變決策」——答案不在找到一個完美的算法，而在構建一個能讓算法、上下文與行動原生交互的生態系統。
ㅤ
以前串接因果結構串到快燒乾，不是方法論不對。
是執行環境從來沒被認真設計過。
ㅤ
現在有人幫你想好了。
剩下的是，你願不願意重新理解自己工作的位置。
