【 這是我第二次被 AI 開發方式震撼到 】
—— 從 Claude Code 到 Genie Code
ㅤ
最近兩周，我持續在 Databricks 上做一件事：把分析能力往真正 Agent 的方向推。
ㅤ
串 Vector Search、table 數據、Genie Code，讓不同 AI 角色分工協作，甚至讓 Agent 用測試數據把完整流程驗過一次。
ㅤ
做著做著，突然意識到：這是我第二次被 AI 的開發方式震撼到。
第一次是 Claude Code，第二次是 Genie Code。
但這兩次震撼的本質非常不同。
ㅤ
ㅤ
▋ 堆 prompt 的年代
ㅤ
回頭看 ChatGPT 3.5 到 4o 那段時間，我們就已經開始構想自動分析流程。
ㅤ
只是那時候的模型，延遲高、成本高、效果普通。
為了撐起來，得堆很多 prompt。
規則越疊越多，改起來不穩。
ㅤ
你會感覺自己不是在建系統，是在維持一堆脆弱的技巧。
ㅤ
現在回想，核心問題不是模型離數據太遠。
真正有價值的推理只佔流程一小部分，其餘全在搬資料、組 context、處理例外。
ㅤ
ㅤ
▋ 第一次震撼：AI 住進了你的 repo
ㅤ
Claude Code 讓我徹底改觀。
ㅤ
它不是在雲端遙遙看你的程式碼，而是真的能進 repo、理解檔案、動手修改。
像一個很強的協作者，直接坐在你旁邊一起寫。
ㅤ
Karpathy 說得好：「AI 應該住在你的電腦上，而不是雲端。」
關鍵不在運算發生在哪裡，而在你環境裡已存在的一切都能被直接利用。
ㅤ
ㅤ
▋ 第二次震撼：AI 住進了你的數據
ㅤ
Genie Code 做的是同一件事，但場景完全不同。
ㅤ
它本來就在你的資料環境裡。
天然能碰到你的 table、schema、catalog，還能搭配 Vector Search、UC Functions、MLflow 一起工作。
ㅤ
Genie Code 出來沒多久，但兩周內我已經在三條線上推進：
ㅤ
→ 把非結構化和結構化資料串在一起做整合判讀。真正的業務分析，很少只靠單一來源就能講出完整答案。
ㅤ
→ 讓 PM Agent 先拋數據狀況，再由 Genie Code 規劃後續分析。不是把所有事情塞給同一個 Agent，而是讓不同角色各做最適合的部分，開始有一點像真正的協作流程。
ㅤ
→ Pipeline 改完後，讓它用測試數據跑端到端驗測。價值從「幫你寫程式」延伸到「幫你確認這個流程到底有沒有真的成立」。
ㅤ
ㅤ
▋ 為什麼「住在哪裡」比「多聰明」更重要
ㅤ
有個數據很有意思：95% 的 AI Agent 在 Production 環境會失敗。
問題不在模型不夠聰明，而在 context 管理、memory 設計這些周邊工程。
ㅤ
大多數人以為自己在做 AI 產品，其實是在做 context selection 系統。
ㅤ
Genie Code 天然坐在數據環境裡，context 不需要額外搬運。
這是整個架構複雜度降低一個量級的起點。
ㅤ
用 AI 應用三層進化來看——L1 建議層、L2 執行層、L3 代理層，

大多數分析工具還停在 L1，只輸出建議不承擔後果。

Genie Code 正在推動從 L1 到 L2 的躍遷：不只告訴你答案，還能動手去查、去比對、去驗證。
ㅤ
ㅤ
▋ 架構判斷
ㅤ
我現在的規劃很清楚：把 Genie Code 放在分析 Agent 的核心位置。
ㅤ
核心推理負責 Observe、Plan、Act。
外圍配上 Vector Search 承接非結構脈絡、table 數據提供事實基礎、UC Functions 和 MLflow 承接操作能力、tracing 機制負責驗證。
ㅤ
我問過自己一個問題：如果明年模型能力翻倍，我的系統會不會自動變得更便宜、更簡單？
Databricks 原生能力會隨平台一起長。這比固化在某個第三方框架上，結構性地更合理。
ㅤ
ㅤ
▋ 誠實說最大的阻礙
ㅤ
Genie Code 目前還不能被動被呼叫。
不能進 job、不能被排程、不能被事件觸發。
很多本來可以自動化的分析流程，最後只能停在半人工階段。
ㅤ
Databricks 的產品團隊，真心拜託：讓 Genie Code 可以被動呼叫，甚至變成 API endpoint。
這東西如果開放到那一步，很多人不是「會想試試看」，而是真的會大量拿去用。
ㅤ
ㅤ
▋結尾
ㅤ
這兩周最強烈的感受不是「又多了一個 AI 工具」。
ㅤ
而是當年的一些想法，終於碰到比較對的時代了。
ㅤ
我們正在目睹分析工具從「只會說」跨進「真的能做」的轉折。
而這個轉折的起點，不是更聰明的模型，是一個終於住對地方的 AI。
