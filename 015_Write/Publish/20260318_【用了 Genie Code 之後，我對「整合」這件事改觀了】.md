【 用了 Databricks Genie Code 之後，我對「整合」這件事改觀了 】

最近實際用了一段時間 Databricks 的 Genie Code。

體感非常好(也有可能新產品流程還沒很好，之前 Debug 很痛苦)

現在來看，它其實沒有什麼驚天動地的新技術。

但用了一段時間之後，我開始注意到一件事：
很多原本看起來只是小摩擦的地方，好像被悄悄拿掉了。

少一次環境切換。
少一次把結果貼回去給 AI 看。
少一次補充「其實我的 table 是這樣設計的」。

這些事情單看都不大。
但累積起來，我開始懷疑一件事：
也許很多 AI 產品讓人覺得「好像還行」，問題不在模型不夠聰明，而在整合深度還不到位。


▋ 以前我對整合的理解，大概停在「有接上就好」

系統 A 能接系統 B，資料可以流動，畫面上可以操作，整體流程跑得通。
大概就算是一種整合。

很多系統確實就是先把流程打通，再慢慢優化細節。
這個做法也沒什麼問題。

但這次用 Genie Code，讓我感覺到一種不一樣的東西。

它的體驗之所以順，很大一部分好像跟 AI 模型本身關係不大。
比較關鍵的是——它幾乎完全「活在」Databricks 的環境裡。
直接接觸到整個資料平台的上下文：運算環境、物件關係、執行結果。
全部在同一個脈絡裡。

之前讀過一個觀點，現在回想覺得蠻精準的：
把「會聊天的 LLM」和「能交付成果的 Agent」分開的，
是 Harness，也就是驅動框架。
模型是原始認知能力，而迴圈、工具、上下文管理、驗證機制，才是讓 AI 真正能做事的外骨骼。

兩個產品用同一個模型，體驗可以像不同物種。
差別就在這一層。


▋ 那些看起來很小的摩擦，加起來其實很大

這種深度帶來的感覺很微妙。

沒有某個功能特別厲害。
就是很多原本的小摩擦不見了。

如果你有用過各種 AI coding 工具，大概知道我在說什麼。
AI 其實很多事情「能做」，但你要花不少力氣幫它理解你的環境。
這裡用什麼命名、那張 table 的欄位長什麼樣、上次那個 query 的結果是什麼。

這些事情每一件都不大。
但它們讓 AI 停留在「輔助工具」的層次。
你還是那個理解全局的人，AI 只是幫你做零碎的事。

當 AI 真正活在你的工作環境裡，情況就不太一樣了。
它能直接接觸到整個平台的 context，開始有機會做更完整的事。

最近讀到一個說法蠻有共鳴的：
「大多數創辦人以為自己在做 AI 產品，其實是在做 context selection 系統。」
多數 Agent 在正式環境失敗，問題在 context 管理、memory 設計、錯誤處理這些看起來不性感的工程問題。

整合深度，本質上就是在解決這件事。


▋ 小團隊做 AI 產品，也許不該怕做太深

這也讓我重新想了小團隊做產品的方向。

以前我的直覺是：資源有限，先做輕、做廣、做快。
整合太深太重，等產品長大了再慢慢補。

現在有點懷疑這個直覺。

一個系統如果只是表面上接了很多東西，AI 其實很難理解整個環境。
很多能力最後還是停留在「好像可以但總差一點」的狀態。
在一個具體場景裡把資料、流程、工具和權限都整得很深，
AI 比較有機會真的進入那個工作環境。
能夠執行、能夠驗證、能夠反覆改進。

換個角度想，專注在一個場景做到極致，回報可能是遞增的。
每多投入一分整合深度，AI 能理解的 context 就多一層，能省下的摩擦就多一塊。
分散力量去接很多東西，回報往往是線性的，甚至因為切換成本而遞減。

有一個判斷框架我覺得蠻值得參考：
評估一個 AI 產品是否「真垂直」，可以看三件事。
有沒有專有數據、有沒有深入工作流程、有沒有對結果負責。
三者缺一，大概就是偽垂直。

Genie Code 之所以讓我有感，也許就是因為它在 Databricks 自己的場景裡，這三件事都做得比較到位。


▋ 所以，整合可能就是產品本身

以前我把「整合」當成產品成熟之後要慢慢補強的部分。
現在越來越覺得，在 AI 產品裡，整合本身可能就是產品。
甚至是最關鍵的一部分。

也許很多 AI 功能讓人覺得「好像還行，但沒有真的改變什麼」，
是因為它被放進了一個太淺的環境。

未來我做 AI 相關的產品，應該會更極端的去思考整合這件事。

很多體驗差異，最後其實都藏在那些看起來很小的摩擦裡。
