【 讀完這份 Skills，我像是偷到了一本資深工程師的備忘錄 】
——agent-skills 把人類工程紀律編譯成 Agent 可執行約束
ㅤ
ㅤ
主管傳來一個連結：addyosmani/agent-skills。
ㅤ
他希望我們看看這份東西，評估看看能不能在團隊內推廣。
ㅤ
順帶說明——Addy Osmani 是 Google Cloud AI Director，這是他的個人開源專案，受 Google 工程實務啟發，但不是 Google 官方產品。
ㅤ
我一打開 repo 就愣了一下。它沒有在教 agent 怎麼寫 code，它在教 agent 怎麼走完一整輪軟體開發週期。
ㅤ
從 DEFINE（定義要做什麼）、PLAN（拆任務）、BUILD（動手寫）、VERIFY（驗證能動）、

REVIEW（合併前把關）、到 SHIP（安全出貨），

每一階段都有對應的 skill，

每個 skill 都寫著觸發條件、步驟、常見偷懶藉口、red flags、驗收證據。


▋ 這一年沒被填滿的那塊，終於有了形

這一整年我花了不少時間看各種 Vibe Coding 的流程資訊，

Lovable 的 Lazar 說要「80% 規劃、20% 執行」、Kent Beck 和幾位架構師主張「Prototype as Spec」、

Karpathy 說 Claude Code 才是正確的 Agent 典範。

讀的時候都很開心，但有一件事始終沒被填滿——那個被強調到爛的「80% 規劃」，到底長什麼樣？
ㅤ
看完 agent-skills，我終於有答案。

資深工程師的軟體開發流程都在裡面，整套 SDLC 被一個 skill 一個 skill 地寫下來，非常受用。
ㅤ
而最讓我印象深刻的，是每個 skill 的最後一節：Rationalizations（合理化辯詞）。
ㅤ
這段裡面列的不是 agent 的話，是資深工程師過去對自己偷懶時的內心獨白：「之後再補測試啦」「這太小不值得開 skill」「先做完這個 sprint 再說」。
ㅤ
讀這份備忘錄的體感，像看一個資深 engineer 把自己跟過去自己對話的紀錄全部公開出來，讓 agent 拿去用。
ㅤ
ㅤ
▋ 我原本對它有的偏見
ㅤ
坦白說，第一眼看到「20 個 skills、3 個 personas、7 個 slash commands」這種數字，

我下意識覺得——這又是一個 prompt pack。

打包一堆 prompt 賣 hype，跟之前看過的很多 AI Coding 模板沒兩樣。
ㅤ
但往下讀的時候，有幾個細節讓我停下來想了很久。
ㅤ
第一個，每個 skill 的格式是「何時觸發 → 步驟 → Red Flags → Verification」。

它寫的是給 agent 走的流程圖，不是給人看的建議條列。
ㅤ
第二個，每個 skill 最後都有一節 Rationalizations，

列出 agent 很可能會用的藉口，然後逐條配好反駁話術。

第一次看到的時候我真的笑出來，

這就是資深工程師在跟自己內心那個「懶惰版的自己」對話。
ㅤ
第三個，它用 Skill × Persona × Slash Command 這三層抽象把「怎麼做」「誰來做」「什麼時候做」拆開。

三件事可以獨立演進、獨立審計、獨立替換。
ㅤ
看到這裡我慢慢收回偏見。

它解決的是 agent 的流程層問題——agent 肯不肯照流程走，

是 agent 懂不懂怎麼寫程式。這兩個層次過去從來沒有被這樣清楚地分開處理過。
ㅤ
ㅤ
▋ 「80% 規劃」這次終於有了可以寫下來的形狀
ㅤ
以前讀「先 spec 後 code」，我知道它是對的，但我始終不知道 spec 要寫到什麼程度才算完。
ㅤ
agent-skills 的 spec-driven-development 把 spec 拆成六大面向：
ㅤ
Objective
Commands
Structure
Code Style
Testing
Boundaries
ㅤ
而且規定一件很硬的事，

未完成 Specify 階段，不得進入 Plan 階段。人類必須審核才能推進。
ㅤ
這叫 gated workflow。

聽起來很像以前那種繁瑣的流程文件，

但差別很明顯：過去的流程寫在 Wiki 上沒人讀，現在的流程被 agent 持續讀、持續驗收、持續反問你有沒有跳步。
ㅤ
我原本以為這會疊床架屋，

靜下來想才發現它在做的是把隱性的工程師直覺顯性化。

直覺寫不下來，agent 就看不到；agent 看不到，你就只能靠 review 事後擋——review 是最貴的人力。
ㅤ
ㅤ
▋ Rationalizations，第一次把「工程師的良心」寫進規格
ㅤ
TDD 的 skill 裡有一句話我看完印象很深：
ㅤ
「A test that passes immediately proves nothing.」
ㅤ
意思是如果你寫了一個測試，它第一次跑就直接 pass，

那它根本沒證明任何東西——你不知道它是在驗證邏輯，還是在驗證空白。
ㅤ
TDD 的 Red-Green-Refactor 裡，RED 那一步要求測試必須先失敗。

每個資深工程師都知道這件事，但我還是常看到 agent 寫完 code 就配一個空測試交差。
ㅤ
agent-skills 是怎麼處理的？它直接把這句話寫進 skill，並且列出 agent 可能會說的所有藉口：
ㅤ
「我等下再補測試」
「這個 edge case 不太可能發生」
「現在時間不夠」
ㅤ
每條藉口後面配好反駁話術。
ㅤ
這段我讀了三遍，因為我一直在想：資深工程師心裡本來就有這些對話，只是從來沒人把它寫下來。
ㅤ
過去這些內心對話只能靠 code review 一次一次地傳給新人。

你這個 PR 沒測試，我要擋；你這個測試沒失敗過，我要擋；你這邊偷偷 catch 例外，我要擋。
ㅤ
agent-skills 做的事是——把這些 review 當下才會出現的話，搬到 agent 執行當下就出現。
ㅤ
當 review 成本從「PR 當下」前移到「工具內建」，工程師的稀缺價值就從寫 code 轉到「寫驗收標準」了。
ㅤ
ㅤ
▋ 三層抽象比 20 個 skills 本身更重要
ㅤ
如果有一天我要把這包 skills 介紹給同事，我不會從 skills 清單講起。我會從三層抽象講起。
ㅤ
Slash Commands：回答「什麼時候做」（when）——例如 /spec、/plan、/build、/review、/ship
ㅤ
Personas：回答「誰來做」（who）——例如 code-reviewer、security-auditor、test-engineer
ㅤ
Skills：回答「怎麼做」（how）——例如 spec-driven-development、TDD、code-review-and-quality
ㅤ
這三層可以獨立演進。

Persona 的角色可以加可以換、skill 的步驟可以改版、slash command 的觸發時機可以調整，彼此不強耦合。
ㅤ
為什麼這件事重要？過去所有「AI Coding 最佳實踐」的建議都是一坨——一坨 system prompt、一坨 rules file、一坨 CLAUDE.md。

當你想改其中一條，你就要重讀整份、重新思考，很容易改出意料外的副作用。
ㅤ
三層拆開之後，行為治理從藝術變成工程。

你可以 diff 一個 skill 的改動、可以 A/B 測試兩個 persona 的 review 風格、可以觀察某個 slash command 的觸發率是不是掉了。
ㅤ
Skill 和 prompt 的本質差異在於可系統化迭代、可轉移、可交易。三層抽象讓 iterate、transfer、trade 這三件事第一次真的能做。
ㅤ
ㅤ
▋ 但我還沒想通的三個地方
ㅤ
寫到這裡本來要下結論了，但誠實一點講，我對這包 skills 還有三個疑惑。如果看到這篇的人剛好有答案，也可以告訴我。
ㅤ
疑惑一：流程多 vs 流程少，到底哪個對？
ㅤ
OpenAI Codex 團隊 40 人只配 1 個 PM，靠工具夠強所以流程可以砍。
ㅤ
但 agent-skills 做的看起來剛好相反——它在加流程。
ㅤ
我目前給自己的和解是：流程多寡本身其實不是重點，「流程可視」才是重點。

agent-skills 把本來就有（但寫在資深工程師腦子裡）的流程搬出來變成可審計物件。

至於每個情境要用多少，它自己也說了： Hackathon、typo、一行改動不該觸發 /spec。
ㅤ
這個和解方案對不對，我還不確定。
ㅤ

疑惑二：模型大跳級以後，這包 skills 會變技術負債嗎？
ㅤ
精心設計的 workflow 本質上是「做空模型進步」的單子。模型跳級時，你之前固化的流程就會變技術負債。
ㅤ
agent-skills 會嗎？我目前的猜測是，三層抽象應該能活下來，細節規則會鬆。

例如「每 PR 不超過 100 行」這條未來可能會放寬，但「PR 要有大小限制」這件事不會消失。

「寫失敗測試」的硬性 RED 步驟可能會被簡化，但「先定義驗收條件」不會。
ㅤ
這是猜測。要真的驗證，只能等半年後回來看。
ㅤ
疑惑三：它在我手上的 codebase 會不會反而有害？
ㅤ
這是最後一個、也是我最擔心的一點。
ㅤ
沒完善測試的 codebase，agent 自主工作會幻覺型產出。
ㅤ
agent-skills 的 TDD、code-review 等 skill 預設「測試基礎建設存在」。

如果你拿這包去一個 legacy 沒測試、沒 CI 的 codebase 跑，agent 不會變資深，

反而會產出一堆你以為在驗證什麼、實際根本跑不起來的假測試。
ㅤ
這是工具/紀律的前置依賴條件問題。不是 agent-skills 的鍋，但如果我沒注意到，就會是我的鍋。
ㅤ
ㅤ
▋ 我打算怎麼用它
ㅤ
寫了這麼多，我其實還沒真的在自己的案子上跑過整套 skills。

這篇不是實戰心得，是讀完之後的理解整理。
ㅤ

先不引入 Persona fan-out，只跑 happy path。
ㅤ
觀察 Rationalizations 在哪些時刻被觸發：

我心裡也會冒出那些藉口——「之後再補測試」「這太小不值得開 spec」，

看看當 agent 照著反駁話術回我時，我會照做、還是會找新藉口。
ㅤ
判斷它適不適合帶到團隊：如果在個人案子上都覺得「開銷 > 收益」，拿到團隊也救不了。這件事我想先自己走過再說。
ㅤ
ㅤ
▋ 最後
ㅤ
讀完這包 skills，我的想法很簡單。
ㅤ
資深工程師的紀律，過去只能靠師徒制傳。
ㅤ
現在第一次，這些紀律可以被寫下來、可以放進 git、可以讓 agent 照著走。
ㅤ
它不會讓 agent 自動變資深。但它讓資深工程師的判斷，第一次有了可分享、可複製、可累積的形狀。
ㅤ
我會先挑 5 個 skills 在自己的案子上試看看，

spec、plan、incremental、TDD、code-review。跑完之後再決定要不要帶到團隊。
ㅤ
如果我自己都做不到這些紀律，我也不會期待 agent 做到。
