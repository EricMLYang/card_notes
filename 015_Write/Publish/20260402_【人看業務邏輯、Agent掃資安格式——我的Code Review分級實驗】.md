【 人看業務邏輯、Agent 掃資安格式 】
——我的 Code Review 分級實驗
ㅤ
最近在規劃團隊導入 AI coding agent 的流程，用內部專案先跑了一輪測試。
ㅤ
過程中碰到的第一個問題蠻出乎意料的。
ㅤ
AI 寫得不好？還好。
真正的問題是 code review 的量暴增到沒人看得完。
ㅤ
以前一個工程師一天產出 200 行 code，你可以仔細看。
Agent 一天產出 2000 行，你根本不可能每行都看。
ㅤ
所以我開始設計分級制度：人只看關鍵，其餘交給 Agent 掃。
這篇是我目前規劃的分級邏輯，和測試過程中踩過的坑。
ㅤ
ㅤ
▋ AI 協作的本質交易
ㅤ
很多人期待「終於不用自己寫 code 了」。
只對了一半。
ㅤ
Agent 產出速度大概是人的 2-5 倍。
但它本質上就是一個很快、不太可靠的 junior。
你省下的寫 code 時間，會直接轉換成 review 時間。
ㅤ
這件事我一開始也沒想清楚。
用內部專案測試的前幾週，看到產出變快，覺得挺好的。
後來才慢慢發現：技術債在堆、資安漏洞沒人抓、業務邏輯錯了卻過了 review。
ㅤ
我目前的理解是，導入 AI coding 之前，第一件該做的事是先設計好 code review 流程。
Prompt template 可以慢慢補，review 流程不對，後面全部白搭。
ㅤ
ㅤ
▋ 我規劃的做法：Code Review 分級
ㅤ
想法很簡單：不是所有東西都值得人看。
ㅤ
以前 review 是平的——每一行都過人的眼睛。
Agent 時代產出量暴增，你必須分級。
ㅤ
我目前分成兩層：
ㅤ
人看（關鍵決策）
→ 業務邏輯是否正確
→ 架構決策是否合理
→ 重大風險：資料權限、金流、隱私
ㅤ
Agent 掃（規則可判斷）
→ 資安漏洞（SQL injection、XSS、敏感資料外洩）
→ 格式一致性、命名規範
→ 測試覆蓋率
ㅤ
說實話，讓 Agent 跑資安掃描一開始只是想試試看能不能減輕負擔。
結果在內部專案測試時，第一輪跑完，抓出的問題比我們手動 review 兩週找到的還多。
這件事讓我蠻驚訝的。
ㅤ
後來想想也合理——Agent 適合處理有明確規則、可自動化判斷的事。
人的注意力應該集中在規則覆蓋不了的地方：業務判斷、架構取捨、風險承擔。
ㅤ
我目前比較傾向這樣看：
高確定性走快車道（Agent），模糊的交給資深工程師推理，真正不確定的回人手決策。
ㅤ
ㅤ
▋ Review 只是冰山一角
ㅤ
把 review 分級規劃好之後，我發現一件更大的事。
ㅤ
Agent 寫 code 快了，但需求不清楚，它只會更快地寫出錯的東西。
架構沒定好，它用最快速度幫你堆技術債。
ㅤ
所以我把流程拆成三層：
ㅤ
探索：PM 負責。AI 幫忙整理資料、生成初版 PRD、找需求矛盾。但決策優先順序仍是人的事。
ㅤ
驗證：架構師、設計師、SA 負責。Agent 可以生成架構草稿、檢查規格衝突。但 trade-off 的判斷是人的責任。SA 在這一層的角色越來越清楚——需求到系統的翻譯者。
ㅤ
實作：Agent 參與度最高。寫 code、跑測試、資安掃描都能幫。但資深工程師仍負責品質和決策，QA 仍負責驗收。
ㅤ
ㅤ
▋ 每個角色都在收斂不確定性
ㅤ
整個流程規劃下來，我觀察到一件蠻有意思的事。
ㅤ
整個團隊其實在做同一件事——逐步降低不確定性。
只是每個角色負責的類型不同：
ㅤ
PM → 產品價值
架構師 → 技術可行性
SA → 系統邏輯
工程師 → 實作品質
QA → 系統風險
ㅤ
AI 的作用？加速產出，但不負責收斂風險。
ㅤ
這裡比較重要的是分清楚「加速」跟「收斂」。
很多團隊導入 AI 出問題，根源就是把這兩件事搞混了。
ㅤ
沒有品質門檻的加速，只會更快地製造垃圾。
以前功能花三個月才做完，大家自然先想清楚值不值得做。
Agent 三天做完，這個天然過濾器消失了。
ㅤ
所以 review、驗收、評估標準，才是 AI 時代真正的控制介面。
ㅤ
ㅤ
▋ 每個人都往上移一層
ㅤ
回頭來看，AI 改變的其實是每個角色在流程中的位置。
ㅤ
工程師核心工作從寫 code 變成 review code。
PM 的決策速度變成新瓶頸——Agent 等著 spec 開工。
SA 從寫文件的人變成 System Designer。
ㅤ
不是減人，是升級責任。
ㅤ
ㅤ
這篇是規劃完、即將正式導入的實驗紀錄。
有些做法在內部專案測試過效果不錯，有些還要等正式上線再觀察。
ㅤ
但有一個判斷我比較確定：
當寫 code 變得便宜，你得重新想清楚——什麼東西是貴的。
ㅤ
答案是判斷。
