【 6 人團隊的 AI Coding 流程：3 級風險分流 + 3 份最小文件，讓 70% 任務不需要額外文件 】
ㅤ
小團隊想導入 AI Coding，通常卡在一個兩難。
ㅤ
流程太重，跑不動。團隊就那幾個人，不可能每個任務都寫 spec、做 review、走簽核。
ㅤ
流程太輕，守不住。Agent 產出的 code 沒人有效把關，上了線才發現問題。
ㅤ
我後來發現，問題出在缺少風險分級。改個按鈕文字跟改個 schema migration，需要的控制深度差很多，但很多團隊讓它們走同一套流程。
ㅤ
這篇整理一套我目前在用的輕量做法——按風險分三級，L0 走短路、L2 才補控制點。
ㅤ
ㅤ
▋ AI Coding 到底改變了什麼
ㅤ
很多團隊導入 AI Coding 的第一個動作是研究怎麼讓 Agent 寫更多 code。
ㅤ
但做了一陣子之後會發現，AI Coding 的本質比較像一場交易——你放棄親手寫 code 的掌控感，換來 2-5 倍的產出速度。代價是你得嚴格審查 Agent 產出的每一行 code，有點像在 review 一個手很快但經驗不足的 junior 的 PR。
ㅤ
規格寫得越清楚，AI 輸出越好。
測試越完整，越敢放手委派。
架構越乾淨，AI 越少做出奇怪的抽象。
ㅤ
還有一件事我覺得容易被忽略：開發成本過去其實是天然的品質過濾器。寫 code 很貴，團隊會被迫先想清楚值不值得做。AI 把這個門檻壓低了，但多數團隊沒有建立新的篩選機制。結果就是用驚人速度產出沒人要的功能、頂尖工程師被低品質 PR 淹沒、帳面上的效率提升沒有轉化成真正的產出。
ㅤ
所以我現在比較關心的問題是：哪些決定不能交給 AI、哪些控制點不能省。
ㅤ
ㅤ
▋ 核心觀點：驗證標準是唯一的控制介面
ㅤ
當 AI 把「寫 code」變成純執行，你能控制的其實只剩評估標準。
ㅤ
TDD 就是一個很好的例子。它的角色已經從開發者的自律工具，變成了 Agent 的控制介面。你用 Interface 定義規格、用 Test 定義驗收條件，鎖住頭尾，中間的實作全交給 Agent 跑到通過為止。骨架打好了，肉再長也歪不到哪裡去。
ㅤ
沒有先定義「怎樣算對」，Agent 只會很有效率地做錯事。
ㅤ
想通這一點，流程該怎麼設計就比較清楚了——重點在把控制點放對位置。
ㅤ
ㅤ
▋ 原則一：角色按判斷權劃分
ㅤ
AI 可以幫忙產出，但不能替人類承擔錯誤決策的後果。所以我在分工時，看的是「誰對這件事有判斷權」：
ㅤ
PM / PO 負責價值判斷——值不值得做？成功長什麼樣？需求切得夠不夠小？
ㅤ
架構師 / Tech Lead 負責邊界判斷——設計邊界在哪？影響哪些模組？哪些 invariant 不能破壞？
ㅤ
驗證標準負責品質判斷——怎麼驗證成功？失敗最可能發生在哪？
ㅤ
Agent 可以整理需求卡初稿、產出設計方案、生成測試骨架——這些它做得很好。但最終的判斷，值不值得做、邊界能不能動、怎樣算對，這些還是人的事。
ㅤ
ㅤ
▋ 原則二：流程按風險分流
ㅤ
這大概是整套做法能不能落地最關鍵的一環。
ㅤ
L0 低風險（UI 文案、CRUD、log、樣板、測試補齊）：需求卡 → Agent → Review。不需要額外文件。
ㅤ
L1 中風險（單服務功能、API 增修、小範圍 schema 調整）：需求卡 → 一頁設計約束 → Agent → Review。
ㅤ
L2 高風險（跨服務變更、schema migration、權限模型、不可逆操作）：需要持續監督、明確 rollback 規劃，合併跟上線是兩個獨立的判斷點。
ㅤ
實際跑起來，大概 70% 的任務落在 L0，需求卡本身就夠了。L2 才需要完整的架構決策記錄。思考的深度對齊風險的高度，該省的省，該補的補。
ㅤ
ㅤ
▋ 原則三：文件只保留最少必要集
ㅤ
必備三份：
ㅤ
需求卡——背景、目標、範圍、驗收標準、反例、風險等級。同時是 Agent 的執行指令和 PM 的價值判斷紀錄。
ㅤ
驗證標準——三層：業務驗證（解決原始問題了嗎）、結構驗證（系統邊界完好嗎）、工程驗證（test、lint、type check 通過了嗎）。
ㅤ
PR / Review 紀錄——人類審查的痕跡，也是團隊知識累積的地方。
ㅤ
視風險再加：L1 加設計約束，L2 加 ADR 和 rollout/backout 記錄。就這些。
ㅤ
ㅤ
▋ 帶走一件事
ㅤ
每次把任務交給 Agent 之前，我會先確認自己答得出這三句：
ㅤ
1. 成功長什麼樣？
2. 怎麼驗證成功？
3. 失敗最可能發生在哪裡？
ㅤ
答得出來，放心讓它跑。答不出來，先停下來想清楚。
ㅤ
軟體開發的價值正在遷移——從「會寫 code 的人」到「能定義驗收標準的人」。這套流程，就是為了讓你站在對的那一邊。
