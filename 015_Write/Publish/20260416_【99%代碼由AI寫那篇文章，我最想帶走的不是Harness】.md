【 我讀完那篇「99% 代碼由 AI 寫」的文章，最想帶走的一件事不是 Harness 】
——如果小團隊也想學，該從哪裡下手？
ㅤ
上週把 Peter Pang 的 AI 優先工程文章看了三遍。
ㅤ
數字很嚇人：CREAO 正式環境 99% 的程式碼由 AI 寫，25 人、10 名工程師，14 天內平均每天 3 到 8 次正式部署。
一項功能從構想到上線一天走完。
三個月前同樣的事要六週。
貼文兩天內超過 41 萬次瀏覽。
ㅤ
看第一次我很興奮，筆記抄了一頁。
ㅤ
看第三次我冷靜下來問自己:「如果我的小團隊想學,該從哪裡下手?」
ㅤ
答案沒有我原本想的那麼技術。
ㅤ
ㅤ
▋ Pang 的答案是 Harness,這我同意
ㅤ
Pang 寫這一串貼文的時候,OpenAI 剛在 2026 年 2 月提出一個詞:Harness Engineering。
ㅤ
他們的 Codex 團隊做了一個超過百萬行的正式環境應用,沒有一行程式出自人手。
所有應用邏輯、測試、CI 設定、文件、可觀測性都是 Codex 代理寫的。
ㅤ
Harness 這個比喻很精準。
馬是 AI 模型——強大、快速,但不知道該往哪跑。
Harness 是韁繩、鞍具、銜鐵,是約束條件、回饋迴圈、文件規範、linter、CI 閘門和架構邊界,讓代理的產出變得可靠。
ㅤ
CREAO 的堆疊具體長這樣:
ㅤ
所有程式碼塞進一個 monorepo。不是為了開發者方便,是為了「代理可讀性」——碎片化的程式碼庫對 Agent 來說等於不存在。
六階段 CI/CD 管線(驗證 CI → 建置 Dev → 測試 Dev → 部署 Prod → 測試 Prod → 發布),沒有跳過、沒有手動覆寫。
每個 PR 觸發三道平行的 AI 審查(Claude Opus 4.6),程式碼品質、安全性、相依性掃描,不是建議是閘門。
每天早上九點自動健康檢查,錯誤分群 → Linear 自動建工單 → 工程師推修復 → 同一管線審查部署 → 錯誤消失自動關單。
ㅤ
Pang 說:「AI 優先工作流程出問題時,解法不是更努力或寫更好的提示詞,而是缺少什麼能力?怎麼讓它對代理可讀、可執行?你不再是在除錯程式碼,而是在除錯那個產出程式碼的環境。」
ㅤ
這句話我特別抄下來。
它把我認知裡的「AI Coding」整個翻過來了——工程的重心從寫 code 移到設計產出 code 的環境。
ㅤ
更早之前我讀到一句也很有感的話:「The single biggest differentiator between agentic engineering and vibe coding is testing.」
Pang 這一串貼文,其實就是把這個道理做到了極端。
ㅤ
所以讀完第一遍,我以為我看懂了:要做 AI-first,先把 harness 建起來。
回去我也該盤一盤自己團隊的測試覆蓋率、環境可複製性、有沒有確定性的 CI。
ㅤ
ㅤ
▋ 但讀完第三遍,我發現 Harness 之前還有一層
ㅤ
第二次讀,我開始困惑一件事。
ㅤ
Pang 在文末寫:「先搞清楚瓶頸在哪。如果 PM 流程比開發時間還長,那才是該先處理的,不是一味加 AI 程式碼工具。」
ㅤ
這句話放在通篇講 CI/CD 閘門和三道 AI 審查的文章結尾,有點違和。
它暗示一件事:就算你把 harness 建得跟 CREAO 一樣完整,如果瓶頸不在開發速度,你只是把非瓶頸環節加速了——整體產線還是卡住。
ㅤ
那 CREAO 為什麼能跑起來?
ㅤ
我覺得答案藏在另一個細節裡:CREAO 本來就是一家 Agent 公司。
ㅤ
他們的產品是 Agent,所以他們用 Agent 建 Agent、用 Agent 測 Agent、用 Agent 監控 Agent。
產品和產線互為飼料——每一次部署不只是上線一個功能,也是一次「我們的 Agent 好不好用」的自我驗證。
這條產線有一個極清楚的主軸:我們要做出好用的 Agent,好用的 Agent 就是能寫 code、能修 bug、能通過我們自己設計的那些閘門。
ㅤ
這就是我讀完第三次,真正卡住的地方——
ㅤ
Harness 不是 AI-first 的起點,產線主軸才是。
ㅤ
你得先回答三個問題:
ㅤ
這條產線要產出什麼?
「好的產出」長什麼樣?
怎麼驗證它產出得好?
ㅤ
這三個問題不釐清,harness 再華麗都是空轉。
你可以建六階段 CI/CD、三道 AI 審查、每天健康檢查——但如果沒人說得清楚「我們為什麼要部署這個功能、它上線後誰會因此變得更好」,你只是用非常先進的方式在產垃圾。
ㅤ
Pang 自己也提到這個風險。
GitClear 分析 1.53 億行程式碼、Carnegie Mellon 對 800 多個 GitHub repo 的研究、Apiiro 對 Fortune 50 的研究,都指向同一個方向——技術債在 AI 輔助下正在指數累積。
ㅤ
AI 生成的程式碼通常看起來乾淨:格式工整,單獨看邏輯正確。
風險在累積。
每次 AI 提交都製造細微不一致——這裡用了不同的身分驗證模式、那裡重複了工具函式、某個查詢能跑但沒遵循團隊的 ORM 慣例。
個別來看微不足道,合在一起就像複利。
ㅤ
Harness 可以攔一部分——CREAO 的三道審查就是幹這個的。
但 harness 攔不住的那一層,是「這個功能根本不該被做」。
這一層的回答不在 harness 裡,在產品主軸裡。
ㅤ
ㅤ
▋ 找一個實體世界的參照:亞馬遜
ㅤ
我覺得亞馬遜提供了一個很好的對照案例。
ㅤ
2022 年 Q1 到 2023 年 Q2,亞馬遜員工從 162 萬降到 146 萬(減少 10%),營收成長 15.4%,人均產值五個季度飆升 28.1%。
機器人員工從 2019 年 20 萬台 → 2022 年 52 萬台 → 2023 年 75 萬台。
北美業務營業利益率從虧損跳升到 6%。
ㅤ
這是實體世界的 AI-first。
ㅤ
但亞馬遜不是因為「買了很多機器人」才變強。
他們是累積了 20 年的零售營運,「撿貨要多快」「搬運要多準」「退貨要幾步完成」這些 KPI 早就被定義、被衡量、被優化過一輪又一輪。
機器人進來的時候,產線主軸已經非常清楚:把貨從 A 搬到 B,錯誤率和時間都有量化基準。
ㅤ
機器人只是接手了產線上「人做不快、做不穩」的那一段。
ㅤ
假設今天一個零售新創,剛成立、沒有 KPI、沒有流程,就買一堆機器人進來,會發生什麼事?
ㅤ
機器人會以最高效率,把錯的貨、搬到錯的地方。
ㅤ
這就是「有 harness 但沒產線主軸」的後果。
ㅤ
ㅤ
▋ 三件可以帶走的事(一個 tech lead 的備忘)
ㅤ
好,那對我這種帶小團隊的技術主管,Pang 這篇文章可以帶走什麼?
ㅤ
我整理了三件。
這不是教戰守則,是我讀完的個人備忘——我自己也還沒全部做到。
ㅤ
ㅤ
第一件:先找瓶頸,再談 AI
ㅤ
這是 Pang 說的,也是我想先提醒自己的。
ㅤ
AI coding 工具很誘人,但它只在「寫 code 是瓶頸」的前提下有效。
對很多小團隊來說,寫 code 根本不是瓶頸——需求不清、驗收條件模糊、PM 和工程沒對齊、決定一個 feature 要不要做的過程比寫它還久——這些才是。
ㅤ
這時候加 AI coding 工具會怎樣?
它會讓原本就不該做的東西,更快地被做出來。
ㅤ
Block CTO Dhanji Prasanna 那句話我很喜歡:「AI 導入成敗首先是組織設計問題,不是工具問題。」
他的意思不是不要用工具,而是用工具之前要先問——這個工具要解哪個瓶頸?
ㅤ
ㅤ
第二件:Interface + Test 是小團隊最負擔得起的 harness
ㅤ
CREAO 的六階段 CI/CD 加三道 AI 審查,對我來說是一個遙遠的北極星。
小團隊短期內不可能堆到那種等級,不代表沒有起點。
ㅤ
前幾天公司一位優秀的同事才丟了一版 coding review 流程草案給我們討論——哪個階段誰 review、用什麼 checklist、AI 跟人類審查怎麼分工。
我當下覺得這其實就是我們在能力範圍內的一版 harness——不華麗,也不完整,但它在回答同一個問題:我們的「好」長什麼樣,誰來替我們守。
ㅤ
我自己最想補上的那一塊,是把規格和驗收條件往 code 裡推:
ㅤ
用 code 定義規格(Interface)
用 code 定義驗收條件(Test)
這兩層親自逐行 review 並保護起來,不讓 AI 改
中間的實作全交給 Agent 跑到通過為止
ㅤ
這是一個小團隊可以今天下午就開始做的事。
它比大肆改造 CI/CD 簡單,做的是同一件事的最小版本——把「好的產出是什麼」用 code 寫清楚,讓 AI 在這個定義之下工作。
ㅤ
測試在 AI 時代的角色不只是防回歸,它變成你跟 AI 溝通「我到底要什麼」的唯一精確語言。
ㅤ
ㅤ
第三件:別把未來的架構師一起砍掉
ㅤ
這件事我原本沒想到,是 Pang 的風險段落提醒我的。
ㅤ
他說 AI-first 的世界裡會出現兩種工程師:
架構師(每個團隊一到兩人,設計標準作業程序、教 AI 怎麼工作、在 AI 方案裡找漏洞)
操作員(其他所有人,在架構師設計的系統內工作,負責審查 AI 的 PR、驗證分流工單)
ㅤ
他也指出:最有價值的技能是架構判斷力——批評 AI、找失敗模式、壓力測試假設的能力。
ㅤ
但這些技能需要多年的實戰編碼、犯錯和正式環境事故才能培養。
ㅤ
你跳不過操作員階段,直接當架構師。
ㅤ
這裡有個時間差陷阱:54% 的工程領導者計畫因 AI 而減少初階聘用。
短期看,初階工程師「產出/成本比」確實打不過 AI。
如果連操作員階段都不讓新人進來做,3 到 5 年後要去哪找下一批架構師?
ㅤ
這是產業級的結構問題,我一個人解決不了。
但我至少可以在自己的小團隊裡:
ㅤ
別一股腦把所有初階位置都砍了
刻意設計 AI 增強的入職流程,讓新人用 AI 協作但仍然接觸到真實的問題
把「找失敗模式、壓力測試假設」變成所有層級的日常訓練,不是只有資深才做
ㅤ
ㅤ
▋ 我真正想找的:自己領域的自我強化迴圈
ㅤ
最後想寫一段比較私人的。
ㅤ
讀完 Pang 的文章,我留下最深的印象其實不是他的 CI/CD 堆疊,是一個畫面:CREAO 的 Agent 在寫 CREAO 的 Agent。
ㅤ
這是一個很純粹的飛輪——產品就是產線,產線就是產品,每一次部署都同時是功能發布和自我驗證。
ㅤ
大模型公司都在期待一件事:模型能自己改進模型。
Anthropic、OpenAI、Google 都有類似的工作在跑,那是他們的終極飛輪。
CREAO 是小規模的同構版本:他們的產品是 Agent,他們用 Agent 去寫、去測、去審查 Agent。
ㅤ
這讓我開始想——在我自己的領域(或你的領域),有沒有一個對應的自我強化迴圈?
ㅤ
不見得要是「我的產品在做我的產品」這種極端形式,可以是更溫和的版本:
ㅤ
我今天產出的一份文件,能不能在下次被我或我的團隊(或 AI)拿去再利用、再改進?
我做的這個決策,能不能留下一個可以被回頭驗證的紀錄?
我今天跟 AI 協作的這一輪,有沒有讓我自己的品味、我的 Interface、我的驗收標準變得更清楚?
ㅤ
如果有,這條迴圈就會複利。
如果沒有,就是每次重來一次。
ㅤ
Pang 的 harness 是一個可見的工程形式。
真正推動那條飛輪的,是 CREAO 對「什麼是好 Agent」這件事的清晰共識——那是產品主軸,不是工程堆疊。
ㅤ
我在想,在自己領域找到這個等價物,可能比複製他們的 CI/CD 更重要。
ㅤ
ㅤ
▋ 收尾
ㅤ
寫完這篇我還是覺得,CREAO 那種極端 AI-first 對我來說很遠。
ㅤ
也許近的部分,是「先想清楚產線主軸」這件事本來就是我的工作,只是過去被別的優先級蓋過去了——AI 只是把這個問題的時間窗口縮短了。
ㅤ
如果你也讀到這裡,然後跟我一樣還沒有答案,我想那是正常的。
比較不正常的,是看完這篇覺得自己立刻就能複製 CREAO 的人。
ㅤ
