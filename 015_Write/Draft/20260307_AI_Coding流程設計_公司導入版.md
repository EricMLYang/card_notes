# AI Coding 流程設計：彈性版

## 我想回答的問題

我想為公司設計一套 AI Coding 流程。不是傳統那種每一步都有厚重文件、層層審批的流程，而是**高度彈性、每一段只保留最關鍵的文件和 Agent 協作方式**的輕量版本。
導入團隊大約 100人，所以不能太死板，要考慮導入阻力，以及不同專案性質不同的彈性

具體來說，我想釐清：

1. **流程骨架該長什麼樣？** 需求文件 → 架構 Review → PRD/Spec → TDD → Agent Coding → Code Review，這六段是否合理？哪些可以合併、哪些不能省？
2. **每一段的「判斷權」歸誰？** AI 拿走了產出的苦力，但每個階段都有一類判斷是 AI 做不了、做錯了代價很高的。誰負責什麼判斷？（參考：PM 判斷值不值得做、架構師判斷結構邊界、開發判斷能不能上線、QE 判斷可不可信、維運判斷風險可不可控）
3. **每一段最少要產出什麼文件？** 不追求完美文件，但要夠讓 Agent 跑、夠讓人類 Review。
4. **Agent 在每一段怎麼參與？** 哪些是 Agent 主導（人類審核）、哪些是人類主導（Agent 輔助）？
5. **最小可行的控制集是什麼？** 不可能一次到位，第一步該先建什麼？（參考 TOC 思維：先打穿最卡的瓶頸）

背景假設：
- 小團隊（6-10 人），不想搞太重的流程
- 已經有人在用 Coding Agent（Claude Code / Cursor / Copilot），但沒有統一的流程規範
- 目標不是「AI 24/7 全自動」，而是「團隊都能安全高效地用 Agent 開發」

### 核心悖論（要誠實面對的）

> 要讓 AI 24/7 有效 → 需要完美的 Spec + 自動化驗證
> 要有完美的 Spec + 自動化驗證 → 需要成熟的工程文化
> 要有成熟的工程文化 → 通常就不缺人力，不需要 AI 24/7

解法不是「先做到完美再開始」，而是**選出最少的控制點，讓 Agent 可以在有限範圍內安全運作，再逐步擴大**。

---

## 流程總覽

```
需求定義 → 架構 Review → PRD/Spec → TDD → Agent Coding → Code Review
```

兩個核心原則：
- **角色按「判斷權」劃分，不按技能劃分**——AI 把技能門檻拉平了，但「為判斷的結果負責」這件事 AI 做不了
- **80% 規劃 20% 執行**——大部分時間花在定義「什麼叫做對」，Agent 負責「把它做出來」

## 一、需求定義（判斷權：PM — 目的判斷）

> PM 的核心判斷：這件事值不值得做？成功長什麼樣子？資源不夠時砍什麼留什麼？

### 素材摘錄

> 「價值鏈重心遷移：從『寫程式』到『寫規格』、從『生成』到『驗證』。稀缺人才畫像改變：能把老闆模糊的話翻譯成精確、完備、邊界條件清晰測試案例的人。」
> —— [[14-軟體開發價值遷移：從生成程式碼到定義驗收條件，人才市場面臨結構性錯配]]

> 「每張交給 AI 的需求卡需包含：明確問題描述、範圍界定、驗收標準。避免模糊詞語。不要僅寫『提升性能』，而是寫『將報表生成時間從 5 秒降到 2 秒，Acceptance: 在 10000 筆資料測試下達成』。」
> —— AgentHub: executable_requirements

> 「先定義『如何驗證成功』的標準再實作。AI 黑盒特性下，不先定義評估標準容易陷入『調參數但不知好壞』的困境。」
> —— [[1-先定義驗證標準再動手的倒推思維]]

> 「PM 要當 Editor 不是 Writer——用 Knowledge-as-Code + Template-Driven Agent 策略，搭配 Coding Agent 持續產出需求文件。」
> —— AgentHub: executable_requirements

> 「PM 判斷的不只是『需求寫清楚』，而是：判斷這件事值不值得讓 AI 做、定義驗收條件的精確度要匹配 AI 的特性、設定 Cost of Delay 優先序——24/7 的 AI 不缺產能，缺的是『先做哪個』的判斷。」
> —— AgentHub: full_ai_coding_keypoint

### 重要文件
- 需求卡（含問題描述、範圍、驗收標準、邊界條件與反例）

### Agent 流程建議
- Agent 可協助將模糊需求（會議記錄、PPT、Slack 訊息）轉成結構化需求卡
- 人類負責確認：邊界條件完整性、驗收標準精確度、優先序判斷
- 進階：建立 Knowledge-as-Code Repo（context/ + templates/ + requirements/），讓 Agent 在充滿上下文的環境中產出需求初稿

### 開發者必練技能
1. 把需求翻譯成可觀測結果（「P95 從 5s → 2s」而非「改善效能」）
2. 定義 DoD（測試、文件、監控、回滾/feature flag）
3. 邊界條件與反例（能寫出 3 個「會失敗的情境」代表你真的懂需求）
4. 切割成可獨立交付的最小單元（一個 PR 一個主題）
5. 規格寫作能力（用固定模板寫 20 次，AI 會越跑越順）

## 二、架構 Review（判斷權：架構師 — 結構判斷）

> 架構師的核心判斷：系統邊界在哪？什麼可以妥協什麼不行？失敗時影響範圍多大？

### 素材摘錄

> 「架構師將 Prototype 當作 Spec：用 vibe coding 做原型驗證收斂規格，Prototype 可直接 RUN、有實際 API/schema、可跟 PO/RD 對焦。」
> —— [[4-Prototype as Spec：架構師用原型代替規格文件的新工作模式]]

> 「用 Interface 定義規格、用 Test 定義驗收條件，這兩層由人類親自 review 並保護，中間實作全交給 Agent。」
> —— [[4-AI Coding 的骨架策略：用 Interface + Test 鎖住頭尾，中間放給 Agent 跑]]

> 「架構師的核心工作是把不確定性轉換成確定性。SPEC 的範圍遠不只畫面、流程和行為，還包括設計決策、跨系統的溝通介面（API）、非功能性需求（效能、安全、成本）、以及失敗模式的處理策略。」
> —— AgentHub: ai_coding_main_role

> 「『什麼時候算足夠確定』本身就是架構師最關鍵的判斷：太早交棒，開發會撞牆返工；太晚交棒，時間全耗在規格上。判斷標準：開發團隊拿到這份 spec，能不能在不回頭問架構師的情況下完成 80% 的工作？」
> —— AgentHub: ai_coding_main_role

> 「核心不是拆 repo，而是拆責任邊界：能力模組、執行流程、環境配置、知識資產各自獨立版本化。」
> —— [[4-Multi-Repo AI Agent 架構]]

> 「權限模型、隔離邊界、沙盒架構應該在系統設計階段就確定，而不是上線後才補。AI 能碰什麼、不能碰什麼、邊界在哪裡——這決定了整個運作的安全框架。」
> —— AgentHub: full_ai_coding_keypoint

### 重要文件
- Interface 定義（API contract / schema）
- 架構骨架（可以是 Prototype 或架構決策記錄 ADR）
- AI 安全邊界（最小權限 + 隔離策略）

### Agent 流程建議
- 架構師可用 Agent 快速產 Prototype 來驗證架構方向（Prototype as Spec）
- 人類鎖住 Interface、模組邊界、安全邊界——這是不可委派的
- Agent 的執行環境隔離（獨立 bot 帳戶 + 只能推 PR 到特定分支 + 沙盒 + 不碰 production）在這一步就要決定

## 三、PRD / Spec（判斷權：架構師 + PM 共同）

> 從「一次做對」轉向「快速迭代」。Spec 是不確定到確定之間的中間產物。

### 素材摘錄

> 「花 80% 時間規劃和對話，20% 執行。先用『腦袋傾倒』釐清思路，找到最有潛力的方向後花一整天做規劃文件，執行變得極簡單。」
> —— [[4-Vibe Coding：80% 規劃 20% 執行的工作流]]

> 「CLAUDE.md 記錄專案的架構風格、命名規範、錯誤處理模式。每花一小時完善，省下好幾天手動修正。每次 AI 犯錯就更新，讓錯誤不再重複。」
> —— [[4-CLAUDE.md 是 AI 助手的腦——投資配置指數回報]]

> 「AI 協作時代，衡量變重要。不要把思考外包給 LLM。用刻意壓縮：研究結果放 research.md，實作步驟放 plan.md 含程式碼和行號，新的 context window 能立即接手。」
> —— AgentHub: learning_resource

> 「落地建議三階段：第一週建 Gate + 隔離 → 第二至四週擴大範圍 + 建立需求標準 → 第二個月補齊剩餘控制點。」
> —— AgentHub: full_ai_coding_keypoint

### 重要文件
- PRD / Spec 文件（含驗收條件、失敗模式、跨系統介面）
- CLAUDE.md / 專案規範文件（給 Agent 的持久 context）
- Task 清單（拆好的最小交付單元，每張都有驗收標準）

### Agent 流程建議
- Agent 可根據需求卡 + 架構骨架，生成 Spec 初稿
- 用 Multi-pass Review：PM Agent 生成 → UX Reviewer 審 → Tech Reviewer 檢查可行性 → 人工 final review
- 人類負責審閱驗收條件、跨系統影響、「足夠確定」的判斷

## 四、TDD / 測試先行（判斷權：QE — 可信度判斷）

> QE 的核心判斷：這個數字能不能對外講？這次變更有沒有破壞已有的正確性？

### 素材摘錄

> 「TDD 從自律工具變成跟 AI 溝通『我要什麼』的精確語言。時間分配翻轉：傳統模式實作佔 50%，AI+TDD 模式中『定義什麼叫做對』膨脹到 70%，實作壓縮到近乎消失。」
> —— [[4-TDD 在 AI 時代的角色轉變：從開發者自律工具到 Agent 的控制介面]]

> 「有測試套件，AI Agent 可迭代直到通過，你能安心委派；無測試套件，AI 會在 broken code 上宣告完成。測試是『將不可靠 agent 轉成可靠系統』的機制。」
> —— [[4-測試套件是 Agentic Engineering 的核心槓桿點]]

> 「Agent 能否自主工作取決於：(1) 測試覆蓋率夠高，Agent 改壞能自己發現修復；(2) 環境可複製，Agent 能在 sandbox 中驗證。」
> —— [[4-AI Coding Agent 自主工作的兩個硬前提：測試覆蓋率 + 環境可複製性]]

> 「當計算轉為純執行，評估標準成了唯一的控制介面。沒有評估就沒有控制，沒有控制就無法落地。」
> —— [[當計算轉為純執行，評估標準成了唯一的控制介面]]

> 「AI 能幫你跑一千個測試，但『測什麼才有意義』的判斷才是關鍵。AI 生成的程式碼有個危險特性——它『看起來對』的機率很高，微妙的錯誤比過去更難發現。QE 需要的不是測試的手速，而是懷疑的直覺。」
> —— AgentHub: ai_coding_main_role

### 重要文件
- 測試案例（含 edge case、反例、失敗情境）
- CI pipeline 設定（自動化測試 + linter + 安全掃描 + secrets scanning）
- Quality Gate 定義（什麼條件下 PR 才能過）

### Agent 流程建議
- 人類寫核心測試案例（定義「什麼叫做對」）
- Agent 可補充 edge case、生成測試骨架
- 測試是 Agent 自主迭代的前提——**沒有測試就不該放給 Agent 跑**
- QE 定義煞車閾值：AI Bug 率超過 X% 時暫停 AI 自主提交

## 五、Agent Coding（判斷權：開發團隊 — 實作判斷）

> 開發者的核心判斷：AI 產出的東西能不能上線？技術債要不要現在處理？三個月後還維護得動嗎？

### 素材摘錄

> 「Agentic Engineering = AI 執行 + 人類架構/審查/決策。Vibe Coding 適合週末 MVP，超出這些場景會導致無人理解代碼在做什麼。」
> —— [[4-Vibe Coding vs Agentic Engineering：審查權是唯一分界線]]

> 「五種協作模式：(1) 截圖式除錯；(2) 非技術人員數據操作；(3) 代碼庫導航；(4) 非同步原型開發（auto-accept 讓 AI 自主編寫 80%）；(5) 同步核心功能開發（商業邏輯需即時監控）。」
> —— [[4-AI Coding Agent 的五種協作模式]]

> 「非同步工作流：晚上描述 5 個實驗方向，早上全部建好，選最好的繼續。工作流程：建 task → Webhook 觸發 Agent 在 Sandbox 改 code / 跑 test / 發 PR → 隔天逐條確認 merge。」
> —— [[4-非同步 Agent 工作模式是必然趨勢]]

> 「把 AI 當『永遠不會學習的 Junior Developer』。三步迭代模型：第一次 95% 垃圾 → 第二次 50% 垃圾 → 第三次可用起點。同時運行多個 Claude 實例如同管理小團隊。」
> —— StaffEngineer 的 ClaudeCode 六週實戰

> 「3-5 個 git worktrees 平行處理；複雜任務先寫計畫再用另一個 Claude 審查；高頻工作封裝成 skill/command。標準：『一天做超過一次就封裝』。」
> —— [[4-Claude Code 十大核心工作流技巧]]

> 「AI 可 100x 速度開發，但人的控制能力有限。有經驗者能到 20x，靠的是用 Interface + Test 建立自動驗證骨架。真正稀缺能力是定義骨架的抽象化能力。」
> —— [[4-AI Coding 的控制能力差距：新手 5x vs 有經驗者 20x，差別在骨架品質]]

> 「開發者的核心能力正在從 writing 轉向 reviewing。最重要的技能是快速識別『看起來對但其實有問題』的程式碼。品味決定 AI 工具的產出上限。」
> —— AgentHub: ai_coding_main_role + 工程師新素養

### 重要文件
- Task 清單（拆好的最小交付單元）

### Agent 流程建議
- 根據任務性質選協作模式（同步 vs 非同步）：
  - 商業邏輯 / 核心功能 → **同步監控**
  - 樣板、CRUD、重構 → **非同步 + 自動測試驗證**
- 每個任務控制在一個 context window 能解決的大小
- Agent 跑完必須通過 CI（測試 + linter + 安全掃描）才能進 Review
- 放下 code ownership 心態——程式碼不珍貴，解決的問題才珍貴

## 六、Code Review（判斷權：開發團隊 + 架構師）

> 核心交易：你用審查時間替代了寫代碼時間。AI 是快速但不可靠的 junior，槓桿前提是你有能力判斷其輸出。

### 素材摘錄

> 「AI 時代 Code Review 從『語法正確』轉向『系統風險控制』。檢視順序：需求對齊 → 失敗模式 → 資料與權限邊界 → 可測試性 → 可觀測性 → 回滾策略。」
> —— [[9-Code Review 檢視面向]]

> 「Agentic Engineering 是『不同性質的難』。你放棄寫代碼的掌控感，換取快速產出，但必須支付嚴格審查的成本。」
> —— [[4-AI 協作的核心交易：你用審查時間替代了寫代碼時間]]

> 「三步 review 流程：Claude 先自審 → 人工審查架構/業務邏輯 → 團隊標準 review。」
> —— StaffEngineer 的 ClaudeCode 六週實戰

> 「風險分四級（R0-R3）。R0：UI 文案/log/註解；R1：單一服務內部邏輯；R2：跨服務協定/schema 變更/效能敏感；R3：DELETE/DROP/權限/憑證/對外曝露面——需四重保護（Dry-run、雙人簽核、備份還原驗證、分段釋出）。」
> —— AgentHub: quality_gates

> 「五大 Quality Gate：(A) Supply Chain — 防幻覺依賴（AI 約 20% 依賴建議指向不存在的套件）；(B) Agent Safety — 防 prompt injection（OWASP LLM #1 威脅）；(C) System Correctness — 防局部完美掩蓋系統脆弱；(D) Code Quality — 防技術債無聲累積（refactoring 從 24% 暴跌至 <10%，duplicate code 增 4 倍）；(E) Agentic Workflow Safety — 防多步驟連鎖失控。」
> —— AgentHub: quality_gates

> 「AI CR 是預選賽，不是最後一關——任何風險等級都需要人類具名簽核。預設不信任——AI 產出與 AI 評語都需要可驗證證據，不接受『AI 說沒問題』。」
> —— AgentHub: quality_gates

### 重要文件
- Code Review Checklist（系統風險導向）
- PR Template（含風險等級、AI 使用揭露、測試證據）

### Agent 流程建議
- 第一層：**Agent 自審**（語法、風格、基本安全、重複碼偵測）
- 第二層：**人工審查**（架構一致性、業務邏輯、系統風險、失敗模式）
- 第三層：**團隊標準 review**（跨人 review，R2/R3 需額外簽核）
- Review 焦點不是逐行看 code，而是：結構性審查（偏離架構意圖？）、品味審查（三個月後能維護？）、風險直覺（感覺不對但說不出為什麼？）

### 合併前自問
1. 跨服務/跨權限/跨環境？→ R2/R3
2. 不可逆？→ 一律 R3 四重保護
3. CI 過了，但只是 mock？→ 補真實整合驗證
4. AI 說沒問題，我有可驗證證據嗎？→ 沒有就不合併
5. 這段程式碼我真的讀懂了嗎？→ 沒有就不合併

## 最小可行控制集（TOC 視角：先打穿什麼？）

如果不能一次到位，按這個順序建立：

**第一：Quality Gate（QE 的判斷）**
沒有它就絕對不能讓 Agent 自主運作。Gate 是人睡覺時唯一還在運作的守門員。
→ 最小實作：CI 中的自動化測試 + linter + 安全掃描 + secrets scanning，全過才能 merge。

**第二：最小權限與隔離（架構師的判斷）**
品質門檻攔品質問題，但 Agent 可能繞過 gate 直接造成損害。
→ 最小實作：獨立 bot 帳戶 + 只能推 PR 到特定分支 + 沙盒執行 + 不碰 production。

**第三：可執行的需求定義（PM 的判斷）**
前兩個保證「不搞破壞」，但需求模糊會讓 Agent 高效地做錯事。
→ 最小實作：維護「AI 可自主執行任務清單」，每項有明確驗收條件。

### 落地節奏

**第一週**：建 Gate + 隔離，選 1-2 個低風險任務讓 Agent 試跑
**第二至四週**：PM 定義任務類型與驗收標準，開發團隊建立 review 節奏（每日 30 分鐘審查 AI PR），開始收集指標
**第二個月起**：補齊成本監控、事故復盤機制、護欄文件化、工程師 review 能力訓練
