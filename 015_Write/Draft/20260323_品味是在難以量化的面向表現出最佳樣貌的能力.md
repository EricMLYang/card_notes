# 品味是在難以量化的面向，表現出最佳樣貌的能力

> 狀態：素材收集
> 建立日期：2026-03-23

---

## 核心定義

**Taste = 在難以量化的面向，表現出最佳樣貌的能力。**

AI 讓「做」的成本歸零之後，唯一能拉開差距的就是 taste。但 taste 這個詞太抽象了。以下用九篇近期文章裡的真實場景，把「品味」拆解成你看得見的東西。

---

## 金句收集：什麼是品味的具體樣貌？

### 一、品味是看見別人看不見的 2px

> 「當一個按鈕有 ring（外框），另一個沒有的時候，有 ring 的按鈕會比沒有的高 2px。」Schoger 說：「這就是我失眠的原因。」
> — Steve Schoger，Tailwind Labs 設計師

> 「當一個元素同時有陰影和實色邊框的時候，陰影和邊框之間會產生一種 muddy（混濁）的效果，視覺上很髒。」
> — Steve Schoger

> 「同一個 prompt 生成的初版，經過大約 50 次設計師主導的迭代，變成了一個看起來完全不同的網站。差異不在技術，在眼光。」
> — Fox Hsiao 評論 Schoger 的 Claude Code 工作流

> 「AI 會用 indigo 當預設色、會把所有東西置中、會用 solid border 配 shadow 製造混濁效果。設計師知道該把它改成什麼。」
> — Schoger 文

AI 的「預設」就是沒有品味的樣子。品味是看見預設的問題在哪。

> 「他甚至自嘲：『我覺得我寫的這些句子根本不太合理，但它每次都聽得懂。』」
> — Schoger 用自然語言指揮 Claude Code

品味到了一定程度，連語言都不需要精確——你說「再緊一點」，AI 就知道該做什麼。因為你的直覺準到可以用模糊的語言表達精確的要求。

> 「partner logo 列不需要標題，直接放 logo 就很清楚。簡單但很多人會過度設計這個區塊。」
> — Schoger

品味不只是加東西的能力，是知道什麼時候不需要加。

> 「我同時感覺自己在跟一個極其聰明的博士生說話，又在跟一個十歲小孩說話，這個組合在人類身上根本不存在。」
> — Karpathy

品味是感覺到 AI 能力的「參差不齊」。多數人只看到它很強的那面，有品味的人同時看見它很弱的那面。

> 「可以被驗證的任務持續被優化，而那些更柔軟、更難評估的能力——理解言外之意、什麼時候該問澄清性問題、幽默感——則停滯不前。」
> — Karpathy

這句話本身就是品味的示範：他看見了 AI 進步曲線裡，別人沒注意到的斷層。

**品味在這裡是什麼**：AI 產出和專業成果之間的差距，不在程式碼，在於你能不能看見那個「哪裡不對」。2px 的偏差、陰影的混濁感——這些沒有任何 linter 會抓，沒有任何測試會報錯。能辨認出這些的能力，就是品味。

### 二、品味是知道什麼不該自動化

> 「重要的是知道『應該自動化什麼』。能做出這個判斷的，只有每天在現場親手操作的你。」
> — 畠山謙人，日本稅務師，0 員工服務 60 家企業

> 「明確劃分『要自動化的部分』和『需要人看的部分』，這樣才能安全地服務 60 家公司。能做出這種劃分，是因為具備稅務師的實務知識。」
> — 畠山謙人

> 「能用關鍵字比對處理的就不呼叫 API。Claude API 只用在『新交易對象』或『判斷模糊的項目』。」
> — 畠山謙人的兩階段系統設計

**品味在這裡是什麼**：不是「全部丟給 AI」，而是知道哪裡該用便宜的規則、哪裡該用貴的推理、哪裡必須留給人。這種分界線沒有教科書會教你，它來自十幾年現場經驗結晶出來的直覺。

---

### 三、品味是知道自己該從哪裡退出

> 「有客觀標準的任務不需要我在迴路裡。」
> — Andrej Karpathy

> 「你不能成為下一個 prompt 的瓶頸，你需要讓自己退出，把事情安排好，讓它們完全自主運作。」
> — Andrej Karpathy

> 「agent 做不到的事，才是你現在的工作。agent 能做的事，它可能已經做得比你好了，所以要謹慎選擇你花時間在哪裡。」
> — Andrej Karpathy

**品味在這裡是什麼**：不是證明自己比 AI 強，而是誠實判斷「哪個環節我已經不是最佳人選」。多數人捨不得放手，因為放手意味著承認自己在那個環節沒有獨特價值。能放手的前提，是你清楚知道自己不可被取代的部分在哪。

---

### 四、品味是在系統設計裡拿捏輕重

> 「與其給代理更多能力，不如給它紀律。」
> — Jesse Vincent，Superpowers 開發者

> 「嘗試用 Kiro 修復小 bug，結果被轉化為四個 user story 和十六個驗收標準。」
> — Martin Fowler，批評 SDD 工具過度設計

> 「工具複雜度應與問題規模匹配，不要用大型框架做小任務。」
> — Martin Fowler

> 「給 AI agent 的工具，不是越多越好，而是越精準越好。」
> — agent-browser 文

Vercel 把 17 個工具砍到 2 個，成功率從 80% 跳到 100%，速度快 3.5 倍。品味是敢砍，而且砍對地方。

> 「少即是多。Unix pipeline 哲學在 2026 年以一種完全意想不到的方式回來了——每個命令做好一件事，組合起來完成複雜任務。」
> — agent-browser 文

品味有時候是認出一個四十年前的老原則，在新場景裡依然是對的。

> 「不是追求 100% 測試覆蓋率。那不現實，也不必要。關鍵是把產品最核心的使用者旅程鎖住。這些測試就像護欄，讓 agent 自由奔跑時不會衝下懸崖。」
> — agent-browser 文

品味是知道「夠用」在哪裡。100% 覆蓋率是沒有品味的人追求的數字；有品味的人先鎖住最痛的 20%。

> 「證據分級：互動類 bug 要完整重現影片，靜態類問題一張截圖就夠。」
> — agent-browser dogfood skill

品味是對不同嚴重程度的問題，給出不同重量的回應。所有事情都用同一套流程處理，就是沒有品味。

> 「micro-GPT 是我多年執念的成果。整個訓練演算法只有 200 行 Python。這不是關於如何讓它跑得快，而是關於演算法本身。」
> — Karpathy

品味是把複雜的東西蒸餾到不能再少的程度。200 行不是偷懶，是花了多年才做到的精簡。

> 「把這些『型』當作 Skill 累積起來，做得越多就越快。」
> — 畠山謙人

品味是知道哪些重複模式值得被固化成系統。不是所有重複都該自動化——只有那些「型」穩定的才值得。

**品味在這裡是什麼**：知道什麼時候該加結構、什麼時候結構本身就是問題。品味的反面不是粗糙，是過度設計——用四個 user story 去修一個 bug，就是沒有品味。品味是恰到好處，不多不少。

---

### 五、品味是區分「技術上正確」和「實務上正確」

> 「工程師用 AI 做出『技術上很厲害的東西』。稅務師用 AI 做出『實務上正確的東西』。這個差別非常大。」
> — 畠山謙人

> 「正因為不是工程師，Claude Code 才能發揮作用。稅務師和會計人員知道『業務的型』。這些實務知識是花了好幾年才累積起來的，AI 無法自己生出來。」
> — 畠山謙人

> 「自動分錄最可怕的是『把不該做分錄的項目也做了』。」
> — 畠山謙人

技術上，AI 可以對每一筆交易做分錄。實務上，有七種情況做了反而是災難。品味是知道「能做」不等於「該做」。

> 「信賴度為 high 或 medium 就登錄，low 則跳過交由我確認。」
> — 畠山謙人的兩階段設計

品味是在系統裡埋一條「不確定就不要動」的底線。技術上你可以讓 AI 全部處理，實務上你知道 low confidence 的東西碰了會出事。

> 「判斷的界線：符合過去模式 → AI 可處理，新出現的 → 先跟我確認。」
> — 畠山謙人的 CLAUDE.md

品味是把自己多年的判斷邊界，寫成 AI 讀得懂的規則。這件事的難度不在「寫」，在於你得先知道那條邊界在哪。

> 「永遠不讀原始碼。你是以使用者身份測試，不是做 code review。」
> — agent-browser 的 dogfood skill 設計

技術上 agent 可以同時看 code 和測 UI。但刻意禁止它看 code，是因為實務上「用起來對不對」跟「寫得對不對」是兩個完全不同的問題。品味是知道要在哪裡劃這條線。

> 「先驗證再收集證據。在開始錄影或截圖之前，先重試一次確認問題確實可重現。」
> — agent-browser dogfood skill

這是從踩過坑學來的。技術上先截圖再說效率更高，但實務上花大量時間記錄一個誤報才是真正的浪費。

> 「平常用 coding agent 用得很開心，因為它直接用自然語言當介面。當你要整合在應用程式內的時候，操作方式就完全不同了。」
> — Andrew 的 MUD agent 實驗

品味是知道「demo 裡能跑」和「產品裡能用」之間有一道巨大的鴻溝。很多人在 demo 階段就以為自己完成了。

**品味在這裡是什麼**：「能跑」和「對」之間的距離。AI 可以寫出語法正確的程式碼，但不知道「餐飲費超過一萬日圓應該歸交際費而不是會議費」。品味是在那些規格書沒寫、測試抓不到、但現場的人一看就知道的地方，做出正確的選擇。

---

### 六、品味是感受到「人格」的差異

> 「Claude 感覺像一個隊友，會和你一起感到興奮。Codex 則非常 dry——它不在乎你在建造什麼，感覺就是：我實作好了。」
> — Andrej Karpathy

> 「很多工具都沒有重視人格設計，而這恰恰是一個關鍵的差異點。」
> — Andrej Karpathy

**品味在這裡是什麼**：產品的功能規格可以一模一樣，但用起來的「感覺」天差地遠。這種感覺沒辦法寫進 PRD、沒辦法用 A/B test 輕易抓到，但它決定了使用者會不會留下來。品味是對這種「說不出來但感覺得到」的品質有要求。

## 品味的六個面向（初步框架）

| 面向 | 一句話定義 | 代表案例 |
|------|-----------|----------|
| **感知力** | 看見別人看不見的差異 | Schoger 的 2px 和混濁陰影 |
| **邊界感** | 知道什麼不該做 | 畠山的自動化分界線 |
| **自知力** | 知道自己該在哪裡退出 | Karpathy 的「退出迴路」 |
| **分寸感** | 結構恰到好處，不多不少 | Fowler 批評過度設計、Vercel 砍 17 工具到 2 個 |
| **現場感** | 區分技術正確與實務正確 | 畠山的「業務的型」、dogfood 的「不讀原始碼」 |
| **質感力** | 對說不出來的品質有要求 | Karpathy 對 agent 人格的觀察 |
