## Draft v1

人看業務邏輯、Agent 掃資安格式——我的 Code Review 分級實驗

---

帶團隊導入 AI coding agent 之後，我遇到的第一個問題不是 AI 寫得不好，而是 code review 的量暴增到沒人看得完。

以前一個工程師一天產出 200 行 code，你可以仔細看。
現在 Agent 一天產出 2000 行，你不可能每行都看。

所以我開始實驗分級制度：人只看關鍵，其餘交給 Agent 掃。
這篇是我目前的分級邏輯和踩過的坑。

---

▋ AI 協作的本質交易

很多人期待「終於不用自己寫 code 了」。
只對了一半。

Agent 產出速度是人的 2-5 倍。
但它本質上是一個「很快但不太可靠的 junior」。
你省下的寫 code 時間，會直接轉換成 review 時間。

大部分團隊沒意識到這個交易。
導入 Agent，看到產出變快，就覺得成功了。
三個月後才發現：技術債堆到天花板、資安漏洞沒人抓。

所以導入 AI coding 之後，第一件該做的事不是寫更多 prompt template，而是重新設計 code review 流程。

---

▋ Code Review 分級制度

核心想法：不是所有東西都值得人看。

以前 review 是平的——每一行都過人的眼睛。
Agent 時代產出量暴增，你必須分級。

**人看（關鍵決策）**
- 業務邏輯是否正確
- 架構決策是否合理
- 重大風險：資料權限、金流、隱私

**Agent 掃（規則可判斷）**
- 資安漏洞（SQL injection、XSS、敏感資料外洩）
- 格式一致性、命名規範
- 測試覆蓋率

說實話，讓 Agent 跑資安掃描一開始只是想減輕負擔。
結果第一輪跑完，抓出的問題比我們手動 review 兩週找到的還多。

效果意外地好。

Agent 適合處理有明確規則、可自動化判斷的事。
人應該把注意力集中在規則無法覆蓋的地方——業務判斷、架構取捨、風險承擔。

高確定性走快車道（Agent），模糊用推理（資深工程師），不確定回人手。

---

▋ 整個流程都要跟著變

Review 分級只是冰山一角。

Agent 寫 code 快了，但需求不清楚，它只會更快地寫出錯的東西。
架構沒定好，它會用最快速度堆技術債。

所以我把流程拆成三層：

**探索**：PM 負責。AI 幫忙整理資料、生成初版 PRD、找需求矛盾。但決策優先順序仍是人的事。

**驗證**：架構師、設計師、SA 負責。Agent 可以生成架構草稿、檢查規格衝突。但 trade-off 的判斷是人的責任。SA 的角色越來越清楚——需求到系統的翻譯者。

**實作**：Agent 參與度最高。寫 code、跑測試、資安掃描都能幫忙。但資深工程師仍負責品質和決策，QA 仍負責驗收。

---

▋ 每個角色都在做同一件事：收斂不確定性

整個團隊其實在逐步降低不確定性，只是類型不同：

PM → 產品價值 / 架構師 → 技術可行性 / SA → 系統邏輯
工程師 → 實作品質 / QA → 系統風險

AI 的作用？加速產出，但不負責收斂風險。

這個區分很重要。
很多團隊導入 AI 出問題，不是 AI 不好用，是搞混了「加速」跟「收斂」。

沒有品質門檻的加速，只會更快地製造垃圾。
高開發成本曾是天然過濾器——功能花三個月才做完，大家自然先想清楚值不值得做。
Agent 三天做完，過濾器消失了。

所以 review、驗收、評估標準，才是 AI 時代真正的控制介面。

---

▋ 每個人都往上移一層

AI 真正改變的不是「誰寫 code」，而是每個角色的作用位置。

工程師核心工作從寫 code 變成 review code。
PM 的決策速度變成新瓶頸——Agent 等著 spec 開工。
SA 從寫文件的人變成 System Designer。

不是減人，是升級責任。

---

這篇不是完美方案，是正在微調的實驗紀錄。
有些做法效果好，有些還在觀察。

但有一個判斷我比較確定：
當寫 code 變得便宜，你必須重新想清楚——什麼東西是貴的。

答案是判斷。

### 鉤子選項（每個標題配 1 個鉤子）

**鉤子 1**（配標題 1）：
上個月，我的團隊開始全面用 Agent 寫 code。第一週的感覺是興奮——產出速度大概快了三到五倍。第二週的感覺是焦慮——因為 code review 變成了地獄。Agent 寫得快，但它不會幫你想清楚邊界條件，不會幫你判斷這個 API 設計三個月後會不會變成技術債。我花在 review 上的時間，比以前自己寫 code 還多。後來我試了一個做法：把 review 分級。關鍵業務邏輯和重大風險由人看，資安格式一致性交給 Agent 掃。結果資安掃描的效果意外地好。這讓我重新想了一件事：AI 時代的開發流程，到底該怎麼設計？這篇是我目前的答案——一份正在實驗、持續微調的初版流程。

**鉤子 2**（配標題 2）：
說實話，我原本以為讓 Agent 跑資安掃描只是應急。結果跑完第一輪，它抓出的問題比我們過去手動 review 兩週找到的還多。那天下午我坐在電腦前，突然意識到一件事：問題不是 Agent 能不能寫 code，而是我一直沒想清楚——哪些事情該讓人做，哪些事情交給 Agent 做，其實比「要不要用 AI」更重要。於是我花了一個月，用內部專案重新設計了整個團隊的開發流程。這篇是我的實驗紀錄：一個 5-10 人小團隊，怎麼把 AI coding agent 真正嵌進探索、驗證、實作三個階段，而且活下來。

**鉤子 3**（配標題 3）：
帶團隊導入 AI coding agent 之後，我遇到的第一個問題不是 AI 寫得不好，而是 code review 的量暴增到沒人看得完。以前一個工程師一天產出 200 行 code，你可以仔細看。現在 Agent 一天可以產出 2000 行，你不可能每行都看。所以我開始實驗一個分級制度：人只看關鍵——業務邏輯對不對、架構決策合不合理、有沒有重大風險。其餘的資安漏洞、格式一致性、常見 pattern 違反，全部交給 Agent 掃。這篇把我目前的分級邏輯、背後的判斷原則、踩過的坑，整理成一份可參考的實驗紀錄。

**鉤子 4**（配標題 4）：
回頭來看，AI coding agent 在我的團隊裡製造的最大衝擊，不是工程師變快了，而是瓶頸轉移了。以前卡在「人手不夠、code 寫不完」。現在 Agent 一口氣可以開好幾條平行線跑，code 不是問題了——問題變成 PM 的需求定義夠不夠清楚、架構師的決策夠不夠快、review 的品質能不能跟上產出的速度。換句話說，瓶頸從手上移到了腦上。這篇是我帶小團隊重新設計開發流程的紀錄。不是一個完美方案，而是一份正在微調中的實驗報告——但裡面有幾個發現，我覺得值得分享。

**鉤子 5**（配標題 5）：
很多團隊導入 AI coding 之後，第一個反應是加文件——寫更多 spec、更多 prompt template、更多 review checklist。我的團隊一開始也這樣。後來我發現，問題不在文件不夠多，而是整個流程的設計邏輯要變。AI 改變的不是「誰寫 code」，而是每個角色在流程中的作用位置。PM 要更早定義清楚驗收條件，架構師要更快給出系統邊界，工程師的核心工作從寫 code 變成 review code。每個人都往上移了一層。這篇是我的實驗筆記：怎麼把這個「往上移一層」的概念，變成一套小團隊可以跑的開發流程。
