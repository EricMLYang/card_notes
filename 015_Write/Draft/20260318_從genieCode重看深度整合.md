## 發想筆記

### 價值維度定位
- **主要：啟發（Inspire）**— 重新定義「整合」在 AI 產品中的角色，從「事後補強」變成「核心策略」
- **次要：共感（Empathize）**— 以個人真實使用體驗出發，很多人都有過「AI 好像還行但沒真的改變什麼」的感受
- **輔助：教育（Educate）**— 提供可參考的思考角度（真垂直三要素、Harness 概念）

### 乾貨維度
- **經驗（最強項）**：Genie Code 的第一手使用體驗 + 對比過去「淺整合」的個人經驗
- **框架（可輕帶）**：真垂直三要素可以作為讀者自我檢查的工具，但不要寫得太像教學
- **知識（自然融入）**：Google 悖論、Agent 95% 失敗率等可以作為「不只是我這樣想」的佐證

### 觀點深化方向
這篇適合 **NicheDown**（W2_06）的路線：
- 從具體故事（Genie Code 使用體驗）出發
- 深入淺出，提煉底層洞察（整合 = 產品本身）
- 但因為定位是「個人感想」，不要過度理論化，保持觀察者的謙虛語氣

### 鉤子選項（前 3 標題 × 3 鉤子）

#### 標題 1「用了 Genie Code 之後，我對『整合』這件事改觀了」

**A-1（真實經驗切入）**
最近實際用了一段時間 Databricks 的 Genie Code。
它不是那種讓你「哇」一聲的東西，也沒有什麼驚天動地的新技術。但用了一段時間之後，我開始注意到一件事：很多原本看起來只是小摩擦的地方，好像被悄悄拿掉了。
少一次環境切換，少一次把結果貼回去給 AI 看，少一次補充「其實我的 table 是這樣設計的」。
這些事情單看都不大，但累積起來，我開始懷疑一件事：也許很多 AI 產品讓人覺得「好像還行」，問題不在模型不夠聰明，而是整合深度還不到位。

**D-1（反直覺觀點）**
以前我對整合的理解，大概停在「系統能互通、資料能流動、流程跑得通」就算及格。
但最近用 Genie Code 的時候，有一種微妙的感覺：它的體驗之所以順，不是因為模型多厲害，而是因為它幾乎完全活在 Databricks 的環境裡——直接接觸到整個平台的上下文、運算環境、物件關係。
這讓我開始想：也許「整合」這件事，要比我原本以為的更深、更極端一些。

**E-1（問題拆解）**
你有沒有遇過這種情況：接了一個 AI 工具，功能都有，但用起來老覺得「差一口氣」？
最近我在用 Genie Code 的時候，開始有一個還在形成中的想法——也許那口氣不是模型的問題，而是整合深度的問題。
這篇文章不是要教你怎麼評估工具，比較像是我自己用過之後的一些觀察和反思。如果你也對 AI 產品的整合策略有興趣，也許可以一起想想。

#### 標題 3「AI 產品真正的差距，藏在那些看起來很小的摩擦裡」

**A-3（真實經驗切入）**
少一次環境切換。少一次補充背景說明。少一次把執行結果複製貼上回來給 AI 看。
這些事情單看都不大。但最近用 Databricks 的 Genie Code，這些小摩擦被拿掉之後，我才意識到它們加起來的影響有多大。
這篇是我用過之後的一些觀察——關於為什麼有些 AI 產品讓人覺得「順」，有些讓人覺得「好像可以但總差一點」。

**B-3（幫讀者做了功課）**
我最近同時在想兩件事：為什麼有些 AI 工具用起來體驗特別好？還有，小團隊做 AI 產品到底該做深還是做廣？
用了一段時間 Genie Code 之後，我開始覺得這兩個問題的答案可能是同一個：整合深度。
這篇會分享我的使用感受，也會聊到幾張卡片筆記裡的觀點——關於 Harness、垂直 AI 護城河、和「窄而深」的策略思考。

**F-3（個人觀點型）**
我越來越覺得，在 AI 產品裡，整合本身可能就是產品，甚至是最關鍵的一部分。
這個想法不是憑空來的。最近用 Genie Code 的體驗讓我很有感——它的順暢不是因為模型聰明，而是因為整合深度把很多小摩擦都消化掉了。
以下是我的一些觀察和反思，不算是完整結論，但也許值得一起想想。

#### 標題 5「整合不是產品成熟以後才要做的事——它本身就是產品」

**D-5（反直覺觀點）**
以前我直覺認為，小團隊資源有限，應該先做輕、做廣、做快，整合是之後再慢慢補的。
但最近我開始動搖。用了一段時間 Genie Code 之後，我感覺到的體驗差異，幾乎都不是來自模型能力，而是來自它和底層環境整合得有多深。
也許在 AI 產品的世界裡，整合不是加分項，而是產品能不能真的起作用的前提條件。

**A-5（真實經驗切入）**
如果你問我，用 Genie Code 最大的感受是什麼？
不是它多會寫程式，也不是它的對話有多精準。是它讓我重新想了一件事：整合這件事，可能要比我原本想的更深、更極端一些。
以前把整合當成產品成熟以後慢慢補的東西，現在覺得它根本就是產品本身。

**E-5（問題拆解）**
我們常聽到一句話：AI 產品的關鍵是模型。
但 Google 有最強的模型、自有的硬體、50 億用戶，為什麼做不出讓人滿意的 AI 產品？Agent 產品有 95% 在正式環境失敗，問題也不在模型不聰明。
那到底差在哪裡？最近用 Genie Code 的一些體驗，讓我開始有一個方向性的猜測。
與其說是它多會寫程式，或是多像一個厲害的聊天機器人，不如說是讓我重新思考了一件事：**整合這件事，可能要比我原本想的更深、更極端一些。**

以前我對產品整合的理解，某種程度上還是比較偏向「有接上就好」。
系統 A 能接系統 B，資料可以流動，畫面上可以操作，整體流程跑得通，大概就算是一種整合。這種做法在很多產品裡其實也沒有問題，很多系統確實就是先把流程打通，再慢慢優化細節。

但這次使用 Genie Code 的過程，讓我開始感覺到另一種差異。
它的體驗之所以順，很大一部分好像不是因為 AI 模型本身，而是因為它幾乎完全活在 Databricks 的環境裡。它不是站在系統外面理解你的資料，也不是只看到幾段 schema 或文件之後再試著推測，而是直接接觸到整個資料平台的上下文、運算環境、物件關係以及執行結果。

這種深度帶來的感覺，其實很微妙。
很多原本看起來只是小摩擦的地方，好像被悄悄拿掉了。

例如少一次環境切換、少一次補充背景說明、少一次把執行結果複製貼上回來給 AI 看。這些事情單看都不大，但累積起來，體驗差異其實蠻明顯的。

所以我最近有一個還在形成中的想法：
**有些 AI 功能最後讓人覺得「好像還行，但沒有真的改變什麼」，也許不一定是因為模型不夠聰明，而是整合深度還不夠。**
這當然只是我目前的一個觀察，但在實際使用之後，這種感覺變得比較強。

這也讓我重新思考小團隊做產品的方向。
以前可能會直覺認為，小團隊資源有限，應該避免做太深的整合，先做比較輕、比較廣、比較快推出去的產品。但現在反而有點懷疑，對 AI 產品來說，**廣而淺可能不一定是最有競爭力的策略，窄而深也許反而更有機會。**

如果一個系統只是表面上接了很多東西，但 AI 其實很難理解整個環境，那很多能力最後可能還是停留在「輔助工具」的層次。
但如果在一個具體場景裡，把資料、流程、工具和權限都整得很深，AI 也許比較有機會真的進入那個工作環境，而不是只是在旁邊提供建議。

對我自己來說，這其實是一個蠻有意思的提醒。
以前我可能比較把「整合」當成產品成熟之後要慢慢補強的部分，但現在越來越覺得，在 AI 產品裡，整合本身可能就是產品的一部分，甚至可能是最關鍵的一部分。

所以如果要把最近這段體驗簡單收斂成一句話，大概會是：

**小團隊在做 AI 產品時，也許可以更認真思考整合這件事，而且整合可能要比原本想的更深一些。很多體驗差異，最後其實都藏在那些看起來很小的摩擦裡。**

# 用了 Genie Code 之後，我對「整合」這件事改觀了

最近實際用了一段時間 Databricks 的 Genie Code。

它不是那種讓你「哇」一聲的東西，也沒有什麼驚天動地的新技術。但用了一段時間之後，我開始注意到一件事：很多原本看起來只是小摩擦的地方，好像被悄悄拿掉了。

少一次環境切換，少一次把結果貼回去給 AI 看，少一次補充「其實我的 table 是這樣設計的」。

這些事情單看都不大，但累積起來，我開始懷疑一件事：也許很多 AI 產品讓人覺得「好像還行」，問題不在模型不夠聰明，而是整合深度還不到位。

---

### 以前我對整合的理解，大概停在「有接上就好」

系統 A 能接系統 B，資料可以流動，畫面上可以操作，整體流程跑得通，大概就算是一種整合。

這種做法在很多產品裡其實也沒有問題。很多系統確實就是先把流程打通，再慢慢優化細節。

但這次使用 Genie Code 的過程，讓我開始感覺到另一種差異。

它的體驗之所以順，很大一部分好像不是因為 AI 模型本身，而是因為它幾乎完全「活在」Databricks 的環境裡。它不是站在系統外面理解你的資料，也不是只看到幾段 schema 或文件之後再試著推測，而是直接接觸到整個資料平台的上下文——運算環境、物件關係、執行結果，全部在同一個脈絡裡。

這讓我想到之前讀過的一個觀點：把「會聊天的 LLM」和「能交付成果的 Agent」區分開來的，不是模型的權重，而是 Harness——驅動框架。模型是原始認知能力，但迴圈、工具、上下文管理、驗證機制這些東西，才是讓 AI 真正能做事的外骨骼。

兩個產品用同一個模型，體驗卻可以像不同物種，差別就在這裡。

---

### 那些「看起來很小的摩擦」，加起來其實很大

這種深度帶來的感覺很微妙。

不是某個功能特別厲害，而是很多原本的小摩擦不見了。少一次環境切換、少一次補充背景說明、少一次把執行結果複製貼上回來。

如果你有用過各種 AI coding 工具，大概知道我在說什麼。很多時候 AI 其實「能做」，但你要花不少力氣幫它理解你的環境——這裡用什麼命名、那張 table 的欄位長什麼樣、上次那個 query 的結果是什麼。

這些事情單看都不大，但它們讓 AI 停留在「輔助工具」這個層次。你還是那個理解全局的人，AI 只是幫你做零碎的事。

反過來，當 AI 真正活在你的工作環境裡，它開始能直接接觸到 context——不只是你餵給它的那幾段文字，而是整個平台的狀態。這時候體驗就不一樣了。

我最近讀到一個說法蠻有共鳴的：「大多數創辦人以為自己在做 AI 產品，其實是在做 context selection 系統。」多數 Agent 在正式環境失敗，問題不在模型不聰明，而在於 context 管理、memory 設計、錯誤處理這些看起來不性感的工程問題。

整合深度，本質上就是在解決這件事。

---

### 小團隊做 AI 產品，也許不該怕做太深

這也讓我重新想了小團隊做產品的方向。

以前我的直覺是：小團隊資源有限，應該先做輕、做廣、做快。整合太深太重，等產品長大了再慢慢補。

但現在反而有點懷疑這個直覺。

如果一個系統只是表面上接了很多東西，但 AI 其實很難理解整個環境，那很多能力最後可能還是停留在「好像可以但總差一點」的狀態。但如果在一個具體場景裡，把資料、流程、工具和權限都整得很深，AI 也許比較有機會真的進入那個工作環境——不只是在旁邊提供建議，而是能夠執行、能夠驗證、能夠反覆改進。

從另一個角度看，專注在一個場景做到極致，回報可能是遞增的。每多投入一分整合深度，AI 能理解的 context 就多一層，能省下的摩擦就多一塊。這跟分散力量去接很多東西不一樣——那種回報往往是線性的，甚至因為切換成本而遞減。

有個觀點我覺得蠻精準的：評估一個 AI 產品是否「真垂直」，可以看三件事——有沒有專有數據、有沒有深入工作流程、有沒有對結果負責。三者缺一，大概就是偽垂直。Genie Code 之所以讓我有感，也許就是因為它在 Databricks 自己的場景裡，這三件事都做得比較到位。

---

### 所以，整合可能是產品本身

這篇不是要幫 Genie Code 背書。它也有自己的侷限，比如離開 Databricks 的生態它就沒那麼好用了，這反而也在印證同一件事。

我真正想說的是一個對我自己蠻有意思的提醒：

以前我把「整合」當成產品成熟之後要慢慢補強的部分。現在越來越覺得，在 AI 產品裡，整合本身可能就是產品的一部分，甚至是最關鍵的一部分。

也許很多 AI 功能讓人覺得「好像還行，但沒有真的改變什麼」，不一定是因為模型不夠聰明，而是因為它被放進了一個太淺的環境。

如果你也在做 AI 相關的產品，也許可以更認真思考整合這件事。

很多體驗差異，最後其實都藏在那些看起來很小的摩擦裡。

---

### 收尾版本 B（備選）

你今天花時間為 AI 搭建的環境、整理的 context、打通的流程，不會白費。隨著模型越來越強，同一套整合架構能產出的價值會非線性增長。

整合不只是現在的產品體驗，更是對未來的投資。

### 收尾版本 C（備選）

在很多 AI 產品裡，真正影響效果的，可能不是模型本身，而是它被放進了什麼樣的環境。

這不是一個很完整的結論，比較像我用過之後的觀察。但至少對我來說，這次體驗讓我重新意識到一件事：那些看起來最不起眼的摩擦，裡面藏的東西比我想的多。
