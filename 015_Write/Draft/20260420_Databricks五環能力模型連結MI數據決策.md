## 原始草稿

這是一份針對「數據與決策斷層」結構性困境的深度解析，結合了早期在決策輔助研究框架中的思考，並對接 Databricks 最新提出的「可組合畫布（Composable Canvas）」架構。

---

### 第一部分：數據與決策斷層的結構性困境

長期以來，數據分析專案普遍面臨「數據豐富，但決策貧瘠」的困境。這種斷層並非源於演算法不足，而是源於三個核心的結構性障礙：

#### 1. 整合技術債的指數成長 ($O(n^2)$ 困境)
傳統 Martech 或數據架構採用「應用程式中心（App-centric）」模式。每引進一個新的 SaaS 工具，為了讓其具備決策所需的數據，必須與現有的 $n$ 個系統進行點對點整合。這導致整合成本隨系統數量呈幾何倍數成長。實務上，IT 團隊 80% 的精力被耗費在維護脆弱的數據管線（Data Pipeline）與解決數據同步延遲，而非優化決策邏輯。這種「同步稅」使得動態決策成為奢望。

#### 2. 上下文語義的破碎化（Semantic Gap）
決策需要完整的上下文。然而，在「僵化堆疊」架構中，客戶行為、企業營運、內容資產等數據被鎖在不同的「紀錄系統（Systems of Record）」中。每個系統對「客戶」或「轉化」的定義不一，導致 AI 或模型在推論時缺乏統一的真理來源（Single Source of Truth）。缺乏語義一致性，所謂的「自動化決策」只會加速傳遞錯誤的判斷。

#### 3. 靜態預測與動態現實的脫鉤
傳統分析多為「事後回溯」或「定點預測」。但在實際商業場景中，決策是連續且滾動的。當一個關鍵指標（如 GDP 修正或終端價格劇變）更新時，系統若無法自動觸發傳導鏈進行修正，該預測模型在產出的瞬間便已過期。

---

### 第二部分：Databricks 白皮書的核心範式轉移

Databricks 提出的《AI 時代的 Martech 革命》白皮書，本質上是在宣告「應用程式堆疊時代」的終結，並提出以「資料情報平台」為核心的**可組合畫布（Composable Canvas）**架構。

#### 1. 數據資產的重新定義（五類資料）
白皮書打破了過去僅關注「結構化交易數據」的限制，將企業資產解構為五類可治理的數據：
* **客戶資料（Customer）：** 行為與軌跡。
* **企業資料（Company）：** 庫存、財務等內部上下文。
* **內容資料（Content）：** 創意與非結構化資訊，這是 AI 代理推論的燃料。
* **代碼資料（Code）：** 模型權重、Prompt 與 Agent 技能。
* **控制資料（Control）：** 語義定義與治理護欄，確保 AI 行為可控。

#### 2. 五環能力模型 (5 Rings of Capability)
這是該白皮書最具建設性的部分，它將數據轉化為價值的路徑標準化：
* **環 1 (Data Core)：** 統一存放上述五類資料，解決孤島問題。
* **環 2 (Semantic Layer)：** 標準化指標與邏輯，讓所有應用程式與 Agent 說同一種語言。
* **環 3 (CaaS, Context-as-a-Service)：** 這是動態決策的關鍵，將零散數據打包成即時上下文。
* **環 4 (Decisioning)：** 獨立的決策引擎，負責在多重目標下進行效用計算與權衡。
* **環 5 (Apps & Agents)：** 最終的執行端，將決策轉化為具體行動。

### 第三部分：過去研究框架與 AI 時代的邏輯對接

回視早期建立的「決策輔助研究框架」，其核心邏輯與五環模型在多年後產生了強烈的共振，並在 AI 代理時代找到了實作的支點。

#### 1. 產生方案：從手動建模到語義驅動的自動模擬
過去強調「因果發現」與「情境模擬」來產生決策選項。在舊架構下，這需要針對每個問題進行繁瑣的特徵工程。
* **現今連結：** 藉由 **Ring 2 (語義層)**，數據被賦予了業務意義。AI 代理不再需要從頭學習數據關係，而是直接在語義層上運行。這使得「因果推論」可以大規模自動化，Agent 能基於統一的語義框架，自動生成多種介入方案。

#### 2. 權衡方案：決策層 (Ring 4) 作為判斷力的載體
過去的研究核心在於「不確定性與風險權衡」，利用價值函數與偏好建模選出最適方案。這正是白皮書中 **Ring 4 (Decisioning)** 的職責。
* **現今連結：** 隨著生成式 AI 降低了「方案產生」的成本，真正的商業壁壘轉移到了「判斷（Judgment）」上。將決策邏輯從特定 App 中抽離，下沉為獨立的 Decisioning Service，使得企業能跨部門、跨渠道套用一致的風險評估標準。這實踐了「以決策輔助為核心」的初衷。

#### 3. 滾動更新願景：CaaS (Ring 3) 實現資訊到推論的即時傳導
過去最理想的願景是「資訊一更新，預測就能跟著滾動修正」。
* **現今連結：** 這一願景在 **Ring 3 (CaaS)** 得到了工程上的解法。當外部新資訊（如市場訊號）進入 **Ring 1 (Data Core)**，CaaS 自動將其打包成最新的上下文提供給 Agent。Agent 隨即觸發推論流程，判斷未來結果（如提前或延後）。這種「自動滾動」不再需要人工介入重新跑數，數據與決策之間的齒輪真正嚙合了。

### 第四部分：個人心得——從工具思維到 Harness Engineering

看到 Databricks 這份白皮書後，最深刻的體悟在於：**決策輔助的終局，不在於更強大的 Prompt，而在於更穩定的「Harness（執行環境）」。**

過去我們在追求「精確預測」時，常陷入單一模型的陷阱。但白皮書揭示了：在 Agent 時代，我們應該建造的是一個「畫布（Canvas）」。在這個畫布上，數據是流動的資產（Ring 1），邏輯是標準化的語義（Ring 2），而 Agent 是在畫布上根據實時上下文（Ring 3）不斷進行決策（Ring 4）並執行（Ring 5）的實體。

這種架構現代化的本質，是將人類的「品味（Taste）」與「判斷（Judgment）」編碼進 **Control Data** 與 **Decisioning Layer**。這讓數據工程師的角色從「水管維護員」轉向「系統架構師（Data Architect）」，重點在於設計一套能讓 Agent 穩定運行的機制，確保當外部世界變動時，系統能自發性地依據我們設定的價值觀，做出邏輯一致的決策。

這不是對過去研究方向的修正，而是對其願景的終極實踐。以前苦思良久的「數據如何變決策」，答案不在於找到一個完美的算法，而在於構建一個能讓算法、上下文與行動原生交互的五環生態系統。

## 發想筆記

### 價值定位
- **主打價值**：啟發（Inspire）+ 教育（Educate）
- **輔助調味**：共感（Empathize）——用第一人稱「以前做因果結構規劃的痛苦」帶入
- **定位理由**：目標讀者（寫 Code 兼 PM 的中階工程師）最需要的是重新定義問題的框架（啟發），以及理解 Databricks 五環能力模型與自己工作的具體連結（教育）。個人經驗作為共感錨點，讓讀者覺得「這個人懂我的痛」。

### 乾貨盤點

**知識素材**：
- Databricks 五環能力模型（五類資料 + 五環架構）
- $O(n^2)$ 整合技術債增長模型
- 因果推論方法論（RCT、DiD、因果森林）
- CaaS (Context-as-a-Service) 概念

**經驗素材**：
- 過去做決策輔助研究時，情境與因果結構規劃非常困難，沒有 Agent 可以協助，要串接非常多東西
- 長期在「資料 + 系統 + 場域」交界處工作（SmartSignage DataHub / VMS / Databricks）

▋ 先說清楚：為什麼「有數據」不等於「能決策」

這件事的結構性障礙有三個，每一個都不是換演算法能解的。

**第一，整合技術債的指數成長。**

傳統架構是「應用程式中心」模式。
每引進一個新的 SaaS 工具，為了讓它具備決策所需的數據，得跟現有的 n 個系統做點對點整合。

整合成本不是線性的，是 $O(n^2)$ 的。

實務上，IT 團隊 80% 的精力被耗費在維護脆弱的 Data Pipeline 與解決同步延遲。
不是在優化決策邏輯。
是在修水管。

**第二，語義的破碎化。**

決策需要完整的上下文。
但在僵化的堆疊架構裡，客戶行為、企業營運、內容資產被鎖在不同的系統中。

每個系統對「客戶」的定義不一樣。
每個系統對「轉化」的定義也不一樣。

沒有統一的語義，所謂的「自動化決策」只會加速傳遞錯誤的判斷。

**第三，靜態預測與動態現實的脫鉤。**

傳統分析多是「事後回溯」或「定點預測」。
但商業場景裡，決策是連續的、滾動的。

當一個關鍵指標更新了——GDP 修正、終端價格劇變——如果系統無法自動觸發傳導鏈修正，那個預測模型在產出的瞬間就已經過期了。

以前我在做決策輔助框架的時候，最痛的就是這裡。
你想做滾動更新，但資訊到推論的傳導鏈全部是斷的。
每次外部條件變了，你得重新手動跑一輪。

▋ Databricks 五環模型：不是又一個新框架，而是終於有人把路徑想清楚了

Databricks 發佈的白皮書，本質上在宣告「應用程式堆疊時代」的終結。
取而代之的是以「資料情報平台」為核心的可組合畫布（Composable Canvas）。

讓我欣賞的不只是架構本身，而是它把企業數據資產重新定義為五類可治理的資料：

- **客戶資料（Customer）**：行為與軌跡
- **企業資料（Company）**：庫存、財務等內部上下文
- **內容資料（Content）**：創意與非結構化資訊——這是 AI Agent 推論的燃料
- **代碼資料（Code）**：模型權重、Prompt 與 Agent 技能
- **控制資料（Control）**：語義定義與治理護欄，確保 AI 行為可控

特別是「控制資料」這一類。
過去我們很少把「規則」和「標準」當成一種需要治理的數據資產。
但在 Agent 時代，你對品質的定義、對風險的容忍度、對決策邊界的判斷——這些東西如果沒有被編碼成可管理的數據，Agent 就是在裸奔。

▋ 五環的內在邏輯：從數據到行動的標準化路徑

五環能力模型是這份白皮書最有建設性的部分。
它把「數據→決策→行動」的路徑拆成五層同心環：

**Ring 1：Data Core**
統一存放五類資料，解決孤島問題。
不再是每個 App 自己存自己的。

**Ring 2：Semantic Layer**
標準化指標與邏輯。
讓所有應用程式與 Agent 說同一種語言。

這一環解決了前面說的「語義破碎化」。
當所有人對「客戶」「轉化」「營收」的定義一致，AI 的推論才有共同基礎。

**Ring 3：CaaS（Context-as-a-Service）**
這是動態決策的關鍵。
把零散數據打包成即時上下文，供 Agent 隨時取用。

過去我最渴望的「資訊一更新，推論就跟著滾動」，在這一環有了工程上的解法。

**Ring 4：Decisioning**
獨立的決策引擎。
負責在多重目標下進行效用計算與權衡。

這一環最被低估，也最重要。
它把決策邏輯從特定 App 中抽離出來，變成一個獨立的服務。

**Ring 5：Apps & Agents**
最終的執行端。
消費前四環提供的數據、語義、上下文與決策結果，轉化為具體行動。

過去要做因果發現和情境模擬，得針對每個問題做繁瑣的特徵工程。
沒有人幫你管理「這個指標到底代表什麼」「這兩個變數有沒有因果關係」。

AI Agent 不再需要從頭學習數據關係，而是直接在語義層上推論。

但沒有一個系統能自動做到這件事。

資訊到推論的傳導不再需要人工介入。
數據與決策之間的齒輪，真正嚙合了。

▋ 最深的體悟：決策輔助的終局，不在 Prompt，在 Harness

看完這份白皮書，最讓我觸動的不是某個技術細節。
而是一個認知的翻轉。

過去追求「精確預測」，常常陷入單一模型的陷阱。
覺得只要模型準一點、數據多一點、演算法新一點，決策品質就會提升。

但五環模型揭示了：重點不在模型本身。
重點在 Harness——那個讓模型真正落地運作的執行環境。

在這個畫布上，數據是流動的資產（Ring 1），邏輯是標準化的語義（Ring 2），Agent 根據即時上下文（Ring 3）不斷進行決策（Ring 4）並執行（Ring 5）。

Harness 的組成不是一段 Prompt。
它是迴圈、工具、上下文管理、持久化、驗證、限制條件的總和。
兩個團隊用同一個模型，做出來的 Agent 體驗卻像不同物種——差別就在 Harness。

更深一層說，Harness 其實是品味的具體實現。

這些選擇反映的是你的價值觀。

▋ 所以這意味著什麼？

架構現代化的本質，是把人類的品味與判斷編碼進 Control Data 與 Decisioning Layer。

這讓做數據的人的角色開始轉變。
不再是「把水管維護好就可以了」。
而是設計一套能讓 Agent 穩定運行的機制。
確保外部世界變動時，系統能依據你設定的價值觀，做出邏輯一致的決策。

對我個人而言，這不是對過去研究方向的修正，而是對那些願景的遲來實踐。

以前苦思良久的「數據如何變決策」，答案不在找到一個完美的算法。
而在構建一個能讓算法、上下文與行動原生交互的五環生態系統。

以前串接因果結構串到快燒乾，不是方法論不對。
是執行環境從來沒被認真設計過。
現在有人幫你想好了。
剩下的是，你願不願意重新理解自己工作的位置。

## Draft v2

【 以前串接因果結構快把人燒乾，現在有人幫你把執行環境想好了 】
——從 Databricks 五環能力模型，重新理解「數據如何變決策」

▋ 以前做決策輔助研究的時候，我遇到一個很挫敗的現實。

理論上很美的東西，實作上串不起來。

因果發現需要特徵工程。
情境模擬需要多個變數同時校準。
滾動更新需要即時的資料傳導鏈。

每一塊單獨做都還行。
但要把它們接成一個能動的決策系統？

沒有 Agent 可以協助。
沒有語義層讓不同系統說同一種語言。
沒有上下文服務自動打包即時資訊。

全靠人工手動串接。

那段經歷讓我很確定一件事。
數據與決策之間的斷層，問題出在架構，不在技術本身。

所以當我看到 Databricks 提出的五環能力模型（Composable Canvas）時，第一反應是——「這就是我以前缺的那套執行環境。」

這篇文章想把五環模型拆開，對接到過去做決策系統時的實戰痛點。


▋ 為什麼「有數據」還是做不了決策

這件事有三個結構性障礙。每一個都跟演算法無關。

**整合技術債的指數成長。**

傳統架構是「應用程式中心」模式。每引進一個新 SaaS 工具，為了讓它拿到決策需要的數據，得跟現有的 n 個系統做點對點整合。

整合成本是 $O(n^2)$ 的。

實務上，IT 團隊 80% 的精力消耗在維護脆弱的 Data Pipeline 和同步延遲上。
真正在優化決策邏輯的時間少得可憐。

**語義的破碎化。**

決策需要完整的上下文。
但客戶行為、企業營運、內容資產被鎖在不同系統裡。

一個系統說的「客戶」，跟另一個系統說的「客戶」，可能根本是不同的東西。
「轉化」也是。

語義不一致的情況下，自動化決策只會更快地傳遞錯誤的判斷。

**靜態預測跟不上動態現實。**

傳統分析多是事後回溯或定點預測。
但商業場景裡的決策是連續的、滾動的。

一個關鍵指標更新了——比方 GDP 修正、終端價格劇變——如果系統沒辦法自動觸發傳導鏈修正，那個預測在產出的瞬間就過期了。

以前做決策輔助框架的時候，最痛的就是這裡。
想做滾動更新，但資訊到推論的傳導鏈全部是斷的。
外部條件一變，就得重新手動跑一輪。


▋ 五環模型：終於有人把這條路想清楚了

Databricks 這份白皮書，本質上在宣告「應用程式堆疊時代」的終結。

取而代之的是以「資料情報平台」為核心的可組合畫布。

讓我覺得有意思的地方，是它把企業數據資產重新拆成五種可治理的資料：

客戶資料（Customer）——行為與軌跡。
企業資料（Company）——庫存、財務等內部上下文。
內容資料（Content）——創意與非結構化資訊，AI Agent 推論的燃料。
代碼資料（Code）——模型權重、Prompt 與 Agent 技能。
控制資料（Control）——語義定義與治理護欄，確保 AI 行為可控。

特別想提「控制資料」這一類。

過去很少有人把「規則」和「標準」當成需要治理的數據資產。
但在 Agent 時代，你對品質的定義、對風險的容忍度、對決策邊界的判斷——這些如果沒有被編碼成可管理的數據，Agent 等於在裸奔。


▋ 五環怎麼運作

五環能力模型把「數據→決策→行動」拆成五層同心環。
我覺得這是白皮書裡最有建設性的部分。

**Ring 1：Data Core。**
統一存放五類資料。不再是每個 App 自己存自己的。

**Ring 2：Semantic Layer。**
標準化指標與邏輯，讓所有應用程式與 Agent 說同一種語言。

這就是前面說的語義破碎化的解法。
當所有人對「客戶」「轉化」「營收」的定義一致，AI 的推論才有共同基礎。

**Ring 3：CaaS（Context-as-a-Service）。**
動態決策的關鍵。把零散數據打包成即時上下文，供 Agent 隨時取用。

過去我最渴望的「資訊一更新，推論就跟著滾動」，在這一環有了工程上的解法。

**Ring 4：Decisioning。**
獨立的決策引擎，負責在多重目標下做效用計算與權衡。

這一環最被低估，也最重要。
它把決策邏輯從特定 App 中抽離，變成獨立服務。

**Ring 5：Apps & Agents。**
執行端。消費前四環提供的數據、語義、上下文與決策結果，轉化為行動。


▋ 以前做不到的事，現在怎麼接上的

回頭看以前做的決策輔助研究，核心邏輯其實沒有錯。
當時缺的是執行環境。

**產生方案——以前靠手動建模，現在靠語義驅動。**

做因果發現和情境模擬，以前得針對每個問題做繁瑣的特徵工程。
沒有人幫你管理「這個指標到底代表什麼」「這兩個變數之間有沒有因果關係」。

有了語義層（Ring 2），數據有了業務意義。
Agent 可以直接在語義層上推論，因果分析可以大規模自動化。

以前這件事完全不可想像。

**權衡方案——以前靠人腦扛，現在有獨立的決策層。**

過去做的研究核心是「不確定性與風險權衡」。
用價值函數和偏好建模，在多個方案中找最適的那一個。
但邏輯綁死在特定應用場景裡，換個部門就得重來。

Ring 4 把決策邏輯下沉為獨立服務。
企業可以跨部門、跨渠道套用一致的風險評估標準。

生成式 AI 把「產生方案」的成本打到極低之後，真正的商業壁壘已經轉移到「判斷」上。
判斷需要的是穩定的決策框架，不是更花俏的 Prompt。

**滾動更新——以前是奢望，現在 CaaS 接上了。**

以前最理想的願景：外部資訊一更新，預測就跟著滾動修正。
但沒有任何系統能自動做到。

Ring 3 提供了工程上的解法。
市場訊號進入 Data Core，CaaS 自動打包成最新上下文提供給 Agent。
Agent 隨即觸發推論，判斷結果是否需要修正。

人工介入不再是必要條件。
數據與決策之間的齒輪，真正嚙合了。


▋ 最深的體悟

看完這份白皮書，最觸動我的不是某個技術細節。

以前追求「精確預測」，常常陷在模型的世界裡。
覺得模型準一點、數據多一點、演算法新一點，決策品質就會提升。

但五環模型揭示的是更根本的東西：重點在 Harness——讓模型真正落地運作的那套執行環境。

在畫布上，數據是流動資產（Ring 1），邏輯是標準化語義（Ring 2），Agent 根據即時上下文（Ring 3）持續進行決策（Ring 4）並執行（Ring 5）。

Harness 包含的是迴圈、工具、上下文管理、持久化、驗證、限制條件。
兩個團隊用同一個模型，做出來的 Agent 體驗可以像完全不同的物種。差別就在 Harness 的設計。

再往深處想，Harness 其實承載的是品味。
你給 Agent 什麼工具、強制什麼順序、用什麼標準衡量成功、怎麼從失敗恢復——這些選擇反映的是你的價值觀。

在所有人都能複製貼上 Agent 的時代，價值觀才是真正的差異化。


▋ 這對做數據的人意味著什麼
