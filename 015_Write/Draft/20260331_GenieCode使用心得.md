## 原始素材

最近持續在嘗試把 Databricks 上的分析能力，往更接近真正 Agent 的方向推進。

目前手上反覆在試的幾件事，大概是這幾條線一起走。

一條是把 Vector Search、table 數據、Genie Code 串在一起。也就是讓它不只會看非結構化的業務資料，還能再把結構化 table 數據一起撈進來，最後交給 Genie Code 做整合判讀。這件事對我來說很重要，因為真正的業務分析，很少只靠單一來源就能講出完整答案。很多時候，文字脈絡、規則文件、異常描述，和實際數據表裡的狀況，本來就該一起看。

另一條是讓 PM Agent 先把數據狀況拋出來，再由 Genie Code 去規劃後續檢查與分析。這背後其實是在試一種分工：不是把所有事情都塞給同一個 Agent，而是讓不同角色先做自己最適合的部分，再把問題、脈絡、懷疑點交給下一個核心去接。這樣的感覺，已經開始有一點像真正的協作流程，而不只是單次問答。

還有一條是 pipeline 修改完後，不只是看 code 過了沒，而是讓 Genie Code 用測試數據把完整流程驗過一次。這對我來說也很有感，因為很多資料流程的問題，不是在你改程式當下就看得出來，而是在資料真的跑完、轉完、落表後，才知道中間哪裡歪掉。若 Agent 能更自然地參與這段驗測，價值就不是只停留在「幫你寫程式」，而是開始碰到「幫你確認這個流程到底有沒有真的成立」。

如果說 Claude Code 這類 coding agent，是我第一次被 AI 開發方式震撼到，那 Genie Code 大概就是第二次。

第一次震撼，主要是來自它真的能進入 repo、理解檔案、動手修改、像一個很強的協作者。第二次震撼，則是因為 Genie Code 這種東西，已經不是單純幫你寫 code，而是它的核心直接坐落在數據旁邊。那種感覺很不一樣。它不是站在外面遙遙看你的資料，而是本來就在你的資料環境裡，還有機會搭配各種操作能力一起工作。這件事一旦成立，很多以前覺得很重、很麻煩、很難產品化的分析流程，突然都變得合理起來。

回頭看 ChatGPT 3.5 到 4o 那一段時間，我們其實就已經因為長期做專案型數據分析，開始構想一套自動分析流程。只是那個年代的模型，坦白說還是有不少現實問題：延遲高、成本高、效果普通，而且為了把事情撐起來，常常要堆很多 prompt。那些 prompt 當下好像能用，但後面維護起來很痛苦，規則越疊越多，改起來也不穩。你會感覺自己花了很多力氣，不是在建一個真的能持續擴張的分析系統，而是在維持一堆脆弱的技巧。

現在再看，整個條件真的變了。

當 Genie Code 這種核心能力開始落在資料環境本身，模型能力也比以前成熟很多，原本那些年想做、但做得很吃力的事情，突然變得輕很多。不是說問題都消失了，而是你終於開始看到一條比較像樣的路：核心推理負責 Observe、Plan、Act，外圍再配上記憶、檢索、工具、驗證、追蹤。整個架構不再只是理想圖，而是越來越接近可以落地的工程系統。

我現在的規劃也越來越清楚，不是把 Genie Code 當成單純的 coding tool，而是把它放在分析 Agent 的核心位置。原本曾經考慮過以第三方框架為中心，現在看起來，若 Databricks 原生能力能繼續長下去，很多決策推演其實可以直接交給 Genie Code。外圍再由 Vector Search 承接非結構脈絡，table 數據提供事實基礎，UC Functions、MLflow、外部工具承接可操作能力，測試資料與 tracing 機制負責驗證。這樣的拆法，比我以前想的更合理，因為核心和外部賦能可以分得更清楚，後續也更容易累積。

當然，現在也不是沒有阻礙。最明顯的限制，就是 Genie Code 目前還不能很自然地被動被呼叫，尤其若沒辦法順暢進入 job 或被當成穩定服務能力使用，那很多本來可以自動化的分析流程，最後還是只能停在半人工階段。這不是小問題，因為一旦你真的把它當成分析引擎來看，就會很希望它不只是互動式工具，而是能被排程、被事件觸發、被其他系統正式串接。

所以我現在最真心的呼籲，大概就是這句：Databricks 的顧問和產品團隊，真的拜託快點讓 Genie Code 可以被動被呼叫，甚至變成 API endpoint。因為這東西如果真的開放到那一步，我相信很多人不是「會想試試看」，而是真的會大量拿去用。

對我來說，這段時間最強烈的感受不是「又多了一個 AI 工具」，而是當年的一些想法，終於碰到比較對的時代了。以前想做的自動分析流程，不是方向錯了，只是那時候的模型能力、產品形態、整合深度，還撐不起來。現在看起來，很多事情開始有機會從概念變成真正可持續推進的系統。這種感覺，確實很讓人興奮。

## 發想筆記

### 價值定位

- **主打價值**：啟發我（Inspire）— 用「兩次震撼」重新定義 AI 在數據分析的位置，把「又一個 AI 工具」翻轉成「架構性的可能」
- **輔助調味**：共感我（Empathize）— 以前堆 prompt 的痛苦、想法跑在時代前面的焦慮與興奮
- **定位理由**：目標讀者（技術主管）最需要的不是操作教學，而是「這東西值不值得投入、方向對不對」的判斷依據

### 乾貨盤點

**知識素材**：
- 三層進化框架（L1 建議 → L2 執行 → L3 代理），可用來定位 Genie Code 正推動的躍遷
- Harness 概念（把模型變成 Agent 的不是模型本身，而是驅動框架）
- Agent 95% Production 失敗原因（context 工程 > 模型選擇）
- Trust Gap 數據（5 小時能力 vs 42 分鐘信任）

**經驗素材（護城河）**：
- 從 ChatGPT 3.5 到 4o 時代就開始構想自動分析流程的親身經歷
- 實際嘗試 Vector Search + table 數據 + Genie Code 串接
- PM Agent → Genie Code 分工協作的實驗
- Pipeline 修改後用 Genie Code 做端到端驗測
- 堆 prompt 時期的維護痛苦（規則越疊越多、改起來不穩）

**框架素材**：
- 「核心推理（Observe → Plan → Act）+ 外圍賦能（記憶、檢索、工具、驗證、追蹤）」的架構概念
- 「兩次震撼」的對比敘事結構（Claude Code = repo 旁 vs Genie Code = 數據旁）

**素材缺口**：
- 缺少具體量化數據（效率提升多少、節省多少時間）
- 缺少其他平台 AI 分析能力的橫向比較
- 建議：不補也行，因為這篇定位是「洞察 + 感想」而非「評測」

### 觀點深化（核心洞察提煉）

這篇文章的非顯然判斷是：**「AI 工具住在哪裡」比「AI 工具多聰明」更重要。**

延伸推論：
1. Claude Code 證明了「住在 repo 旁邊」的價值 → Genie Code 證明了「住在數據旁邊」的價值
2. 以前的自動分析流程之所以失敗，不是模型不夠好，而是模型離數據太遠
3. Genie Code 不是更好的 copilot，它是第一個「原生坐在數據上」的分析核心
4. 這也暗示了一個產品方向：未來的 AI 工具競爭，不只是比模型能力，更是比「誰離場景最近」

**鉤子 ①**（配標題①，反直覺觀點型）

如果 Claude Code 這類 coding agent 是我第一次被 AI 開發方式震撼到，那 Genie Code 大概就是第二次。

這件事一旦成立，很多以前覺得很重、很難產品化的分析流程，突然都變得合理起來。

這篇文章，我想拆開這個「住在數據旁邊」為什麼比「模型更聰明」更關鍵。

## Draft v1

### 這是我第二次被 AI 開發方式震撼到——從 Claude Code 到 Genie Code

最近兩周，我持續在嘗試把 Databricks 上的分析能力往真正 Agent 的方向推進——串 Vector Search、table 數據、Genie Code，讓不同的 AI 角色分工協作，甚至讓 Agent 用測試數據把完整流程驗過一次。

做著做著，突然意識到一件事：這是我第二次被 AI 的開發方式震撼到。
第一次是 Claude Code，第二次是 Genie Code。
但這兩次震撼的本質非常不同。

這篇文章，我想談談為什麼 Genie Code 改變了我對「數據分析該怎麼走」的判斷。

---

▋ 回顧：堆 prompt 的年代，不是在建系統，是在維持技巧

回頭看 ChatGPT 3.5 到 4o 那段時間，我們其實就已經因為長期做專案型數據分析，開始構想一套自動分析流程。

只是那個年代的模型，坦白說還是有不少現實問題。
延遲高、成本高、效果普通。
而且為了把事情撐起來，常常要堆很多 prompt。

那些 prompt 當下好像能用，但後面維護起來很痛苦。
規則越疊越多，改起來不穩。
你會感覺自己花了很多力氣，不是在建一個真的能持續擴張的分析系統，而是在維持一堆脆弱的技巧。

現在回想，那段時間的核心問題不是模型不夠聰明。
而是模型離數據太遠。
你得花大量工程去搬資料、組 context、設計 prompt、處理例外——真正有價值的推理，反而只佔整個流程的一小部分。

---

▋ 第一次震撼：Claude Code 讓 AI 住進了你的 repo

如果要選一個讓我對 AI 開發方式徹底改觀的起點，大概就是 Claude Code。

它不是在雲端遙遙看你的程式碼。
它真的能進入 repo、理解檔案結構、動手修改、跑測試。
像一個很強的協作者，直接坐在你旁邊一起寫。

Karpathy 說得好：「AI 應該住在你的電腦上，而不是雲端。」
關鍵不在於運算發生在哪裡，而在於你的環境裡已經存在的所有東西——檔案、配置、上下文、密鑰——都可以被直接利用。

這件事對我的衝擊很大。
不是「又多了一個 AI 工具」，而是開發方式本身被改變了。

---

▋ 第二次震撼：Genie Code 讓 AI 住進了你的數據

如果 Claude Code 是住在你的 repo 旁邊，那 Genie Code 就是住在你的數據旁邊。

這兩件事本質一樣，但場景完全不同。

Genie Code 不是站在外面遙遙看你的資料，然後靠你把資料搬過去。
它本來就在你的資料環境裡。
它天然能碰到你的 table、你的 schema、你的 catalog。
它還有機會搭配 Vector Search、UC Functions、MLflow 這些操作能力一起工作。

這件事一旦成立，很多以前覺得很重、很麻煩、很難產品化的分析流程，突然都變得合理起來。

Genie Code 出來沒多久，但我在這短短兩周的實驗裡，已經在三條線上同時推進。

---

▋ 三條實驗線：從串接、分工到端到端驗測

**第一條線：把非結構化和結構化資料串在一起。**

讓 Vector Search 負責非結構化的業務資料（規則文件、異常描述、文字脈絡），table 數據提供結構化事實基礎，最後交給 Genie Code 做整合判讀。

真正的業務分析，很少只靠單一來源就能講出完整答案。
文字脈絡和數據表裡的狀況，本來就該一起看。

**第二條線：讓不同 Agent 角色分工協作。**

PM Agent 先把數據狀況拋出來，再由 Genie Code 去規劃後續檢查與分析。
不是把所有事情都塞給同一個 Agent，而是讓不同角色先做自己最適合的部分，再把問題、脈絡、懷疑點交給下一個核心去接。

這樣的感覺，已經開始有一點像真正的協作流程，而不只是單次問答。

**第三條線：pipeline 修改後的端到端驗測。**

不只是看 code 過了沒，而是讓 Genie Code 用測試數據把完整流程驗過一次。
很多資料流程的問題，不是在你改程式當下就看得出來，而是在資料真的跑完、轉完、落表後，才知道中間哪裡歪掉。

若 Agent 能更自然地參與這段驗測，價值就不只停留在「幫你寫程式」，而是開始碰到「幫你確認這個流程到底有沒有真的成立」。

---

▋ 為什麼「住在哪裡」比「多聰明」更重要

有一個數據很有意思。
大多數 AI Agent 進到正式環境後，有 95% 會失敗。
問題不在模型不夠聰明，而在於 context 管理、memory 設計、錯誤處理這些周邊工程。

換句話說，大多數人以為自己在做 AI 產品，其實是在做 context selection 系統。

這就是 Genie Code 的核心優勢：它天然坐在你的數據環境裡，context 不需要額外搬運。
這不是小事。
這是整個架構複雜度降低一個量級的起點。

如果用 AI 應用的三層進化來看：L1 建議層（生成洞見）、L2 執行層（完成任務）、L3 代理層（持續負責結果）——大多數分析工具還停在 L1，因為只輸出建議不承擔後果。
Genie Code 正在推動的，是從 L1 到 L2 的躍遷。
它不只告訴你答案，它能動手去查、去比對、去驗證。

---

▋ 架構判斷：把 Genie Code 放在分析 Agent 的核心位置

我現在的規劃越來越清楚了。

不是把 Genie Code 當成單純的 coding tool，而是把它放在分析 Agent 的核心位置。
核心推理負責 Observe、Plan、Act。
外圍再配上：
- Vector Search 承接非結構脈絡
- table 數據提供事實基礎
- UC Functions、MLflow、外部工具承接可操作能力
- 測試資料與 tracing 機制負責驗證

原本曾經考慮過以第三方框架為中心。
但我問了自己一個問題：如果明年模型能力翻倍，我的系統會不會在不做大改的情況下變得更便宜、更簡單、更可靠？

如果答案是否定的，你就把太多「智慧」塞進了腳手架，而不是留給模型。

Databricks 原生能力的好處是，它會隨著平台一起長。
模型進步的時候，你的系統不需要重寫，而是自動變好。
這比固化在某個第三方框架上，結構性地更合理。

---

▋ 誠實說：現在最大的阻礙

當然不是沒有阻礙。

最明顯的限制，就是 Genie Code 目前還不能很自然地被動被呼叫。
如果沒辦法順暢進入 job，或被當成穩定服務能力使用，那很多本來可以自動化的分析流程，最後還是只能停在半人工階段。

這不是小問題。
一旦你真的把它當成分析引擎來看，就會很希望它不只是互動式工具，而是能被排程、被事件觸發、被其他系統正式串接。

AI 能做 5 小時的工作，但如果人類不敢放手讓它跑，信任的落差就變成最大的瓶頸。
Genie Code 現在有能力，但缺少被正式串接的介面——這個 Trust Gap 一天不解決，真正的自動化就一天走不完最後一步。

所以我現在最真心的呼籲就是這句：
Databricks 的產品團隊，真的拜託快點讓 Genie Code 可以被動被呼叫，甚至變成 API endpoint。
因為這東西如果真的開放到那一步，很多人不是「會想試試看」，而是真的會大量拿去用。

---

▋ 這不是又多了一個工具，而是當年的想法碰到對的時代

對我來說，這兩周最強烈的感受不是「又多了一個 AI 工具」。

而是當年的一些想法，終於碰到比較對的時代了。

以前想做的自動分析流程，不是方向錯了。
只是那時候的模型能力、產品形態、整合深度，還撐不起來。

現在再看，整個條件真的變了。
很多事情開始有機會從概念變成真正可持續推進的系統。

如果要用一句話收束：
**AI 工具「住在哪裡」比「多聰明」更重要。Claude Code 證明了住在 repo 旁邊的價值，Genie Code 正在證明住在數據旁邊的價值。**

這種感覺，確實很讓人興奮。

### 收尾版本選項

**收尾 A（Trust Gap 呼籲型）**：
誰能設計出讓人安心放手的信任機制，誰就能釋放 AI 的真正產能。Genie Code 有能力，現在只差一個被正式串接的入口。

**收尾 B（時代碰撞型，目前 Draft v1 使用）**：
AI 工具「住在哪裡」比「多聰明」更重要。Claude Code 證明了住在 repo 旁邊的價值，Genie Code 正在證明住在數據旁邊的價值。

**收尾 C（三層進化型）**：
我們正在目睹分析工具從「只會說」跨進「真的能做」的轉折。而這個轉折的起點，不是更聰明的模型，是一個終於住對地方的 AI。
