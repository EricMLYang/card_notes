# 一份 API 遷移文件教我的事：評估不該是外掛

說實話，「評估」這件事在多數 AI 專案裡的待遇都差不多。
大家都知道重要，但永遠排在「先讓它跑起來」後面。

我自己也是。

最近在 Databricks 上嘗試建一個 coding agent。
一開始的注意力全放在 prompt 怎麼寫、工具怎麼接、context 怎麼管理。
直到 agent 真的能跑了，我才開始認真想一個問題：
我該怎麼評估它到底做得好不好？

不是「跑起來沒有 error」就好，而是它產出的東西，有沒有真的到位？

為了回答這個問題，我去查了 Databricks 上的 evaluation 工具，然後讀到了 MLflow 3 的遷移文件。
一開始以為只是把舊 API 換成新 API 的技術指南。
但讀完之後，我覺得它透露的東西比表面多很多。

以下是我讀完之後整理的幾個筆記。

---

▋ 表面上是搬家，實際上是收編

MLflow 3 做的第一件事，是把原本獨立的 Agent Evaluation 整進 `mlflow.genai` 命名空間。

聽起來只是換個 import path。
但你仔細看它一起搬進來的東西：evaluation API、tracing backend、human feedback 收集、內建的 LLM judges/scorers、還有統一的 UI。

我覺得這比較像是收編。

Databricks 把 trace、eval、labeling、scorer 全部拉進同一個地方。
跑完 agent、看 trace、評分、收人工回饋，不用再東拼西湊。

對我這種還在學的人來說，光是搞清楚「我該用哪個工具來評估」就夠頭痛了。
現在少一層拼裝的功夫，學習門檻直接降了一截。

---

▋ 最讓我停下來想的改動

這份文件裡有很多 API 更名。
`mlflow.evaluate()` 改成 `mlflow.genai.evaluate()`。
`@metric` 改成 `@scorer`。
欄位名稱從 `request/response` 統一成 `inputs/outputs/expectations`。

這些我都能理解，就是命名規範統一。

但有一個改動讓我想了比較久：
**MLflow 3 不再自動幫你跑 judge。**

舊版的邏輯是，只要你的資料欄位符合格式，它就自動跑一批預設的 judge。
新版要求你必須自己指定要跑哪些 scorer——`Correctness()`、`Safety()`、`Guidelines()`、`RetrievalGroundedness()`，一個一個明確列出來。

第一反應是：這不是變麻煩了嗎？以前自動跑不是很方便？

但想一想，這背後的邏輯其實很合理。

自動跑 judge 的問題是：你不知道它幫你評了什麼、用什麼標準評的。
對一個還在探索的 side project 來說也許無所謂。
但如果你認真想讓 agent 上線，你得自己定義「什麼叫做對」。

說白了，就是從「平台幫你評」變成「你自己要想清楚評什麼」。

乍聽之下是多了工作。
但我後來想想，如果連自己都說不清楚什麼叫好，那自動跑再多 judge 也只是製造一種安心的錯覺。

---

▋ 評估結果存成 trace，這件事很值得注意

還有一個我覺得想得蠻周到的改動：
evaluation 的結果現在直接存成 traces with assessments。

以前跑完評估要去翻一個獨立的 result table。
現在 evaluation 結果跟 production trace 用的是同一套資料模型。

這樣一來，debug、monitoring、offline eval、human review，都走同一套 trace infrastructure。

老實說，對我來說最實際的好處就是不用先搞懂四套工具再把它們接起來。
一開始就在同一個地方看，學的時候少繞很多路。

---

▋ 讀完這份文件之後

回頭看，我讀這份文件的初衷只是想幫 coding agent 加個評估功能。
但讀完之後覺得，Databricks 想的事情比我原本預期的大很多。

它把 tracing、grading、human review、production monitoring 全部收進同一套基礎設施。
不只是做工具，而是在建一整個 agent lifecycle 的底盤。

我之前在筆記裡寫過一句話：**評估能力是區分「玩票」跟「認真」的分水嶺。**

讀完這份文件，我對這句話的感受更深了。
沒有評估，你根本不知道 agent 是在進步還是退步，優化就變成憑感覺在調。

Databricks 這次做的事，就是把「評估」從一個你得自己想辦法外掛的東西，變成平台直接內建的能力。

這篇文件改變了什麼嗎？
對我來說，大概就是讓我少走了一段「自己拼湊評估工具」的彎路。

有時候讀文件最大的收穫，不是學到新功能，而是發現別人已經把你模糊覺得重要的事情，做成了系統。
