## 原始素材

### 來源文章摘要
素材來自 `003_Capture/20260408_AnalysisAgent價值案例.md.md`，核心分析：

**核心論點**：Agent 架構的核心商業價值不在取代演算法，而在降低「Time-to-Insight」和「分析到行動」的摩擦成本。

**五大案例**：
- **Dentsu**：predictive analytics copilot → analysis time -80%, time-to-insight -90%（Access/Delivery 型）
- **causaLens**：Causal AI + digital workers → decision intelligence（Analytical Engine 型）
- **Salesforce**：Agentforce → lead revenue +60%, resolution rate 85%（Action Layer 型）
- **S&P Global**：M365 Copilot → data extraction 95% faster（Access/Delivery 型）
- **Shell**：AI 管線模擬 → 106x simulation speed（工業 Action Layer 型）

**雙重分類框架**：
1. 商業模式分類：A 內部自助分析 / B 外部資料產品 / C 工業流程自動化 / D 驅動業務行動
2. 系統能力分類：Access/Delivery Layer → Analytical Engine Layer → Action Layer

**個人判斷**：Dentsu 型先做（最快產生可見價值）→ causaLens 型是分析理想 → Salesforce 型是系統終局。

### 個人經驗
- Databricks Genie / Code Agent 分析體驗
- 產學合作計量分析：分析很技術性，但找到真正落地決策的應用很難
- 帶團隊導入 AI 工具的第一線經驗

### 個人色彩
- 寫作定位：從資料平台到決策系統
- 核心問題：哪些分析值得做成產品能力（productize）、哪些工作可以交給 Agent
- 價值觀：技術價值要回到真實場景檢驗；AI 的真正威力是進入工作閉環

> 假想讀者：帶 5-10 人小團隊、正在導入 AI 工具的技術主管。他在意的是「我的團隊該先做什麼？」「怎麼避免踩坑？」「怎麼跟老闆說明價值？」

### (a) 開場素材

**推薦方案 1**：「接水管」比喻 × 個人經驗
- 來源卡片：[[新創的核心價值是接水管—連接企業數據與雲端AI]]
- 使用方式：用產學合作計量分析的經歷開場——「分析能力從來不缺，缺的是把分析結果接到決策行動的那根水管」。再延伸到 Dentsu 案例：他們做的本質就是「接水管」。

**推薦方案 2**：Databricks Genie 體驗 × 回憶對比
- 使用方式：「打開 Databricks Genie 問了第一個問題的時候，我突然想起當年跟教授做計量分析的日子。」從個人體驗切入，引出核心問題：分析結果很漂亮，但從分析到決策之間的鴻溝，才是真正的商業問題。

### (b) 核心框架

**文章骨幹——三層能力 × 案例對照**：
1. **Access/Delivery Layer**（讓分析更快被拿到）← 案例：Dentsu + S&P Global + [[從 GenAI 功能到可收費 Agent 的三層演進]]
2. **Analytical Engine Layer**（讓分析更深更準）← 案例：causaLens + [[DecisionOps]] + [[因果推論商業應用：從相關性到指導性分析]]
3. **Action Layer**（讓分析結果驅動行動）← 案例：Salesforce + Shell + [[三層進化：從 GenAI 圖表建議到 Agent]]

**文章主軸**：這三層不是升級路線，而是三種不同的能力焦點。技術主管的第一步應該是 Dentsu 型（最快可見價值），但心裡要知道終局是 Salesforce 型（分析接行動）。

### (d) 反面論述

- 讀者可能質疑：「既然 Dentsu 型最快，直接全力做就好了？」→ 回應：Access/Delivery 再快，如果底層資料治理沒做好，Agent 只會「加速產出有說服力的錯誤結論」。SSOT 是 Agent 上線的絕對前置條件。來自 [[Context Graph 捕獲組織隱性知識]]。

### 鉤子 × 5（每個標題搭配一個鉤子）

**鉤子 1**（搭配標題 1 — 真實經驗切入型）

> 幾年前跟教授做產學合作的計量分析，模型跑了好幾輪，變數調了又調，回歸結果終於漂亮了。拿去給業務主管看，對方盯著螢幕看了三秒，問了一句：「所以我下週一的會議該講什麼？」
>
> 那一刻我才意識到，分析能力從來不是瓶頸。真正的瓶頸是：分析結果跟業務行動之間，中間隔著一條看不見的鴻溝。
>
> 最近用 Databricks Genie 做了一輪數據分析，那個體驗讓我重新想起這件事。AI 讓分析變快了、門檻變低了、產出變好看了。但「所以我該怎麼做」這個問題，並沒有因此消失。
>
> 我花了一段時間拆解 Dentsu、causaLens、Salesforce、S&P Global、Shell 這五家在 Agent × 數據分析上走得最前面的企業，發現一件事：Agent 的核心商業價值，不在取代演算法，而在解決三種完全不同層次的問題。
>
> 這篇文章會把這三層拆開來看，並且告訴你——如果你帶一個 5-10 人的團隊，第一步該從哪裡開始。

**鉤子 2**（搭配標題 2 — 反直覺觀點型）

> 多數人聽到「Agent 做數據分析」，直覺想到的是：更聰明的模型、更強的預測、更準的建議。
>

## Draft v1

# 分析能力從來不缺，缺的是接到行動的那根水管

幾年前跟教授做產學合作的計量分析，模型跑了好幾輪，變數調了又調，回歸結果終於漂亮了。

拿去給業務主管看，對方盯著螢幕看了三秒，問了一句：
「所以我下週一的會議該講什麼？」

那一刻我才意識到，分析能力從來不是瓶頸。
真正的瓶頸是：分析結果跟業務行動之間，中間隔著一條看不見的鴻溝。

最近用 Databricks Genie 做了一輪數據分析，那個體驗讓我重新想起這件事。
AI 讓分析變快了、門檻變低了、產出變好看了。
但「所以我該怎麼做」這個問題，並沒有因此消失。

我花了一段時間拆解 Dentsu、causaLens、Salesforce、S&P Global、Shell 這五家在 Agent × 數據分析上走得最前面的企業，發現一件事：

Agent 的核心商業價值，不在取代演算法，而在解決三種完全不同層次的問題。

這篇會把這三層拆開來看，並且從帶小團隊的視角，判斷第一步該從哪裡開始。

---

▋ Agent 的價值，不在讓分析更聰明

先說結論。

目前看到的趨勢很明確：Agent 在數據分析上的核心價值，通常不在取代 forecasting、optimization、causal inference 這些進階分析能力本身。

它真正解決的，是把這些能力從「產出分析結果」推進到「更快被取得、被理解、被接上流程與行動」。

所以企業公開強調的成果，常常不是模型準確率。
而是分析時間、time-to-insight、比較分析速度、resolution rate、ROAS、工時節省。
這些指標的共同點是——它們都在衡量交付與執行的效率，而不是分析本身的品質。

我把這件事拆成三層來看。

---

▋ 第一層：Access / Delivery — 讓分析更快被拿到

這是目前最多企業在做、也最容易產生可見價值的那一層。

Dentsu 是全球行銷與媒體服務集團，他們用 Azure AI Foundry 做了一個 predictive analytics copilot。
做的事情其實不複雜：讓 media insights 更容易被員工和客戶取得。

但成果很驚人——analysis time 減少 80%，time-to-insight 減少 90%。

S&P Global 做的也是類似的事。
他們把旗下的能源與商品資料接上 Microsoft 365 Copilot，讓客戶可以用自然語言查詢高價值的市場數據。
結果是 data extraction 快了 95%，comparative analysis 快了 98%。

這兩家的共通點是什麼？

底層的分析能力本來就在。
資料本來就有、模型本來就跑、報表本來就出。
Agent 改變的不是分析引擎，而是分析結果的「交付介面」。

用一個比喻來說：這一層做的事情，就是「接水管」。
水源（資料）和淨水廠（分析引擎）都在了，但水龍頭（取用介面）離用戶太遠。
Agent 做的是把水龍頭搬到用戶的桌上。

如果你帶一個 5-10 人的小團隊，正在考慮怎麼導入 Agent 做數據分析，這一層幾乎一定是最適合先切入的。

原因不是它最強，而是它最快產生可見價值。
它直接解決一個非常常見的痛點：分析結果明明存在，但取得太慢、太靠專家、太難被業務端消費。

---

▋ 第二層：Analytical Engine — 讓分析更深更準

第一層解決的是「拿得到」。
第二層要解決的是「分析本身能不能更深」。

causaLens 是做 Causal AI 和 decision intelligence 的公司。
它的主軸已經擴展到以 causal reasoning 為基礎的 digital workers，強調在金融服務等高決策價值場景的可用性。

這家公司代表的不是單純的問答型 Agent，而是更接近「進階分析引擎 + Agent 調用」的方向。

傳統 BI 回答的是「發生了什麼」（描述性分析）。
進階一點可以回答「會發生什麼」（預測性分析）。
但 causaLens 想走到的是「為什麼發生」和「改什麼最有用」（因果推論 + 指導性分析）。

說實話，這是我最想達到的分析理想。

但它不是從 Dentsu 型自然升級而來的下一站。
它是另一條更高要求的分析能力線。
它更依賴結構化資料、因果設計、評估機制與高品質的業務建模。

MLOps 管的是「預測模型的部署」——會發生什麼。
DecisionOps 管的是「決策模型的部署」——該做什麼。
這兩者的工程差異很大，不是換個 prompt 就能解決。

對大多數帶小團隊的技術主管來說，這一層是值得知道、但不急著做的。
除非你的業務場景天生需要因果推論（例如定價策略、行銷投放歸因、醫療決策），否則投資報酬率不見得最高。

---

▋ 第三層：Action Layer — 讓分析驅動行動

第一層讓你拿得到分析結果。
第二層讓分析更深。
第三層要做的是：讓分析結果真的去推動一件事情發生。

Salesforce 是最典型的代表。
他們用 Agentforce + Data 360，把數據、分群、預測、規則，直接接進 sales、marketing、service 流程。

成果不是「分析更快」這種效率指標了。
而是直接的商業產出：
lead revenue 和 contract value per lead 增加 60%。
resolution rate 達到 85%。
projected 減少 98,000 個 support cases。
透過 AI-powered Slack notifications 發現 2,300 萬美元潛在續約收入機會。

另一邊，Shell 展示的是工業端的 Action Layer。
AI 用在管線腐蝕模擬，比傳統 CFD 方法快 106 倍。
CCS storage site assessment 流程加速超過 60%。

這告訴你一件事：Action Layer 不只是前台 CRM 的專利。
後台的工程評估、監控、模擬、排程，也都是 Agent 可以驅動行動的場域。

大多數 AI 應用其實停在 L1——建議層。
能說、會分析、給建議，但不承擔後果。
要進到 L2（執行層）和 L3（代理層），需要補齊的不是模型能力，而是工具調用、狀態記憶、驗證與回饋迴圈。

如果系統最後要有真正的商業影響，終局通常不是停在更好的分析介面。
而是走到能夠推動 sales、marketing、service 或 operational workflow 的 Action Layer。

---

▋ 但有一個隱藏的死穴

講完三層能力，有一件事必須說清楚。

Agent 毫無資料清洗能力。

這聽起來像廢話，但很多人在規劃 Agent 導入的時候，潛意識裡會有一個錯誤前提：覺得 AI 夠聰明，就能自己處理髒數據、自動統一異質系統、解決資料孤島。

事實恰好相反。

若缺乏嚴謹的 Data Catalog、中繼資料（Metadata）與統一業務邏輯定義，Agent 強大的自然語言生成能力只會做一件事——加速產出具高度說服力的錯誤結論。

這就是我說的「數據幻覺」。
它比模型幻覺更危險，因為它看起來格式正確、邏輯通順、用了真實的表名和欄位名，但算出來的數字根本不對。

95% 的 Agent 在 Production 失敗，問題不在模型不夠聰明。
在於 context 管理、memory 設計、錯誤處理、評估回饋機制。
「大多數創辦人以為自己在做 AI 產品，其實是在做 context selection 系統。」

Agent 跨越了人機互動的最後一哩路。
但底層資料工程的第一哩路沒做好，那條水管接上去只會流出汙水。

嚴格的資料治理，是 Agent 上線的絕對前置條件。
不是 nice-to-have，是 must-have。

---

▋ 我的判斷：帶小團隊該從哪裡開始

回到帶 5-10 人小團隊的視角。
三層能力不是一條單一路徑的先後升級，而是三種不同的投資方向。

**近期最適合落地的，是 Dentsu 型。**
把現有分析能力的交付介面做好。
用 Semantic Layer、NL-to-query、RAG 讓業務端可以直接取用分析結果。
這是最快能讓老闆看到成果、最容易量化的切入點。

**但同步要投資的，是資料治理。**
不需要一步到位，但至少要有 Data Catalog、統一的業務邏輯定義、基本的資料品質監控。
否則 Agent 越強，錯誤越有說服力。

**心裡記住終局是 Salesforce 型。**
分析結果最後要能接到行動——推動 outreach、resolution、campaign、opportunity follow-up、或工程評估。
這才是真正有商業影響的地方。

至於 causaLens 型？
它是分析理想。
如果你的場景天生需要因果推論，值得投資。
但對大多數小團隊來說，這不是第一步。

---

▋ 分析成本降低之後，價值去了哪裡

最後想多說一件事。

Agent 正在讓「分析交付」的成本大幅降低。
過去需要資料分析師花一週整理的報告，現在可能 10 分鐘就有了。

但價值不是消失了。
它往上游和下游轉移了。

往上游——資料治理、語意統一、Metadata 管理變得更值錢。
因為 Agent 放大了好資料的價值，也放大了壞資料的傷害。

往下游——行動執行、流程接口、系統整合變得更值錢。
因為分析做完只是開始，真正的商業影響在行動。

所以，如果你問我 Agent 時代什麼能力最值得投資？

不是讓分析更聰明。
而是讓資料更乾淨，讓行動更順暢。

真正值得追的，不是讓 AI 分析更強。
而是讓人類更敢放手——誰能設計出讓人安心的信任機制，誰就能釋放 AI 真正的產能。

AI 用在管線腐蝕模擬，比傳統 CFD 方法快 106 倍。
CCS storage site assessment 流程加速超過 60%。

這告訴你一件事：第三層的應用場域很寬。
前台的 CRM 和後台的工程評估、監控、模擬，都是 Agent 可以接上行動的地方。

我目前的判斷是，如果系統最後要有真正的商業影響，終局通常會走到這裡。
光是有更好的分析介面還不夠，分析結果要能推動 outreach、resolution、campaign，或者工程評估流程。

但老實說，大多數 AI 應用目前還停在建議層——能說、會分析、給建議，但沒有真的承擔後果。
要從「建議」走到「執行」甚至「代理」，需要補齊的是工具調用、狀態記憶、驗證機制和回饋迴圈。

⠀

▋ 隱藏的死穴：資料治理

講完三層，有一件事必須直說。

Agent 毫無資料清洗能力。

這聽起來像廢話，但很多團隊在規劃 Agent 導入的時候，潛意識裡會有一個假設：AI 夠聰明，應該能自動處理髒數據、統一異質系統。

實際做下去，完全不是那麼回事。

若缺乏 Data Catalog、中繼資料管理和統一的業務邏輯定義，Agent 強大的自然語言生成能力只會做一件事——加速產出具高度說服力的錯誤結論。

這就是「數據幻覺」。
比模型幻覺更麻煩，因為它格式正確、邏輯通順、用了真實的表名和欄位名，但算出來的數字根本不對。

有一個觀察讓我印象很深：多數 Agent 在正式環境失敗，問題出在 context 管理、memory 設計、錯誤處理、評估回饋機制。
「大多數創辦人以為自己在做 AI 產品，其實是在做 context selection 系統。」

Agent 跨越了人機互動的最後一哩路。
但底層資料工程的第一哩路沒搞好，那條水管接上去只會流出汙水。

嚴格的資料治理，是 Agent 上線的前置條件。

⠀

▋ 我目前的判斷

回到帶小團隊的視角。
三層能力不是升級路線，是三種不同的投資方向。

**先做第一層。**
把現有分析能力的交付介面做好。
用 Semantic Layer、NL-to-query、RAG 讓業務端可以直接取用分析結果。
這最快能讓老闆看到成果。

**同步投資資料治理。**
不需要一步到位，但至少要有 Data Catalog、統一的業務邏輯定義、基本的資料品質監控。
否則 Agent 越強，錯誤越有說服力。

**心裡記住終局是第三層。**
分析結果最後要能接到行動。
推動 outreach、resolution、campaign、opportunity follow-up，或工程評估流程。
這才是真正產生商業影響的地方。

至於第二層？
如果你的場景天生需要因果推論，值得投資。
但對大多數小團隊來說，這不會是第一步。

⠀

▋ 分析成本降低之後

最後多說一件事。

Agent 正在讓「分析交付」的成本大幅降低。
過去需要分析師花一週整理的報告，現在可能 10 分鐘就有了。

但價值沒有消失，它往兩個方向轉移了。

往上游——資料治理、語意統一、Metadata 管理變得更值錢。
因為 Agent 放大了好資料的價值，也放大了壞資料的傷害。

往下游——行動執行、流程接口、系統整合變得更值錢。
因為分析做完只是開始，商業影響在行動端。

如果你問我 Agent 時代什麼能力最值得投資？

我目前的看法是：讓資料更乾淨，讓行動更順暢。

分析這件事本身，正在快速變便宜。
真正稀缺的，是分析結果兩端的基礎設施——上游的資料品質，和下游接到行動的那根水管。
