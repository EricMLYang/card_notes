# AI-first 工程心得：產線主軸與 Harness

## 作者的三個起心動念（來自原始 capture）

1. **產品主軸 / 價值主軸很重要，產線才滾動得起來**——Peter Pang 的 CREAO 能把「一天 8 次部署」跑通，我直覺認為不只是因為 harness 建得好，而是因為他們本來就有一條清晰的產品主軸去餵這條產線；沒有主軸，harness 只是一條空轉的輸送帶。
2. **這是有人花資源做的極端實驗**——對一般公司不見得能 1:1 複製，但概念可以拆出來學。
3. **CREAO 的產品本身就是 Agent，產線和產品相輔相成**——就像大模型公司都在期待「模型能自己改進模型」，我想在自己的領域找到類似的自我強化迴圈，才是真正該學的東西。

### 核心卡片（直接相關）

1. **[[Harness 是把模型變成 Agent 的關鍵（不是模型本身）]]** — 角色：框架
   - 關鍵摘錄：「把『會聊天的 LLM』和『能交付成果的 Agent』區分開來的，不是模型的權重，是 Harness。」
   - Harness 包含六件事：迴圈/協調器、工具、上下文管理、持久化、驗證、限制條件。

2. **[[Harness 是品味的具體實現]]** — 角色：框架
   - 關鍵摘錄：「AI 讓平庸的執行變得廉價；長期來看，AI 讓平庸的決策變得致命。品味就是『什麼值得存在』的判斷；Harness 是品味被具體實現的方式。」
   - 你的 Harness 就是一面鏡子，映照出你的價值觀。

3. **[[測試套件是 Agentic Engineering 的核心槓桿點]]** — 角色：論據
   - 關鍵摘錄：「The single biggest differentiator between agentic engineering and vibe coding is testing.」
   - 有測試 → AI 自己迭代到正確；沒測試 → AI 會愉快地宣告完成在 broken code 上。

4. **[[AI Coding Agent 自主工作的兩個硬前提：測試覆蓋率 + 環境可複製性]]** — 角色：論據

6. **[[AI 協作的核心交易：你用審查時間替代了寫代碼時間]]** — 角色：論據

### 料感盤點（價值 × 乾貨）

**外部價值（讀者得到什麼）**
- 主要：**啟發**——讓 tech lead 意識到「AI-first 的真正門檻是產線主軸，不是 harness」

- 輕度：**教育**——把 Harness 的六要素講清楚

**內部乾貨（我有什麼可以講）**
- 框架：Harness 六要素、Interface+Test 骨架、系統瓶頸思維
- 案例：CREAO、Codex、Stripe、Cloudflare、亞馬遜、Block
- 類比：產線主軸 → 數據飛輪 → 亞馬遜機器人
- 個人觀點（我想帶入的三個起心動念）：
  1. 產線主軸先於 harness
  2. 極端案例要翻譯成可遷移概念
  3. 自己領域的「自我強化迴圈」才是真正要找的東西

**缺口（誠實承認）**
- 我自己沒跑過 CREAO 這種規模的 AI-first
- 小團隊落地 checklist 不夠具體
- 「主軸不清會怎麼崩」的戲劇化案例比較弱

### 核心觀點深化（三層遞進）

**Layer 1：表層觀察**
Peter Pang 的實驗讓人心動。99% 代碼由 AI 寫、一天 8 次部署、一週做完過去六週的事。這很炫。

**Layer 2：往下一層——Harness**
但 Pang 自己講得很清楚：這不是「寫更好的 prompt」，而是建出一個讓 Agent 可讀、可執行、可回饋的環境。Harness 才是主角。

**Layer 3：再往下一層——產線主軸（這篇文章的觀點）**
Harness 也不是終極答案。你得先回答：「這條產線到底要產什麼？怎麼驗證產出是好的？」——CREAO 的答案很清楚，因為他們本來就是 Agent 公司，產品和產線互為飼料；亞馬遜的答案很清楚，因為 20 年累積的撿貨、搬運 KPI 一直在那。一般公司卡住，大多不是卡在技術的 harness，是卡在**產品的主軸**：我到底要這條產線產出什麼？我的「好」長什麼樣？這個問題不釐清，harness 再華麗，都是空轉。

### 謙虛口吻的鋪陳策略

- 不說「大多數公司搞錯重點」，改說「我看完也很興奮，但冷靜下來問自己能學什麼」
- 不說「你應該建 harness」，改說「我讀到這裡也在想，我自己的工作有什麼可以試的」
- 用「我」不用「你」作為主要人稱
- 承認自己也還沒做到，但這篇是想把學到的東西寫下來備忘

### 角度摘要

**核心主張**：AI-first 工程的真正門檻不是 harness，而是「產線主軸」——你必須先回答「這條產線要產什麼、好的長什麼樣、怎麼驗證」。CREAO 能跑通是因為他們本來就是 Agent 公司（主軸最清楚的極端情況），亞馬遜能跑通是因為 20 年零售 KPI 早就鎖定。一般公司卡住，大多不是卡在技術 harness，而是卡在產品主軸不清。

**給 5–10 人 Tech Lead 的可帶走的三點**：
1. 先找瓶頸，再談 AI——瓶頸若在 PM/需求/驗收，加 AI coding 工具只會更亂
2. 把 Interface + Test 當成「給 AI 看的規格」——最小成本的 harness
3. 刻意保留初階工程師的養成路徑——未來架構師的來源，不能因為短期效率砍掉

### 一、Pang 的答案是 Harness，這我同意

Pang 寫這一串貼文的時候，OpenAI 剛在 2026 年 2 月丟出一個詞：**Harness Engineering**。

他們的 Codex 團隊做了一個超過百萬行的正式應用，沒有一行程式出自人手。所有應用邏輯、測試、CI 設定、文件、可觀測性都是 Codex 代理寫的。

Harness 這個比喻很精準。馬是 AI 模型——強大、快速，但不知道該往哪跑。Harness 是韁繩、鞍具、銜鐵，是約束條件、回饋迴圈、文件規範、linter、CI 閘門和架構邊界，讓代理的產出變得可靠。

CREAO 的堆疊具體長這樣：

- 所有程式碼塞進一個 monorepo，不是為了開發者方便，是為了「代理可讀性」——碎片化的程式碼庫對 Agent 來說等於不存在
- 六階段 CI/CD 管線（驗證 CI → 建置 Dev → 測試 Dev → 部署 Prod → 測試 Prod → 發布），沒有跳過、沒有手動覆寫
- 每個 PR 觸發三道平行的 AI 審查（Claude Opus 4.6），程式碼品質、安全性、相依性掃描，不是建議是閘門
- 每天早上九點自動健康檢查，錯誤分群→ Linear 自動建工單 → 工程師推修復 → 同一管線審查部署 → 錯誤消失自動關單

Pang 說：「AI 優先工作流程出問題時，解法不是更努力或寫更好的提示詞，而是缺少什麼能力？怎麼讓它對代理可讀、可執行？你不再是在除錯程式碼，而是在除錯那個產出程式碼的環境。」

這句話我特別抄下來。它把我認知裡的「AI Coding」整個翻過來了——**工程的重心從寫 code 移到設計產出 code 的環境**。

我之前寫過一張卡片叫〈測試套件是 Agentic Engineering 的核心槓桿點〉，引用的是更早一句話：「The single biggest differentiator between agentic engineering and vibe coding is testing.」——Pang 這一串貼文其實就是把這個道理做到了極端。

所以讀完第一遍，我覺得我看懂了：**要做 AI-first，先把 harness 建起來。** 回去我也該盤一盤自己團隊的測試覆蓋率、環境可複製性、有沒有確定性的 CI。

### 二、但讀完第三遍，我發現 Harness 之前還有一層

第二次讀，我開始困惑一件事。

Pang 在文末寫：「先搞清楚瓶頸在哪。如果 PM 流程比開發時間還長，那才是該先處理的，不是一味加 AI 程式碼工具。」

這句話放在通篇講 CI/CD 閘門和三道 AI 審查的文章結尾，有點違和。它暗示一件事：**就算你把 harness 建得跟 CREAO 一樣完整，如果瓶頸不在開發速度，你只是把非瓶頸環節加速了——整體產線還是卡住。**

那 CREAO 為什麼能跑起來？

我覺得答案藏在另一個細節裡：**CREAO 本來就是一家 Agent 公司。**

他們的產品是 Agent，所以他們用 Agent 建 Agent，用 Agent 測 Agent，用 Agent 監控 Agent。產品和產線互為飼料——每一次部署不只是上線一個功能，也是一次「我們的 Agent 好不好用」的自我驗證。這條產線有一個**極清楚的主軸**：我們要做出好用的 Agent，好用的 Agent 就是能寫 code、能修 bug、能通過我們自己設計的那些閘門。

這就是我讀完第三次，真正卡住的地方——

**我忽然意識到：Harness 不是 AI-first 的起點，產線主軸才是。**

你得先回答三個問題：

1. 這條產線要產出什麼？
2. 「好的產出」長什麼樣？
3. 怎麼驗證它產出得好？

這三個問題不釐清，harness 再華麗都是空轉。你可以建六階段 CI/CD、三道 AI 審查、每天健康檢查——但如果沒人說得清楚「我們為什麼要部署這個功能、它上線後誰會因此變得更好」，你只是用非常先進的方式在產垃圾。

我想到另一張卡片〈高開發成本曾是天然過濾器：AI 降低門檻後的四大組織副作用〉裡一句話：高開發成本以前是品質過濾器，過濾器消失後，組織會用驚人速度產出大量無人要的垃圾功能。Pang 自己也提到這個風險：GitClear 分析 1.53 億行程式碼、Carnegie Mellon 800 多個 GitHub repo、Apiiro 對 Fortune 50 的研究，都指向同一個方向——技術債在 AI 輔助下正在指數累積。

AI 生成的程式碼通常看起來乾淨：格式工整，單獨看邏輯正確。風險在累積。每次 AI 提交都製造細微不一致，個別來看微不足道，合在一起就像複利。

Harness 可以攔一部分——CREAO 的三道審查就是幹這個的。但 harness 攔不住的那一層，是「這個功能根本不該被做」。這一層的回答不在 harness 裡，在**產品主軸**裡。

### 三、找一個實體世界的參照：亞馬遜

我覺得亞馬遜提供了一個很好的對照案例。

2022 年 Q1 到 2023 年 Q2，亞馬遜員工從 162 萬降到 146 萬（減少 10%），營收成長 15.4%，人均產值五個季度飆升 28.1%。機器人員工從 2019 年 20 萬台 → 2022 年 52 萬台 → 2023 年 75 萬台。北美業務營業利益率從虧損跳升到 6%。

這是實體世界的 AI-first。

但亞馬遜不是因為「買了很多機器人」才變強——他們是**累積了 20 年的零售營運**，「撿貨要多快」「搬運要多準」「退貨要幾步完成」這些 KPI 早就被定義、被衡量、被優化過一輪又一輪。機器人進來的時候，產線主軸已經非常清楚：把貨從 A 搬到 B，錯誤率和時間都有量化基準。

機器人只是接手了產線上「人做不快、做不穩」的那一段。

假設今天一個零售新創，剛成立、沒有 KPI、沒有流程，就買一堆機器人進來，會發生什麼事？

機器人會以最高效率，把錯的貨、搬到錯的地方。

這就是「有 harness 但沒產線主軸」的後果。

### 四、三件可以帶走的事（一個 tech lead 的備忘）

好，那對我這種帶 5–10 人小團隊的技術主管，Pang 這篇文章可以帶走什麼？

我整理了三件。這不是教戰守則，是我讀完的個人備忘——我自己也還沒全部做到。

**第一件：先找瓶頸，再談 AI**

這是 Pang 說的，也是我想先提醒自己的。

AI coding 工具很誘人，但它只在「寫 code 是瓶頸」的前提下有效。對很多小團隊來說，寫 code 根本不是瓶頸——需求不清、驗收條件模糊、PM 和工程沒對齊、決定一個 feature 要不要做的過程比寫它還久——這些才是。

這時候加 AI coding 工具會怎樣？它會讓原本就不該做的東西，**更快地被做出來**。

Block CTO Dhanji Prasanna 那句話我很喜歡：「AI 導入成敗首先是組織設計問題，不是工具問題。」他的意思不是不要用工具，而是用工具之前要先問——這個工具要解哪個瓶頸？

**第二件：Interface + Test 是小團隊最負擔得起的 harness**

CREAO 的六階段 CI/CD 加三道 AI 審查，對我來說是一個遙遠的北極星。小團隊短期內不可能堆到那種等級，但不代表沒有起點。

我覺得最便宜的起點是 **Interface + Test** 這個骨架策略：

- 用 code 定義規格（Interface）
- 用 code 定義驗收條件（Test）
- 這兩層你親自逐行 review 並保護起來，不讓 AI 改
- 中間的實作全交給 Agent 跑到通過為止

這是一個小團隊可以今天下午就開始做的事。它比大肆改造 CI/CD 簡單，但它做的是同一件事的最小版本——**把「好的產出是什麼」用 code 寫清楚，讓 AI 在這個定義之下工作。**

測試在 AI 時代的角色不只是防回歸，它變成你跟 AI 溝通「我到底要什麼」的唯一精確語言。

**第三件：別把未來的架構師一起砍掉**

這件事我原本沒想到，是 Pang 的風險段落提醒我的。

他說 AI-first 的世界裡會出現兩種工程師：**架構師**（每個團隊一到兩人，設計標準作業程序、教 AI 怎麼工作、在 AI 方案裡找漏洞）和**操作員**（其他所有人，在架構師設計的系統內工作，負責審查 AI 的 PR、驗證分流工單）。

他也指出：**最有價值的技能是架構判斷力**，也就是批評 AI、找失敗模式、壓力測試假設的能力。

但——這些技能需要多年的實戰編碼、犯錯和正式環境事故才能培養。

你跳不過操作員階段，直接當架構師。

這裡有個時間差陷阱：54% 的工程領導者計畫因 AI 而減少初階聘用。短期看，初階工程師「產出/成本比」確實打不過 AI。但如果連操作員階段都不讓新人進來做，3 到 5 年後你要去哪找下一批架構師？

這是產業級的結構問題，我一個人解決不了。但我至少可以在自己的小團隊裡：

- 別一股腦把所有初階位置都砍了
- 刻意設計 AI 增強的入職流程，讓新人用 AI 協作但仍然接觸到真實的問題
- 把「找失敗模式、壓力測試假設」變成所有層級的日常訓練，不是只有資深才做

### 五、我真正想找的：自己領域的自我強化迴圈

最後想寫一段比較私人的。

讀完 Pang 的文章，我留下最深的印象其實不是他的 CI/CD 堆疊，是一個畫面：**CREAO 的 Agent 在寫 CREAO 的 Agent。**

這是一個很純粹的飛輪——產品就是產線，產線就是產品，每一次部署都同時是功能發布和自我驗證。

大模型公司都在期待一件事：**模型能自己改進模型。** Anthropic、OpenAI、Google 都有類似的工作在跑，那是他們的終極飛輪。

CREAO 是小規模的同構版本：他們的產品是 Agent，他們用 Agent 去寫、去測、去審查 Agent。

這讓我開始想——在我自己的領域（或你的領域），有沒有一個對應的自我強化迴圈？

不見得是「我的產品在做我的產品」這種極端形式，但可以是更溫和的版本：

- 我今天產出的一份文件，能不能在下次被我或我的團隊（或 AI）拿去再利用、再改進？
- 我做的這個決策，能不能留下一個可以被回頭驗證的紀錄？
- 我今天跟 AI 協作的這一輪，有沒有讓我自己的品味、我的 Interface、我的驗收標準變得更清楚？

如果有，這條迴圈就會複利。
如果沒有，就是每次重來一次。

Pang 的 harness 是一個可見的工程形式。但真正推動那條飛輪的，是 CREAO 對「什麼是好 Agent」這件事的清晰共識——那是產品主軸，不是工程堆疊。

我在想，在自己領域找到這個等價物，可能比複製他們的 CI/CD 更重要。

## 一、Pang 的答案是 Harness，這我同意

Pang 寫這一串貼文的時候，OpenAI 剛在 2026 年 2 月提出一個詞：**Harness Engineering**。

他們的 Codex 團隊做了一個超過百萬行的正式環境應用，沒有一行程式出自人手。所有應用邏輯、測試、CI 設定、文件、可觀測性都是 Codex 代理寫的。

Harness 這個比喻很精準。馬是 AI 模型——強大、快速，但不知道該往哪跑。Harness 是韁繩、鞍具、銜鐵，是約束條件、回饋迴圈、文件規範、linter、CI 閘門和架構邊界，讓代理的產出變得可靠。

我整理了三件。這不是教戰守則，是我讀完的個人備忘——我自己也還沒全部做到。

### 第一件：先找瓶頸，再談 AI

這是 Pang 說的，也是我想先提醒自己的。

AI coding 工具很誘人，但它只在「寫 code 是瓶頸」的前提下有效。對很多小團隊來說，寫 code 根本不是瓶頸——需求不清、驗收條件模糊、PM 和工程沒對齊、決定一個 feature 要不要做的過程比寫它還久——這些才是。

這時候加 AI coding 工具會怎樣？它會讓原本就不該做的東西，**更快地被做出來**。

Block CTO Dhanji Prasanna 那句話我很喜歡：「AI 導入成敗首先是組織設計問題，不是工具問題。」他的意思不是不要用工具，而是用工具之前要先問——這個工具要解哪個瓶頸？

### 第二件：Interface + Test 是小團隊最負擔得起的 harness

CREAO 的六階段 CI/CD 加三道 AI 審查，對我來說是一個遙遠的北極星。小團隊短期內不可能堆到那種等級，不代表沒有起點。

前幾天公司一位優秀的同事才丟了一版 coding review 流程草案給我們討論——哪個階段誰 review、用什麼 checklist、AI 跟人類審查怎麼分工。我當下覺得這其實就是我們在能力範圍內的一版 harness——不華麗，也不完整，但它在回答同一個問題：**我們的「好」長什麼樣，誰來替我們守。**

我自己最想補上的那一塊，是把規格和驗收條件往 code 裡推：

- 用 code 定義規格（Interface）
- 用 code 定義驗收條件（Test）
- 這兩層親自逐行 review 並保護起來，不讓 AI 改
- 中間的實作全交給 Agent 跑到通過為止

這是一個小團隊可以今天下午就開始做的事。它比大肆改造 CI/CD 簡單，做的是同一件事的最小版本——**把「好的產出是什麼」用 code 寫清楚，讓 AI 在這個定義之下工作。**

測試在 AI 時代的角色不只是防回歸，它變成你跟 AI 溝通「我到底要什麼」的唯一精確語言。

### 第三件：別把未來的架構師一起砍掉

這件事我原本沒想到，是 Pang 的風險段落提醒我的。

他說 AI-first 的世界裡會出現兩種工程師：**架構師**（每個團隊一到兩人，設計標準作業程序、教 AI 怎麼工作、在 AI 方案裡找漏洞），和**操作員**（其他所有人，在架構師設計的系統內工作，負責審查 AI 的 PR、驗證分流工單）。

他也指出：**最有價值的技能是架構判斷力**——批評 AI、找失敗模式、壓力測試假設的能力。

但這些技能需要多年的實戰編碼、犯錯和正式環境事故才能培養。

你跳不過操作員階段，直接當架構師。

這裡有個時間差陷阱：54% 的工程領導者計畫因 AI 而減少初階聘用。短期看，初階工程師「產出/成本比」確實打不過 AI。如果連操作員階段都不讓新人進來做，3 到 5 年後要去哪找下一批架構師？

這是產業級的結構問題，我一個人解決不了。但我至少可以在自己的小團隊裡：

- 別一股腦把所有初階位置都砍了。
- 刻意設計 AI 增強的入職流程，讓新人用 AI 協作但仍然接觸到真實的問題。
- 把「找失敗模式、壓力測試假設」變成所有層級的日常訓練，不是只有資深才做。

## 五、我真正想找的：自己領域的自我強化迴圈

最後想寫一段比較私人的。

讀完 Pang 的文章，我留下最深的印象其實不是他的 CI/CD 堆疊，是一個畫面：**CREAO 的 Agent 在寫 CREAO 的 Agent。**

這是一個很純粹的飛輪——產品就是產線，產線就是產品，每一次部署都同時是功能發布和自我驗證。

大模型公司都在期待一件事：**模型能自己改進模型。** Anthropic、OpenAI、Google 都有類似的工作在跑，那是他們的終極飛輪。CREAO 是小規模的同構版本：他們的產品是 Agent，他們用 Agent 去寫、去測、去審查 Agent。

這讓我開始想——在我自己的領域（或你的領域），有沒有一個對應的自我強化迴圈？

不見得要是「我的產品在做我的產品」這種極端形式，可以是更溫和的版本：

- 我今天產出的一份文件，能不能在下次被我或我的團隊（或 AI）拿去再利用、再改進？
- 我做的這個決策，能不能留下一個可以被回頭驗證的紀錄？
- 我今天跟 AI 協作的這一輪，有沒有讓我自己的品味、我的 Interface、我的驗收標準變得更清楚？

如果有，這條迴圈就會複利。
如果沒有，就是每次重來一次。

Pang 的 harness 是一個可見的工程形式。真正推動那條飛輪的，是 CREAO 對「什麼是好 Agent」這件事的清晰共識——那是產品主軸，不是工程堆疊。

我在想，在自己領域找到這個等價物，可能比複製他們的 CI/CD 更重要。

---

## 收尾

寫完這篇我還是覺得，CREAO 那種極端 AI-first 對我來說很遠。

也許近的部分，是「先想清楚產線主軸」這件事本來就是我的工作，只是過去被別的優先級蓋過去了——AI 只是把這個問題的時間窗口縮短了。

如果你也讀到這裡，然後跟我一樣還沒有答案，我想那是正常的。比較不正常的，是看完這篇覺得自己立刻就能複製 CREAO 的人。
