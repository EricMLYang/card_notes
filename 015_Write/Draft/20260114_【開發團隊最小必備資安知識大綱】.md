# 【開發團隊最小必備資安知識大綱】



開發外部產品一直被提醒需要注意資安，

資安這是必備要求但也不能無限度上綱，

因此整理一份 開發者(非 Infra 單位) 必備的最小知識，

期望達到以下效果:

- 清單檢查:

- 搭配後續 ADP 購買的課程，但更具焦必要範圍，增加學習效果

- 配合 AWS 的認證考試部分內容準備



### 主題:

**開發團隊資安學習大綱** &

**AWS SAA-C03 Domain 1: Design Secure Architectures (30%)**

**整合版**

- 目標: 開發安全系統 + 準備 AWS Solutions Architect Associate 認證

- 原則: 實務導向、避免過深理論、以開發者視角學習



## 📚 學習路徑總覽

- 第一階段 (2週): 資安基礎 + AWS 身份與存取管理

- 第二階段 (2週): 雲端架構安全 + AWS 網路防護

- 第三階段 (2週): API/後端安全 + AWS 資料保護

- 第四階段 (1週): 前端安全 + 自動化工具整合

## 🔰 第一階段:資安基礎思維 + AWS 身份管理 (2週)

### 📖 Part 1: 資安核心概念 (3天)

**這部分在學什麼?**

建立「安全思考方式」,了解攻擊者會從哪些角度攻擊你的系統



**學習內容:**

- CIA 三要素

   - Confidentiality (機密性): 資料不被未授權的人看到

   - Integrity (完整性): 資料不會被竄改

   - Availability (可用性): 系統隨時可用,不會被打掛



- Zero Trust 原則:

   - 不相信任何人,每次存取都要驗證

   - 最小權限原則:只給需要的權限,不多給



- 常見攻擊類型 (知道是什麼就好,不用學怎麼攻擊)

   - SQL Injection: 在輸入框塞惡意指令

   - XSS: 在網頁植入惡意程式碼

   - CSRF: 偽造使用者的操作

   - Token 被偷: 別人拿你的登入憑證冒充你



- 威脅模型簡化版

   - 畫出系統架構圖

   - 標出「哪些地方有資料」、「誰可以存取」

   - 找出「如果被攻擊會怎樣」的弱點



**實作練習:** 在 VMS 或廣告平台架構圖上,標出 5 個可能的攻擊點

### ☁️ 新增: AWS 責任共擔模型 (Shared Responsibility Model)

**這部分在學什麼?**

了解「哪些安全是 AWS 負責,哪些是你負責」,避免誤以為上雲就自動安全



**學習內容:**

- AWS 負責「雲端本身」的安全 (Security OF the Cloud)

   - 實體機房安全 (門禁、監控)

   - 硬體設備維護

   - 網路基礎設施

   - 底層虛擬化平台

   - 託管服務的安全 (例如: RDS 的資料庫軟體更新)



- 你負責「雲端上」的安全 (Security IN the Cloud)

   - IAM 權限設定

   - 資料加密 (at rest / in transit)

   - 作業系統與應用程式的更新

   - 網路設定 (Security Group, NACL)

   - 應用程式層的安全 (API 驗證、Input Validation)



- 責任分界線

   - IaaS (EC2): 你負責作業系統以上的所有東西

   - PaaS (RDS, Elastic Beanstalk): AWS 管理更多,你管理應用層

   - SaaS (S3): AWS 管理平台,你管理存取權限與加密



**為什麼重要:**

很多資安事件是因為「以為 AWS 會幫忙做,結果沒做」

例如: S3 bucket 設定錯誤導致資料外洩 → 這是你的責任,不是 AWS 的責任



**實作練習:**

畫一張你們系統的「責任分界圖」,標示哪些是 AWS 負責,哪些是團隊負責

### 🔐 Part 2: AWS 身份與存取管理 (IAM) (4天)

**這部分在學什麼?**

控制「誰可以做什麼」,這是雲端安全的基礎



**對應 SAA-C03 Domain 1:**

- 1\.1 Design secure access to AWS resources

- 1\.2 Design secure workload and application architectures



**學習內容:**

- IAM 核心概念

   - User: 個人帳號 (例如:工程師 Eric)

   - Group: 一群人 (例如:開發團隊)

   - Role: 臨時身份 (例如:讓 EC2 可以讀 S3)

   - Policy: 權限規則 (例如:可以讀取 S3 但不能刪除)



- IAM Policy 結構

   - Effect: Allow (允許) 或 Deny (拒絕)

   - Action: 可以做什麼 (例如: s3:GetObject)

   - Resource: 針對哪個資源 (例如: 特定的 S3 bucket)

   - Condition: 什麼條件下才生效 (例如: IP 來自公司網路)



- IAM Best Practices

   - Root 帳號只用來開帳號,平常不用

   - 開啟 MFA (多因素驗證)

   - 用 Role 而不是 Access Key (避免金鑰外洩)

   - 定期檢查權限,移除不需要的



- 跨帳號存取 (Cross-Account Access)

   - 為什麼需要: 開發環境和正式環境分開

   - 怎麼做: 用 Role + Trust Policy



- 服務相關角色 (Service Roles)

   - EC2 要讀 S3: 給 EC2 一個 Role

   - Lambda 要寫 DynamoDB: 給 Lambda 一個 Role

 - 避免把 Access Key 寫在程式碼裡



**實作練習:**

建立一個 IAM Role,讓你的 EC2 可以讀取特定 S3 bucket,但不能刪除

---

### 🔍 新增: IAM Access Analyzer

**這部分在學什麼?**

自動找出「權限給太多」的問題,避免安全漏洞



學習內容:

- 什麼是 Access Analyzer

   - AWS 提供的免費工具

   - 自動分析 IAM Policy,找出過寬的權限

   - 找出「可以被外部存取」的資源 (例如: 公開的 S3 bucket)

- 常見發現的問題

   - S3 bucket 被設定為 Public

   - IAM Role 的 Trust Policy 允許任何 AWS 帳號

   - Lambda 函數可以被外部呼叫

   - KMS Key 可以被其他帳號使用

- 如何使用

   - 在 IAM Console 開啟 Access Analyzer

   - 定期檢查發現 (Findings)

   - 修正高風險的權限設定

- 建議頻率

   - 每季檢查一次

   - 新增重要資源時立即檢查

   - 有外部稽核前務必檢查

**實作練習:**

開啟 Access Analyzer,檢查你們的 AWS 帳號有沒有「可被外部存取」的資源

### 🔑 Part 3: AWS Secrets 與金鑰管理 (3天)

**這部分在學什麼?**

安全地存放密碼、API Key、資料庫連線字串等機密資訊



**對應 SAA-C03 Domain 1:**

- 1\.1 Design secure access to AWS resources



**學習內容:**

- AWS Secrets Manager

   - 用途: 存放資料庫密碼、API Key

   - 功能: 自動輪換密碼 (定期換新密碼)

   - 使用方式: 應用程式啟動時從 Secrets Manager 讀取



- AWS Systems Manager Parameter Store

   - 用途: 存放設定參數 (可以是機密或非機密)

   - 與 Secrets Manager 的差異: 比較便宜,但沒有自動輪換功能

   - 何時用: 存放非關鍵的設定值



- AWS KMS (Key Management Service)

   - 用途: 管理加密金鑰

   - Customer Managed Keys (CMK): 你自己管理的金鑰

   - 自動輪換: 每年自動換新金鑰

   - 搭配其他服務: S3、RDS、EBS 都可以用 KMS 加密



- 實務原則

   - 絕對不要把密碼寫在程式碼裡

   - 絕對不要把 Access Key 放在 Git 裡

   - 用環境變數 + Secrets Manager



- CMK vs SSE-S3 的選擇

   - SSE-S3 (AWS 管理金鑰): 簡單方便,AWS 自動管理

   - CMK (客戶管理金鑰): 可以控制誰能用這把金鑰,適合敏感資料

   - 建議: 一般資料用 SSE-S3,機密資料用 CMK

- 實作練習: 把資料庫連線字串存到 Secrets Manager,讓你的應用程式讀取

## ☁️ 第二階段:雲端架構安全 + AWS 網路防護 (2週)



### 🏗️ Part 4: AWS 網路安全架構 (5天)

**這部分在學什麼?**

設計「防火牆」,控制哪些流量可以進出你的系統



**對應 SAA-C03 Domain 1:**

1\.2 Design secure workload and application architectures



**學習內容:**

- VPC (Virtual Private Cloud)

   - 用途: 你在 AWS 裡的私人網路區域

   - 概念: 就像你的辦公室有自己的內部網路



- Subnet (子網路)

   - Public Subnet: 可以連到網際網路 (例如:放 Web Server)

   - Private Subnet: 不能直接連到網際網路 (例如:放資料庫)



- Security Group (安全群組)

   - 概念: 虛擬防火牆,控制進出流量

   - Stateful: 允許進來的流量,回應可以自動出去

   - 預設: 全部拒絕,只開放你允許的

   - 實例: 只允許 HTTPS (443) 進來



- Network ACL (網路存取控制清單)

   - 概念: Subnet 層級的防火牆

   - Stateless: 進出要分別設定

   - 與 Security Group 的差異: NACL 是 Subnet 層,SG 是 Instance 層



- Private Endpoint (VPC Endpoint)

   - 用途: 不透過網際網路存取 AWS 服務

   - 為什麼需要: 更安全,流量不出 AWS 內部網路

   - 實例: Private Subnet 的 EC2 要存取 S3



- NAT Gateway

   - 用途: 讓 Private Subnet 的機器可以連外部網路 (例如:下載更新)

   - 但外部不能主動連進來



- 網路分層架構

   - Public Subnet: 放 Load Balancer、Bastion Host

   - Private Subnet: 放應用程式伺服器

   - Data Subnet: 放資料庫 (最嚴格保護)



**實作練習:**

畫出你們系統的 VPC 架構圖,標示哪些元件在 Public/Private Subnet

### 🏢 新增: 多帳號與環境隔離 (基本概念)

**這部分在學什麼?**

了解為什麼公司要分多個 AWS 帳號,以及這如何保護你的系統



學習內容:

- 為什麼要分帳號

   - Dev 環境出問題不會影響 Prod

   - 不同團隊/專案的資源完全隔離

   - 帳單分開,容易追蹤成本

   - 降低「誤刪正式環境資源」的風險



- 常見的帳號架構

   - Sandbox: 工程師實驗用 (可以亂搞)

   - Dev: 開發環境

   - Staging: 測試環境 (接近正式環境的設定)

   - Production: 正式環境 (最嚴格的權限控管)



- AWS Organizations

   - 用途: 統一管理多個 AWS 帳號

   - 功能: 集中付費、統一權限政策

   - 你只需知道: 公司的 AWS 帳號是用 Organizations 管理的



- 環境隔離的好處

   - 開發者在 Dev 有較大權限,可以快速測試

   - 到 Staging/Prod 權限變嚴格,需要審核

   - 即使 Dev 帳號被入侵,駭客也無法碰到正式環境



**開發者需要知道的:**

- 你在哪個環境工作

- 為什麼某些操作在 Prod 會被擋 (這是保護機制)

- 需要 Prod 權限時找誰申請



**不需要知道的:**

- 如何設計 SCP (Service Control Policy) ← 這是雲端管理員的工作

- 如何規劃帳號架構 ← 這是架構師的工作



**實作練習:**

畫一張圖,標示你們公司有哪幾個 AWS 環境,以及你在每個環境的權限差異

### 🛡️ Part 5: AWS 資料加密與保護 (5天)

**這部分在學什麼?**

確保資料「存放時」和「傳輸時」都是安全的



**對應 SAA-C03 Domain 1:**

1\.3 Determine appropriate data security controls



**學習內容:**

- 加密基本概念

   - Encryption at rest (靜態加密): 資料存在硬碟上要加密

   - Encryption in transit (傳輸加密): 資料在網路上傳要加密 (HTTPS)

   - Encryption in use (使用中加密): 記憶體裡的資料也加密 (進階)



- S3 安全設定

   - Bucket Policy: 控制誰可以存取 bucket

   - Block Public Access: 預設關閉公開存取

   - Versioning: 保留檔案歷史版本,防止誤刪

   - S3 Encryption: SSE-S3 (AWS 管理) / SSE-KMS (你管理金鑰)

   - S3 Object Lock: 防止檔案被刪除或修改 (合規用)



- RDS 安全設定

   - 加密: 啟用 encryption at rest

   - 備份加密: 自動備份也要加密

   - IAM Database Authentication: 用 IAM Role 登入資料庫,不用密碼

   - Private Subnet: 資料庫不要放在 Public Subnet



- 資料分類與保護

   - Public: 可以公開的資料 (產品目錄)

   - Internal: 公司內部資料 (員工名單)

   - Confidential: 機密資料 (客戶個資、財務資料)

   - Restricted: 最高機密 (核心商業機密)



- 資料遮蔽與匿名化

   - Masking: 隱藏部分資料 (信用卡號顯示後四碼)

   - Pseudonymization: 用假名取代真名 (測試環境用)

   - Tokenization: 用亂數取代敏感資料



- 法規基本認知 (不用深入,知道是什麼就好)

   - GDPR: 歐盟個資法 (使用者有權要求刪除資料)

   - 台灣個資法: 收集個資要有目的,不能亂用



**實作練習:**

設定 S3 bucket 的加密與存取控制,確保只有授權的 IAM Role 可以讀取

---





### 💾 新增: 備份與復原基礎概念

**這部分在學什麼?**

確保資料不會因為誤刪、故障或攻擊而永久消失



學習內容:

- RPO 與 RTO (基本概念)

   - RPO (Recovery Point Objective): 可以接受遺失多少資料

      - 例如: RPO = 1 小時,代表最多遺失 1 小時的資料

   - RTO (Recovery Time Objective): 多久要復原系統

      - 例如: RTO = 4 小時,代表 4 小時內要讓系統恢復運作

   - 開發者需要知道: 你們系統的 RPO/RTO 目標是多少



- RDS 自動備份

   - 預設功能: RDS 每天自動備份

   - 保留期限: 可設定 1-35 天

   - 如何還原: 可以還原到「某個時間點」

   - 建議設定: 保留至少 7 天



- S3 版本控管 (Versioning)

   - 功能: 保留檔案的所有歷史版本

   - 好處: 誤刪可以復原、被竄改可以還原

   - 建議: 重要資料的 S3 bucket 都要開啟



- 跨 AZ 部署 (高可用性)

   - RDS Multi-AZ: 主資料庫故障時自動切換到備援

   - S3 自動跨 AZ: 資料會自動備份到多個資料中心



**開發者需要知道的:**

- 備份有開啟

- 誤刪資料時知道怎麼還原 (或找誰處理)

- 了解系統的 RPO/RTO 目標



**不需要知道的:**

- 如何設計完整的災難復原計畫 ← SRE 職責

- 如何進行 DR 演練 ← SRE 職責

- 跨區域備份架構 ← 架構師職責



**實作練習:**

檢查你們的 RDS 是否開啟自動備份,S3 bucket 是否開啟版本控管

## 🔒 第三階段:API/後端安全 + AWS 應用層防護 (2週)

### 🧑‍💻 Part 6: 安全 API 開發實務 (7天)

**這部分在學什麼?**

保護你的 API 不被未授權存取或攻擊



**對應 SAA-C03 Domain 1:**

1\.2 Design secure workload and application architectures



**學習內容:**

- **身份驗證 (Authentication)**

   - OAuth 2.0: 授權流程標準 (例如:用 Google 帳號登入)

   - JWT (JSON Web Token): 一種 token 格式

   - JWT 結構: Header + Payload + Signature

   - Token 過期機制: 設定有效期限 (例如: 15 分鐘)

   - Refresh Token: 用來換新的 Access Token



- **API Gateway 安全機制**

   - Throttling: 限制請求次數 (防止 DDoS)

   - API Key: 簡單的存取控制

   - WAF (Web Application Firewall): 擋掉惡意請求

   - CORS 設定: 控制哪些網站可以呼叫你的 API



- Input Validation (輸入驗證)

   - 為什麼重要: 防止 SQL Injection、XSS

   - 怎麼做: 檢查資料格式、長度、特殊字元

   - 白名單 vs 黑名單: 只允許預期的格式 (白名單較安全)



- 安全 Headers

   - Content-Security-Policy: 防止 XSS

   - X-Frame-Options: 防止 Clickjacking

   - Strict-Transport-Security: 強制使用 HTTPS



- 加密與雜湊

   - 加密 (Encryption): 可以解密回來 (例如: AES)

   - 雜湊 (Hash): 不可逆 (例如: 密碼用 bcrypt)

   - 不要用: MD5、SHA-1 (已被破解)

   - 該用: bcrypt、PBKDF2、Argon2



- 安全日誌

   - 記錄什麼: 登入失敗、權限拒絕、異常操作

   - 不要記錄: 密碼、信用卡號、完整的 token

   - 保留多久: 至少 90 天 (依法規需求)



**實作練習:**

撰寫一個受 JWT 保護的 API,加上輸入驗證與錯誤處理

### 🔐 Part 7: AWS 應用層安全服務 (3天)

**這部分在學什麼?**

使用 AWS 提供的安全工具保護你的應用程式



**對應 SAA-C03 Domain 1:**

1\.2 Design secure workload and application architectures



**學習內容:**

- AWS WAF (Web Application Firewall)

   - 用途: 保護 Web 應用程式免受常見攻擊

   - 可以擋: SQL Injection、XSS、DDoS

   - 規則來源: AWS Managed Rules (預設規則集) 或自訂規則



- AWS Shield

   - Shield Standard: 自動防護基本 DDoS (免費)

   - Shield Advanced: 進階 DDoS 防護 (付費)



- Amazon Cognito

   - 用途: 使用者註冊、登入、權限管理

### 🔍 新增: 監控與偵測基礎

 **這部分在學什麼?**

了解「系統被攻擊時會發生什麼」,以及如何快速發現問題



**學習內容:**

- CloudTrail (操作記錄)\*\*

   - 用途: 記錄所有 AWS 操作 (誰在什麼時候做了什麼)

   - 為什麼重要: 被入侵時可以追查「駭客做了什麼」

   - 用途: 自動偵測異常行為 (例如: 非常規地區登入)

   - Root 帳號被使用 (應該永遠不用 Root)

   - S3 bucket 被設為 Public (資料外洩風險)

   - IAM Policy 被大幅修改 (可能是權限被濫用)

**開發者需要知道的:**

- 這些服務存在,而且應該保持開啟

- 遇到安全告警時,知道去哪裡看 log

- 知道找誰處理告警 (通常是 SRE 或資安團隊)

**不需要知道的:**

- 如何設計完整的告警矩陣 ← SRE 職責

- Security Hub 的深入整合 ← 資安團隊職責

- AWS Config 的漂移偵測 ← DevOps 職責

**實作練習:**

確認你們的 AWS 帳號有開啟 CloudTrail 和 GuardDuty,並找到在哪裡查看 log

## Draft v1（Step 5: BODY+CLOSE — 2026-03-16）

# 開發者（非 Infra）必備資安知識：4 個階段、9 週、21 個檢查點

S3 bucket 公開設定錯誤、IAM 權限給太寬、Access Key 存在 GitHub。

這三件事每年讓企業損失數十億美元。

全都不是駭客有多厲害——而是開發者以為「這是 AWS 在管的」。

---

▋ **責任的空白地帶**

開發外部產品的過程中，我一直被提醒「要注意資安」。

但沒有人告訴我具體是什麼。

要學到哪個程度算夠？哪些是開發者的責任、哪些是 Infra 的、哪些是 AWS 的？

直到某次上線前的資安審查報告回來，有一半的問題出在應用層——不是 Infra 沒設好，是我們開發的時候沒想到。

從那次之後我才意識到：**不是沒有清單，是沒有人整理過給開發者看的那一份。**

這篇就是那份清單。

---

▋ **AWS 教你的最重要一課**

AWS 有一份文件，每個雲端開發者都應該讀一遍：**Shared Responsibility Model（責任共擔模型）**。

它說的很清楚——

**AWS 負責「雲端本身」的安全**（Security *of* the Cloud）：實體機房、硬體、虛擬化層、底層網路。

**你負責「雲端上」的安全**（Security *in* the Cloud）：IAM 設定、資料加密、作業系統更新、應用層的所有東西。

這個模型最重要的啟示是：**上了 AWS，不代表你就安全了。**

90% 的雲端資安事件，不是因為 AWS 被攻破。

是因為開發者沒做「自己那份」。

S3 bucket 設為公開、IAM 權限開太寬——這些是你的責任，不是 AWS 的。

---

▋ **最小可行資安（MVS）**

資安是個無底洞。

越學越深，越深越覺得不夠。你可以學到 CISSP，你可以學到滲透測試，你可以永遠找到新漏洞去修。

但你需要的不是這個。

你需要的是 **MVS（Minimum Viable Security，最小可行資安）**——

> 不是要做到銅牆鐵壁，而是要做到「不是最容易被打的那個」。

就像 Code Review，你不需要把每一行都審到完美，但任何會進正式環境、影響資料安全的變更，都不應該跳過系統層的審查。

道理一樣：有邊界，有底線，其他的依資源排優先。

這份清單的設計原則，就是 MVS：4 個階段、9 週、21 個必做檢查點。

---

▋ **第一階段：你是誰？（2 週）**

**身份管理，是雲端安全最核心的一層。**

幾乎所有大型入侵事件，追溯到最後都有一個共同起點：**某個不該有的人，拿到了不該有的憑證。**

IAM 的核心只有三個原則：

**Root 帳號鎖起來**，只用來開帳號，平常加上 MFA 封存不用。

**最小權限原則**，只給需要的權限，需要什麼開什麼，不多給一分。

**用 Role 而不是 Access Key**，服務之間的通訊用 Role，不要讓金鑰在程式碼裡流動。

Access Key 是最常洩漏的東西。

.env 檔案、config.json、Git history——只要出現在這些地方，就是公開的密碼。

正確做法是 **AWS Secrets Manager**：應用程式啟動時動態讀取，密碼不落地。

這兩件事做好，你已經比 80% 的開發者安全。

---

▋ **第二階段：流量怎麼進來、資料怎麼保護？（3 週）**

**網路分層，是防止入侵擴散的關鍵。**

VPC 的設計邏輯只有一個：越敏感的東西，距離外網越遠。

```
Internet
  ↓
Public Subnet（Load Balancer）
  ↓
Private Subnet（應用程式伺服器）
  ↓
Data Subnet（資料庫 — 完全不對外）
```

Security Group 是每個 instance 的虛擬防火牆。

預設拒絕全部，只開放必要的 port。對外服務通常只需要 443。

**資料加密有兩個基本要求：**

Encryption at rest（靜態加密）：S3、RDS 的資料儲存時要加密，在服務設定中開啟即可，幾乎是零成本。

Encryption in transit（傳輸加密）：強制 HTTPS，設定 HSTS header，不讓明文在網路上傳輸。

S3 特別要注意一件事：**Block Public Access 要保持開啟**，不要手動關掉。

這個開關被關掉導致資料外洩的案例，比你想像的多得多。

---

▋ **第三階段：你的 API，能被怎麼打都安全嗎？（3 週）**

**應用層是開發者最能控制的地方，也是漏洞最密集的地方。**

身份驗證用 **OAuth 2.0 + JWT** 是現代 API 的標配。

Access Token 短期（15 分鐘）、Refresh Token 長期，有明確的過期機制。

Token 不要存在 localStorage——容易被 XSS 偷走，改用 HttpOnly Cookie。

輸入驗證，是防止 Injection 攻擊的唯一方法。

```python
# 錯誤：直接拼接 SQL
query = f"SELECT * FROM users WHERE name = '{user_input}'"

# 正確：參數化查詢
query = "SELECT * FROM users WHERE name = %s"
cursor.execute(query, (user_input,))
```

白名單原則：只接受預期格式，其餘一律拒絕。

如果你用 AI 寫了這段 code，你知道要審查這裡嗎？

**這是 AI 時代開發者的新課題：** AI 讓你的開發速度快了 3 倍，但「架構、安全、邊界案例」的判斷責任，沒有因此轉移。資深工程師的競爭優勢之一，就是能看出 AI 寫的 code 哪裡安全、哪裡有坑。

最後，在 API Gateway 前面加上 **AWS WAF**，開啟 AWS Managed Rules——SQL Injection、XSS、常見 DDoS 模式，自動擋掉，不需要自己寫規則。

---

▋ **第四階段：讓 CI/CD 幫你守門（1 週）**

**自動化安全掃描，是最高 CP 值的投資。**

設定一次，每次 PR 自動執行，問題在進 main branch 之前就被攔截。

前端的常見陷阱先確認：

- `v-html` / `dangerouslySetInnerHTML` 直接植入未驗證的 HTML → XSS 開門
- Token 存在 localStorage → 被 XSS 偷走就全部洩漏
- 沒有設 CSP（Content-Security-Policy）Header → 外部腳本可以任意執行

CI/CD pipeline 加三件事：

```yaml
步驟一：Secret Scanning（有沒有把密碼 commit 進去）
步驟二：Dependency Check（Dependabot / Snyk，套件有沒有已知漏洞）
步驟三：SAST（SonarCloud，程式碼有沒有安全漏洞模式）
```

這三件事，GitHub 上幾乎都有免費方案。

沒有設定的團隊，每次上線都在賭。

▋ 起點

S3 bucket 公開設定錯誤、IAM 權限給太寬、Access Key 存在 GitHub。

這三件事每年讓企業損失數十億美元。

駭客的水準不是重點，重點是：開發者以為這些事情「有人在管」。

外部產品開發的過程裡，「資安要注意」這句話我聽過很多次。

但沒有人告訴我具體是什麼。

要學到哪個程度？哪些是開發者的責任、哪些是 Infra 的、哪些是 AWS 的？

直到某次上線前資安審查報告回來，有一半的問題出在應用層。

不是 Infra 沒設好。是我們開發的時候沒有把這些東西列進去。

這篇整理的，就是「開發者那份」的清單。

---

▋ AWS 教給我最重要的一課

AWS 有一份文件，雲端開發者都應該讀一遍：**Shared Responsibility Model（責任共擔模型）**。

**AWS 負責「雲端本身」的安全**（Security *of* the Cloud）。

實體機房、硬體、虛擬化層、底層網路——這些 AWS 扛。

**你負責「雲端上」的安全**（Security *in* the Cloud）。

IAM 設定、資料加密、作業系統更新、應用層的所有東西——這些是你的。

很多資安事件的起點，是開發者以為上了雲就安全了。

S3 bucket 設為公開、IAM 權限給太寬、Access Key 出現在 Git——

這些全部都是 Security *in* the Cloud 的問題。責任在開發者這邊，AWS 管不到。

這張責任地圖，是學資安之前最重要的一個認知校正。

---

▋ 最小可行資安（MVS）

資安是個無底洞。

學完 IAM 還有 VPC，學完 VPC 還有 WAF，學完 WAF 還有滲透測試。

你可以無限往下挖，但挖到最後會發現大多數知識跟你日常開發的關係越來越遠。

這份清單的出發點是 **MVS（Minimum Viable Security）**。

一個實際的目標：讓你的系統在安全性上，不是最容易被找到漏洞的那個。

Code Review 的邏輯也是這樣——你不需要把每一行都審到完美，

但任何會進正式環境、影響資料安全的變更，不應該跳過系統層的審查。

有邊界，有底線，其他的依資源排優先。

4 個階段、9 週、21 個檢查點，這是我目前認為「夠用」的版本。

---

▋ 第一階段：身份管理（2 週）

幾乎所有大型入侵事件，追溯到最後都有一個共同起點：

某個不該有的人，拿到了不該有的憑證。

IAM 的實務核心只有三件事：

**Root 帳號鎖起來。** 只用來開帳號，加上 MFA 之後封存不用。

**最小權限原則。** 需要什麼開什麼，不多給一分。每次開權限前問一個問題：這個 Role / User 最少需要什麼？

**服務間通訊用 Role，不用 Access Key。** Access Key 是最容易洩漏的東西——`.env` 檔案、`config.json`、Git history，只要出現在這些地方就等於公開密碼。

機密資訊的正確存放方式是 **AWS Secrets Manager**：

應用程式啟動時動態讀取，密碼不落地、不在 code 裡流動。

這兩件事做好，已經比大多數沒特別注意的系統安全一個量級。

```
Internet
  ↓

  ↓

  ↓

```
