▍整體架構：我事務所的系統構成
先看整體架構。在我的事務所裡，以下這些機制每天都在自動運作。
Claude Code 是指揮塔，將 freee、Gmail、Google 日曆、Notion 與 Slack 全部串連起來。
重點是「不需要每次都有人下指示才能動」。有兩種類型：排程任務每晚自動執行的，以及我手動輸入指令才啟動的。
每晚 21 點自動執行 freee 自動分錄，每晚 22 點記錄 X 的粉絲數據。MCP 連接了 freee API、Gmail、Google 日曆、Notion、Slack。

━━━━━━━━━━━━━━━━━━━━
▍① 每晚 21 點，60 家公司的分錄自動處理完畢
這是影響力最大的一套機制。
每晚 21 點，Claude Code 的排程任務會自動啟動，處理 freee 中所有公司已登錄但未處理的交易明細。

處理流程如下。
Phase 1：取得未處理明細
從 freee API 取得各事業所的帳戶明細。只抽出狀態為「未處理」且為「出金」的項目，再篩選在進行中會計年度內的項目。

Phase 2：自動判定會計科目（兩階段）
這裡是最重要的部分。會計科目的判定用兩個階段進行。

第一階段：關鍵字辭典比對
針對 14 個會計科目類別，每個類別登錄超過 100 個關鍵字。
舉例：「Suica」「JR」「計程車」→ 旅費交通費。「Amazon」「Yodobashi」「Bic Camera」→ 消耗品費。「AWS」「Google Cloud」「ChatGPT」→ 通訊費。餐飲店名且金額 1 萬日圓以下 → 會議費。餐飲店名且金額超過 1 萬日圓 → 交際費。

交易摘要欄會先做全形轉半形、大寫轉小寫的正規化再比對。日語、英語、片假名、半形片假名全部支援。另外也納入了特殊模式。「匯款＋專業人士名稱」→ 支付報酬（自動抽取交易對象名稱）。「匯款＋片假名人名」→ 外包費（發生日自動改為前月月底）。但對員工的匯款屬於薪資所以跳過（從 freee 人事勞務 API 取得員工姓名並排除）。

第二階段：Claude API 做為備援
只有關鍵字無法判定的交易才會送給 Claude API。提示可用的 14 個會計科目，要求以 JSON 格式回答。信賴度為 high 或 medium 就登錄，low 則跳過交由我確認。
這個兩階段設計是關鍵。能用關鍵字比對處理的就不呼叫 API，所以速度快。Claude API 只用在「新交易對象」或「判斷模糊的項目」。

Phase 3：重複檢查與登錄
登錄前會以既有交易的日期、金額與摘要前 40 個字做重複檢查，防止重複登錄。ATM 提款則視為帳戶轉帳另行處理。沒問題就用 freee API 登錄交易。若交易對象是新的，也會透過 API 自動建立交易對象主檔。

Phase 4：安全性（這點很重要）
因為要處理 60 家公司的資料，事業所之間的資料隔離非常徹底。
各事業所的處理以 company_id 為單位完全分離。交易的詳細內容（摘要、金額）僅記錄在日誌檔案中，畫面上只顯示件數與狀態。日誌檔案也依事業所獨立。

A 公司在處理分錄時混入 B 公司資料這種事，在設計上絕對不會發生。既然受託管理顧問客戶的資料，這點絕不能妥協。

處理時間：全 60 家以串列方式處理，30 到 50 分鐘。每家公司約 20 到 40 秒。
以前是用自動操作瀏覽器的方式，要花 5 個小時。我請 Claude Code「改寫成直接呼叫 API 的方式」，就變成現在的樣子。

5 小時縮短到 50 分鐘。而且每晚自動執行。早上起來，前一天的會計就已經處理好了。
光靠這一點，我的工作面貌就徹底改變了。

▍② 排除規則：「不該做分錄的項目」自動過濾
自動分錄最可怕的是「把不該做分錄的項目也做了」。
我的系統內建了 7 種排除規則。

內容不明的借記卡交易（只有「借記＋數字」沒有店名的項目）
借款的償還（公庫、融資、貸款等關鍵詞）
社會保險費與稅金（年金、健保、扣繳稅）

薪資支付
投資與資產運用（證券帳戶）
ATM 提款與餘額調整（以帳戶轉帳另行處理）
公共事業費（因為沒有對應科目所以排除）

這些會在科目判定之前就被篩掉，交由我目視確認。
「全部自動化不是很危險嗎？」常被這樣問。我的做法是明確劃分「要自動化的部分」和「需要人看的部分」，這樣才能安全地服務 60 家公司。能做出這種劃分，是因為具備稅務師的實務知識。

▍③ 130 張發票，15 分鐘轉成 freee 匯入資料
有一家顧問客戶說「想把一整年的發票資料匯入 freee」。
發票 130 張。營業額 3,000 萬日圓。消費稅 300 萬日圓。扣繳稅 xxx 萬日圓。
我跟 Claude Code 說「讀取這些發票 PDF，轉換成 freee 的交易匯入用 Excel」，它就寫了一支 Python 腳本，15 分鐘全部轉換完畢。

營業額、消費稅、扣繳稅額、交易對象名稱、日期，全部正確。手工處理要整整一天。Claude Code 只要 15 分鐘。而且腳本留著，明年同樣的處理一瞬間就搞定。

━━━━━━━━━━━━━━━━━━━━
▍④ 會計軟體搬家的自動化
有些顧問客戶想從 MF 會計（Money Forward）轉換到 freee。會計軟體的搬家其實很繁瑣。要做科目對應表、匯出歷史資料、轉換格式、匯入。手工處理每家公司要花好幾天。

只要跟 Claude Code 說「做一個把 MF 會計匯出的 CSV 轉成 freee 匯入格式的腳本」就行了。
另外也做了從 MF 會計瀏覽器畫面用 Playwright 自動登錄分錄的腳本。不只 freee，MF 會計的顧問客戶也能對應。

━━━━━━━━━━━━━━━━━━━━
▍⑤ MCP 串接外部工具，轉記作業歸零
MCP 是 Claude Code 連接外部工具的機制。我串接了哪些工具，分別在做什麼？freee MCP：即時查看特定事業所的交易資料。「算一下上個月的通訊費總額」這種問題也能馬上回答。

Gmail MCP：幫顧問客戶草擬提醒郵件。「對還沒交月報資料的顧問客戶寄信」就能一次批量草擬。

Google 日曆 MCP：列出下週的會議預定。自動產生會議前的準備工作項目。

Notion MCP：讀取會議記錄，自動生成下次會議議程。按顧問客戶管理個別的待辦事項。

Slack MCP：掌握 TODO 與 Slack bot。

稅務師工作大概八成的體感是資料的轉記和確認。把 A 的資料搬到 B、把 C 的內容反映到 D，不斷重複。

用 MCP 全部串起來之後，這個轉記作業幾乎歸零。

━━━━━━━━━━━━━━━━━━━━
▍⑥ 排程任務：「忘記」這回事消失了
我現在都用 Claude Code 的排程功能來跑每天的固定流程：
每晚 21:00：freee 自動記帳（幫 60 間公司跑會計分錄）。每晚 22:00：抓取 X（Twitter）的追蹤人數並自動存進 Notion。

人難免會忘事，但 AI 不會。光是能把「不小心漏掉」這件事徹底歸零，那種心理上的踏實感真的差很多。設定也很簡單，告訴 Claude Code「建一個排程任務，每晚 21 點執行 freee 自動記帳」就好。

━━━━━━━━━━━━━━━━━━━━
▍⑦ 用 Skill 累積「業務的型」
Claude Code 的 Skill 功能可以把常用指示定型化並儲存，用斜線指令一鍵啟動。
我做了一些 Skill：
/freee-check → 檢查 freee 的未處理明細。/mtg-followup → 會議後的記錄整理與後續行動建立。/ipo-analysis → 讀取並分析新上市公司的登記簿與有價證券申報書。

稅務師的工作是重複同樣的模式。決算、申報、月報、會議。把這些「型」當作 Skill 累積起來，做得越多就越快。而且 Skill 的內容也是 Claude Code 幫你寫，我只需要告訴它「想自動化什麼」。

━━━━━━━━━━━━━━━━━━━━
▍⑧ CLAUDE.md：給 AI 的「業務手冊」
Claude Code 有一個叫 CLAUDE.md 的檔案，可以寫對 AI 的永久指示。等於是給 AI 的業務手冊。

我的 CLAUDE.md 裡寫了這些：分錄分類規則（餐飲費 1 萬日圓以下 → 會議費，超過 1 萬日圓 → 交際費）。稅務分類規則（海外 SaaS → 通訊費，稅務分類代碼 0）。安全政策（絕對不可輸出 API 金鑰或個人編號）。輸出路徑規則（顧問客戶的資料要放到指定資料夾，不要亂放在 Downloads）。判斷的界線（符合過去模式 → AI 可處理，新出現的 → 先跟我確認）。

寫好之後，Claude Code 每次作業前都會讀取這些指示，能維持處理的一致性。就像新人到職的交接資料。不過跟人不同的是，AI 不會因為重複教同一件事而不耐煩，也不會忘記。

━━━━━━━━━━━━━━━━━━━━
▍⑨ 業務日誌的自動記錄
每當任務完成，Claude Code 會自動記錄到日誌檔案。記錄項目：日期、類別、任務概要、如果手動操作估計要花多少時間、AI 實際花費的時間、節省下來的時間、能不能當作 X 貼文素材。按月彙整成摘要後，就能用數字看出「本月用 AI 自動化了多少小時的工作」。

實際測量下來，每月減少超過 24 小時。換算成工作日約每天超過 1 小時。全年約 300 小時。
看到這個數字，就再也回不去手動作業了。

▍為什麼「後勤專業人員」更應該用 Claude Code，畠山說他最想傳達的是：正因為不是工程師，Claude Code 才能發揮作用。稅務師和會計人員知道「業務的型」。分錄規則、申報流程、月結的檢查重點、對顧問客戶的應對模式。這些實務知識是花了好幾年才累積起來的，AI 無法自己生出來。

Claude Code 缺的正是這種「現場知識」。反過來說，只要有現場知識，程式碼全部可以交給 Claude Code 寫。這次他為了做這套系統所寫的程式碼幾乎是零。全部都是 Claude Code 寫的。他做的只是說「我想要這樣的機制」「這種情況應該這樣判定」。

工程師用 AI 做出「技術上很厲害的東西」。稅務師用 AI 做出「實務上正確的東西」。這個差別非常大。

▍我不僱用員工的理由
「為什麼不僱人？」常被這樣問。答案很簡單，不僱人的好處太大了。

人事成本歸零，顧問費就能壓低，直接回饋給客戶。資訊管理風險大幅降低，一個人資安最簡單，決策速度快，新工具當天就能導入。空出來的時間拿去繼續推進 AI 應用，形成正向循環。當然極限總會到來。但我感受到的是，AI 每個月都在幫我把那個極限往上推。半年前我還覺得 40 家大概是極限。現在 60 家還綽綽有餘。
