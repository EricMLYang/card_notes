# 兩個月 21,000 顆星：Vercel 的 agent-browser 為什麼讓 AI 工程師瘋狂？

---

2026 年 1 月 19 日，Vercel Labs 在 GitHub 上推了一個 repo，叫 `agent-browser`。一個瀏覽器自動化的命令列工具，用 Rust 寫的，專門設計給 AI coding agent 用。

兩個月後，這個 repo 累積了超過 21,000 顆 GitHub star。它的核心 skill 被安裝超過 79,000 次在 OpenCode 上、52,600 次在 Codex 上、51,100 次在 Gemini CLI 上。Claude Code、Cursor、GitHub Copilot、Amp——幾乎所有你叫得出名字的 AI coding agent，都在用它。

但這不是一篇介紹工具的文章。

**這是一篇關於「為什麼測試，而不是寫 code，才是 agentic coding 真正的瓶頸」的文章。** 而 agent-browser 之所以爆紅，恰恰是因為它戳中了這個瓶頸的核心。

---

## AI 說「完成了」，然後呢？

如果你在 2026 年初用過任何一款 AI coding agent 開發前端功能，你一定經歷過這個場景：agent 花了三分鐘幫你寫好一個元件，信心滿滿地說 "done"。然後你打開瀏覽器，點了兩下，發現按鈕根本沒反應。

這不是模型能力的問題。這是**驗證迴圈（validation loop）的問題**。

Anthropic 在 2026 年初發布的 Agentic Coding Trends Report 裡提到，TELUS 團隊用 AI agent 節省了超過 50 萬小時的工程時間，Zapier 達到 89% 的全組織 AI 採用率，Stripe 的 agent 每週送出超過 1,000 個 PR。這些數字很漂亮。但報告裡沒有大聲說出來的另一面是：**當 agent 每週產出 1,000 個 PR，誰來驗證這 1,000 個 PR 沒有搞壞東西？**

傳統軟體開發有一個運作了幾十年的安全網：CI/CD pipeline 裡的自動化測試。Agent 寫完 code，跑一遍單元測試和整合測試，如果全過就合併。問題是，對於大多數**既有專案（brownfield project）**，這個安全網根本不存在——或者千瘡百孔。

這才是真正的瓶頸。

---

## Brownfield 的測試債：agentic coding 的阿基里斯腱

Greenfield 專案很好辦。你讓 agent 從第一天就把測試寫進去，建立乾淨的 TDD 迴圈，一切井然有序。

但現實世界裡，絕大多數軟體團隊面對的是 brownfield——已經跑了三年、五年、十年的產品，背著十幾萬行的技術債，測試覆蓋率可能只有個位數百分比。這些專案不是不想有測試，而是補測試的成本高到沒有人願意優先做。

問題是，**agentic coding 把這個技術債的代價放大了十倍。**

為什麼？因為 agentic workflow 的核心邏輯是一個迴圈：**Plan → Execute → Verify → Iterate。** Agent 規劃變更、執行修改、驗證結果、根據反饋再修改。這個迴圈轉得越快、人類介入越少，產能提升就越大。

但「Verify」這一步，需要有東西可以跑。如果沒有自動化測試，Verify 這一步就只能靠人類打開瀏覽器手動點擊。**迴圈斷在這裡**。Agent 寫了 code，等人類測試；人類測完回報問題，agent 再改；改完又等人類測。這不是 agentic workflow，這是一個有 AI 幫忙打字的傳統開發流程。

所以我的判斷是：**如果你的團隊正在導入 agentic coding，第一優先不是挑選哪個 agent 最聰明，而是趕快把 E2E testing 基於產品原始定義的 user journey 給捕捉起來。**

這裡說的不是追求 100% 測試覆蓋率。那不現實，也不必要。關鍵是把產品最核心的使用者旅程——註冊、登入、結帳、搜尋、設定變更——用 end-to-end test 鎖住。這些測試就像是一組護欄，讓 agent 在迴圈裡自由奔跑的時候，不會衝下懸崖。

護欄就位之後，每一次 iterate 都可以讓 agent 獨立完成：改 code、跑測試、看結果、再改。**人類從「每一步都要檢查」變成「定期 review PR」。** 這才是 agentic workflow 真正的加速點——不是讓 agent 寫得更快，而是讓整個迴圈不需要人類擋在中間。

## 當 Playwright 遇上 AI：一場 context window 的戰爭

好，你同意測試很重要。那工具呢？

過去兩年，Playwright 幾乎是瀏覽器自動化的同義詞。它強大、靈活、生態系成熟。但當你試著把 Playwright 塞進 AI agent 的工作流程時，一個致命的問題浮現了：**它不是為 AI 設計的。**

Playwright MCP（Model Context Protocol）在每次 session 啟動時，需要把 26 個以上的 tool 定義載入 agent 的 context window，成本大約 13,700 個 token。Chrome DevTools MCP 更誇張，要 17,000 個 token。這代表什麼？agent 還沒開始做任何事情，光是「知道自己可以做什麼」就已經吃掉了 200K context window 的將近 9%。

對於短任務，這無所謂。但 agentic workflow 的核心是長程、多步驟的自動化——填表單、切換頁面、擷取資料、驗證互動、記錄結果。每一步操作，Playwright 都會回傳完整的 DOM accessibility tree，動輒數千行。Agent 還沒搞清楚該做什麼，就已經被 HTML 淹沒了。

**agent-browser 做了一件根本性不同的事：它不走 MCP，它走 CLI。**

Agent 透過 shell command 操作瀏覽器。`agent-browser open example.com`、`agent-browser snapshot -i`、`agent-browser click @e2`。每個頁面的快照只回傳 accessibility tree 裡的互動元素，用簡潔的 ref 標記（@e1、@e2、@e3），一個典型頁面大概 200 到 400 個 token。

Vercel 自己的 D0 text-to-SQL 專案驗證了這個設計哲學的威力。他們原本用 17 個專用 tool，成功率 80%，平均耗時 274.8 秒，消耗約 102,000 個 token。最差的案例花了 724 秒、燒掉 145,463 個 token，還是失敗。後來他們砍到只剩 2 個 tool——ExecuteCommand 和 ExecuteSQL。成功率跳到 100%，平均耗時 77.4 秒（快 3.5 倍），token 消耗降到約 61,000（減少 37%）。

這不只是效能優化。這是一個深刻的設計洞見：**給 AI agent 的工具，不是越多越好，而是越精準越好。** 少即是多。Unix pipeline 哲學在 2026 年以一種完全意想不到的方式回來了——每個命令做好一件事，組合起來完成複雜任務。

## Dogfood：讓 agent 像人一樣測試，用機器的速度

在這個基礎之上，agent-browser 推出了一個叫 **dogfood** 的 skill。這是我認為最值得關注的部分。

你對任何支援 skill 的 AI agent 說一句 `dogfood vercel.com`，它就會自己啟動瀏覽器，系統性地走過整個 app，像一個資深 QA 工程師一樣逐頁點擊、填表單、開 dropdown、測邊界條件、檢查 console error，然後把每一個發現的問題寫成結構化報告，附上截圖和重現影片。

它的工作流程分六步：初始化環境、處理登入驗證、拍全局截圖理解 app 結構、系統性探索所有頁面和互動元素、即時記錄問題、收尾彙整報告。

幾個設計決策很聰明。

**證據分級。** 互動類的 bug（按了按鈕東西壞了），dogfood 要求完整重現：先開始錄影，每步之間停頓 1 秒讓人類看得懂，逐步截圖，最後停在壞掉的畫面上多停 2 秒。靜態類問題（錯字、排版跑掉），一張標注截圖就夠。這種區分省了大量 agent 的 token 預算和時間。

**即時記錄。** 每發現一個問題當場停下來寫進報告，不是全部掃完再回頭整理。因為 session 隨時可能中斷，先寫先贏。

**永遠不讀原始碼。** Skill 裡明確規定：你是以使用者身份測試，不是做 code review。這把 dogfood 的角色框得很清楚——它測的是「用起來」的體驗，不是「寫得對不對」的邏輯。

**先驗證再收集證據。** 在開始錄影或截圖之前，先重試一次確認問題確實可重現。這是從實際使用中學到的教訓——有個團隊發現 agent 花了大量 turn 在記錄一個最後被證明是誤報的問題上。

## 真正的範式轉移：Agent-native 工具

退一步看更大的圖景。

過去十年，軟體開發的工具鏈都是**為人類設計的**。VS Code 有漂亮的 GUI，Playwright 有直覺的 API，Jira 有拖拉的看板。這些工具假設使用者是一個坐在螢幕前、用滑鼠和鍵盤操作的人類。

但當 agent 成為開發流程中的主要執行者，這個假設就不成立了。Agent 不需要 GUI，它需要**結構化的文字輸出**。Agent 不需要 26 個 tool 定義，它需要**精簡到極致的命令集**。Agent 不需要漂亮的截圖，它需要**用最少 token 傳達最多資訊的 accessibility tree snapshot**。

**agent-browser 的核心洞見是：AI agent 需要專屬於自己的工具，而不是人類工具的 wrapper。**

這就是為什麼它用 CLI 而不是 MCP。這就是為什麼它用 Rust 寫核心（快速的 daemon 常駐在背景，不需要每次操作都重新啟動瀏覽器）。這就是為什麼它把 snapshot 精簡到 200 個 token。每一個設計決策，都是在回答同一個問題：**怎麼讓 agent 在有限的 context window 裡，最有效率地完成瀏覽器操作？**

有趣的是，Google 的 Chrome DevTools 團隊現在也開始開發自己的 CLI 工具。當 MCP 領域最大的玩家開始轉向 CLI 介面，這本身就說明了一件事：**agent-native 工具，正在成為新的標準。**

## 把迴圈接起來：agentic workflow 的完整閉環

回到文章開頭的核心論點。

Agentic coding 的效能上限，取決於**整個迴圈能多快轉完一圈，而且中間不需要人類介入**。寫 code 這一步，LLM 已經做得夠好了。部署這一步，CI/CD 已經自動化了。但「驗證 code 有沒有搞壞東西」這一步，長期以來卡在人類和不完整的測試套件之間。

agent-browser 和 dogfood skill 的意義在於，它們開始填補這個缺口。不是完美地填補——dogfood 每次跑大概找 5 到 10 個問題，它不會替代完整的 QA 流程——但它讓 agent 可以在寫完 code 之後，**自己打開瀏覽器，自己跑一遍核心 user journey，自己確認沒有明顯的 regression**。

把這件事的意義放大來看：

**對 brownfield 專案來說，正確的導入順序應該是這樣的——**

先用 agent-browser 這類工具，快速捕捉產品核心 user journey 的 E2E 測試。不需要完美，不需要覆蓋每一個邊界條件，先把「使用者最常走的路」錄下來。這些測試變成 agent 的護欄。

接著，把 agentic workflow 的每一個 iterate 迴圈盡量設計成不需要人類介入。Agent 拿到任務、改 code、跑 E2E 測試、如果失敗就自己修、修完再跑一次。只有當 agent 連續失敗、或者改動範圍超出測試覆蓋時，才 escalate 給人類。

最後，人類的角色從「逐行審查 code」變成「設計護欄、定義品質標準、處理 agent 解決不了的問題」。這不是被取代，這是被升級。

## 台灣的機會與結構性挑戰

台灣能從這個趨勢裡學到什麼？

先看一個結構性事實：台灣絕大多數的軟體公司，特別是新創和中小型團隊，**根本沒有專職 QA**。工程師自己寫、自己測、自己部署。測試覆蓋率？很多團隊連這個指標都沒有在追蹤。

這意味著 agentic coding 對台灣的衝擊可能比矽谷更劇烈。矽谷的大型軟體公司通常已經有完整的 CI/CD pipeline、有 QA 團隊、有測試基礎設施。他們導入 agentic workflow 是在既有基礎上加速。但台灣很多團隊的起點不同——他們連基礎護欄都還沒建好。

弔詭的是，這反而可能是一個機會。正因為沒有沉重的 legacy 測試框架要維護，台灣團隊可以直接跳到 agent-native 的測試工具。不用先花三個月寫 Playwright 腳本，直接用 agent-browser 的 dogfood skill 開始捕捉核心 user journey。不完美，但夠用。

更關鍵的是工程文化的轉變。台灣工程師長期以來習慣「全端包辦」，從前端到後端到部署到測試一個人搞定。這種模式在 agentic coding 時代反而是優勢——因為你最了解自己的產品，你最知道哪些 user journey 是核心、哪些邊界條件最容易出事。**你不需要成為 QA 專家，你需要成為「agent 的教練」**：告訴它要測什麼、怎麼判斷成功或失敗、什麼情況下要通知你。

## 這場賽跑的終點在哪裡？

2026 年 2 月 3 日，Apple 發布 Xcode 26.3，正式支援 agentic coding。Claude Agent 和 Codex 可以在 Xcode 裡搜尋文件、探索檔案結構、修改專案設定、甚至截取 Xcode Preview 來視覺驗證自己的成果。同一時間，GitHub 推出 Agentic Workflows，把 AI agent 嵌入 CI/CD pipeline。Anthropic 的報告預測，2026 年的定義性挑戰不是 AI 能不能參與軟體開發，而是組織怎麼刻意地為 AI 參與而設計流程。

軟體開發正在經歷自 GUI 以來最大的範式轉移。工程師的角色從「寫 code 的人」變成「協調 agent 的人」，從直接生產轉為系統設計與品質把關。

但在這個宏大敘事底下，有一個很務實的問題：**你的 agentic workflow 迴圈裡，「Verify」那一步到底靠什麼在跑？**

如果答案是「靠我自己手動點」，那你的 agent 再聰明也快不了多少。如果答案是「我們測試覆蓋率還很低」，那現在就是補起來的時候——不是用傳統方式花三個月寫測試，而是用 agent-native 的工具，先把最重要的路徑鎖住。

Vercel 用 agent-browser 在兩個月內拿下 21,000 顆星，不是因為它解決了一個很酷的技術問題。是因為它解決了一個很痛的實際問題：**AI agent 終於可以自己驗證自己的工作了。**

這場 agentic coding 的軍備競賽，最後可能不是比誰的模型更聰明，而是比誰的迴圈轉得更快、更完整、更不需要人類擋在中間。而測試——那個長期以來被工程師視為苦差事的環節——正在成為整個迴圈裡最有價值的一環。
