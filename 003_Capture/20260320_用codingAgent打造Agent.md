今天貼點有意思的 side project: MUD + Agent... 我為了研究虛擬角色 Agent 的開發, 打造了 MUD Like 虛擬世界來養 Agent 的故事 XD

我覺得 coding agent (我用通用名詞, 不用 claude code, 因為各家的 coding agent 其實都有一定的水準了) 大致上已經是公認打造 agent 最理想的做法了, 看看 claude code 那麼多樣的衍伸應用, 連 openclaw 也是基於他所創造出來的開源專案, 大致上沒有懸念了

我關注的重心, 在於它如何進一步變成能 embed 在我的應用服務內的 "常駐" agent, 因此從兩年多以前, 我就不斷的嘗試 Semantic Kernel, Lang Chain, ADK 這類開發框架來延伸產品的處理能力...

這次我把重心移到:

1. 用 coding agent 來自訂我要的 agent 行為 ( 包含 skill, tool, sandbox 等 )
2. 用 SDK 將 agent 與我的應用程式高度整合

實際要動手時, 才發現問題沒那麼簡單 XDD, 平常用 coding agent 用得很開心, 因為它直接用自然語言當介面, 直接對應到任務執行 (coding) 了。當你要整合在應用程式內的時候, 操作方式就完全不同了, 你得面對這些狀況:

1. 操作方式不同
你的輸入輸出可能不再是單純的 "對話", 而是透過其他方式, 輸入輸出會是結構化的資料, 而非自然語言

2. 觸發方式不同
通常 coding agent 都是默默地等待你的指令, 而應用程式則不一定, 可能需要透過你的操作驅動, 或是在背景等待事件驅動 (Event Driven), 或是靠定期排程驅動 (Heartbeat)

3. 緊急程度不同
有些狀況需要快速回應, 你指定的 SKILL / Tools, 甚至 Model 都需要選擇, 重要的任務則需要深度思考判定才能回應, 然而這些不同狀況的需求可能又共享同樣的 context / knowledge ..

這些狀況其實不大好下手啊, 於是我決定先簡化問題, 建立一個單純的模擬環境, 在裡面驗證我想像的 Agent 開發情境。於是我決定自己花點下班的念書時間, 來嘗試一下這個 side project: MUD + Agent ..., MUD 我用我熟悉的 .net 來開發, 而 Agent 我用 GitHub Copilot SDK 來開發, 人物設定就是 wiki 抓來的簡介, 存成 .md 檔, 丟在各自角色的 workspace 內...
