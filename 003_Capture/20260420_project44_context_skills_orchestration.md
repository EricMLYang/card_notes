## 摘要
project44 在 decision44 發表一組供應鏈 AI agent，背後主打的不是模型，而是三層基礎：context、skills、orchestration。它把 context 定義成來自全球物流網路與即時事件流的 operating picture，把 skills 定義成「明確工作 + 經驗證資料 + semantic meaning」，再用 orchestrator 管理各 agent 的順序、升級與稽核線索。這其實是在用產品語言重述一個你熟悉的命題：agent 成敗主要取決於 domain context 與 execution design。

## 為什麼值得看
這篇值得你看，不是因為 project44 的 use case 本身，而是它把 Decision Intelligence × agent 的商業敘事講得很完整。它幾乎在公開告訴市場：單靠模型升級無法複製 domain context、semantic precision 與 orchestration。這與你的 DecisionOps 語言高度相容，尤其適合作為「為什麼 agent 不等於 chatbot，也不等於單一模型能力」的案例。

## 可能偏誤或限制
這是公司 press release，業務敘事很強，數據與成效由公司自行陳述。文中提到的成長與效益沒有完整方法說明，也缺少失敗案例。它仍然很有訊號價值，但比較適合拿來觀察 category messaging，而不是直接當成中立研究結論。

## 重點譯摘
- project44 明確說，多數 enterprise agent 失敗不是模型太弱，而是底層基礎不夠。
- 它把可複製的核心拆成三層：context、skills、orchestration。
- 所謂 skill，不只是 task automation，而是結合任務範圍、驗證資料與語意定義後，才足以支撐決策。
- orchestration 的角色像資深營運主管：決定誰先做、資訊怎麼傳、何時升級與如何留 audit trail。
