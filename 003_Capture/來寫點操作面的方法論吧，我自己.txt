來寫點操作面的方法論吧，我自己在過去幾個月用 AI 會作的幾個我覺得算核心的判斷跟事情：

1. 通常我會先判斷這個事情處於什麼狀態，分列如下：

a. 從無到有的起步階段，比方說我要作個報名系統，但我現在啥都沒有的狀態，這時候根據我人類的知識，我會知道我需要一個我會知道我需要一個伺服器、一個資料庫、一個網站。

這是比較容易的，我相信多數新進來的夥伴們應該都在這個階段，但這個階段其實是個坑，難題並不在這個階段。

那這時候我會拆幾個角度處理：
i. 先蓋資料庫跟後端，叫他先幫我寫一個 前後端後端資料庫的整體概略規劃

ii. 寫完之後看一下資料庫結構是不是我喜歡的，不是，我從資料庫先改。(data first ，跟一般專案一樣，因為資料怎麼存本質上就決定了系統要怎麼寫的一半）

iii. 再來評估看看我要不要給他後端跟前端的框架（最近是蠻常在 python 跟 c# 選一個的，有趣的是 python 如果不是AI的話其實不會在我選項內）

iv. 再來是前端（或app）的框架，現在app 我是肯定都用 react native，前端還在 react 跟 vue 之間測試，各有討厭的地方。

v. 再來 local 環境，通常我會寫 docker compose 

vi. 再來驗證邏輯

我會切主線驗證，把70％核心的流程（比方說 註冊/報名/事後查詢報名）作為主場景先驗，至於 form validation ，退款、取消報名等負向流程其次，最後是管理流程（如後台系統跟取消報名這些）放最後。

基本上先抓絕對會死的部分，放掉錯誤會有點丟臉（但不至於失敗）的事情。隨著時間慢慢分層逼近。

vii. 建制主機，這個我通常不太選server less，主要是我個人偏好問題，我偏好用VM解決。

viii. 寫 ci ，反覆驗證幾次

其中這裡面，我會認真盯著的，跟我會盡量選擇最好的模型的，是 1,2,6 。其他的部份我基本上都抱著隨便他作，結果看過沒問題就行的。這段 sonnet 啥的我不太在意。

然後 api 面我基本上會盡量先把 input /output 先讓他弄好，確保輸出入的流程跟我想的一樣，而這點也是非常倚賴我個人過去的開發經驗跟知識。

所以談流程的時候，其實人類本身的回饋能力、回饋速度，依舊是體系中重要的一環。當然你說人不參與行不行，可以，他其實還是有能力跑出一堆東西。

但對我來說，我就很難有能力去監督裡面是否失控，所以我個人不會這樣選。

但能不能讓有能力的人事後進來透過 AI audit 呢，這個我倒也常幹，不是不行。就是你要留意中間往返的時間划算不划算。

我目前測起來是前期高度介入還是非常划算的狀態。

這是從無到有的情況，也是比較容易的情況。但是現實世界，通常不是這樣的，會有別的情況。比方說從十年老系統開始的情況。
