# 從寫作型知識庫到外部認知系統：repo 升級計畫

日期：2026-03-17  
參考草稿：`015_Write/Draft/20260317_從軟體開發流程思考_知識管理只是在幫寫文章_是否太浪費了.md`

## 一句話判斷

這個 repo 已經有不錯的「Capture -> 卡片 -> 寫作」骨架，但如果要回應草稿中的核心追問，下一步不應只是再增加寫作 prompt，而是把 repo 升級成能承載判斷規則、工作流程、失敗模式、專案記憶與自我校正的外部認知系統。

## 目前 repo 的優勢

1. 已經有清楚的目錄結構：`PARA + CardNotes + Write`。
2. 已經有個人化設定：`000_MyContext/` 內含 `capture_profile.md` 與 `break_cards_profile.md`。
3. 已經有 skill / agent / prompt 生態，可作為未來能力編排入口。
4. 卡片系統已有分類索引，具備知識積累與重組基礎。
5. 使用者偏好已明確偏向「可操作、可取捨、可驗證」的高訊號內容。

## 目前最值得處理的缺口

### 1. repo 的重心仍偏向「內容輸出」

目前主流程仍以整理、拆卡、寫作為主。這很有價值，但距離「可被 AI 穩定調用的判斷系統」還差幾層：

- decision rules
- workflow templates
- failure patterns
- project memory
- evaluation rubrics
- experiment logs

### 2. 缺少「情境與邊界」作為一級資產

現有卡片多為高品質概念卡，但未必都具備以下欄位：

- 何時適用
- 何時不適用
- 觸發訊號
- 反例
- 驗證方式
- 會影響哪些決策

如果沒有這層，AI 比較容易幫忙重述觀點，不容易幫忙做判斷。

### 3. 缺少對「實際自己」的記錄

草稿裡最重要的一點是：知識庫不應只保存理想化的自己，也要保存真實決策偏誤、常犯錯誤、在什麼情境下會失真。  
這一層目前還沒有明確結構承接。

### 4. 缺少閉環與 eval

如果 repo 想從知識庫升級成外部認知系統，就不能只有收集與輸出，還需要：

- 事前推演
- 事中檢查
- 事後回收
- 成效評分
- 規則更新

### 5. 指令層已有漂移風險

`AGENTS.md` 內列出的 skills 與 `.github/skills/` 實際內容已有落差。例如：

- `process-inbox`、`teardown-article`、`sync-agent-instructions` 出現在 `AGENTS.md`
- `.github/skills/` 內實際存在的是 `enrich-capture`、`copilot-sync`、`repo-healthcheck`、`refine-cards` 等

這代表 repo 的規範層與實作層開始不同步。若不先修，未來新增能力會持續累積維護成本。

## 升級北極星

把這個 repo 從「方便整理與寫作的知識庫」，升級成「可被 AI 檢索、組合、驗證、反駁、追蹤結果的外部認知系統」。

更具體地說，未來希望它不只支援：

- 寫文章
- 找卡片
- 產生題目

而能進一步支援：

- 專案前的風險推演
- 架構與商業取捨評估
- 學習缺口偵測
- 專案過程中的工作流提醒
- 事後回顧與規則修正
- 針對使用者盲點的反向提醒

## 建議的六條開發主線

## A. 先修 repo 的 instruction drift

目標：讓規格、技能、目錄、命名重新對齊，避免未來 agent 行為越來越不可靠。

建議動作：

1. 盤點 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md`、`.github/skills/` 的對應關係。
2. 建立「技能註冊表」文件，明確列出：
   - skill 名稱
   - 實體路徑
   - 觸發語句
   - 輸入輸出
   - 維護狀態
3. 把不存在或已改名的 skill 全數修正。
4. 為 repo health 增加定期檢查項：指令文件是否引用不存在的檔案或 skill。

這條主線是第一順位，因為它是後續所有自動化的基礎。

## B. 新增「可調用 context」的資產類型

目標：讓 repo 儲存的不只是概念，而是能參與思考與決策的結構。

建議新增一層資產類型，先不一定要大改現有主目錄，但至少要明確定義模板。

優先資產類型：

1. `decision_rule`
   - 一條判斷規則
   - 包含適用情境、前提、反例、代價、觀察訊號
2. `evaluation_rubric`
   - 用來評估提案、文章、專案、架構選項
3. `workflow_template`
   - 把可重複工作流程標準化
4. `failure_pattern`
   - 常見失敗模式、誤判訊號、補救方式
5. `project_memory`
   - 記錄專案脈絡、重要決策、未解問題、已驗證假設
6. `experiment_log`
   - 記錄試驗、觀察、結果、下一步

關鍵原則不是多建資料夾，而是讓這些資產變成一級物件，而不是散落在文章或卡片裡。

## C. 升級卡片格式，讓卡片能參與判斷

目標：保留原本卡片系統，但增加足夠 metadata，讓 AI 能更可靠地調用。

建議為新卡片或高價值卡片逐步加入可選欄位：

- 類型：principle / rule / anti-pattern / checklist / rubric / case
- 適用情境
- 不適用情境
- 觸發條件
- 依賴前提
- 驗證方式
- 關聯決策
- 信心等級
- 來源性質：一手經驗 / 外部資料 / 推論整理

目前卡片已具備不錯的概念、重要性、邊界結構，這是優勢。下一步是補上更強的「可運算欄位」。

## D. 把入口從「主題分類」升級成「任務路由」

目標：讓 repo 不只是按主題找資料，而是按工作情境調用資產。

現況強項是主題索引，例如：

- `03_ai_applications_and_productization.md`
- `09_software_engineering.md`
- `13_writing_communication_and_expression.md`

下一步建議增加「任務型入口」，例如：

- 我要做架構評估
- 我要準備提案
- 我要開會前整理風險
- 我要拆解學習缺口
- 我要回顧專案失敗原因

實作方式可以是：

1. 在 `010_CardNotes/01_Index/` 外，再新增任務索引。
2. 或在 `.github/prompts/` / `.github/skills/` 層建立 task router。
3. 或在 `000_MyContext/` 定義個人常見工作情境與對應資產路由。

這一步會直接把 repo 從知識分類器推向工作系統。

## E. 建立 eval 與自我校正閉環

目標：避免 Personalized AI 只會越來越像你，卻不一定越來越接近現實。

建議新增一條最重要但現在缺席的閉環：

1. 事前：用既有規則推演風險與選項
2. 事中：根據 workflow / rubric 做檢查
3. 事後：記錄結果、偏差、反例、失敗模式
4. 更新：修正 decision rule 與 rubric

優先做法：

1. 為專案新增簡短 postmortem 模板。
2. 為關鍵判斷新增 prediction / outcome 對照欄位。
3. 建立「我常犯的錯」清單與案例索引。
4. 建立「哪些原則最近失效」的季度回顧機制。

這會讓 repo 開始具備 personal eval system 的雛形。

## F. 將現有 skills 從「內容處理」擴展到「決策輔助」

目標：讓 skill 生態從寫作助手，往工作與判斷助手擴張。

優先可以新增或改造的能力：

1. `decision-review`
   - 對提案、架構、策略做 trade-off 檢查
2. `project-memory`
   - 自動整理專案脈絡、決策紀錄、未解問題
3. `premortem`
   - 在行動前先列出風險與失敗路徑
4. `retrospective-eval`
   - 專案結束後更新 failure patterns 與 rules
5. `workflow-orchestrator`
   - 根據情境切換 prompt、checklist、模板
6. `bias-check`
   - 扮演不盲從使用者的反駁角色

這些能力比新增更多寫作 prompt 更貼近草稿中的長期方向。

## 建議的實作順序

## Phase 1: 打底期（1-2 週）

目標：先把 repo 的規範層與資產模型穩住。

交付物：

1. 修正 `AGENTS.md` 與 `.github/skills/` 的對應關係。
2. 新增技能註冊表與 repo health checklist。
3. 定義 5-6 種新資產模板。
4. 為卡片補一版最小必要 metadata 規格。

## Phase 2: 結構化期（2-4 週）

目標：讓高價值內容開始被轉成可調用 context。

交付物：

1. 從現有高品質卡片中挑 20-30 張做結構升級。
2. 為 2-3 個進行中專案建立 `project_memory`。
3. 為 1-2 個常見工作情境建立任務型入口。
4. 新增至少一個 `decision-review` 或 `premortem` skill。

## Phase 3: 閉環期（4-8 週）

目標：讓 repo 開始具備校正能力，而不只是儲存能力。

交付物：

1. 建立 postmortem / experiment log / failure pattern 的回收流程。
2. 新增季度回顧：哪些規則有效、哪些規則失效。
3. 建立 prediction vs outcome 的簡單追蹤機制。
4. 讓至少一條 workflow 可由 agent 協助完整跑完。

## Phase 4: 編排期（8 週以上）

目標：把 repo 變成真正可工作的外部認知系統。

交付物：

1. 任務型 router 成形。
2. 技能會依專案階段切換使用不同模板與 context。
3. Agent 不只會順著使用者，也能在關鍵處提出反例、風險與證據缺口。
4. 知識系統開始支援實際專案推進，而不只是文章產出。

## 建議先做、但先不要做太大的事

先做：

- 修正 instruction drift
- 定義資產模板
- 為高價值卡片補 metadata
- 補上 project memory 與 postmortem
- 做一個能用的 decision-review skill

先不要急著做：

- 大規模重整所有資料夾
- 一次改寫所有舊卡片
- 為每種情境都建立 agent
- 追求過度自動化

先讓 1-2 條工作流真的跑通，比建立一整套華麗結構更重要。

## 建議用來判斷是否真的進步的指標

不是看：

- 卡片數量
- prompt 數量
- skill 數量
- 文章產量

而是看：

1. 新專案開始時，能否更快拉出風險、假設與待驗證點。
2. 同類型錯誤是否減少重複發生。
3. 是否能更快把閱讀內容轉成 decision rule / checklist / workflow。
4. AI 是否開始能在關鍵地方提出有價值的反駁，而不只是附和。
5. repo 是否開始支援真實工作推進，而非只支援寫作輸出。

## 最後的判斷

這個 repo 下一階段最值得做的，不是把「第二大腦」做得更大，而是把它做得更像一個有記憶、有路由、有規則、有評估、有自我修正能力的工作系統。

如果只把知識管理做到方便寫文章，已經不差；但若要真正吃到 AI 時代的複利，repo 應該開始保存與調用的，不只是內容，而是你的判斷方式、工作方法、失敗教訓與修正迴路。
