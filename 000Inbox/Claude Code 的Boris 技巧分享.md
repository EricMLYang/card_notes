Claude Code 的Boris 休假就有時間寫長文了？以下是他剛分享的 Claude Code 使用技巧整理。

👉

你可以把 Claude Code 當成一套工作系統，而不只是一個聊天視窗。這套系統的目標很明確：同時推進多條任務線，每條都有清楚的規格與驗證方式，而且你累積的規則會持續生效，不必每次從零開始。Boris 這次分享的核心就在這裡。每個團隊的用法不盡相同，所以你要做的是多嘗試幾次，再把最順手的流程固定下來。

第一個關鍵是平行處理。 團隊習慣同時開啟 3 到 5 個獨立的工作階段，搭配 git worktrees，讓多項任務同步推進。你可以把一個工作階段用來做功能實作，另一個用來重現與修 bug，再留一個做重構或清理技術債，甚至再加一個專門跑資料查詢與指標驗證。這樣做的價值在於把等待時間全部轉換成有效產出。很多人會替每個 worktree 命名，並設定 shell alias，一鍵切換到不同任務線。也有人保留一個專門用於分析的 worktree，查 log、跑 BigQuery 查詢，避免搞亂開發用的上下文。相關工作流程文件在這裡。

第二個關鍵是複雜任務一定先從計畫模式開始。 你把心力花在規劃上，Claude 就比較容易一次到位。團隊有人會先讓一個 Claude 寫計畫，再開第二個 Claude 用資深工程師的角度審查，挑出設計漏洞、遺漏的驗證、風險與取捨。更重要的是，一旦發現方向不對，就要立刻切回計畫模式重新規劃，不要硬撐著做下去。團隊也會要求 Claude 在驗證步驟時進入計畫模式，先規劃要跑哪些測試、要比對哪些行為差異、回歸範圍應涵蓋到哪裡，避免做到最後才臨時亂試。

第三個關鍵是用心經營 CLAUDE.md，讓錯誤不再重複。 團隊很實用的一句話是：每次你糾正 Claude 之後，就要求它更新自己的 CLAUDE.md，確保下次不再犯同樣的錯。Claude 在替自己寫規則這件事上表現很好，你等於是把「教它一次」變成「永久降低錯誤率」。隨著使用時間拉長，你也要毫不留情地刪改精簡 CLAUDE.md，讓規則保持簡短、可檢查、可執行。有些工程師會讓 Claude 為每個任務或專案維護一個 notes 目錄，每次 PR 後更新，再從 CLAUDE.md 引用 notes。這樣 CLAUDE.md 維持乾淨，細節留在 notes 裡，閱讀與維護成本低很多。

第四個關鍵是把高頻工作封裝成 skill 或 slash command，並 commit 到 git，跨專案重複利用。 團隊的判斷標準很直接：如果一天會做超過一次，就應該封裝成 skill 或指令。例如做一個 /techdebt 指令，每次工作階段結束都跑一次，找出重複程式碼與可以合併的地方並清掉。也可以做一個彙整上下文的指令，把過去 7 天在 Slack、Google 文件、Asana、GitHub 的重要變更整合成一份上下文資料，讓你在開工前快速對齊狀態。還有人打造類似 analytics engineer 風格的 agent，用來撰寫 dbt models、審查程式碼，並在開發環境測試變更，讓流程更像一條可重複執行的產線。

第五個關鍵是「多數 bug 讓 Claude 自己修」，你只要下達結果。 團隊的做法是把 Slack 上的 bug 討論串整段貼給 Claude，只說「修」，讓它自己讀上下文、找重現步驟、提出修法。或者你直接說「去修好那些失敗的 CI 測試」，不要指揮每一個步驟。這種做法的重點在於，你要把成功條件講清楚（例如 CI 全綠、特定測試通過、行為比對一致），其餘細節交給 Claude 自行收斂。團隊也提到讓 Claude 看 docker log 來排查分散式系統問題，它在這方面的表現常常超出預期，因為它能快速把零碎線索串成推論路徑。

第六個關鍵是提升 prompting 技巧，把 Claude 變成你的 reviewer 與證明機器。 你可以直接挑戰它，要求嚴格審查變更，在通過測試之前不准發 PR。你也可以要求它「證明給我看這個是對的」，讓它比對 main 與 feature branch 的行為差異，逼它把驗證過程說清楚。當修復結果不理想時，你可以要求它用目前掌握的資訊砍掉重練，給出更乾淨、更優雅的方案。還有一個很實際的原則：交付工作之前先寫好詳細規格，越具體越好，模糊空間越少，產出越穩定。

第七個關鍵是終端機與環境設定，讓你在多個工作階段之間切換時不失焦。 團隊很愛用 Ghostty，看重的是同步渲染、24 位元色彩與完整 Unicode 支援，長時間使用的穩定度更好。為了在多個 Claude 之間切換，團隊也會用 /statusline 自訂狀態列，把 context 使用量和目前的 git branch 顯示在畫面上，避免在錯的分支改錯的東西。還有一個常被低估的技巧：語音輸入。說話速度通常比打字快，內容也更容易完整，對於撰寫規格、描述重現步驟、列驗證清單很有幫助。在 macOS 上，連按兩下 fn 即可啟用語音輸入。更多設定資訊在這裡。

第八個關鍵是 subagent 的使用方式。 你可以在任何希望 Claude 投入更多運算資源的請求後面加上「use subagents」，讓它用更完整的分工去處理問題。你也可以把個別任務分派給 subagent，讓主 agent 的 context window 保持精簡聚焦。例如把「掃描 repo 找相關程式碼」、「整理變更點」、「列回歸清單」交給 subagent，主 agent 只保留決策資訊與結論。團隊也提到利用 hook 把權限請求轉交給 Opus 4.5 做攻擊行為檢查，再自動放行安全的請求。相關文件在這裡。

第九個關鍵是資料分析。 團隊讓 Claude Code 透過 bq CLI 隨時撈取與分析指標，並把 BigQuery skill commit 到 codebase，讓每個人都能直接在 Claude Code 裡跑查詢。你可以把這個概念擴大：只要你的資料庫或分析平台有 CLI、MCP 或 API，就能用同樣的方式把查詢與分析交給 Claude，自己專注在指標定義、判讀與決策。要讓這套流程穩定運作，你需要要求它把查詢條件寫清楚、結果附上可重跑的描述，並確保指標口徑一致，避免每次分析都像在猜。

第十個關鍵是把 Claude 用在學習上，讓每次改動都變成你能累積的理解。 團隊會在 /config 啟用「Explanatory」或「Learning」輸出模式，要求 Claude 說明每次修改背後的原因。你也可以讓 Claude 產生視覺化的 HTML 簡報，解說你不熟悉的程式碼，或請它畫 ASCII 圖表說明協定與架構，快速補齊腦中的模糊區塊。這些做法的共同點是：把你原本只得到「改好了」的結果，變成「你知道為什麼要這樣改」的理解。

把這些要點串成一個日常節奏，就會有很強的落地感。開工先用計畫模式寫清楚目標、範圍與驗證方式，再用 3 到 5 條平行任務線推進功能、bug、重構、分析。遇到卡關就回計畫模式縮小問題，修正一次就更新 CLAUDE.md，並把高頻流程封裝成 skill 或 slash command。最後用 /techdebt 之類的指令收尾，把技術債留在今天清掉。
