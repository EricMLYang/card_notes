# 當 TDD 不再是品質保證，而是 AI 的操控介面

> 來源：回應 Simon Willison 關於 Red/Green TDD 用於 coding agent 的文章
> 核心主張：軟體開發的價值正在從「生成程式碼」遷移到「定義驗收條件」

## 表層技巧 vs 深層結構變化

Simon Willison 上週寫了一篇關於 Red/Green TDD 用在 coding agent 上的短文，核心主張很直覺：先寫測試、確認失敗、再讓 AI 寫到通過，這樣 coding agent 就不會亂寫一通。文章寫得清楚，觀察也對。但我覺得它卡在一個常見的盲點：它把 TDD 當作一種「品質保證手段」在講，卻沒追問一個更根本的問題，**當你用測試來驅動 AI 寫程式，你其實正在重新定義「寫程式」這件事本身的價值結構。**

表面上看，這是一篇關於 prompt engineering 的技巧文。但從產業經濟的角度看，這更像是一個訊號：**軟體開發的價值正在從「生成程式碼」遷移到「定義驗收條件」。**

## TDD 角色轉變：從自律工具到控制介面

先拆解 TDD 在 AI 語境下到底改變了什麼。傳統的 TDD 是開發者自律的工具，你先想清楚行為規格，再動手實作，避免寫出不必要的東西。但當執行實作的不是人而是 coding agent，**TDD 的角色就從「自律工具」變成了「控制介面」。測試不再只是驗證程式對不對，它變成了你跟 AI 溝通「我到底要什麼」的唯一精確語言。**

這件事的意義比看起來大得多。過去十年，軟體工程師的核心產出是程式碼。程式碼是資產，寫得好的程式碼有維護性、可擴展性，這些是工程師薪資的錨點。但如果 coding agent 能在幾秒內生成程式碼，而且只要測試寫得夠好就能保證品質，那**「寫程式碼」這個動作的邊際價值會被壓到趨近於零。剩下來有價值的，是寫測試的能力——更精確地說，是把模糊的商業需求翻譯成可執行驗收條件的能力。**

## 時間分配的結構性翻轉

用一個粗暴的算法來感受這件事：

### 傳統模式下，一個功能的開發時間分配：
- 需求釐清 20%
- 寫測試 15%
- 寫實作 50%
- debug 15%

### AI 輔助模式下（沒有 TDD）：
- 需求釐清 20%
- 寫實作 0%（AI 生成）
- review AI 產出 + debug 60%

### AI + Red/Green TDD 模式：
- 需求釐清 30%
- 寫測試 40%
- AI 實作 5%
- 驗證通過 25%

**注意那個結構性的翻轉：實作時間從佔一半壓縮到近乎消失，但「定義什麼叫做對」的時間從 35% 膨脹到 70%。開發者的工作內容沒有消失，但重心徹底搬家了。**

## 三個值得追問的問題

這帶出真正值得追問的幾件事：

### 1. 如果「寫測試」變成核心技能，軟體人才的估值邏輯會怎麼重組？

現在市場上最貴的工程師，通常是能寫高效能、高擴展性系統架構的人。但在 AI + TDD 的世界裡，架構實作可以被 agent 處理，真正稀缺的是那些能把「老闆說的那句模糊的話」翻譯成一組精確、完備、邊界條件清晰的測試案例的人。

**這不是傳統意義上的「QA 工程師」，這更接近系統分析師加上領域專家的混合體。** 如果市場還在用舊的框架給「寫程式的人」定價，那就會出現一段錯配期：真正產生價值的能力被低估，正在被替代的能力被高估。

### 2. Coding agent 的可靠性一旦被 TDD 框架兜住，軟體開發的成本結構會發生什麼？

這是錢的問題。一個 junior 工程師年薪 80 萬到 120 萬台幣，主要產出是在 senior 指導下寫實作程式碼。如果 coding agent 加上 TDD 能覆蓋掉這塊產出的七成，那企業面對的選擇不是「要不要用 AI」，而是「省下來的人力成本要拿去幹嘛」。

是拿來擴大產品線？還是直接變成利潤？還是轉投入更多 AI 工具的訂閱費用？**省下的錢搬去哪裡，決定了這波 AI coding 工具到底是「產業升級」還是「勞動替代」。**

### 3. Red/Green TDD 讓 coding agent 變得可控，但它同時也把 AI 鎖在一個極度狹窄的創造空間裡但這是特性還是限制？

TDD 的本質是「你先定義對的樣子，AI 負責抵達」。這對明確規格的功能開發極度有效。但軟體裡最有價值的工作，往往是那些連規格都還不確定的探索性開發——你不知道要什麼，寫著寫著才發現方向。

**Red/Green TDD 在這種場景下是失效的，因為你寫不出測試。** 如果團隊因為 TDD + AI 太好用而過度傾斜向「有明確規格的任務」，忽略了探索性的工程投入，長期來看可能反而壓縮了創新的空間。

## 結論：產業價值鏈的重心遷移

Willison 的文章把 Red/Green TDD 定位成一個 prompting 技巧，這沒有錯，但只講了最表層。底下真正在發生的，是**軟體開發價值鏈的重心遷移：從「寫程式」到「寫規格」，從「生成」到「驗證」，從「動手的人」到「定義標準的人」。**

能看到這層結構變化，才是在談 AI 對軟體產業的真正衝擊。否則就只是在分享一個好用的 prompt 技巧，然後錯過整個產業權力重組的訊號。

# 拆解結果

## A. 主脈絡與個人映射
- **論證骨架**：從 Simon Willison 的 Red/Green TDD 技巧文出發 → 指出表層技巧下的深層結構變化 → TDD 從自律工具變成控制介面 → 時間分配結構性翻轉（定義標準從 35% → 70%）→ 三個值得追問的問題（人才估值、成本結構、創新壓縮）→ 收束到「軟體開發價值鏈重心遷移」。
- **個人映射**：對系統建造者最重要的洞見是「寫測試的能力 = 把模糊商業需求翻譯成可執行驗收條件的能力」——這正是系統分析師 + 領域專家的混合體，也是不可壓縮的能力。

## B. 卡片（Zettel）

序號 1
- 標題：TDD 在 AI 時代的角色轉變：從開發者自律工具到 Agent 的控制介面
- 類型：Model
- 概念（50–300 字）：傳統 TDD 是開發者自律工具——先想清楚行為規格再動手。當執行實作的是 Coding Agent，TDD 變成「控制介面」——測試不再只驗證對不對，它是你跟 AI 溝通「我到底要什麼」的唯一精確語言。時間分配發生結構性翻轉：傳統模式（需求 20% + 寫測試 15% + 寫實作 50% + debug 15%）→ AI+TDD 模式（需求 30% + 寫測試 40% + AI 實作 5% + 驗證 25%）。實作從佔一半壓縮到近乎消失，「定義什麼叫做對」從 35% 膨脹到 70%。開發者工作沒消失，但重心徹底搬家了。
- 重要性（1 句）：重新定義了「寫程式」的價值結構——寫程式碼的邊際價值趨近於零，寫測試（定義驗收條件）才是核心產出。
- 邊界/反例（1–2 句）：TDD 在探索性開發場景失效——你不知道要什麼時寫不出測試。過度依賴 TDD + AI 可能讓團隊傾斜向「有明確規格的任務」，壓縮創新空間。
- 知識鉤子：與 #Interface+Test骨架策略 形成實踐互補——一個講方法論、一個講產業分析；是 #軟體開發價值遷移 的底層機制說明。

序號 2
- 標題：軟體開發價值遷移：從生成程式碼到定義驗收條件，人才市場面臨結構性錯配
- 類型：Trend
- 概念（50–300 字）：軟體開發價值鏈正在重心遷移：從「寫程式」到「寫規格」、從「生成」到「驗證」、從「動手的人」到「定義標準的人」。稀缺人才畫像改變——不是傳統 QA 工程師，而是能把「老闆那句模糊的話」翻譯成一組精確、完備、邊界條件清晰的測試案例的人（系統分析師 + 領域專家混合體）。市場正在經歷錯配期：真正產生價值的能力（需求翻譯）被低估，正被替代的能力（寫實作 code）被高估。Junior 七成產出可被 AI + TDD 覆蓋——省下的成本流向（研發擴張 vs 利潤回收）決定這波是「產業升級」還是「勞動替代」。
- 重要性（1 句）：這是判斷「軟體工程師如何在 AI 時代重新定位」的核心趨勢分析。
- 邊界/反例（1–2 句）：「需求翻譯」能力本身也可能被未來的 AI 壓縮（如果模型能直接從模糊描述生成可靠測試）。目前 AI 生成測試的品質仍不穩定，人工定義仍是必要的。
- 知識鉤子：與 #AI Coding控制能力差距 直接呼應——控制能力的差距就是這個遷移的具體表現；與 #AI取代SaaS錢搬家了 形成跨領域映射——成本消失後的資金流向邏輯。
