# AI Agent 的被忽視的瓶頸：CPU 才是關鍵

> 來源：Richard's Research Blog, Vik's Newsletter
> 核心研究：喬治亞理工與英特爾 2025 年 11 月論文
> 主題：CPU 在 Agentic AI 中從配角變主角

## 核心發現

1. **AI Agent 任務中 CPU 是瓶頸**
2. **多核心的高階 Server CPU 市場 TAM 增加**
3. **雲端背景 Python 直譯執行大量應用**，GPT、Claude、Gemini 都內建 Python 直譯（含眾多函式庫）反覆生成程式碼在內部沙盒執行，編譯或直譯都是靠 CPU

## 計算範式的轉變

### 過去兩年：GPU 主導

「過去兩年，CPU 在 AI 基礎建設中一直是次要考慮，而 GPU 則成為訓練焦點，近年來更是推論焦點。在過去六個月中，代理型人工智慧的崛起成為人工智慧推論所需的「殺手級應用」。

**隨著代理處理與 AI 互動的工作，計算的基本需求正在轉變，而 CPU 正迅速成為被忽視的瓶頸。** 代理型 AI 系統串連工具呼叫、API 請求、記憶體查詢、編排邏輯，這些都由 CPU 處理，而 GPU 則閒置不被充分利用。在代理的新時代，CPU 核心數、每個核心的任務以及記憶體階層/頻寬決定了 GPU 利用率、整體吞吐量與 TCO。

**代理的工作負載特性會使計算托盤、機架或叢集中的 CPU 與 GPU 比例升高，甚至可能需要比 GPU 更多的 CPU，顯著增加 CPU TAM**

### 關鍵數據

喬治亞理工與英特爾於 2025 年 11 月發表的一篇論文估計，**CPU 上的工具處理佔代理工作負載總延遲的 50% 至 90%。**

雖然普遍的說法是 CPU 短缺，且代理式 AI 需要更多 CPU，但本文的具體框架是 **CPU 其實是瓶頸。**

## 人機循環到代理循環

### 過去：人類是瓶頸

過去兩年，標準的大型語言模型工作流程一直是人機介入循環：你輸入請求，等待回應，閱讀，思考，然後行動。這個循環本質上很慢，因為人類是瓶頸。

### 現在：代理自主循環

在代理人時代，人類退出循環，代理人自己接手這個循環。根據廣泛的人類輸入，每位代理會閱讀回應、制定計畫、啟動工具呼叫或新的推論請求，並等待結果。

**關鍵差異在於每位使用者有數百個次代理，且同時循環於此循環。** 這個循環完成得越快，特工完成任務的速度也越快。更多的代理允許同時執行多個平行任務，這些輸出必須協調一致。**代理程式的處理與編排全都在 CPU 上進行。**

## CPU 角色轉變：從配角到指令層

### 純推論時代：CPU 是配角

CPU 作為「主節點」的角色僅限於將請求傳送給 GPU，並將結果回傳給使用者。計算密集的工作（矩陣乘法、注意力計算、符號生成）都在 GPU 上完成。這也是為什麼過去兩年來，GPU 一直主導著關於 AI 基礎設施的討論。在純推論架構中，CPU 負載較輕，且很少成為瓶頸。

### 代理協調：CPU 變成指令層

**在編排工作流程中，CPU 從輔助角色轉變為指令層。CPU 效能最終決定了 GPU 的使用量。**

工作流程一開始也是這樣：使用者提出請求，CPU 會標記它。但第一個推論呼叫不是產生回應，而是產生執行計畫。這個計畫由多個子代理並行執行，每個子代理呼叫不同的工具，等待的時間也不一。CPU 協調這些子代理之間的關係：哪些是獨立的，哪些依賴於其他代理的結果，以及它們完成的順序。每個子代理也可能根據思路鏈推理運行自己的推論迴圈。

## 成功的代理協調需要三點

### 1. 高時脈頻率

一個具備足夠高時脈頻率的 CPU 能以最小延遲監控每個令牌，並判斷子代理何時完成，並視需要將結果整合到其他子代理中。

### 2. 高核心數量

高核心數量，讓多個代理能同時執行多項任務。

### 3. 低延遲記憶體存取

低延遲存取記憶體以取得上下文與中介狀態，使快取設計與記憶體階層變得不可或缺。

## 完整工作流程

當所有子代理完成後，CPU 會將其結果整合並送入反射推論迴圈，以評估原始請求是否被滿足。如果沒有，則又要開始新的規劃、推論與工具執行循環。此過程持續進行，直到最終結果被解除代幣化並返回給使用者。

## 推理與行動：決定 CPU 與 GPU 比例的因素

### 評分標準：九項指標

任何伺服器 CPU 是否適合執行代理工作負載，皆可依下列九項指標進行評分：

#### 1. 每核心效能與時脈
CPU 需要快速做出編排決策，以維持 GPU 的供能。它還需要持續監控每個子代理的標記，以了解何時完成以及下一步該做什麼，且尾延遲極低。

#### 2. 核心數量
每個核心假設處理一個代理，或上下文在幾個代理間切換以協調推理迴圈。像是同時多執行緒（SMT）等功能允許每個核心擁有更多代理程式。根據工作負載，單一代理甚至可能使用多個核心或執行緒。

#### 3. CPU-xPU 互連頻寬
決定 CPU 傳送資料或讀取 xPU 的速度。

#### 4. CPU 記憶體頻寬與容量
較高的記憶體頻寬意味著能以較低的延遲完成更大的上下文、KV 快取與推理預填充。

#### 5. 快取設計與大小
更多 L3 快取意味著隨機資料存取突發時記憶體延遲較不明顯，因為 CPU 不必離片。

#### 6. 每瓦效能
當部署數千個代理時，每代理單位的經濟效益取決於它為達到所需效能所消耗的能源。

#### 7. PCIe 產生與通道數量
代理程式會呼叫 API、存取儲存、爬取網頁。I/O 連接網路與 NVMe 儲存決定工具呼叫吞吐量。

#### 8. 指令集架構（ISA）
x86 與 ARM 的差異很重要，因為它決定了工具呼叫的支援範圍。ARM 較簡單的 RISC ISA 提供每瓦效能的提升。此外，ISA 的偏好會決定你被鎖定在哪個生態系統（NVIDIA 的 Grace（ARM）+GPU、Google 的 x86+TPU，甚至是 x86 + NVIDIA）。

#### 9. 非統一記憶體存取（NUMA）域
現代伺服器 CPU 幾乎總是由多個晶片組成，以容納多個核心。根據晶片組與實體配置之間的互連技術，每個 CPU 對記憶體的存取延遲可能不均勻。這表示不同核心上的代理可能會看到不同的記憶體效能。

## 案例：NVIDIA Vera CPU

一個適合重推理工作的 CPU 範例是 NVIDIA 的 Vera CPU，原因如下：

### 每核心效能
88 核心高速客製化 Olympus ARM 核心，支援多執行緒。核心數不是最高，但每核心效能不錯。

### CPU-xPU 互連
**NVLink-C2C 雙向 1.8 TB/s。** 一致性記憶體共享讓 Rubin GPU 能直接從 CPU 記憶體讀取，就像讀取自己的記憶體一樣。這是讓 Vera 成為推理端 CPU 的關鍵特徵。沒有其他作品能比得上。

### 記憶體頻寬與容量
1.5 TB LPDDR5 於 8 個 SOCAMM 模組中，速度為 1.2 TB/s。具備龐大的 KV 快取擴充容量。

雖然選擇 Vera 讓你被鎖定在 NVIDIA 生態系統，且 ARM 架構使工具相容性存疑，但它龐大的 GPU 互連頻寬、高速核心和大量 DRAM 讓它在所有推理重負載的重要指標上都得分很高。

## 研究發現：工具處理的巨大成本

### 關鍵數據

- **CPU 上的工具處理可佔用高達 90.6% 的總延遲**
- 代理吞吐量會被 CPU 因素（核心一致性、同步與過度訂閱）或 GPU 因素（主記憶體容量與頻寬）所限制
- **CPU 動態能量在大批次中消耗高達 44% 的總動態能量**

### 工具處理的內容

雖然 AI 模型大多運行於 GPU 或加速器上，但 CPU 也被用於工具處理，包括：
- Python 或 Bash 執行
- 網頁搜尋
- URL 擷取
- 詞彙摘要

過去 AI 效率的策略多聚焦於 GPU 核心與 KV 快取排程，而代理式 AI 則在執行流程中引入大量以 CPU 為中心的工具。

## 多執行緒 vs 多處理

### 共享記憶體模型（多執行緒）

多執行緒會在程序中產生多個執行緒，這些執行緒在不同核心上平行執行，且全部存取相同的記憶體空間。

### 多處理

使用多個獨立記憶體的程序；此方法常用於 Python 中，以繞過 CPU 限制工作負載的全域直譯器鎖（GIL）。

### 權衡

執行緒與程序的選擇涉及權衡：
- **執行緒**：產生與切換開銷較低，但需謹慎同步以避免競逐條件
- **程序**：開銷較高，卻消除任務間直接記憶體干擾的可能性

透過運用這些技術，代理型 AI 工作負載中 CPU 密集型部分能大幅加速，更有效率地利用 CPU 資源並更有效率地將資料傳送至 GPU。

# 拆解結果

## A. 主脈絡與個人映射
- **論證骨架**：從 GPU 主導的純推論時代 → 代理型 AI 崛起改變計算範式 → CPU 工具處理佔總延遲 50%-90%（喬治亞理工+英特爾論文）→ CPU 從配角變指令層 → 成功的代理協調三要素 → CPU 評分九項指標 → NVIDIA Vera 案例。
- **個人映射**：對系統建造者的核心啟示——設計 Agent 架構時，CPU 效能（而非 GPU）可能才是決定系統吞吐量的瓶頸。這改變了基礎設施選型和成本優化的方向。
