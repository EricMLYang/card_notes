# AWS 提案策略



**用「最小閉環」把 AWS 納入雙雲：**

Azure 續當內部效率與既有數據主軸；AWS 用來打外部專案、降 BI 成本、承接 AI Agent 商機，12 個月內把團隊定位成「可對外收費的解決方案供應單位」。



---

## **為什麼現在要做（3 大理由）**

1. **市場商機**：中大型客戶常指定 AWS；沒有交付力＝錯失案源。

2. **品牌與議價力**：雙雲形象更專業，降低被單一供應商綁死的風險。

3. **單位定位**：以 AWS 形成對外可售的模板與服務，從成本中心轉為營收單位。





---





## **引入 AWS 的三支槓桿（短｜中｜長）**

1. **短期：報表切入（QuickSight vs Power BI）+ 現有 AWS 專案**

   - **何時選 QuickSight**：閱讀者多、成本敏感、資料在 AWS。

   - **何時選 Power BI**：深綁 M365/Teams/Excel，既有授權多。

   - **行動**：挑 1–2 份高流量、低進階功能報表做遷移試點；同時把**現有 AWS 客戶專案**模板化（IaC、SOP）。

   

2. **中期：AI Agent 產品化**

   - **內部**：開發「知識問答 / 程式碼助手」Agent；

   - **外部**：把 Agent 嵌入 VMS / 智慧空間方案，做成可售的「GenAI 套件」。

   

3. **長期：夥伴等級**

   - 明確走 **APN 路徑（Select → Advanced / Premier）**，形成與 AWS 共賣與背書。

   





---



## **協調行動（3 段、各不超過 3 條）**

**最小閉環立起來**

- SSO：Entra ↔ AWS IAM Identity Center（共用身分、最小權限）。

- **報表試點**：QuickSight 與 Power BI 各做 1 份可比對的 PoC（含成本/體驗）。

- **AWS 專案模板**：將現有 AWS 專案打包成「一鍵部署」與運維手冊。





**可複製與可售賣**

- 擴到 2–3 個報表遷移；產出「BI 成本優化清單」。

- 上線 1 個 AI Agent PoC（內部用例），撰成標準化組件。

- 啟動 APN 要求：完成首批培訓/認證，整理 1 個客戶成功案例。





**品牌與營收**

- 上架 2 個「可售方案」：① BI 成本優化包、② VMS GenAI Agent。

- 申請 **Select / Advanced**，啟動與 AWS 共賣；

- 以方案包帶動 1–2 個新客戶訂單。





---





## **成功指標（KPI，精準可量）**

- 2 份報表 PoC 上線、AWS 專案模板完成、SSO 打通。

- 報表遷移≥3 份、1 個 AI Agent PoC、≥3 張 AWS 初級/進階認證。

- 2 個可售方案、1 個新簽單、達成 APN 等級（Select/Advanced）。





---





## **不做清單（避免分心）**

- 不做大規模遷離 Azure；雙雲＝互補，不是取代。

- 不全面鋪開 AWS 服務；只投資在 **報表、AI、既有專案交付**。

- 不一次性替換所有 BI；先高流量、低複雜度場景試點。





---

## **對主管的三句話**

1. **不做雙雲＝持續錯失 AWS 指定案**；我們用最小閉環在 90 天內補上。

2. **先用報表省錢、再用 AI 增值**；短期見效、中期開新線。

3. **走 APN 夥伴路徑**；把部門從成本變營收，品牌力一起上來。
