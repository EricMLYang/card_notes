# **系統需求規格書**

-  **(System Requirements Specification, SRS)**

預設對象：4人小團隊開發團隊
工作目的：一臺可裝接CMS與多種場域感測器的智慧廣告推播平臺

---

## 一、系統介紹

開發一個智慧廣告推播平臺，能根據現場人流、位置、時段等資訊，自動推薦最適合廣告給電子看板播放，並集成系統監控、分析與進階優化能力。

---

## 二、功能需求 (Functional Requirements)

1. 感測器資料接收與篩理

2. 廣告素材上傳與元資訊編譯

3. 觀看行為分析與場域行為報告

4. 廣告推薦機制：執行標籤化策略推播

5. 推播內容排程與CMS互連

6. 加值功能：價值揭示與優惠發送

---

## 三、非功能需求 (Non-functional Requirements)

1. 性能：1s 內推薦結果，CMS連線在 3s 內啟動播放

2. 安全：網路通信 HTTPS、Azure AD 認證

3. 相容性：支援 3 種主要 CMS API 種類

4. 可用性：系統 uptime 目標 99.5%

---

## 四、使用者角色與操作流程 (User Stories)

1. **PM/SA**：設定場域與標籤組合策略

2. **FE/BE Dev**：設計CMS適配器與API伺服

3. **Data Engineer**：擱理感測器資料，建立Audience Profile

4. **Data Analyst**：對場域行為進行檢視、增值分析

---

## 五、系統限制與邊界 (Constraints)

1. **執行環境**：Azure 上的 App Service + SQL Database + Databricks

2. **電子看板播放操作**：不能直接控制播放系統，必須透過 CMS 互動

3. **MVP 限定**：先實作時間/人流為基礎的平常推播

---

## 六、資料流與 API 需求

### 資料流程圖簡要

感測器資料 → 轉成 Audience Profile → 推播分析 → 推播配置 → CMS 播放並回報

### 關鍵API (操作預算)

1. `POST /sensor-data`：傳送感測器資料(JSON Schema)

2. `GET /recommendation`：根據場域與時間請求廣告推薦

3. `POST /push-to-cms`：傳送推播指令給 CMS
