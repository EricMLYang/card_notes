前幾天的 PO 文, 聊到 Agent MVP 的做法, 其實我心裡真正想的是: AI 的進步, 工作流程應該要有根本上的改變才對, 那麼架構師 (或是資深的技術人員) 本身該做什麼調整? 我從之前提過, Kent Beck 三年前講了一段話開始:
原文:
"The value of 90% of my skills just dropped to $0. The leverage for the remaining 10% went up 1000x. I need to recalibrate."
翻譯:
"AI 讓我 90% 的技能價值變成 $0, 但是剩下的 10% 的價值卻放大 1000x, 我需要重新調整策略"
聽起來很有道理, 但是除非你真的認清那 10% 是什麼, 並且想清楚你該怎麼調整, 否則這段話對你來說只是心靈雞湯而以. 實際上什麼都不會改變。要想通該怎麼做, 很多基礎知識在這時就浮現出來了, 我試著拆解一下:
限制理論 提到這麼一個觀念, 整個系統的效率取決於最弱的一環 (瓶頸), 換個角度來說, 主動 "讓自己成為瓶頸" (指的是那 10%) 就是優化到極致的表現, 其他 90% 都優化到極致了, 剩下 10% 非你不可的部分影響了整個系統 (團隊) 的效率, 就是這句話的意思。這意味著你要將有限的時間與精力, 聚焦在真正影響目標的核心問題 (10%) 上, 避免在非瓶頸處 (90%) 浪費資源。
於是我重新思考了這兩年來 AI 在軟體開發領域的改變, 拿了一個 side project 重新用 AI 做了一次, 嘗試了我自己的角色 (架構師) 該怎麼運用新工具才能發揮最大的效益。如果要問我軟體開發工具最大的變革, 那我會挑兩個當代表:
➡️ Claude Code:
或其他 Coding Agent, 它們代表了 AI 運用工具及撰寫程式碼的能力
➡️ SDD:
或是及其他相關的 Toolkits (例如 Spec-Kit or OpenSpec), 它們代表了可執行的 Spec 逐步往 "source code" 的用途演進
架構師最重要的任務是:
了解商業需求與技術要求, 替產品單位與決策者翻譯需求, (找出) 並轉換成開發團隊能執行的做法 (規格)。我把整個流程簡化成兩大階段:
① 處理不確定性的問題
② 執行確定性的任務
而這兩者中間的產物就是明確的 SPEC. 架構師面對的規模, SPEC 不是只定義畫面, 流程, 以及行為而已, 還包括背後的設計, 跨系統的溝通介面 (通常是 API)。
這些規格可能不只用在一個專案, 可能是龐大系統的重要 API, 會影響多個團隊, 多個專案之間的協作。目前的 AI 開發, 大多都還是在單一團隊或是單一專案的範圍內, 跨多個專案的設計, 需要人來把關的比例就高得多。
面對不確定性的問題, 我的做法就是善用 vibe coding 來做 Prototype, 靠 Prototype 來驗證 & 收斂 SPEC。這 Prototype (包含 code) 對我而言, 是個更精準描述需求的 "SPEC", 他敘述了最終系統的行為 (因為你可以直接 RUN), 設計 (因為 Prototype 有實際的 API spec, database schema 可以參考), 整合 (多個專案的需求, 都可以用 Prototype 的方式來呼叫 API, 確保單一規格 API 能同時滿足多樣需求)
如果我同時將需求, 設計, 驗證, 跨團隊溝通等等議題, 都簡化成 Prototype , 其實就是建構系統的模型了 (過去我談抽象化的系統建模就是指這個)。然而建模也是很花時間的，過去不大有人願意為了這些最終不需要交付的東西投人力，但是現在有 vibe coding 了啊, 時間與人力成本壓低到極限, 靠架構師自身的人力就足以應付, 拿來做這件事剛剛好。
SDD 要有效, 良好的 SPEC 是必要的。SPEC 要寫得好很不容易, 通常要包含非常多的設計跟驗證才有可能做到。因此我的想法是: 先做好 Prototype, 再從 Prototype 中萃取設計, 寫成 SPEC。我自己實際嘗試, 用 Prototype (code) 來當作 SPEC (document) 的前段作業, 附加的好處還不少。我舉幾個過去維護 SPEC 很吃力的環節:
➡️ SPEC 異動很難維護 (即使有 AI 也容易漏掉)
若改為 Prototype 則可以靠重構的工具, 明確的把相依的設計都改對
➡️ SPEC 異動很難驗證
若改為 Prototype 則可以靠大量的測試工具 ( BDD, TDD, Auto Testing ), 明確的確認設計異動符合每個系統要求
➡️ SPEC 異動很難溝通
若改為 Prototype 則有具體的設計跟畫面操作流程, 可追蹤, 執行過程也可觀察, 可以很容易的跟 PO / RD 對焦設計細節
從結論來看, 其實就是架構師在建立大型系統早期階段, 都用模型來模擬跟溝通, 極大化的在早期就避免各種設計疏失造成的風險。借助軟體工程的知識 + AI 的工具, 模型的建立跟維護的成本降到極限, 效益也提升到極限。最終的瓶頸就是: 你能多快的構思大型系統運作的樣貌? 你能多有效率的跟多少團隊溝通設計?
我在  #webconf2025 談 MVP 的時候, 用了這張圖來說明, 當時是說 PO 用 prototype 來溝通跟驗證需求的場景。現在我拿同樣的結構, 來說明 Architect 如何用 Prototype 來跟整個系統的開發團隊來溝通跟驗證。而這些過程, 能得到的成果是被驗證過的高品質 SPEC (所謂的 "高品質", 就是你照著規格來做, "一定" 能達成目標的質化指標。因為這些設計早已經過驗證, 排除了大部分的設計風險, 就像建築師都會用力學模擬建築物的結構, 航太工程師都會用風洞來測試模型的流體力學一樣)。
這想法，是我認為工程師若要在未來 AI 的時代還保有價值, 應該要思考的方向。其實這方法我已經嘗試了一段時間了, 我會整理實作的案例 (有 demo code + vibe coding 過程示範) 讓大家參考。
趁過年的假期, 整理了一下我的想法, 相關參考資訊我放留言, 先預祝大家新年快樂 😃
