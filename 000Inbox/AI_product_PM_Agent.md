
# AI 時代軟體開發的角色分工與策略

---

## 起點：重新校準自己的價值

Kent Beck 說過一句讓我很有感的話：「AI 讓我 90% 的技能價值歸零，但剩下 10% 的槓桿放大了 1000 倍。我需要重新校準。」

我用限制理論（Theory of Constraints）來理解這句話——整個系統的效率取決於最弱的一環。當 AI 把 90% 的工作優化到極致，剩下那 10% 非你不可的部分，就成了決定整個團隊效率的瓶頸。主動讓自己成為那個瓶頸，意味著你把有限的時間和精力，聚焦在真正影響目標的核心問題上，不在非瓶頸處浪費資源。

於是我重新思考了這兩年 AI 在軟體開發領域的改變，拿了一個 side project 重新用 AI 跑了一遍，嘗試站在自己的角色（架構師）找出運用新工具的最大效益。如果要我挑兩個最具代表性的變革：

- **Claude Code**（或其他 Coding Agent）：代表 AI 運用工具及撰寫程式碼的能力。
- **SDD / Spec-Kit / OpenSpec**：代表「可執行的 Spec」正逐步往 source code 的用途演進。

---

## 整體思路

1. **PM** 產出有價值、可使用、行得通的產品需求。
2. **架構師** 用全局視野把需求中的不確定性轉換成高品質 SPEC。
3. **開發團隊** 憑藉自身知識駕馭 AI，產出高品質程式碼並嚴謹驗證。

## 一、產品經理（PM）：找出值得做的事

### 核心定義

產品經理最根本的工作是「找出有價值、可使用且行得通的產品」（to discover a product that is valuable, usable and feasible）。這是「產品探索」（Product Discovery）的核心任務，必須解決四個關鍵風險：

- **價值風險（Value Risk）**：顧客會買單或決定使用嗎？
- **易用性風險（Usability Risk）**：用戶搞得懂怎麼用嗎？（需與設計師合作）
- **實行性風險（Feasibility Risk）**：工程師做得出來嗎？（需與工程師合作）
- **商業可行性風險（Business Viability Risk）**：對公司的財務、法務、銷售等面向行得通嗎？

另一個值得記住的框架是 **FLOP**——Failure in Launch, Operation, Premise，提醒我們產品可能在上市、營運、前提假設這三個階段失敗。

### PM 必須帶來的四項貢獻

1. **深入了解顧客**：成為顧客專家，透過質化（訪談）與量化（數據）掌握痛點、渴望與決策模式。
2. **深入了解資料**：每天第一件事查看數據，了解顧客如何使用產品、銷售數據、A/B 測試結果。
3. **深入了解事業**：理解產品如何幫公司獲利，掌握內部利害關係人（業務、行銷、財務、法務）的限制。
4. **深入了解市場與產業**：熟悉競爭對手、科技趨勢、顧客行為變化，為「未來的市場」開發產品。

### 收斂與交付

PM 的工作是基於用戶理解，把商業目標收斂成可驗證的問題與取捨，定義成功指標與護欄（Guardrails），切出團隊能穩定交付的最小方案。具體來說就是：

- 定義要贏什麼（Outcome）
- 定義真正要解的問題（Problem Framing）
- 做取捨並形成可交付的賭注（Bet / Slice）
- 把成功條件交給工程（Acceptance / Guardrails）

Marty Cagan（SVPG）一直在強調：產品團隊要花更多力氣在 Discovery（判斷與選擇），而不是只把 Delivery（流程與交付）做快。AI 讓 Delivery 變更快之後，團隊的時間會集中在「找出能改善 Outcomes 的解法」。

優先序的本質是經濟取捨。Donald G. Reinertsen 的觀點是：不要只看利潤或重要性，要看「延遲會讓價值怎麼變化」（Cost of Delay），這才是決定先做哪個 bet 的核心邏輯。

### 驗證與前型（Pretotyping）

產品空想地帶的失敗有四個交互作用的根源：溝通曲解、不擅預測、不付代價、確認偏誤。這就是為什麼光靠空想或一般訪談很難產出好產品。

驗證 Premise 的技巧：用「至少 X% 的 Y 會 Z」這個句型。X 是目標市場的特定百分比，Y 是目標市場的明確描述，Z 是你期望市場對構想的接受情況。

前型（Pretotyping）要回答的核心問題：我會用嗎？怎麼用、多常用、什麼時候用？別人會買嗎？願意花多少錢？前型的三個原則：必須讓目標客群願意付出某種代價以產生自我數據；可迅速執行；成本低廉。

### AI 輔助做法

- 用 Coding Agent 搭配充滿相關資訊的 Repo 來協助產品探索。
- 參考《Ideaflow》的做法：在 AI 時代源源不絕地產生和測試創意。

## 二、架構師：把不確定性轉換成確定性

架構師的角色是了解商業需求與技術要求，替產品單位與決策者翻譯需求，轉換成開發團隊能執行的規格。主要處理的問題：

### 1. 不確定性

架構師面對的規模遠不只畫面、流程和行為。SPEC 還包括背後的設計、跨系統的溝通介面（通常是 API）。

### 2. 原型（Prototype）

善用 Vibe Coding 來做 Prototype，靠 Prototype 來驗證並收斂 SPEC。Prototype 是把需求的不確定性轉換成確定性的工程化方法。

對我而言，Prototype（包含 code）就是更精準描述需求的 SPEC：

- 它敘述了最終系統的**行為**——因為你可以直接 RUN。
- 它定義了**設計**——因為有實際的 API spec 和 database schema 可參考。
- 它驗證了**整合**——多個專案的需求都可以透過 Prototype 呼叫 API，確保單一規格能同時滿足多樣需求。

原型要回答的核心問題：我們能打造出來嗎？能按預期運作嗎？怎樣盡可能做得小、大、便宜、節能？

### 3. SPEC

不確定到確定之間的重要中間產物。

### 4. 執行確定性的任務

當 SPEC 足夠確定，後續就是可預期的執行工作。

## 三、開發團隊：確定性規格下的高品質開發

Andrew Ng 教授分享過一個例子：他用 MidJourney 時，發現自己無法精準描述想要的結果。而他的一位合作夥伴，因為具備藝術史背景，熟悉不同藝術風格與流派，能精準描述需求，產出的成果遠比 AI 專家更好。

這個例子凸顯了**品味**的重要性。對藝術有品味的人能更好地掌握圖像生成 AI；同理，軟體工程師要掌握程式生成 AI，就要培養對程式設計的品味——了解程式設計的歷史脈絡、各種範式、設計模式與撰寫風格。唯有具備這些背景知識，才能有效與 AI 互動，精確表達你希望程式碼如何撰寫。


## 點子流：
- 點子比率 (Idea Ratio): 創意是數量的累積，優秀的成果來自於大量的嘗試與修正。能夠持續創造優異成果的人，通常都符合這種高產出的模式。
* **關鍵數字：** **2,000 : 1**
* **定義：** 每產生 2,000 個可能性，才能得出 1 個成功的解決方案。
* **誤區澄清：** 這 2,000 個點子並非憑空想像的獨立創意，而是包含創新過程中的所有**組合、變化與反覆修改**。

- 創新的漏斗效應 (以 IDEO 案例為例)
1. **原始點子：** 4,000 個
2. **可行原型：** 200 個
3. **商業化產品：** 12 個
4. **最終成功：** 2 ~ 3 個

- 多元的觀點：學習圈學習圈、筆友、顧客委員會、異花授粉、初體驗、留意新手說的話、互補的合作夥伴、實話空間、
- 顛覆既有觀點：What if、如果 ... 不是很好嗎？、同理心訪談、翻轉假設、觀察久一點、
- 讓直覺導航、讓創意續航
- 視角：目的、不同人的角度、縮放視野、連續為為什麼、極端假設
- 產生組合框架，調整旋鈕：尺度規模、品質、情緒、嚴述程度、預期、相似度
- 減法、回溯法、反過來想、觀察模仿發散
