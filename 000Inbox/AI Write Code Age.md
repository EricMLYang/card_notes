# AI Write Code Age

- Reference:

   - **[Y Combinator Youtube: Vibe Coding Is The Future](https://www.youtube.com/watch?v=IACHfKmZMr8)**

   - [Anthropic CEO - Dario Amodei in  The Future of US AI Leadership](https://www.youtube.com/live/esCSpbDPJik?si=47Yq21Jx5v4ZDSE2) 

   - **[Mosky Liu's Post:](https://www.facebook.com/share/1BwHAtUoj9/)** [AI 時代的軟體開發方向：生成程式勢不可擋，不如學會如何管理！](https://www.facebook.com/share/1BwHAtUoj9/)



大約今年農曆年前，跟團隊 Member 在討論 AI 輔助程式時候，他們認為很多時候 AI 寫的 Code 在現有系統會有很多風險，當時公司專屬 Azure 網域上的 LLM 還是早期版本的 ChatGPT 4o，而我已經認知到今年 LLM 成本會是持續降低的一年（沒想到過年期間 DeepSeek 新聞讓 AI 使用成本降低速度加快），當時我只是很模糊的回覆說: 

「其實外面最新模型能力都提升滿多的，而且 AI 只是工具，就看你怎麼用他，在 Demo 情境你就可以放心讓 AI 生成模型，在既有系統你還是可以讓 AI 去檢查、優化你的 Code，生成文件…等 」



我自己覺得當時講完沒有太多說服力，但當下也沒有持續去探討這件事，直到看到 Mosky Liu 用 “可維護性” 的主軸來貫穿 AI 寫程式的後續發展，加上 Y Combinator 探討 Vibe Code 和 Anthropic CEO 的影片，想法似乎看到一些收斂的方向，以下內文就從這三個參考資訊讓自己的想法更具體一點



主要三個思考主軸：

1. AI 100% 寫程式這句話要更多層次考量，並不是模型突破的瞬間大家就瘋狂採用，現實世界不是這樣運作的

2. Vide Code: 重要的是區分你現在產品/專案是在  Zero to One, One to N, 維運的哪一階段

3. 可維護性:  DevOps 基礎應該是每個擁抱 AI 寫程式的人，最重要的功課



## \[ AI 12 個月內寫 100%的 Code \]

前陣子 Thread 和 FB 被 Dario 的話洗版，他提到



「 幾個月內 AI 就可以寫 90% 的程式， 12 個月內 AI 可以寫 100%的 Code 」



這樣的身份講出這樣的話，一定會引起很多討論，但我認為大家的反應太過直覺，忘記這個世界在真實演進的複雜性，就像你看到統計說台灣房價平均上漲 30%，你不看到底都是台積宅漲 80% 其他持平，還是全台平均都在漲，然後就直接在你家附近買房準備坐等升值一樣好笑，



以下是我對 Dario 這句話的解讀：

1. Dario 對 AI 絕對是最樂觀的那一群，他會這麼說我們應該要做一下偏差校正，也許是時間上會往後延，或是最後只有 95% 之類的



2. 所謂 AI 12 個月後可以 100% 寫 Code，我認為比較正確的解讀是，他認為 12 個月後的 LLM 模型，應該會進步到足以寫出很大比例且錯誤極少的高品質程式碼，但絕對不是大家會用 AI 寫 Code 了，但光想像到那天的發生，還是可以預期軟體世界運作會被迫去因應這個趨勢

3. 當 LLM 程式能力超強了，這技術多久才會普及到大部份的公司，其實才是我對這句話後續想要探討的重點，因為這包含了不同產業對軟體的重視程度、不想碰雲端 AI 資安考量、原本公司 infra 配合考量、整體成本考量 …等，這一部分才是所謂 100% 的最後挑戰

4. 更不用提你是，新產品或是既有龐大程式碼庫的專案，你是開發很底層的軟硬體整合或是純 SaaS 系統…等重要差異，這一點就帶到下一個主題要討論的重點，Vibe Code



## \[ Vibe Code \]

由 Andrej Karpathy 貼文所引起的討論，我自己解讀是 AI 讓更目的或業務導向寫程式方式變得好像有可能發生，畢竟你靠描述能產生的程式碼其品質應該會越來越好！

![CleanShot 2025-03-15 at 11.01.27@2x.png](./AI%20Write%20Code%20Age-assets/CleanShot%202025-03-15%20at%2011.01.27@2x.png)



老實說這個是我自己最期待的事情，我個人算是半路轉深度學習領域，回學校念一、兩年書後，趁著當年深度學習正紅，就到竹科去工作了，因為自己經歷的關係，很快的就進入帶小 Team 進行專案的角色，有一段時間比較少寫程式，直到 GenAI 爆發後，我發現我寫 Code 生產力又回來了，雖然程式內容還是比不上專門寫程式同事，但因為我非常業務、目標導向，因此要比生產力其實是不輸他們太多！



回到 YC 這個影片的討論，我非常有感他們對於目前軟體開發的兩階段分類，

- Zero to One：要趕快找到市場的高度不確定階段，最重要的是 Product Market Fit

- One to N：找到市場後的擴張時期，是否能讓所有潛在客戶都已穩定的體驗使用你的產品

- 維運(我自己加的)： 其實還有很大一部分已經過了擴張期的成熟系統，需要持續維運，

因此在討論 AI 能不能 100％ 寫程式之前，至少先看看你在開發的內容是屬於哪個階段會比較實際一點，



Zero to One

從之前 Perplexity 創辦人提到，他認為懂產品的軟體開發人才會越來越重要，到本部片提到他們訪問很多新創發現，不少新創說他們 95% 的程式碼都是 AI 生成的，而且強調講這句話的人其技術能力都是非常強的，但這部分很大原因是，他們都處於 Zero to One 階段，這階段最重要的事不是你把 Code 寫的多棒，而是要再錢燒完之前，趕快推出市場會買單的產品，因此速度絕對會是最重要的考量，



當上週 Code 有哪邊怪怪的，沒關係我全部叫 AI 重生一遍，



當某段邏輯比較難，沒關係我就讓 AI 一直產生到我可以接受為止，



以上程式碼不好維護或是 AI 出錯的代價很小，因為本來就沒多少用戶，反之你不快一點推出東西去看看用戶要不要的代價就非常的大



One to N

這時候就可以開始探討第二階段了，當用戶越來越多，AI 出錯的代價越來越高，對於能夠設計出因應業務擴張需求的複雜系統高手，其價值還是一樣會非常高，這時候應該沒有 CEO 敢說，AI 現在只有 1% 機率會出錯，雖然出什麼錯或是影響多嚴重我們無法預估，但為了速度還是都讓 AI 處理吧，反正就是 200萬用戶會受影響而已



也因此我這兩年都持續訂閱 O’REILLY 的 Online Learning，持續看一些比較原理的書，例如近期就是在研究 Databricks 底層的儲存工具 - Delta Lake 的資料結構理解一番，因為當牽扯到大量數據的效率、費用、可靠性時，你出狀況唯一解方是請 AI 把脈的話，那你壓力應該會非常的大



回到出錯的代價，其實這可以放到很多面向，例如當你是開發硬體產品，出錯代價相對高，其實你自然就會開始去限縮 AI 的自由度，直到負責人覺得比較放心為止，而維運也用這個原則考量的話，就可以針對把每一次的新功能放到以上兩階段光譜的某一個中間值，當這功能出錯代價小，ok 請AI 做，反之，請把公司大神請出來確認一下



而出錯代價這個思考維度，帶到了 Mosky Liu 用 “可維護性” 的主軸，讓我覺得非常的呼應





## \[AI 寫 Code 的可維護性\]

Mosky Liu  這篇文章提到

「Vibe Coding？Vibe Coding 我覺得很 chill，我喜歡這個概念，只是就像很多軟體工程師的體驗一樣，對於正經的系統來說，AI 還沒有這麼強。

我認為還是要回歸到「可維護性」。

軟體工程師學會寫程式之後的下一步是什麼？」



後續她提到了好多動作:  typing / type hints、具自我解釋性的資料結構（例如 dataclass）、註解、文件、函數、模組越小越好、完整的測試集、設計污染區、依風險決定人與 AI 的比例



以上不就是 DevOps 的重要核心目標嗎，

DevOps 透多很多機制，讓開發和運維之間隔閡可以被打破，結果是軟體更新變得更加快速且安全，也就是說大家可以很放心的在共同的專案中，快速接單更新你自己的那一部分，大部分狀況都會在流程被識別阻擋，那當所謂的快速開發移轉到 AI 時代，不就只是更加的極端而已



這讓我開始思考，以前因為專案期限的關係，團隊對於 DevOps 的投入始終在偏淺，這樣的行為應該要開始反轉了，因為只有 DevOps 思維跟基礎很成熟時，自然就不會煩惱 AI 到底要寫多少的 Code，因為你會有一套自己標準去審查，讓 AI 可以在可控的範圍近讓去發揮他不爽就重構的優勢，



這樣就帶到一個我認為很值得去思考的重點，當科技巨頭說他們用 AI 做了好多事而你們公司卻沒有感受到的時候，其實某總程度你應該要認知到，也許在他們很強大的 infra 下，確實可以讓 AI 做更多的發揮，而你們公司還只是停在訂閱 GitHub Copilot 的程度，如果軟體影響你公司很大的話，你應該要感到害怕才是





## \[結論\]

我覺得從很早期看了黑天鵝一書所開啟的概念可以用在生活、工作、投資…等很多面向，因為當一件事下檔有限上檔無限，你就可以儘量去下注，而 AI 寫程式也被我用上了，以下簡單的總結

1. AI 全面取代寫程式會有更多層次的考量，要去看分佈，而不是平均值

2. 先區分目前軟體開發是哪一階段，重點會是什麼，重速度還是整體系統架構

3. 透過整體 Infra 機制讓 AI 在可控的範圍內盡量去發揮他們專長，這通常和微服務想達成的效果一樣，越接近終端的變化會越快，那你應該盡量讓 AI 在那一層發揮所長
