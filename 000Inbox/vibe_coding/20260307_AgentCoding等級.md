# Agent Coding 自駕等級（微調版）

我建議改成：

* **Level 0：手排**
* **Level 1：駕駛輔助**
* **Level 2：受控代駕**
* **Level 3A：Agent 設計**
* **Level 3B：AI Coding Review**
* **Level 3C：自動驗測 / 資安檢查**
* **Level 4：受控自治交付**

這樣有幾個好處：

* 保留「Level 4 是明顯高一階」的象徵感
* 中間有三個可以逐步升級的 3A / 3B / 3C
* 很符合你們這種 6–10 人小團隊慢慢擴散到更多團隊的現實路徑

---

# Level 0：手排

## 定義

幾乎沒有正式 Agent Coding 流程。
工程師主要自己寫，AI 只偶爾拿來問問題、查錯、補語法。

## 特徵

* 沒有共同規範
* 沒有共用 prompt / context
* 沒有固定驗收格式
* 沒有區分哪些任務可以交給 Agent

## 目標

先讓團隊開始穩定使用至少一種 Agent 工具，不要求流程成熟。

---

# Level 1：駕駛輔助

## 定義

Agent 已經成為日常開發輔助工具，但還是 **人類全程主導、同步監督**。

## Agent 可做

* 補函式
* 補測試
* 解釋 code
* 修小 bug
* 幫忙重構局部邏輯
* 寫文件初稿

## 特徵

* 每個人會用，但團隊沒統一流程
* 成效高度依賴個人能力
* Agent 沒有獨立任務概念

## 升級關鍵

開始建立團隊共同最小標準，而不是只靠個人習慣。

---

# Level 2：受控代駕

## 定義

Agent 已經可以接手 **低風險、範圍清楚、可驗證** 的任務，但仍以單人受控執行為主。

## Agent 可做

* CRUD / 樣板
* 小範圍重構
* 測試補齊
* 明確 bug 修正
* 文件整理
* 單模組內局部功能

## 必備控制點

* 最小需求卡
* 最小驗收標準
* 基本 CI（build / test / lint）
* main branch 保護
* Repo 級基本 context

## 代表意義

這表示團隊已經不是「亂用 AI」，而是開始進入 **可控委派**。

## 升級關鍵

從「Agent 幫我寫」進到「Agent 幫我一起設計與分解」。

---

# Level 3A：Agent 設計

## 定義

Agent 不只參與寫 code，開始參與 **需求整理、方案展開、設計約束、task 切分**。

## 核心意義

這一級代表團隊開始把 AI 用在「寫之前」，而不只是「寫的時候」。

## Agent 可做

* 把會議內容轉成需求卡
* 根據需求產出 Spec 初稿
* 展開 2–3 個設計方案
* 產出簡版設計約束
* 協助拆 task 與驗收條件
* 產出 prototype 幫助對焦

## 必備控制點

* 固定需求卡模板
* 固定設計約束模板
* 明確 L0 / L1 / L2 分級
* PM / Tech Lead 對「值與邊界」有最後判斷權

## 升級標誌

團隊開始出現這種情況：

* 需求不是直接丟一句話給 Agent
* Agent 會先幫忙整理與分解
* 人類 review 的對象不只是 code，也包括 spec / task /約束

## 為什麼很重要

很多團隊卡住，不是因為 Agent 不會寫，
而是因為**寫之前沒有把東西定清楚**。

---

# Level 3B：AI Coding Review

## 定義

Agent 已經可以比較獨立地完成 coding 任務，甚至在背景執行、開 PR、自審，但 **最終合併仍由人類主導**。

## 核心意義

這一級是從「AI 幫我寫」跨到「AI 幫我完成一整段開發工作」。

## Agent 可做

* 根據 task 背景實作
* 自己跑 lint / test
* 自己修到可提交狀態
* 自動生成 PR 說明
* 先做第一輪自審
* 根據 review comment 再修正

## 必備控制點

* PR template
* AI 使用揭露
* Review checklist
* CODEOWNERS 或至少核心 owner
* 人類具名 review
* 風險任務不能無人 merge

## 升級標誌

出現以下現象：

* Agent 開的 PR 品質開始穩定
* Review 不再只是抓語法，而是抓風險、邊界、可維護性
* 人類主要在做判斷，不再做大量機械修補

## 這一級的本質

> Agent 開始像 junior developer + reviewer assistant 的綜合體。

---

# Level 3C：自動驗測 / 資安檢查

## 定義

團隊已把「驗證」從人工習慣提升到 **可自動執行的守門機制**。
這表示 Agent 的輸出不只是有人看，而是會先被系統性檢查。

## 核心意義

這一級的重點不是寫更多測試，而是建立：

> 沒過驗證，就不能往前走

## 自動檢查範圍

至少包括：

* unit test
* integration / smoke test（依需要）
* lint / type check
* secrets scanning
* dependency / vulnerability check
* 基本 security check
* 必要時 perf baseline 驗證

## 必備控制點

* CI gate 完整接上 PR 流程
* fail fast 機制
* 不通過不得 merge
* L2 任務開始要求 rollout / rollback 考量
* 對高風險區域有額外保護

## 升級標誌

團隊已經不太會說：

* 「看起來沒問題先 merge」
* 「這次先跳過測試」
* 「這只是小改先不要掃」

而是開始說：

* 「先看 gate 有沒有過」
* 「這個 PR 的證據夠不夠」
* 「這個風險等級要不要補真實驗證」

## 這一級的重要性

這是從「AI 可用」走向「AI 可規模化」的關鍵門檻。

---

# Level 4：受控自治交付

## 定義

在明確範圍內，Agent 已能完成從任務接收、設計展開、實作、檢查、PR 到 ready-for-merge 的大部分工作。
人類主要負責：

* 高風險判斷
* 例外處理
* merge / release 最終決策
* 流程優化與治理

## 核心意義

這不是「完全自動上線」，而是：

> 在受控範圍內，Agent 已成為穩定的交付單位。

## 適用範圍

通常先發生在：

* L0 任務
* 標準化模組
* 樣板型功能
* 文件 / 測試 / refactor 類型工作
* 固定框架內的開發

## 必備控制點

* AOD 很清楚
* Task routing 已成熟
* 需求 / 設計 / 驗證模板穩定
* Review 與 CI 完整
* 量測指標固定追蹤
* 有事故與回顧機制

## 代表意義

這時候團隊已經不是「會用 Agent」，而是 **有能力管理 Agent 生產力**。

---

# 為什麼這個版本比原本更好

## 原本問題

如果只有：

* 0
* 1
* 2
* 3
* 4

那所有有點進展的團隊都會卡在 3，
因為 3 裡面其實混了三種完全不同的成熟度：

1. Agent 參與設計
2. Agent 可獨立 coding + PR review
3. 自動化驗測與資安守門

這三者不是同一件事。

---

## 拆開後的好處

你現在可以明確分辨：

* 團隊是卡在 **需求與設計前置**
* 還是卡在 **AI PR review 能力**
* 還是卡在 **CI / security / validation 系統化**

這對內部推動非常重要，因為不同卡點要補的東西完全不同。

---

# 我建議的公司用法

## 公司層級

不要要求所有團隊都追 Level 4。
比較合理的是：

* **基線目標：Level 2**
* **進階目標：Level 3A → 3B → 3C**
* **成熟試點團隊：Level 4**

---

## 小團隊自評

你可以要求每個團隊每季評一次：

1. 我們現在在哪一級
2. 下一級缺什麼控制點
3. 這季只升一小級，不求一次到頂

這樣比喊「今年升到 Level 4」務實很多。

---

# 最後濃縮版

你可以把整套成熟度模型濃縮成這樣：

> Agent Coding 成熟度，不應只看 AI 能不能寫 code，而要看團隊是否已建立從需求、設計、review 到驗測與資安的控制能力。
> 因此成熟度分成：Level 0 手排、Level 1 駕駛輔助、Level 2 受控代駕、Level 3A Agent 設計、Level 3B AI Coding Review、Level 3C 自動驗測 / 資安檢查、Level 4 受控自治交付。
