# AI Coding 流程設計：輕量可擴散版

## 核心原則

這套流程不是為了把每一步都文件化，而是為了保留幾個 **不能交給 AI 決定的控制點**，讓小團隊能先跑起來，之後再擴散到更大的組織。

三個原則：

1. **角色按判斷權劃分，不按產出劃分**
   AI 可以幫忙產出，但不能替人類承擔錯誤決策的責任。

2. **流程按風險分流，不是一條路走到底**
   低風險任務走短流程，高風險任務才補更多控制點。

3. **文件只保留最少必要集**
   只保留能讓 Agent 執行、也能讓人類 review 的最小文件。

---

# 1. PM / PO：價值判斷

## 核心責任

PM / PO 負責回答這些問題：

* 這件事值不值得做
* 成功長什麼樣
* 哪些先做、哪些不做
* 需求切得夠不夠小，能不能交給 Agent 處理

這一段的本質不是「寫文件」，而是把模糊想法變成 **可執行的需求**。

## 最小產物

**需求卡**

建議至少包含：

* 背景 / 問題描述
* 目標結果
* 範圍 / 不做什麼
* 驗收標準
* 反例 / 邊界情境
* 風險等級（L0 / L1 / L2）

## Agent 參與方式

Agent 可以協助：

* 把會議記錄、聊天紀錄、簡報內容整理成需求卡初稿
* 補出模糊需求的缺口
* 協助把需求切成更小的任務

人類要負責最後確認：

* 這件事是否值得做
* 驗收標準是否明確
* 需求是否切得夠小

---

## 1-1. 設計：UI / UX

## 核心責任

設計流程不是每次都必須獨立存在，而是由 PM 根據需求價值與任務性質決定是否引入。

適合引入設計流程的情況：

* 牽涉主要使用者流程
* 介面互動改動較大
* 對轉換率、操作效率、理解成本有明顯影響
* 需求本身還不夠具體，需要用畫面幫助收斂

## 最小產物

視任務大小選一種即可：

* 線框稿
* 簡單 flow 圖
* 高保真畫面
* 可執行 prototype

## Agent 參與方式

Agent 可以：

* 根據需求卡生成 wireframe / UI 初稿
* 協助整理設計規範
* 幫忙將設計稿轉為前端元件草稿

人類設計 / PM 要負責：

* 使用者流程是否合理
* 是否符合產品價值與品牌一致性
* 哪些互動細節不能被 AI 自由發揮

---

# 2. 架構師 / Tech Lead：邊界判斷

## 核心責任

架構師 / Tech Lead 負責回答：

* 最基本的設計邊界是什麼
* 這次改動會影響哪些模組
* API / schema / domain boundary 能不能這樣改
* 哪些不變條件不能破壞
* 權限、資料隔離、非功能需求是否足夠

這一段的本質是：**把不確定性縮小到團隊可以安全執行的範圍**。

## 最小產物

依風險決定：

* L0：不一定需要獨立文件
* L1：1 頁設計約束
* L2：簡版 ADR / 架構決策記錄

內容至少要涵蓋：

* 影響模組
* 介面 / schema 是否變更
* 不可破壞的 invariant
* 權限 / 資料隔離
* 非功能需求（效能、安全、穩定性）
* rollout / rollback 基本策略（L2 必須明確）

## Agent 參與方式

Agent 可以：

* 幫忙整理現有系統影響面
* 產出不同設計方案草稿
* 快速做 prototype 驗證方向

人類要負責最後判斷：

* 哪個方案能採用
* 哪些邊界不可破
* 哪些風險不能接受

---

# 3. 架構師 / PM：風險等級判斷（L0 / L1 / L2）

這一步是流程能不能輕量化的關鍵。
不是所有任務都走同一套，而是先分級。

---

## L0：低風險任務

適用範圍：

* UI 文案
* 單點 CRUD
* log / 註解
* 純樣板
* 低風險重構
* 測試補齊

### 骨架

```text
需求卡 → 驗證 → Agent → Review
```

### 特點

* 不需要長文件
* 可讓 Agent 高度參與
* 人類保留 merge 前 review

---

## L1：中風險任務

適用範圍：

* 單服務功能開發
* 一般報表邏輯
* 既有 API 增修
* 小範圍 schema / flow 調整
* 中度業務邏輯變更

### 骨架

```text
需求卡 → 設計約束（簡版） → 驗證 → Agent → Review
```

### 特點

* 增加 1 頁設計約束
* 讓 Agent 在明確邊界內執行
* Review 仍由人類主導

---

## L2：高風險任務

適用範圍：

* 跨服務協定變更
* schema migration
* 權限模型
* 敏感資料處理
* 效能敏感改動
* 對外 API / workflow / infra 變更
* 不可逆操作

### 骨架

```text
需求卡 → 架構/ADR → 驗證 + rollout/backout → 受監督 Agent → 多人簽核 → 分段釋出
```

### 特點

* 不能只靠 Agent 自跑
* 必須有人持續監督
* 必須明確規劃 rollback / backout
* 合併與上線是兩個獨立判斷點

---

## 3-1. 彈性合併原則

這套流程要能落地，重點不是多一段少一段，而是 **哪些可以合併，哪些不能省**。

### 可合併一：需求定義 + PRD / Spec

對 70% 任務，不需要兩份文件。

只要需求卡已包含：

* 目標
* 範圍
* 驗收
* 反例
* 風險級別

就已足夠讓 Agent 開始工作。

---

### 可合併二：架構 Review + Spec

對小團隊，很多時候架構 review 本質上只是 review 一份 1 頁設計約束。

所以不需要再另外要求一份長篇 spec。

---

### 不能省的東西

不管任務大小，以下幾個不能省：

* 明確的需求目標
* 最基本的設計邊界
* 驗證標準
* 人類 review
* 高風險任務的 rollout / rollback 思考

# 4. 共同驗收標準

這一段是整套流程的核心。
不先定義「怎樣算對」，Agent 只會很有效率地做錯事。

共同驗收標準分三層：

---

## 4-1. PM 的業務驗證

確認這次改動是否真的達成原始商業目的。

例如：

* 功能是否符合需求卡定義
* 成功條件是否真的滿足
* 邊界情境是否被處理
* 是否符合優先序與範圍控制

這是「值不值得做、做完有沒有價值」的驗證。

---

## 4-2. 架構的驗證

確認這次改動沒有破壞系統邊界與結構品質。

至少要檢查：

* 介面是否一致
* API / schema 是否符合設計約束
* 權限與資料隔離是否安全
* perf baseline 是否明顯退化
* security checks 是否通過

這是「系統能不能承受這次改動」的驗證。

---

## 4-3. 程式的驗證

確認這次改動具備最基本的工程可信度。

至少應包含：

* unit tests
* lint / type check
* 可執行驗證
* 必要時的 integration / smoke test

這是「這段 code 是不是只是看起來對」的驗證。

---

# 4.Q：驗證標準

這裡我幫你整理成一個獨立段落，因為這其實是整個流程最重要的控制介面。

## 定義

**驗證標準 = 在任務完成前，先定義什麼證據能證明它是對的。**

重點不是「有沒有測試」，而是：

* 測了什麼
* 為什麼這樣測足夠
* 哪些錯誤一定要被擋住

---

## 驗證標準的三個層次

### A. 業務驗證

由 PM / PO 主導

回答：

* 有沒有解決原始問題
* 有沒有達成業務目標
* 使用者看到的結果是否正確

例子：

* 報表查詢時間從 5 秒降到 2 秒內
* 使用者能在 3 步內完成提交
* 新流程不影響既有關鍵任務完成率

---

### B. 結構驗證

由架構師 / Tech Lead 主導

回答：

* 有沒有破壞架構邊界
* API / schema / domain model 是否一致
* 安全、效能、隔離性是否維持住

例子：

* API contract 不變或有版本策略
* schema migration 可回滾
* 不增加跨模組耦合
* P95 latency 不超過 baseline 的 10%

---

### C. 程式驗證

由開發 / QE 主導

回答：

* 程式是否穩定可執行
* 關鍵邏輯是否有測試保護
* Agent 改壞時能不能被自動發現

例子：

* 單元測試通過
* integration test 通過
* lint / type check 通過
* secrets scanning / security check 通過

---

## 驗證標準的最低要求

對每張可交給 Agent 的任務，至少要回答這三句：

1. **成功長什麼樣？**
2. **怎麼驗證成功？**
3. **失敗最可能發生在哪裡？**

如果這三句答不出來，就不該交給 Agent 自主跑。

# 5. 最小文件集

為了保持輕量，我建議整套流程固定只維護這幾種文件：

## 必備

1. **需求卡**
2. **驗證標準**
3. **PR / Review 紀錄**

## 視風險增加

4. **設計約束（L1）**
5. **ADR / rollout-backout 記錄（L2）**

## 專案級共用

6. **AI 協作規範檔**

   * 專案 context
   * coding 規範
   * review 規範
   * Agent 可做 / 不可做範圍

---

# 6. Agent 在這套流程中的定位

## Agent 主導、人類審核

適合：

* 需求卡初稿整理
* 規格初稿整理
* UI 草稿 / prototype
* 測試骨架生成
* 一般程式實作
* 重構 / 樣板 / CRUD
* PR 說明整理
* 第一輪自審

---

## 人類主導、Agent 輔助

適合：

* 價值判斷
* 需求優先序
* 架構邊界
* 驗證標準定義
* 高風險改動
* rollout / rollback 決策
* 最終 merge / release 判斷
