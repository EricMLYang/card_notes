# AI Agent 框架的投资陷阱：做空模型进步 vs 押注计算能力

> 来源：ihower 电子报，引用 Minh Pham 文章

## 核心论点：大多数 AI Agent 框架在做空模型进步

大多數 AI Agent 框架可能犯一個經典的投資錯誤：把資源全壓在人工設計的「腳手架」上，而不是押注在模型本身的能力成長曲線。

看 #ihower 電子報裡提到 Minh Pham 這篇文章講了一個我挺有感的觀點：當你的 Agent 系統靠的是「加更多節點、加更多子代理、加更多工作流程」來提升可靠性，你其實不是在 scale 計算能力，你是在 scale 人力。

## The Bitter Lesson：手工知識 vs 通用計算

這跟 Rich Sutton 1919 年那篇 The Bitter Lesson 說的一模一樣：手工嵌入人類知識的系統，短期看起來聰明，長期永遠輸給 scale computation 的通用方法。

### 投資語言的映射

映射到投資語言就很清楚了：大多數 Agent 框架是在做空模型進步的速度。

你花三個月精心設計的 workflow，本質上是一張做空波動率的單子：
- **模型沒進步時**：你賺穩定的小錢（系統照常運行）
- **模型能力一旦跳升**：你之前固化的架構不會自動變簡單，反而成了技術負債。你爆倉了。

## 正確的架構設計檢驗標準

真正 Bitter Lesson Pilled 的架構長什麼樣？文章給了一個極好的檢驗標準：

**如果明年模型能力翻倍，你的系統會不會在不做大改的情況下變得更便宜、更簡單、更可靠？**

如果答案是否定的，你就是把太多「智慧」塞進了腳手架，而不是留給模型。

## Caveat：Thin Harness ≠ No Harness

當然也是有 Caveat：這不是說今天就該把所有 workflow 拆掉。產品需要確定性、可稽核性和安全控制，thin harness 不等於 no harness。

但你的長期押注方向要對：**結構應該從學習中湧現，而不是從設計中強加。**

## 結論

市場對精巧但無法 scale 的設計從不仁慈，但對押注計算能力的人，回報向來慷慨。

# 拆解結果

## A. 主脈絡與個人映射
- **論證骨架**：引用 Minh Pham 文章，將 AI Agent 框架設計映射到投資語言——加節點加子代理 = scale 人力 = 做空模型進步。用 Rich Sutton 的 Bitter Lesson 做底層支撐，最後給出「模型能力翻倍時系統會不會自動變簡單」的檢驗標準。
- **個人映射**：對系統建造者最直接的衝擊——你在設計 Agent 架構時，每多加一層人工設計的 workflow，就多下了一張做空模型進步的單子。判斷框架可直接套用到所有技術架構決策。

## B. 卡片（Zettel）

序號 1
- 標題：Agent 架構的做空檢驗：模型翻倍時你的系統會不會自動變簡單？
- 類型：Heuristic
- 概念（50–300 字）：大多數 AI Agent 框架靠加更多節點、子代理、工作流程提升可靠性，本質是在 scale 人力而非計算能力。用投資語言講：花三個月精心設計的 workflow 是一張做空波動率的單子——模型沒進步時賺穩定小錢（系統照常運行），模型能力跳升時你之前固化的架構不會自動變簡單，反而成了技術負債，你爆倉了。檢驗標準極簡：**如果明年模型能力翻倍，你的系統會不會在不做大改的情況下變得更便宜、更簡單、更可靠？** 答案是否定的，你就把太多「智慧」塞進了腳手架而不是留給模型。
- 重要性（1 句）：這是設計 Agent 架構時最核心的一個自我檢驗問題，能避免在即將過時的人工設計上過度投資。
- 邊界/反例（1–2 句）：Thin Harness ≠ No Harness——產品需要確定性、可稽核性和安全控制，不是今天就拆掉所有 workflow。短期內模型進步可能停滯，此時 workflow 投資有正回報。
- 知識鉤子：與 #BitterLesson 形成投資語言版的實踐映射；可作為 #從工程問題到領域問題 的架構決策前置判斷。

序號 2
- 標題：The Bitter Lesson 投資版：手工知識系統做空模型進步，通用計算做多
- 類型：Principle
- 概念（50–300 字）：Rich Sutton 的 The Bitter Lesson 指出：手工嵌入人類知識的系統短期看起來聰明，長期永遠輸給 scale computation 的通用方法。映射到投資語言：你在押注腳手架（做空模型進步）還是押注計算能力（做多通用方法）？核心區分是「你在 scale 人力還是 scale 計算能力」——加更多人工設計的節點 = scale 人力，讓模型用更多 compute 解決問題 = scale 計算。市場對精巧但無法 scale 的設計從不仁慈，但對押注計算能力的人回報向來慷慨。結構應該從學習中湧現，而不是從設計中強加。
- 重要性（1 句）：這是跨 AI 時代所有技術架構決策的底層判斷框架——面對新技術時，先問自己在做空什麼、做多什麼。
- 邊界/反例（1–2 句）：在受監管場景（醫療、金融）下，pure compute scaling 缺乏可解釋性和可稽核性，人工設計的 guardrails 仍有不可替代的價值。
- 知識鉤子：是 #Agent架構做空檢驗 的底層理論支撐；與 #不可壓縮才是護城河 形成互補——Bitter Lesson 說哪些會被壓縮，Naval 說哪些壓縮不了。
