# 用 Interface + Test 驯服 AI Coding 的不确定性

> 作者经验分享：如何用软件工程基础技巧控制 AI Coding 的质量

## 核心挑战

過年前聊了 prototype, 今天來聊我閃開 ai coding 不確定性的對策吧。用 coding agent 要有效率最大的挑戰有兩個, 一個是需要交代清楚需求 (要給 AI 明確的規格), 另一個是要控制不確定性 (要 review AI 產出的成果), 兩件事其實都是吃力不討好的任務..

其實把 AI 換成 "人" 來看, 這些事情是有方法可以面對的, 過去軟體工程的技巧都告訴你了, 只是過去要 "寫 code" 成本很高, 沒什麼人願意為了品質多寫一點 "不是最終會被交付" 的 code... ( 例如 poc, unit test .. )

## 方法論：用 Code 定義邊界

最近我手上有個專案, 我就正式的跑了一次, 效果意外的好, 簡單分享一下我的想法跟步驟:

### 1. 用 code 來寫規格（定義 interface）

### 2. 用 code 來驗收規格（替 interface 寫 test case）

這些 code, 我會親自逐行 code review, 確保跟我想像的一樣｡這些 code 驗證完, 我會特別保護起來 (區隔不同的 git repo, 或是拿掉權限, 總之這兩包 code 沒有我同意我不會讓 AI 亂改)

## 工作流程

這時整個開發的頭跟尾都已經準備好, 中間的路就讓 coding agent 盡量發揮, 可以放著讓 ai 慢慢跑, 直到完成為止. 

- ai 沒有符合規格 (interface), 理論上 build 就會失敗
- ai 寫的 code 沒有符合預期的行為, 理論上 test 就會失敗
- ai 寫完的 code 要是跟我想像的行為不同, 我會回到 (2) 把測試補齊
- 如果最後發現設計有缺失, 我會回到 (1) 把 interface 調整到位, 然後重複上述步驟

真正需要 "人" 介入的, 是開發的需求 (1) 跟驗收條件 (2), 規格開的好不好, 設計跟架構的品味好不好, 就是看 (1) 跟 (2)。

## 直接 vs 間接 Code Review

去年底 .NET Conf 最後的爐邊會談, 董大偉老師問了我一個問題：

**"你會不會 review AI 寫的每一行 code?"**

當時我說 "會", 不過當時我講完後面還有一段話沒講... 我會 "直接或間接" 的 review 每一行 code ...

實際上就是:

### 1. 直接 Review：Interface + Unit Test

(1) 跟 (2) 我會親自 code review, 因為他代表我的設計, 我甚至會親手寫部分的 interface / unit test code, 因為寫的同時, 我也在檢視 "這 spec 開的好不好? 這 interface 定的好不好?", 藉由親自寫 unit test, 我可以抓到 "其他 developer 用這 interface 的 dx 好不好", 這是手感, 也是大家所謂的 "品味"

### 2. 間接 Review：交給編譯器和測試

因為這些東西都是用 code 來定義的, 因此搭配編譯器, 他就能 "間接" 的幫我 review 其他 ai 產出來的 code 了｡ 骨架打好了, 肉再長也歪不到哪裡去 (頂多肥了一點), 以後鍛鍊一下減肥就沒事了

## 過去的應用場景

其實這些習慣, 過去我很早就這樣做了, 原因很多, 只因為我有 coding 能力, 而且我抽象化能力還算不錯, 寫這種 code 寫的很快, 過去我通常拿這方法來:

1. 要求外包商, 用系統化的方式持續驗證每次的交付都符合我的期待
2. 要求配合的開發團隊, 用系統化的方式持續驗證每次的交付都符合我的期待
3. 用 code + comments 來表達意圖, 修正到確認可行後再變成正式文件

## Code 作為精準的設計語言

雖然很多人都說 "不要再寫 code" 了, 寫不過 AI 的, 但是我覺得 **code 仍舊是表達你對系統的設計跟意圖最精準的語言**, 你不需要為了實現功能寫那麼多 code, 但是你仍然可以拿 code 有效的跟 AI / 其他工程師溝通。

因為多寫了這些看似非必要的 code, 事實證明這讓我事後省了很多麻煩事, AI 可以一次到位, 因為寫錯了他自己會知道, 然後就會自己修正了。

過去這方法其實推不大動, 其他人看到要多寫非必要的 code 就止步了, 直到現在 ai 大幅降低 coding 的成本, 一方面多寫這些 code 不再是麻煩事, 同時 ai 的速度也凸顯了很多 "外包" 給 ai 開發帶來的各種不確定性問題, 我過去的習慣意外的適合在這個年代 ai coding 的需求

## 軟體工程基礎的重要性

這個案例凸顯的是, **如果你對基礎的軟體工程技巧越熟練, 你越知道如何應付新的工具**｡沒有借力使力 (用少量的 interface + unit test code 來替你驗證更多的 code), 並且再來源階段就確保品質, 建立一個能持續驗證跟改善的 pipeline, 你就越難發揮 ai 帶來的生產力提升

### 生產力提升的差異

現實的狀況是: **ai 其實可以用 100x 速度開發, 但是你的控制能力有限, 加速到 5x 就無法再快了** (review 不完, 或是狀況開始失控), 有經驗的人就能發揮到 20x, 差別就在這裡

## 過去累積的技術基礎

從這角度來看, 其實我過去都在朝這路線累積, 例如:

1. api first / contract first, 用狀態機來收斂 interface 的設計
2. contract -> test
3. decision table + vibe testing

## 不要放棄設計和檢核

AI coding, 就不用跟他拚寫 code 了, 但是**該掌握好的設計跟檢核是少不得的，一旦放棄了，你在開發流程中的必要性也不存在了**。

同時, 在設計還未完全定案的時候, 改來改去是免不了的, **code 的修改 遠比修改 docs 容易得多**, 你不會改了這邊忘了那邊 (有重構功能輔助, 也有測試幫你留意)。

另外你寫的 unit test 也沒有浪費, 除了用 test 定義你的期待之外, **這也是給 AI 參考很有幫助的 "sample code"**, 告訴 AI 你期待怎麼使用他, 後面它寫出來的 code 會更精準 (這些都是單純文件無法做到的境界)

## 影響力擴大：從 100x 速度到 100x 影響力

我平日的任務, 更常見的是定義好一組規格後, 可能有多個專案或是團隊需要同時參考, 這時這樣做的效益就更大了。

**我沒辦法做到 100x 的速度, 但是我可以讓影響力擴大到 100x**, 因為我同時也替多個專案準備好高品質且一致性高的 spec ( interface + test + docs ), 讓他們能有效率的交給 AI 進行開發 (SDD)

## 總結

這些都是當年誤打誤撞, 但是有把基礎弄好, 現在突然拿到 coding agent 這種神兵力氣, 用起來意外的順手, 這也算是個收穫吧! 這邊的做法, 其實過去在談 API First 都有聊過, 我就把過去在 .NET Conf 談這題目的連結放在第一則留言~

# 拆解結果

## A. 主脈絡與個人映射
- **論證骨架**：從 AI Coding 的兩大挑戰（需求不清 + 品質不可控）出發 → 提出用 Interface 定義規格 + Test 定義驗收條件 → 區分直接 Review（Interface/Test）vs 間接 Review（編譯器+測試）→ 論證 Code 仍是表達設計意圖最精準的語言 → 收束到「放棄設計和檢核 = 放棄開發流程中的必要性」。
- **個人映射**：直接命中系統建造者的核心——用少量高品質 code（Interface + Test）驅動大量 AI 產出，是典型的槓桿操作。「從 100x 速度到 100x 影響力」的觀點挑戰了只追求速度的思維。

## B. 卡片（Zettel）

序號 1
- 標題：AI Coding 的骨架策略：用 Interface + Test 鎖住頭尾，中間放給 Agent 跑
- 類型：Heuristic
- 概念（50–300 字）：AI Coding 的核心挑戰是需求不清和品質不可控。解法：(1) 用 code 定義規格（Interface）；(2) 用 code 定義驗收條件（Test）；(3) 這兩層親自逐行 review 並保護起來（不讓 AI 改）；(4) 中間的實作全交給 Agent 跑到通過為止。直接 Review Interface + Test（代表你的設計意圖和品味），間接 Review 靠編譯器和測試自動驗證其餘 code。骨架打好了，肉再長也歪不到哪裡去。Unit Test 還兼任 AI 的 Sample Code——告訴 AI 你期待怎麼使用它，比文件更精準。Code 修改也比 Docs 容易（重構工具 + 測試保護），在設計未定案階段尤其有優勢。
- 重要性（1 句）：這是目前最實用的 AI Coding 品質控制方法——用最少的人工投入（Interface + Test）獲得最大的品質保證。
- 邊界/反例（1–2 句）：前提是開發者有足夠的抽象化能力來定義好 Interface；對於 UI/UX 密集型或探索性原型開發，Interface 定義成本可能過高。
- 知識鉤子：與 #TDD作為AI控制介面 形成完美互補——一個是實踐方法，一個是產業分析；可作為 #Agent架構做空檢驗 的軟體工程層實踐。

序號 2
- 標題：AI Coding 的控制能力差距：新手 5x vs 有經驗者 20x，差別在骨架品質
- 類型：Leverage
- 概念（50–300 字）：AI 可以用 100x 速度開發，但你的控制能力有限——加速到 5x 就 review 不完或狀況失控。有經驗的人能發揮到 20x，差別在於：(1) 能否用 Interface + Test 建立自動驗證的骨架；(2) 能否借力使力——用少量 code 間接 review 大量 AI 產出。更大的槓桿在影響力：一套高品質規格（Interface + Test + Docs）可以服務多個專案或團隊，從 100x 速度升級到 100x 影響力。過去 API First / Contract First 方法推不動，因為「多寫非必要 code」成本高；現在 AI 降低了 coding 成本，同時凸顯了品質控制的需求，這套方法意外地完美契合。
- 重要性（1 句）：揭示了 AI Coding 時代的真正稀缺能力不是寫 code 速度，而是定義骨架的抽象化能力和軟體工程基礎。
- 邊界/反例（1–2 句）：20x 的上限隨模型進步可能繼續提升；但控制能力的差距可能反而擴大，因為更強的模型需要更好的規格才能發揮。
- 知識鉤子：與 #軟體開發價值遷移 直接呼應——價值從「寫實作」遷移到「定義標準」；可作為 #AI是槓桿你是支點 的具體技術實踐版本。

序號 3
- 標題：Code 是表達系統設計意圖最精準的語言，放棄它就放棄了必要性
- 類型：Stance
- 概念（50–300 字）：AI Coding 時代不需要為實現功能寫那麼多 code，但 code 仍是表達系統設計和意圖最精準的語言。原因：(1) code 有編譯器和測試保護，改了這邊不會忘那邊；(2) code 可重構，比改文件穩定；(3) Unit Test 同時是 spec 和 sample code，純文件做不到。放棄設計和檢核，你在開發流程中的必要性也不存在了——不是不寫 code，而是寫的 code 從「實作」轉向「規格和驗收」。
- 重要性（1 句）：這是對「不用再寫 code」論調的有力反駁——重點不是寫不寫，而是寫什麼。
- 邊界/反例（1–2 句）：若 AI 未來能可靠地從自然語言生成 Interface + Test，這個主張的強度會減弱。目前 AI 生成的 Test 品質仍不夠可靠，需要人工校準。
- 知識鉤子：與 #不可壓縮才是護城河 形成呼應——設計意圖和品味是不可壓縮的部分；與 #從工程問題到領域問題 形成互補——兩者都指向「人的價值在設計層而非實作層」。
