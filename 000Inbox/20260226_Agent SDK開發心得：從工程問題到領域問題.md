# Agent SDK 開發心得：從解決工程問題到解決領域問題

> 作者經驗：從 LLM function calling / Multi-Agent 轉向 Coding Agent + SKILLs + MCP
> 參考演講：Anthropic "Don't Build Agents, Build Skills Instead"

## 方法論轉變

最近的主力, 都押在 Agent 的開發跟研究, 而我放棄了過去手刻程式碼的方式來打造 Agent ( 就是 LLM function calling / Multi-Agent + MCP ), 改用通用的 Coding Agent + SKILLs + MCP 的做法快速的驗證, 之後再用 Coding Agent 的 SDK 將這些設計整合到正式的服務上。

這方法我嘗試了幾個月, 效果很不錯, 正好今天看到 ihower 的龍蝦整理了這場演講內容, 另外也看到 Claude Code 今天發布的新功能: Remote Control (連結我放留言), 剛剛好都跟我最近在研究的主題有沾上邊, 就寫了這篇來聊一下 Agent SDK 這主題吧

## 工作流程

前面的 POST 聊到 (連結我放留言), 我先拿 Coding Agent 當作通用的 Agent, 先在上面寫 SKILL, 寫 MCP, 快速打造 MVP 驗證後, 再花點時間用 SDK 整合到既有服務上, 而我採用的 Coding Agent 有兩個: 一個是 Claude Code, 另一個是 GitHub Copilot, 而這兩套 Agent 正好都有對應的 SDK 可以用。

## 兩種 SDK 設計哲學：加法 vs 減法

這兩套 SDK 的設計哲學挺不一樣的, 最大的差異在於一個是加法, 另一個是減法, 衍伸出不一樣的使用方式跟適用場景。分別簡單介紹一下:

### 1. Claude Agent SDK（減法）

從 Claude Code 的成功, 把 CLI 中處理 Agent 的核心邏輯抽取出來而成的 SDK, 簡單的說官方的 CLI 就是用自家的 SDK 包裝出來的, 剝掉 CLI 這層外殼拿到的 SDK, 你可以拿 SDK 去做客製化調整來配合你的應用, 理論上你會有能力在你的服務內重現 Claude Code 的各種能力

**特性**：
- 抽取 CLI 核心邏輯而成
- SDK 就是核心，純 Code Library
- 整個 Agent Core 在你的 Process 內運行
- 原生支援 Node / Python

### 2. GitHub Copilot SDK（加法）

這套 SDK 則走另一個路線, 不去拆解 CLI 抽出核心, 反過來是把 CLI 加上一層通訊協定, 讓 SDK 能更容易的 "遙控" GitHub Copilot CLI 來替你做事｡你一樣有能力再你的服務內重現 GitHub Copilot CLI 的各種能力, 因為背後就是靠真正的 CLI 在替你做事

**特性**：
- CLI + 通訊協定
- SDK 遙控真正的 CLI
- 先天分離部署（獨立 Process）
- 內建 server mode（`--headless --port 1234`）

## 部署架構差異

因為這些差異, 也導致兩者在運行 Agent 跟部署方式都有很大的不同。

**Claude Agent SDK**：
- 單純的 SDK / Code Library
- Agent Core 在你的 Process 內運行
- 主體怎麼部署就怎麼運行

**GitHub Copilot SDK**：
- SDK 負責「遠端遙控」CLI
- 分別部署 code 跟 CLI
- 先天就是獨立 Process
- 更容易做分散式處理

## 為何選擇 GitHub Copilot SDK（.NET 開發者視角）

為了因應這樣的設計, GitHub Copilot CLI 內建了 server mode ( 參數加上 --headless --port 1234 就可以了 ), 這樣的設計對想偷懶的我而言是大加分, 原因有幾個:

### 1. 多語言支援更友善

我基本上都是 .NET 為主, Claude 的 SDK 原生只支援 Node / Python, 要支援 .NET 通常得封裝一層 API server, 而 Copilot 的做法內建這樣的支援了

### 2. 環境一致性

Claude 的作法給我更大的彈性去自訂 Agent 的組態, 但是我想要無腦的搬移, 這時不論是 Prototype 或是 Production 都用同一套 CLI 來運行, 只差在我直接操作或是透過 SDK, 環境更一致

### 3. 官方多語言支援

因為架構上的差異, GitHub Copilot SDK 更容易支援多種語言, 我慣用的 .NET 也在官方支援清單內, 省了我不少麻煩

### 4. 部署更單純

部署更單純, 我可以直接把 CLI 封裝在 container 內, 由於沒有我自己封裝的程式碼, 部署與更新更容易了, 跟我服務的版本更新沒有關聯, 只跟 Agent / CLI 有關

## 最大的感想：從工程問題回到領域問題

經歷過這些, 突然發現這跟我過去兩年, 嘗試過各種不同的技術手段實作的 Agent 差異很大啊, 從 GPTs 版本的安得魯小舖開始, 我也用 Semantic Kernel 實作過 CLI 版本的 Shopping Agent, Google ADK / A2A 的版本也嘗試過 (不過這個沒公開), 到現在用 GitHub Copilot SDK 來實作, 差別實在很大, 最大的感想就是:

**我真的開始在「解決領域問題」了, 而不是寫了一堆 code 在解決 Agent 的工程問題｡**

我花了 80% 的時間在評估：
- 我的問題該準備什麼樣的 SKILL, Agent 才能確實的完成任務? 
- 我該配給 Agent 哪些 Tools ( MCP, scripts ), Agent 才能代替我處理外部系統的整合? 

這些都處理完之後, SDK 只是把這些功能整併到既有的 service, UI / API 做好串接, 工程方面的問題就解決了, 這些 Agent 處理問題的能力馬上就融入了我的服務中｡

## 軟體開發的典範轉移

這其實顛覆了過去軟體開發的流程跟觀念, 從分析 / 設計 / 驗證 / 部署其實都不一樣了, 對我來說這才是把整個軟體開發翻過來的變革, AI 寫 code 是在角色跟效率上也把整個軟體開發都翻過來了, 但是本質上還是在 "寫 code", 寫的是現在這個時代的 code, 只是效率提升好幾倍, 變革的地方不一樣

# 拆解結果

## A. 主脈絡與個人映射
- **論證骨架**：作者從自身 Agent 開發經驗出發，論證放棄手刻 Agent（LLM function calling / Multi-Agent）改用 Coding Agent + SKILLs + MCP 的效率差異，再比較兩種 SDK（Claude 減法 vs Copilot 加法）的設計哲學，最後收束到核心觀點——真正的變革是從解決工程問題到解決領域問題。
- **個人映射**：直接挑戰了系統建造者容易陷入的陷阱——花太多時間在 Agent 的工程基礎設施上，而非思考 SKILL 和 Tools 的領域設計。「80% 時間在領域問題」是對 Bitter Lesson 的實踐版呼應。

## B. 卡片（Zettel）

序號 1
- 標題：Agent SDK 兩種設計哲學：減法（抽取核心）vs 加法（遙控 CLI）
- 類型：Model
- 概念（50–300 字）：Claude Agent SDK 是減法——從 CLI 成功中抽取核心邏輯為 SDK，Agent Core 在你的 Process 內運行，純 Code Library，原生 Node/Python。GitHub Copilot SDK 是加法——CLI + 通訊協定，SDK 遙控真正的 CLI，先天分離部署，內建 server mode（`--headless --port`）。關鍵差異在部署架構：Claude SDK 是同 Process（彈性高但語言綁定），Copilot SDK 是獨立 Process（語言無關但多一層通訊）。選擇取決於：語言生態（.NET 只有 Copilot 官方支援）、環境一致性需求（Prototype 和 Production 用同一套 CLI）、分散式部署需求。
- 重要性（1 句）：為「用哪套 SDK 整合 Agent 到既有服務」提供明確的選型框架。
- 邊界/反例（1–2 句）：SDK 生態快速演進，這個對比的保鮮期可能只有 6-12 個月。若需要深度客製化 Agent 行為（如自定義 tool execution），Claude SDK 的同 Process 架構更有彈性。
- 知識鉤子：與 #Agent架構做空檢驗 形成架構選型的互補——一個判斷「要不要做」，一個判斷「用什麼做」。

序號 2
- 標題：Agent 開發的典範轉移：80% 時間從工程問題搬到領域問題
- 類型：Leverage
- 概念（50–300 字）：作者經歷 GPTs → Semantic Kernel → Google ADK/A2A → Coding Agent SDK 的完整路徑後，最大感想是：用 Coding Agent + SKILLs + MCP 做法後，80% 時間花在評估「該準備什麼 SKILL」和「該配什麼 Tools/MCP」——這些都是領域問題，不是工程問題。工作流程變成：先用 Coding Agent 當通用 Agent 寫 SKILL/MCP 驗證 MVP → 確認可行後用 SDK 整合到既有服務。SDK 只負責「把功能整併」，工程量極小。核心呼應 Anthropic 理念：Don't Build Agents, Build Skills Instead。軟體開發的分析/設計/驗證/部署典範全部翻轉。
- 重要性（1 句）：對系統建造者的關鍵提醒——如果你還在花大量時間寫 Agent 的工程 code，你可能正在用舊範式解決新問題。
- 邊界/反例（1–2 句）：前提是通用 Coding Agent 的能力已經夠強；對於需要極致延遲或特殊安全要求的場景，仍可能需要手刻 Agent 核心。
- 知識鉤子：是 #BitterLesson投資版 的實踐案例——把「智慧」從腳手架搬到 SKILL 設計；與 #軟體開發價值遷移 形成呼應，都指向「寫 code 本身的價值在下降」。
