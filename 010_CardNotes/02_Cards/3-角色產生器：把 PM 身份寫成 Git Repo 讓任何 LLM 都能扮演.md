# 角色產生器：把 PM 身份寫成 Git Repo 讓任何 LLM 都能扮演

**類型**：Pattern

## 概念

將一個角色的完整決策能力結構化為 Git Repo，包含四層檔案：`identity.yaml`（身份與決策標準）、`principles.md`（決策邏輯與優先序）、`context/`（產品背景、數據架構、客戶情境）、`product/`（進行中的 OPP 與 Backlog）。這四層加在一起就是「角色產生器」——餵給任何 LLM，它就能用該角色的視角、語言和優先序回答問題。角色從此存在於 Repo 而非某個人的腦子裡。關鍵設計：用 Git 管理意味著角色有版本歷史，三個月後回來看訪談記錄，能知道當時的 PM 處於什麼 Backlog 狀態下做了什麼判斷。這把「人走茶涼」的隱性知識問題轉化為可追溯的結構化資產。

## 重要性

讓組織角色從「綁定個人」變成「可實例化的模板」，是 Agent 時代組織知識資產化的基礎設施。

## 邊界/反例

角色 Repo 的維護成本不低——context/ 和 product/ 需要持續更新，過時的 Repo 比沒有更危險（Agent 會用過期資訊做出看似合理的錯誤判斷）。適合決策頻繁、需要跨團隊協作的角色，不適合職責模糊或快速變動的探索期角色。

## 知識鉤子

與 [[4-CLAUDE.md 是 AI 助手的腦——投資配置指數回報]] 形成「從單一配置文件到完整角色 Repo」的演進路徑；可作為 [[3-Context Graph 捕獲組織隱性知識]] 的個人級實踐——Context Graph 捕獲組織行為模式，角色 Repo 則捕獲個人決策模式。與 [[3-Agent 開發的典範轉移：80% 時間從工程問題搬到領域問題]] 呼應：定義角色就是在做領域建模。
