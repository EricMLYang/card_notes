# 20–50 個真實失敗就能啟動 Eval — 完美覆蓋率是延遲投資的藉口

- 狀態：KEEP
- 類型：Heuristic
- 分類：10-
- 索引：010_CardNotes/01_Index/10_testing_operations_and_reliability.md
- 來源：005_PARA/02_Areas/AI_Software_Data/Agent/20260228_Anthropic_拆解Agent評估的迷霧.md（#3）

## 核心命題
等覆蓋率夠了再做 eval 是組織常見的拖延藉口；早期 agent 變動大、effect size 大，從 bug tracker 與 support queue 撈 20–50 個真實失敗，就足以啟動評估循環。

## 卡片內容
團隊常以為 eval 要上百題才有意義，所以一直不開始；但等越久越貴——你會被迫從已上線系統反推成功標準，那時 ground truth 早被舊行為污染。20–50 個真實失敗的好處是：來自真用戶情境、effect size 顯著到可以早早分辨改動好壞、且每題都能講清楚「為什麼這算失敗」。這是把 eval 從研究級任務拉成小團隊也能起步的最小可行起點。題目要求：無歧義、附 reference solution、來自真實 failure mode。

## 使用情境
- 小團隊要做 agent productization 但沒有 dedicated eval team
- 面對「我們資料不夠不能做 eval」的拒絕
- 為新功能定義第一版驗收標準

## 邊界 / 失效條件
- 20–50 題容易過擬合，需設計擴張節奏
- 若失敗模式集中在幾個 root cause，樣本可能誤導 prioritization
