# 架構師隨筆：當計算轉為純執行，評估標準成了唯一的控制介面

最近在檢視幾個內部與客戶的 AI 專案落地狀況，有一個很深的感觸。隨著大語言模型的能力提升，我們正經歷一個工程本質的轉變：**「計算」正在變成純粹的「執行」。**

過去，軟體工程師敲鍵盤寫邏輯、數據分析師刻 SQL 撈資料，這些「生成」的動作是我們工作的核心。但現在，AI 生成程式碼或分析報告的邊際成本趨近於零。當 AI 可以在三秒內吐出幾百行邏輯時，工程實務上的真正瓶頸已經不是「怎麼把它寫出來」，而是「我怎麼知道它寫得對不對」。

這意味著，人機協作的溝通介面發生了根本性的轉移。我們不再透過「給予實作指令（Prompt）」來控制系統，而是**以「評估標準（Evaluation）」作為真正的控制介面**。

### 1. 軟體工程：TDD 從「除錯工具」變成「確定性介面」

在軟體開發中，這個現象最直觀的體現就是測試驅動開發（TDD）的定位轉換。

以前我們寫測試，是為了防止未來改 code 時把系統弄壞。但現在面對 Coding Agent，自然語言的指令充滿歧義（例如：「幫我寫個購物車打折邏輯，特價品除外」），AI 很容易漏掉邊界條件。

此時，TDD 的紅綠燈測試（Red/Green）就成了最精確的介面。我們不告訴 AI 該用什麼演算法，而是直接給出斷言（Assert）。當測試寫下 `assert(calculate_discount(cart) == 100)` 時，這就是一道無法妥協的契約。AI 的產出被徹底黑盒化了，只要能通過測試，裡面的實作細節並不重要。開發者的核心價值，變成了「將商業需求無損轉譯為自動化測試案例」的能力。

### 2. 數據分析：從探索未知到「資料合約與常識斷言」

身為數據架構師，我每天面對的不是單純的應用程式碼，而是龐大且髒亂的資料管線（Data Pipeline）。數據分析的本質是探索，我們不可能預先寫下「測試：下個月流失率必須是 5%」。那麼，在 Business Data Science 中，我們的「TDD 介面」是什麼？

答案是**資料合約（Data Contracts）與商業常識斷言（Sanity Checks）**。

當我們讓 AI 去做多張資料表的 Join 或特徵工程時，用來控制它的介面是預先設定好的約束條件。例如：

* **資料品質斷言**：處理後的 `user_id` 必須是唯一值（防堵 AI 寫錯 Join 導致資料發散）。
* **商業邏輯約束**：各區分公司的營收加總，必須等於總公司的帳面總額；本月活躍用戶數絕對不可能大於總註冊人數。

這些統計邊界與恆等式，就是數據分析的介面。我們不需要逐行去 Review AI 產出的 Pandas 或 SQL，只要資料流能通過這些斷言卡點，我們就能信任其計算軌跡。

### 3. AI Agent：從結果盲測到「軌跡評估與 LLM-as-a-Judge」

當任務複雜度拉高到讓 AI Agent 自主執行多步驟任務（例如：蒐集資料、調用 API、整理報告）時，既沒有編譯器可以報錯，也沒有絕對的數學恆等式。這也是目前多數企業專案卡關的地方——Prompt 再長，也防不住 Agent 產生幻覺或呼叫錯工具。

在 Agent 的開發實務中，介面變成了更系統化的 **Evals（評估框架）**。

* **軌跡評估（Trajectory Evaluation）**：我們不再只看最終報告寫得好不好，而是對 Agent 的執行步驟下斷言。它在寫入資料庫前，有沒有先呼叫權限檢查的 API？如果沒有，在「這一步」就直接攔截並要求重試。
* **LLM-as-a-Judge**：對於語意和邏輯的判斷，我們將人類的評分量表（Rubrics）寫成明確的指引，交由另一個專門負責審查的裁判模型來驗證。這其實就是將「Code Review」的機制自動化並抽象到語意層面。

### 結語：Databricks 視角下的基礎設施收斂

這也是為什麼在 Databricks 的數據架構演進中，我們現在將極大的精力投入在「評估（Evaluation）」的基礎設施上。

市場上太多 GenAI 的 PoC 最終無法上線（Production），根本原因就是團隊缺乏系統化的評估標準，導致系統行為不可控。這也是我們在 MLflow 3 中深度整合 Mosaic AI Agent Evaluation 的原因。

不管是推動 Judge Builder 讓團隊能將 Domain Knowledge 具現化為裁判模型，還是完善 Tracing（軌跡追蹤）與 Review App 來收集人類專家的真實回饋，這些都不是為了盲目追逐最新的 AI 噱頭，而是為了解決一個樸素的工程問題：**沒有評估，就沒有控制；沒有控制，就無法落地。**

從軟體的 TDD、數據的合約，到 Agent 的 Evals，背後的邏輯完全一致。未來，最頂尖的工程師與架構師，未必是寫出最漂亮演算法的人，而是那些最懂得如何制定「評估標準（尺）」的人。
