# 結構化訪談取代開會：跨 Agent 知識交換協議

**類型**：Heuristic

## 概念

跨部門會議低效的根因：問題即興、回答靠記憶、紀錄不完整。替代方案是結構化訪談協議：(1) 發起方 Agent 先讀完雙方角色 Repo，理解提案背景；(2) 設計分層問卷，按「現況→痛點→合作價值→可行性→優先度」層層遞進；(3) 受訪方帶著自己的 Repo 上下文逐題回答；(4) 回答直接寫進同一份文件成為永久記錄。關鍵差異在於回答品質——帶 Repo 上下文回答時，答案能精確到數據鏈路和技術架構層次（例如「完全沒有物流資訊，數據鏈路是 AI Camera / CMS → MongoDB → Databricks → 報表」），而傳統會議只會得到「沒有」。訪談文件本身就是最有價值的產出：它是跨角色的知識轉移協議，任何後續參與者或 Agent 都能讀到完整討論脈絡，連推理過程都保留。

## 重要性

把會議從「消耗時間的線性對話」轉化為「可累積、可複用的結構化知識資產」，是 Agent 時代跨團隊協作的新基礎設施。

## 邊界/反例

結構化訪談適合有明確提案方向的探索，不適合需要自由發散的腦力激盪場景。問卷設計品質決定產出品質——問題太封閉會遺漏意外洞察。且需要受訪方有維護良好的角色 Repo 才能發揮精確回答的優勢。

## 知識鉤子

與 [[3-角色產生器：把 PM 身份寫成 Git Repo 讓任何 LLM 都能扮演]] 是配套組件——角色 Repo 是「身份」，結構化訪談是「溝通協議」，兩者加起來才構成完整的 Agent 協作系統。可作為 [[7-B2B PM 的三步用戶反饋框架]] 的 Agent 化升級版。與 [[1-一切都是 Context：對 AI 和對人的底層能力是同一個]] 互為印證：結構化訪談本質上是讓 Context 精確傳遞的協議設計。
