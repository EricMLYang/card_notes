# 先具體後抽象的認知搭建順序

**類型**: Principle

## 概念

傳統教學是「先定義後應用」（先講 API 是 Application Programming Interface，再講用途），這違反人類認知習慣。

**有效順序**：先給具體案例建立直覺，再引入抽象定義和符號。

**對比範例**：

❌ **傳統寫法**：
「什麼是 API？API 是應用程式介面，定義了軟體元件之間的互動...」（讀者已睡著）

✅ **好吸收的寫法**：
「想像餐廳：你是顧客，廚房是系統，服務生是 API。你透過服務生點餐，他去廚房拿菜送回來」，然後才說「這個服務生的角色，就是 API」。

**機制**：類似 3Blue1Brown 從最簡單範例開始，建立直覺後才引入數學符號。這是「鷹架效應」(Scaffolding)：先建立認知支點，再逐步增加複雜度。

## 重要性

顛覆傳統教學順序，提供了一個可檢驗的寫作結構原則，可大幅提高首次理解率。

## 邊界/反例

對已有基礎的讀者可能顯得冗長；某些情境下（如速查文件）需要先給定義。過度依賴案例可能犧牲理論完整性。
