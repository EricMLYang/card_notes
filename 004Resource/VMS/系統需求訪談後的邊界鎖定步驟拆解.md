# 系統需求訪談後的邊界鎖定步驟拆解

<任務>

在系統開發前期的需求作業階段，請建議初幾個主要階段，讓將發散的紀錄，可以轉成客戶認可的需求邊界



<本案件現況描述>

- 電動貨車 Vehicle Management System 開發，已經進行了兩組主要使用單位的訪談，想將目前較發散的紀錄，彙整成後續實際規格前的對焦資訊

- 本系統主要功能是  管理頁面、報表、通知、數據傳輸服務

- 系統開發分成數據分析 (databricks)、應用系統，應用系統又區分報表和其他

- 這份步驟，專注在 "應用系統的 其他"，包含首頁、登入、管理頁面、分頁、權限管理、通知、數據傳輸 ....等，會影響應用層前後端開發的內容，





<Other Context>

- 本分資料是要跟內部工程師討論用，掌控流程用，非對外討論

- 重要目的:  讓分析、報表以外的系統功能，能夠鎖住邊界，未來應用系統開發不要大改

- 報表僅牽扯金質 table 和報表流程，只需先訂出誰看、看什麼、要能匯出 ...等

- 本階段規劃是到正式 規格 定案前的流程，UI 設計可初一些稿，但還不到細部設計





<reference>

- 產出格式: 階段名稱，目的/目標，輸入，活動內容，輸出，注意事項與備註

- 用清單，容易觀看確認的方式產出，但不要表格

- 階段不要太細，頂多2-3 次，例如:  流程圖初稿對焦 ( 頁面稿跟流程 )
