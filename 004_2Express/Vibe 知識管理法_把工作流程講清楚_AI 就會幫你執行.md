【Vibe 知識管理法：把工作流程講清楚，AI 就會幫你執行】

前陣子彙整 AI 在我個人知識管理扮演的角色，
當時的結論，
是繼續用 Heptabase 做深度主題研究，
卡片盒筆記，
就用 Obsidian 搭配 Gemini CLI 處理，
但大概不到半年，
我決定要把 Heptabase 退訂了。


主因是 Coding Agent 這半年的進化速度實在太快了，
快到讓我覺得，
分散在兩個工具之間的摩擦力，
已經大過 Heptabase 白板帶來的好處了。


▋ 為什麼退訂 Heptabase
之前我提到留在 Heptabase 的主因是：
在 Heptabase 寫的東西，在我腦袋比較有留下痕跡。

這個觀點沒有改變，
因為白版很符合我腦袋思考模式。
很適合單一主題研究，
但是漸漸的那些主題的更新越來越麻煩。

反之今年 Coding Agent 能力的提升實在是太有感，
讓人不得不重新思考工作模式，
對比用 Claude code 或 GitHub copilot 短短幾句話就可以做各種處理，
會越來越無法忍受大量內文只能透過迂迴的方式跟 AI 互動。


最關鍵的是，
當我想要調整任何流程只要改一個 Markdown 檔案就好，
不需要等官方開發新功能，
不需要用別人設計的 workaround，
加上各種自動化的誘因。

其他平台的好處已經不足以抵銷無法輕易跟 Coding Agent 互動的痛點了。


▋ 我做了什麼
我目前的知識庫是一個 Git Repo，
同時也是一個 Obsidian Vault，
目前有 200 多張卡片盒筆記的卡片、14 個索引分類、草稿區、發佈區、問題意識區，
結構就是結合打造第二大腦 + 卡片盒筆記的流程，

然後我用 Agent Skills 機制，
規劃基本流程，
每個 Skill 就像一個有 SOP 的小助手，

例如：

．卡片處理類
- break-cards：丟一篇長文進去，自動拆成原子化卡片
- link-cards：幫新卡片找到知識庫裡最相關的既有卡片
- process-inbox：一鍵處理 Inbox → 拆卡片 → 建檔 → 更新索引 → 歸檔原文

．寫作類
- gather-cards：給一個主題，自動從 200+ 張卡片裡撈出最相關的素材
- write-pipeline：從選題到發佈的全自動 pipeline，中間只有 3 個人工檢查點
- resume-draft：隨時可以看所有草稿的進度，從中斷的地方繼續

資訊搜集類
- scout-news：根據我的卡片、草稿、問題意識，自動上網搜相關新聞放進 Inbox

這些 Skill 就是用 Markdown 寫的 SOP 文件，
告訴 Agent 遇到什麼情境該怎麼做、用什麼順序執行、在哪裡暫停讓我決策。

但組合起來就形成了一個完整流程：
- 根據個人情境 > 搜集新聞 > Inbox > 拆卡片 > 建立知識
- 連結卡片 > 新增主題庫 > 萃取知識 > 建立草稿 > 更新知識庫
- 從主題或草稿出發 > 文章流程 > 發文

知識越多，搜集越精準；寫越多文章，卡片越多；卡片越多，下一篇文章素材越豐富。




▋ Coding Agent 趨勢下的一點觀察
Karpathy 2025 回顧中認為，
Claude Code 是第一個令人信服的 LLM Agent 展示，
關鍵在於已存在的所有東西：電腦環境、安裝、上下文、資料、密鑰、配置，以及低延遲互動。


我寫的這些 Skill 檔案，本質上就是 Markdown，
裡面沒有任何程式碼，
只有清楚的步驟、判斷邏輯、輸入輸出格式。

換句話說，
只要你能把工作流程講清楚 Coding Agent 就能幫你執行。

所以知識管理也可以 Vibe，
你不需要會寫程式，
你只需要知道你想要什麼樣的知識處理流程。

而且隨著 Agent 越來越強，
你今天寫的 SOP 會讓明天的 Agent 執行得更好，
你不需要改任何東西，
它自己就會進步。

這大概是我目前覺得最划算的投資：
花時間把流程想清楚，寫成 Skill，
然後坐等 Agent 能力升級帶來的免費紅利。


▋ 小結
全面擁抱 Coding Agent 做知識管理，
對我來說是自己設計知識處理的自動化流程，

轉變的核心邏輯很簡單：
Coding Agent 的趨勢是明確的，
能力只會越來越強，
而我現在投入的每一個 Skill、每一個流程設計，
都會隨著 Agent 進步而自動升值。
