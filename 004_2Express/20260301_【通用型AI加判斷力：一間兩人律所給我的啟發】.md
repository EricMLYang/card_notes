【 通用型 AI + 判斷力：一間兩人律所給我的啟發 】

▋ 曼尼的分享，幫我把好幾張卡片串起來了

今天早上讀到曼尼分享的一篇文章，原作者是美國小型律師事務所創辦人 Zack Shapiro，講他怎麼用 Claude 處理法律實務。

曼尼在導讀裡直接點出幾個關鍵字：Skill、Agent、Harness、Skill as a Service。這些詞對一直在關注 AI 工作流的人來說不陌生，但曼尼厲害的地方是他把這些概念串成一條清晰的脈絡——從 prompt 到 skill，從 skill 到 agent，從 agent 到 harness，最後落在「人類負責判斷與負責」。

讀完之後的感覺比較像是「對，就是這個意思」。因為平常整理筆記的時候，陸續收了一些相關的卡片，但一直停在抽象原則。這篇文章加上曼尼的解讀，直接補上了一個完整的實戰案例。

▋ 通用型 AI + 你的判斷力 > 專門 AI

市面上一堆法律 AI 產品：Harvey、Spellbook、CoCounsel。賣點都是「我們幫法律人打造專用工具」。

Zack 的觀點很直接：那些只是套殼。

真正的競爭優勢在怎麼「運用」模板。每間事務所的保密協議、股權購買協議都大同小異。區分好律師和平庸律師的，是發現對方埋在第 14(c) 條裡的陷阱，判斷哪場爭端值得堅持、哪場該讓步。

這是判斷力。而判斷力存在於個人，存在於十年執業的累積。

所以他的做法是：拿通用型 AI（Claude），用自己十年執業經驗寫成 Skill 指令，讓 AI 套用「他個人的」框架，帶著他的分析偏好、溝通語氣、驗證流程。

▋ Build Skills，重點在領域知識

之前讀過一個觀點：Agent 開發 80% 的時間應該從工程問題搬到領域問題。換句話說，重點在 Build Skills。

讀 Zack 的文章才真正理解這句話的意思。他沒有去買法律 AI SaaS，而是花時間把自己的「合約審查邏輯」、「客戶溝通語氣」、「研究驗證流程」編碼成 Skill。這些 Skill 可迭代、可轉移、可規模化，跟寫一次就丟的 prompt 完全不同層級。

文章裡有一個場景很有說服力：收購案交割前夕，對方律師晚上七點丟來一封重組條款的信。他把所有文件餵進 Claude，幾分鐘內就交叉比對出對方自己的揭露清單和提議條款互相矛盾。到晚上十一點，反制方案完成，隔天順利交割。

大型事務所的三人初級律師團隊可能要熬夜才能做完的分析，他兩小時搞定。

但重點在於他知道要讓 AI 看什麼、怎麼看、看完之後怎麼判斷。速度只是判斷力被放大後的副產品。

▋ 讀完才更理解自己為什麼在做 PM Agent

上週分享過一個小嘗試：用 Coding Agent 建了一個 PM Agent，想解決「需求想不清楚」的問題。當時只是覺得寫 Code 越來越便宜，瓶頸會往上游移動，所以先試試看。

讀完這篇才比較能說清楚背後的邏輯——Zack 在律師領域做的事，其實跟產品領域遇到的問題很像：

律師這邊：
・把合約審查邏輯寫成 Skill
・AI 套用資深律師的法律框架
・判斷哪條值得爭、哪條可以讓
・初級律師從文件製作轉向判斷力

PM 這邊：
・把需求探索流程寫成 Skill
・Agent 套用 PM 的產品思維框架
・判斷哪個需求值得做、哪個該放
・工程師從寫 Code 轉向架構判斷

共同點是：專業判斷力被 AI 放大了。之前只是隱約感覺到這件事，Zack 的案例讓這個想法變得具體很多。

▋ Skill as a Service

曼尼在導讀裡提到一個很精準的觀察：我們正從「模型即服務（MaaS）」轉向「技能即服務（Skill as a Service）」。

文章裡有一段話把這件事講得很透：

「需要多年指導才能傳遞的知識，現在變成了一個從初稿開始就有效運作的指令文件。」

想像一位資深律師把十年的判斷力編碼成 Skill，裝到五十個初級律師的機器上，每個人從第一天就能用資深律師的框架產出初稿。這已經是知識傳遞方式的根本改變。

之前讀到「Harness」這個概念時覺得有道理但有點抽象。現在回頭看，Zack 做的就是 Harness——他駕馭模型的那套系統：迴圈、工具、約束、驗證，每一層都灌入了領域判斷力。模型的能力是基礎，但真正產生差距的是這套系統。

▋ 帶走的三件事

第一，Skill 需要磨。Zack 花了一兩個小時跟 Claude 反覆校準，才產出六個可用的 Skill。丟一句「幫我審查合約」是不夠的。這點跟做 PM Agent 的經驗一樣——每個流程步驟的指令品質，直接決定產出能不能用。

第二，Zack 說得很直白：「AI 沒有在執業，是你。」不管工具多強，最後簽名的、承擔後果的，還是人。

第三，通用型 AI 的進步速度會超過垂直產品。這個判斷蠻大膽的，但有道理。用前沿模型，每個新功能第一天就到手；用套殼產品，得等別人的工程團隊決定下一步開發什麼。

▋ 同樣的邏輯，適用於任何「判斷力密集」的職務

讀完之後忍不住想：還有哪些職務符合這個模式？整理了一下，條件其實很明確——產出以文本為主、需要防禦性判斷（知道何時該堅持、何時該讓步）、流程能在文件環境中跑完。

符合的比想像中多：專利工程師寫請求項要在涵蓋範圍和前案之間抓界線、地政士處理複雜繼承要在地政潛規則和國稅局紅線之間找最佳路徑、資安稽核師做 Gap Analysis 要判斷哪些缺失能用補償性控制過關。甚至都更計畫團隊的容積獎勵取捨、醫藥法規專員的查驗登記文件，本質上都是同一件事：通用型 AI 負責跑量，人負責判斷。

▋ 小結

這篇文章加上曼尼的導讀，幫我把最近零散在想的東西收攏了。

AI 時代的護城河是「編碼過的判斷力」。Zack 用法律實務把這件事講得非常具體，曼尼再把它拉高到 Skill、Agent、Harness 的框架裡。

最近在嘗試的 PM Agent，大概也是在摸索同一件事——怎麼把產品判斷力系統化，讓 AI 放大它。還在很早期，但方向應該沒偏太遠。
