【放下鍵盤後,才發現談需求比 Debug 還難】

外部客戶的需求溝通,是一門真正的藝術。

我原本以為在大企業內部開發 AI 相關專案,洽談與構想需求已經不容易。

沒想到與外部客戶溝通,才發現這才是真正的挑戰。

從客戶角度來看,他們是付錢的那一方。

他們有義務讓公司的錢付得最值得。

不同公司之間本質上就缺少同公司的溝通速度。

開發成本的考量至少要低於收入。

我們開發的案子還帶有一點創新性質——有案例可循,但我們跟客戶都沒做過。

在這種情況下,以往內部專案的洽談方式都需要重新考慮。

經過一陣摸索後,我整理出幾個自己覺得比較受用的方法。

但我必須強調,我並非 PM 專長,所以這些並不是最好的解法,僅是我個人的經驗談。





▋ 專案邊界的嚴格界定

接洽到簽約前的這段時間,專案邊界必須時時謹記在心。

這是最基本也最容易被忽略的關鍵。

我手上多個專案大多是數據管線、治理、以及分析後的應用。

使用情境不脫離 BI 報表、預警通知、數據傳輸服務等範疇。

只要談到任何進階的系統操作,例如地圖上的動態路線,我內心都會警鈴大作。

因為那已經超出我們的專案邊界。

明確的邊界能保護團隊不被無限擴張的需求拖垮,也能讓客戶清楚知道什麼在服務範圍內。

這是雙方建立信任的第一步。





▋ 情境式的訪談

逐一展開需求訪談時,我傾向用情境式的訪談方式。

我會預先查資料,理解客戶業務最有可能的情境。

所有問題都圍繞著實際情境出發。

最後才帶到系統怎麼操作使用。

因為我已經嚴格界定專案邊界,這時候的訪談可以比較發散一點。

最重要的目的是深挖使用者最實際的問題。

而不是要聽到「我希望系統是 xxx」這種表面需求。

因為此刻的需求往往在後面會發現常常要變更。

真正的需求藏在使用者的日常工作流程裡,不是藏在他們的想像中。





▋ 先解決最麻煩的風險

需求範圍底定之後,我個人最忌諱先逃避最麻煩的工作。

然後埋頭去進行其他事情,並自我說服說事情有進展。

這種鴕鳥心態只會讓風險在專案後期爆發。

例如訪談完我發現我們的報表工具可能無法應付需求。

我第一時間就會請優秀同事想辦法解決。

等這些麻煩一一變成可控範圍,再往下進行其他工作。

先處理最麻煩的風險,能讓整個專案的不確定性大幅降低。





▋ 對焦三大工具之一:User Story

User Story 就是以單一使用者角度,描述使用者要完成的工作是什麼。

再加上可驗證的條件。

這能把「做什麼」和「為什麼」綁在一起。

並讓需求變成最小可驗證單位。

工程師能直接拆成任務與測試案例。

通常會使用標準句型:As a〈角色〉, I want〈行為〉, so that〈價值〉。

再搭配 Acceptance Criteria,也就是驗收條件。

可以用 Given/When/Then 的條件式,或是清單的方式呈現。

例如:Given 我是某某角色且已登入,When 我在進行某某動作時,Then 會新增一筆通知並在通知中心顯示,狀態顯示為待處理。

這種格式讓需求變得具體、可測試,也讓開發與驗收有了共同語言。





▋ 對焦三大工具之二:Wireflow

Wireflow 是一條「從入口到完成」的動作流程圖。

它描述一個核心任務的完整路徑。

節點(Node)是「頁面/狀態/事件」。

邊(Edge)是「使用者操作/系統跳轉」。

Wireflow 能把「跨頁面、跨模組」的端到端行為講清楚。

避免只看單頁 Wireframe 卻漏了權限、例外、資料依賴等重要環節。

這是從整體視角檢視使用者旅程的最佳工具。

能提早發現流程斷點或邏輯漏洞。





▋ 對焦三大工具之三:Wireframe

Wireframe 是灰階介面草圖,呈現版位與資訊階層。

它不含視覺設計,也就是不包含色彩、品牌等元素。

這是為了先去除設計雜訊,對齊欄位、操作控件、可視元素的優先順序。

避免工程後期返工。

從 UI 也要能看出必需的資料與權限,回饋給 API 或資料表設計。

產出應該包含每頁標出主區塊、欄位或表格欄、CTA 按鈕、空狀態或錯誤狀態。

關鍵互動要加上 3 到 5 條註解。

例如「切換角色後,卡片清單會自動過濾」。

只做關鍵頁面,例如首頁、車輛明細、通知中心、權限配置,每頁畫一個畫面就好。

Wireframe 讓團隊在投入視覺設計與開發前,就能確認功能邏輯與資訊架構是否正確。





▋ 重要性簡單粗暴地講清楚:MoSCoW

想像老闆或 PM 丟給你一份長長的功能開發清單。

上面列了 20 個要做的新功能。

但時間和人力永遠不夠用。

這時候你心裡一定會想:「所以,到底要先做哪個?」

MoSCoW 就是一種幫助團隊回答這個問題的「分類帽」。

它把所有功能需求分成四個等級。

讓大家,尤其是工程師,一眼就知道什麼最重要、什麼可以先放著。

這個詞是四個英文單字的縮寫。

讓我們用一個「開發使用者登入功能」的例子來說明。

M 代表 Must Have,必做。

白話文就是:「沒有這個,產品就等於廢了,根本不能上線。」

工程師聽得懂的說法是:這是「上線的最低標準」,也就是 Minimum Viable Product,簡稱 MVP。

如果死線快到了,其他功能都可以砍,但這個不行。

這是你晚上可能要加班也要生出來的東西。

例如使用者能用「帳號密碼」成功登入系統。

沒有這個,使用者連第一步都進不去。

S 代表 Should Have,應做。

白話文是:「非常重要,能讓產品好用很多,但真的沒有它,產品還是可以運作。」

工程師聽得懂的說法是:這是讓產品從「堪用」變成「好用」的關鍵。

通常是第一版上線後,要優先排入開發的功能。

如果這次開發時間夠就做,如果時間不夠,下次更新一定要補上。

例如「記住我」的勾選框。

雖然每次手動輸入帳密也能登入,但有個「記住我」會方便很多,使用者會很開心。

C 代表 Could Have,可做。

白話文是:「有了會更好,但沒有也完全不影響核心功能,屬於錦上添花。」

工程師聽得懂的說法是:這是「行有餘力再說」的項目。

通常是 PM 的夢想清單。

或是開發到最後發現還有點時間,可以順手加上去的小功能。

如果最後沒做,沒人會怪你。

例如使用「Google 或 Facebook 第三方登入」。

這功能很酷很方便,但對核心的帳號密碼登入完全沒影響。

W 代表 Won't Have,這次不做。

白話文是:「這次的開發範圍『確定不包含』的功能。」

工程師聽得懂的說法是:這不是說這個功能很爛或永遠不做。

而是「現在、這個版本」不做。

把它明確寫出來是為了讓大家有共識。

避免有人,通常是 PM 或老闆,開發到一半突然跑來說:「欸那個某某功能順便做一下吧?」

你可以直接回他:「老闆,我們當初說好這個是 Won't Have 的。」

它最大的作用是幫你擋掉一堆臨時安插的需求。

例如「用指紋或臉部辨識登入」。

這個技術很棒,但可能需要更多研究和時間。

所以「這次」先不做,未來再考慮。

MoSCoW 讓優先順序變得透明,也讓範圍管理有了明確的對話基礎。





▋ 風險:未決定或不確定清單

最後一個工具是維護一份「未決定或不確定清單」。

專案進行中一定會有灰色地帶。

可能是技術可行性還在評估,可能是客戶內部還沒決定要用哪種流程。

把這些不確定性明確列出來,能避免團隊埋頭做了半天,結果做錯方向。

這份清單要定期更新,並在每次會議中追蹤進度。

讓所有人都知道哪些事情還懸在半空中,需要盡快釐清。

外部客戶溝通需求,比內部開發更需要結構與紀律,因為雙方的默契與信任都還在建立中。





▋ 重點回顧

1. 外部客戶溝通涉及付費關係與創新不確定性,需重新思考洽談方式。

2. 專案邊界從接洽到簽約前就嚴格界定,避免需求無限擴張。

3. 情境式訪談圍繞實際情境深挖真正問題,而非收集表面功能想像。

4. User Story 用標準句型加驗收條件,讓需求成為最小可驗證單位。

5. Wireflow 描繪端到端流程,揭露跨頁面的權限、例外與資料依賴。

6. MoSCoW 將需求分為必做、應做、可做、不做四級,讓優先順序透明。

7. 維護「未決定清單」定期追蹤灰色地帶,避免團隊做錯方向。
