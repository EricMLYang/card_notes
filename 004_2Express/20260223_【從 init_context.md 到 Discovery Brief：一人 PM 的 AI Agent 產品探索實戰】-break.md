【從 init_context.md 到 Discovery Brief：一個下午、8 個信號、6 個 Opportunity——一人 PM 的 AI Agent 產品探索實戰】
ㅤ
ㅤ
PM 最常犯的錯，是用形容詞描述問題。
ㅤ
「很複雜」「很麻煩」「風險很高」——這些話在會議上沒有人會反對，但也沒有人知道下一步該做什麼。
ㅤ
我在建 AI Agent 產品探索管線時發現一件事：AI 不接受形容詞。你說「爬蟲很麻煩」，它會要求你定義「麻煩」。你說「風險很高」，它會要求你給出假說和驗證方法。
ㅤ
這個「被迫清晰」的過程，是我從整個實作中學到最值得分享的一件事。
ㅤ
以下是一個人的 PM，如何在一個下午用 AI Agent 建出一條產品探索管線的完整紀錄。
ㅤ
ㅤ
▋ 起點：一份 Context 文件，和八個說不清楚的擔憂
ㅤ
MI 2.0 是一個內部系統改版專案——把十年前的 Drupal + Tableau + SQL Server 架構，遷移到 Vue.js + ASP.NET Core + PostgreSQL + Databricks，同時從一個事業群擴展到三個。
ㅤ
專案啟動時，我手上只有一份 `init_context.md`，裡面記錄了專案背景、利害關係人、技術現狀、以及我對 PM 方法論的整理（四大風險框架、FLOP、Cost of Delay、Pretotyping）。
ㅤ
這些方法論看起來很完整，但實際要怎麼落地？
ㅤ
一個人的 PM，面對跨事業群的系統遷移，沒有第二雙眼睛幫你檢查決策，沒有人幫你追蹤哪個風險最致命。腦袋裡有八個擔憂同時在轉——Tableau 遷移風險、CSV 人工瓶頸、爬蟲維護成本、集團雲連線未知——每個都覺得該處理，但就是排不出優先序。
ㅤ
所以我花了一個下午，用 Claude 建了一個 PM Agent。目的很單純：把我腦中的模糊，翻譯成有結構的東西。
ㅤ
ㅤ
▋ Agent 做量，人做判斷
ㅤ
整個 Agent 的定位很明確：Product Discovery 助手，專注在產品開發的最上游。
ㅤ
產出邊界畫在 Discovery Brief——也就是工程團隊接手寫 PRD 之前的所有東西。信號捕獲、機會識別、驗證實驗設計、Backlog 排序，這些交給 Agent。至於「賭哪個方案」、「砍什麼留什麼」，這些判斷由人來做。
ㅤ
這個分工是務實的。AI 可以源源不絕產出選項，但它不能為結果負責。AI 降低了執行門檻，卻同時提高了判斷門檻——Agent 可以在十分鐘內幫你生出六個 Opportunity 和對應的驗證實驗，但「該不該賭這個方向」，只有人能決定。
ㅤ
PM 的價值在於：從 Agent 給你的十個方案裡，選一個去賭，然後扛住後果。
ㅤ
ㅤ
▋ 從空 Repo 到可運作的管線：四個步驟
ㅤ
整個建構花了一個下午。
ㅤ
ㅤ
▋ 第一步：定義結構
ㅤ
參考了另一個較成熟的專案 Repo 的模式，但刻意保持最簡——只建立真正需要的資料夾和檔案：
ㅤ
`context/`：5 份文件構成專案的活快照（產品背景、儀表板、決策記錄、Story Map、方法論底本）
`product/`：產品工作流的所有產出（信號收件匣、Opportunity Solution Tree、驗證實驗、Backlog、Discovery Brief、溝通彙整）
`.github/prompts/` 和 `.github/skills/`：Agent 的能力定義
ㅤ
這裡有個細節值得提：context 文件不是一次寫完的。它們會隨著每次操作逐步豐富——儀表板在 Triage 後自動更新風險追蹤、實驗進度、Now/Next/Later 優先序。與其說是文件，更像活的狀態機。
ㅤ
Agent 的產出品質完全取決於你餵給它的 context 品質。垃圾進、垃圾出，在 AI 時代這件事更直接。
ㅤ
ㅤ
▋ 第二步：建立工作流
ㅤ
五個階段的管線：
ㅤ
信號捕獲 → Triage & OST → 驗證實驗 → Backlog 排序 → Discovery Brief
ㅤ
每個階段都有對應的 Prompt 或 Skill。這裡有一個設計我自己覺得最關鍵：每個 Opportunity 必須至少有一個驗證實驗，通過才能進 Backlog。
ㅤ
聽起來很理想化，但 MI 2.0 是內部專案，使用者不到 20 人，而且就坐在隔壁。走廊測試的成本趨近於零。
ㅤ
多數內部專案從不做驗證，因為「反正是內部的，做了再說」。結果就是做了三個月才發現方向錯了。
ㅤ
強制每個 Opportunity 綁定驗證實驗，是這套系統最重要的護欄。
ㅤ
ㅤ
▋ 第三步：餵入真實信號，執行 Triage
ㅤ
把實際的 8 個痛點信號餵進系統，讓 Agent 執行 Triage。
ㅤ
結果：8 個信號收斂為 6 個 Opportunity，其中 3 個相關信號被合併成一個複合型機會（爬蟲後處理 + Pipeline 遷移 + 數據進雲路徑）。每個 Opportunity 都配上了驗證實驗，包含假說、方法、成功標準。
ㅤ
這個過程最有價值的地方，是被迫把模糊的擔憂結構化。
ㅤ
「爬蟲很麻煩」變成了「EXP-003：1 條完整爬蟲數據路徑可在 2 天內跑通 POC」。有了明確的假說，工程團隊就知道要驗證什麼、花多少時間、成功長什麼樣。
ㅤ
說到底，AI 要求你把輸入結構化。你不能丟一句「爬蟲很麻煩」給它，你得說清楚：誰受影響、影響什麼 Outcome、假說是什麼、怎麼驗證。
ㅤ
這個「被迫清晰」的過程，本身就是 PM 最該做的事。以前沒人逼，現在 AI 逼了。
ㅤ
ㅤ
▋ 第四步：產出溝通文件
ㅤ
最後一步是產出工程團隊的討論彙整——一份可以直接在瀏覽器開啟的 HTML 頁面。裡面包含技術棧對照、Blocker 警示、實驗優先序、候選方案比較、待決策事項。
ㅤ
這份文件讓一件事變得可能：會議開始前，所有人已經看到結構化的議題。不用再花前 20 分鐘在「我們今天要討論什麼」上打轉了。
ㅤ
ㅤ
▋ 一個下午之後
ㅤ
目前的狀態：
ㅤ
8 個信號已 Triage，6 個 Opportunity 已登錄
6 個驗證實驗已設計，2 個是 Blocker 等級（集團雲部署、爬蟲數據路徑）
第一份工程團隊溝通彙整已產出
任何人 Clone Repo 後可直接用 `ask-context` Skill 查詢專案資訊
ㅤ
下一步是執行那兩個 Blocker 實驗。如果集團雲部署不了，其他都是空談。
ㅤ
這也是 Triage 最直接的價值：搞清楚什麼不通的話其他都不用做。好策略的第一步永遠是診斷——找到真正的瓶頸，然後把所有資源集中到那一點。
ㅤ
ㅤ
▋ 收尾
ㅤ
PM 的工作是降低不確定性，文件只是副產品。
ㅤ
Agent 不會讓你變成更好的 PM。但它讓「不做 PM 該做的事」變得更難逃避——你不能用形容詞敷衍它，你不能跳過驗證直接進開發，你不能把八個模糊的擔憂同時丟在腦子裡假裝自己在思考。
ㅤ
定義問題的能力，比找到答案更稀缺。Agent 幫你找答案，但問什麼問題，還是你的事。

# 拆解結果

## A. 主脈絡（3–6 行）
- 作者（一人 PM）面對內部系統遷移專案，腦中有 8 個模糊擔憂但排不出優先序，於是用 Claude 在一個下午建了一條 Product Discovery Agent 管線。
- 核心設計原則是「Agent 做量、人做判斷」——Agent 負責信號 Triage、Opportunity 識別、驗證實驗設計，但「賭哪個方向」的決策權留給人。
- 管線中最關鍵的護欄：每個 Opportunity 必須綁定至少一個驗證實驗才能進 Backlog，強制消滅「反正是內部的，做了再說」的慣性。
- 作者挑戰的預設：多數人認為 AI 的價值是「幫你找答案」，但作者主張 AI 最大的價值是「逼你把問題定義清楚」——AI 不接受形容詞，這個「被迫清晰」的副作用才是真正的槓桿點。
- Context 文件不是靜態文件而是活的狀態機，Agent 產出品質完全取決於 context 品質，這與主流「AI 會自己變聰明」的期待相反。

## B. 卡片（Zettel）

序號 1
- 標題：AI 的「被迫清晰」效應——形容詞殺手作為 PM 思考品質的護欄
- 類型：Principle
- 概念（50–300 字）：AI Agent 不接受形容詞。你說「爬蟲很麻煩」，它會要求你定義「麻煩」；你說「風險很高」，它會要求假說和驗證方法。這個特性表面上是 AI 的「笨」，實際上是最有效的思考品質護欄。PM 最常犯的錯就是用形容詞描述問題——「很複雜」「很麻煩」在會議上沒人反對，但也沒人知道下一步。作者的實戰經驗：「爬蟲很麻煩」被 Agent 逼成「EXP-003：1 條完整爬蟲數據路徑可在 2 天內跑通 POC」，有了假說、時間框、成功標準，工程團隊立刻知道要做什麼。底層邏輯：「被迫清晰」本身就是 PM 最該做的事，以前沒人逼，現在 AI 逼了。這不是 AI 的附加功能，而是把結構化輸入當作前置條件的系統，天然就會淘汰模糊思考。
- 重要性（1 句）：這是「AI 作為思考夥伴」最具操作性的機制——不靠 AI 給你答案，靠 AI 的輸入要求逼你升級問題定義的精確度。
- 邊界/反例（1–2 句）：當問題本身處於極早期探索、連方向都不確定時，過早結構化反而會限制發散；此時需要的是「先發散再收斂」而非直接餵給 Agent。另外，如果 PM 本身沒有領域知識，Agent 逼出來的結構化只是「看起來精確的垃圾」。
- 可連結關鍵詞：#被迫清晰 #AI人機協作 #問題定義 #結構化思考 #PM方法論

序號 2
- 標題：Agent 做量、人做判斷——AI 時代的人機分工邊界畫法
- 類型：Heuristic
- 概念（50–300 字）：作者為 PM Agent 畫了一條清晰的分工線：Agent 負責信號捕獲、機會識別、驗證實驗設計、Backlog 排序——所有「可以量產選項」的工作。人負責「賭哪個方案」「砍什麼留什麼」——所有需要承擔後果的決策。這條線畫在 Discovery Brief，也就是工程團隊接手寫 PRD 之前。背後的判斷邏輯：AI 降低了執行門檻，卻同時提高了判斷門檻。當 Agent 十分鐘內能生出六個 Opportunity，選擇成本不是降低了，而是升高了——因為選項變多，每個都看起來合理，判斷力的稀缺性反而凸顯。PM 的價值重新定義為：從 Agent 給你的十個方案裡，選一個去賭，然後扛住後果。
- 重要性（1 句）：這是一個可直接套用的人機分工判斷式——凡是「可量產且不需負責」的環節交 Agent，凡是「需要下注且承擔後果」的環節留給人。
- 邊界/反例（1–2 句）：當決策可逆且成本極低時（例如 A/B 測試的變體選擇），人的判斷環節可以進一步壓縮甚至自動化。另外，「判斷」本身也需要 context，如果人沒有足夠的領域浸泡，拿到再多選項也選不好。
- 可連結關鍵詞：#人機分工 #AI_Agent架構 #判斷力 #ProductDiscovery #決策權歸屬

序號 3
- 標題：內部專案的驗證護欄——強制 Opportunity 綁定實驗才能進 Backlog
- 類型：Warning
- 概念（50–300 字）：多數內部專案從不做驗證，理由是「反正是內部的，做了再說」。結果就是做了三個月才發現方向錯了。作者在 MI 2.0 管線中設了一個硬規則：每個 Opportunity 必須至少有一個驗證實驗，通過才能進 Backlog。這聽起來理想化，但作者指出內部專案其實是驗證成本最低的場景——MI 2.0 使用者不到 20 人，就坐在隔壁，走廊測試成本趨近於零。真正的問題從來不是「驗證太貴」，而是「沒有機制強制你去驗證」。這個護欄的效果：8 個信號 Triage 後收斂為 6 個 Opportunity，其中 2 個被標為 Blocker 等級（集團雲部署、爬蟲數據路徑），直接決定了「什麼不通其他都不用做」的資源配置。
- 重要性（1 句）：這是一個反直覺的設計——在看似最不需要驗證的場景（內部專案），強制驗證反而回報最高，因為驗證成本最低但方向錯誤的代價一樣致命。
- 邊界/反例（1–2 句）：如果內部專案的利害關係人分散在不同時區或組織，走廊測試的低成本假設就不成立，需要更正式的驗證設計。當 Opportunity 本身是技術基礎建設（如資料庫遷移），驗證實驗的設計會退化成 POC spike，這時護欄的價值更多在「逼你定義成功標準」而非「驗證用戶需求」。
- 可連結關鍵詞：#驗證實驗 #內部專案 #護欄設計 #Pretotyping #ProductDiscovery

序號 4
- 標題：Context 文件是活的狀態機，不是靜態文件——Agent 品質的瓶頸在輸入端
- 類型：Model
- 概念（50–300 字）：作者建構 PM Agent 時，context 資料夾裡的 5 份文件（產品背景、儀表板、決策記錄、Story Map、方法論底本）不是一次寫完的靜態文件，而是隨著每次操作逐步豐富的活狀態機——儀表板在 Triage 後自動更新風險追蹤、實驗進度、Now/Next/Later 優先序。這個設計背後的判斷：Agent 的產出品質完全取決於你餵給它的 context 品質，垃圾進垃圾出在 AI 時代更直接。與其花時間調 prompt，不如花時間維護 context。這也意味著 Agent 管線的真正建設成本不在 prompt engineering，而在 context engineering——持續維護一個準確反映專案現狀的活文件系統。
- 重要性（1 句）：這把 AI Agent 的品質問題從「怎麼寫 prompt」重新定位到「怎麼維護 context」，是一個更有槓桿的介入點。
- 邊界/反例（1–2 句）：當專案節奏極快（每天多次狀態變更），手動維護 context 文件的成本會超過收益，此時需要自動化同步機制。如果團隊成員不止一人，context 文件的版本衝突和一致性維護會成為新問題。
- 可連結關鍵詞：#ContextEngineering #AI_Agent架構 #狀態機 #知識管理 #垃圾進垃圾出

序號 5
- 標題：Triage 的終極價值：找到「不通則全不用做」的 Blocker
- 類型：Leverage
- 概念（50–300 字）：作者把 8 個信號餵進管線做 Triage，收斂為 6 個 Opportunity 後，最關鍵的產出不是排出優先序，而是識別出 2 個 Blocker 等級的實驗——集團雲部署和爬蟲數據路徑。作者的判斷：「如果集團雲部署不了，其他都是空談。」這就是 Triage 最直接的價值——搞清楚什麼不通的話其他都不用做。這與 Richard Rumelt 的好策略思維一脈相承：好策略的第一步永遠是診斷，找到真正的瓶頸（The Crux），然後把所有資源集中到那一點。多數 PM 的優先排序是「哪個價值最高先做哪個」，但更有效的排序是「哪個不通則全盤皆輸先驗證哪個」——這是風險驅動而非價值驅動的排序邏輯。
- 重要性（1 句）：這是一個可遷移的排序啟發式——在充滿不確定性的專案初期，先驗證「致命假設」比先追求「最大價值」更有槓桿。
- 邊界/反例（1–2 句）：當所有 Opportunity 的 Blocker 風險都很低時，這個排序邏輯退化為一般的價值排序，此時不需要特別強調 Blocker-first。如果 Blocker 實驗本身需要很長時間（例如等待第三方審批），團隊可能需要同時平行推進非 Blocker 項目以避免空轉。
- 可連結關鍵詞：#TheCrux #Triage #槓桿點 #風險驅動排序 #好策略壞策略

## C. 連結建議（組裝藍圖）
- 卡片 1 + 卡片 2 可組成：**「AI 時代 PM 能力模型重定義」框架**——AI 逼你清晰（卡片 1）+ AI 讓判斷力成為稀缺資源（卡片 2），共同指向「PM 的核心能力從執行力遷移到問題定義力和下注判斷力」。
- 卡片 3 + 卡片 5 可組成：**「不確定性專案的護欄系統」框架**——強制驗證（卡片 3）確保每個方向都有實驗支撐，Blocker-first 排序（卡片 5）確保先驗證致命假設，兩者共同構成一個完整的風險管理迴路。
- 卡片 4 與現有卡片的連結：與 AI Agent 工程化落地系列卡片直接相關——context engineering 是 Agent 架構中容易被忽略但決定品質上限的環節。
- 卡片 1 與現有卡片的連結：「被迫清晰」效應可連結到 SCQA 故事線、結構化思考等既有知識節點，作為「為什麼結構化有效」的實戰佐證。
