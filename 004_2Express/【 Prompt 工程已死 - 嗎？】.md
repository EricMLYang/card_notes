【 Prompt 工程已死 - 嗎？】

最近因為填寫滿紅的網路作家 周加恩的問卷，

而陸續收到一些 Email，

今天第 10 封還佛心附上 《精準提詞》：快速啟動指南，

( 可惡，一直誘惑我買 AI 電馭寫作的課 ）

老實說如果你是工程師，

你也滿常靠 AI 輔助開發，

裡面的內容其實不會讓人感到太驚訝，

但是有人整理好給你很快上手應用又是一回事了，

這段讓我想起很多人在說 prompt 工程已死的論調，

我覺得 是，也不是．



▋ 早期很多 Prompt 技巧已死

我是從 ChatGPT 3 就開始碰的人，

當時要在一個市場分析案子裡應用，

的確那時候 LLM 不是很容易駕馭 (又慢又貴），

Prompt 很容易越寫越長，

當時還有很多奇怪技巧，

例如：幫我想 10 個然後只顯示最好的那 3 個，

這時期也是 Prompt 工程師吹噓最大得時候



▋ Prompt 還是重要 - 脈絡並不是模型變聰明就能精進的

一個問題身處於何種上下文情境，

這個絕對不是模型變聰明就能理解的，

因為這些資訊沒有絕對對錯，

完全就是依照你的需求提供，

如何不要太多廢話或重複資訊下，

給與精準必要的上下文，

我認為是短時間不太會消失的，

但的確也可能因為 prompt 模板化而變的更簡單



可能是自己最近也跟設計師在討論客變，

裡面有一段滿有感，

「 

脈絡優先：跟 AI 互動的真正難題是脈絡

你不會：跟他說一大堆你要很華麗、很典雅、很優美、很格局方正….等形 容詞（工作指引描述過多容易互相矛盾）

▋ Google 的 TCREI 提詞模板

TCREI 是周加恩最常用的，

主要是：

\-Task 任務目標: 角色、做什麼、條件、遵循格式、語氣風格 

\-Context 脈絡：我是誰、目標理想、 要解決的問題、什麼有這個問題、資源環境限制

\-Reference 參照 步驟、 格式、範例、 思考架構

\-Evaluate 評估 → 檢視產出是否符合你要的

\-Iterate 修正 → 進行多輪迭代把提詞改成你要的樣子。



▋快速試用 - **程式科技知識吸收用摘要 Prompt**

**Task**：請幫我摘要以下 YouTube 影片內容，讓我能快速吸收重點。摘要需分成三個部分。

**Context**：

- 我是一個想吸收知識的觀眾，關注程式與科技。

- 我偏好以下三種面向：

   1. 最新發展（趨勢、版本更新、未來方向）

   2. 扎實技術說明（原理、架構、程式細節）

   3. 技術延伸反思（應用場景、優缺點、對開發者/產業的啟發）

**Reference**：

請用條列或三段式結構輸出，每個部分 2–3 句即可，簡明扼要。



以上產生的摘要已經非常接近我最有印象的片段，



**影片： 從 Lisp 到 AI：Stripe 創辦人的程式設計哲學，如何影響一家千億美元公司**

1\.最新發展（趨勢、版本更新、未來方向）

**\-AI 驅動開發：** 影片強調 AI 在軟體開發中扮演的角色日益重要。討論中提到 Cursor 這類工具如何整合 AI 協助編寫程式碼，預示著未來開發者可能會使用更高層級、較不正式的程式語言，將重點從「如何做」轉移到「做什麼」。

**\-整合開發環境（IDE）：** Stripe 共同創辦人 Patrick Collison 支持將執行環境、文本編輯與程式碼執行結合的完整整合開發環境。他認為 VS Code 和 Cursor 是朝正確方向邁進的工具，但仍有潛力進行更深度的整合，並設想未來的 IDE 能在編輯器中直接提供程式碼的豐富即時資訊，例如效能分析、日誌和錯誤資訊。

2\.扎實技術說明（原理、架構、程式細節）

**\-基於 Continuation 的網路框架：** Patrick 解釋他早期偏好基於 Continuation 的網路框架，這也讓他接觸到 Smalltalk。這類框架能帶來更具互動性且高效的開發過程，例如在一個網路請求過程中編輯程式碼，並從堆疊中較高的位置恢復執行。

**\-資料模型與 API：** 影片中很大一部分討論聚焦在良好設計的資料模型與 API 的重要性。Patrick 強調這些元素不僅影響組織架構（康威定律），也影響商業策略與成果。他反思 Stripe 的 V2 API，其目標是統一核心抽象概念，並為未來提供更具彈性的基礎。新 API 的設計過程包含了廣泛的客戶驗證，以及編寫實際整合程式碼以確保它們實用且不過度設計。

3\.技術延伸反思（應用場景、優缺點、對開發者/產業的啟發）

**\-程式語言選擇：** 影片觸及到最初技術選擇的影響，例如 Stripe 採用 Ruby 和 MongoDB。儘管這些決策帶來了長期的後果，但也形塑了公司的技術軌跡。Patrick 指出，儘管 Stripe 已經成長，他們仍會選擇性地用像 Java 這樣的語言重寫服務，以滿足特定的效能需求。

**\-AI 對生產力的影響：** 影片探討 AI 的好處是否已反映在經濟生產力數據上。雖然樂觀情緒很高，但數據尚無定論。Patrick 提到一篇近期論文表明語言模型並未帶來顯著的生產力提升，但也承認新技術的普及需要時間。

**\-程式設計與生物學的未來：** 展望未來，影片討論 AI 自動化程式設計任務的潛力，這能讓更多人創造軟體，並可能為設計師等非技術職位帶來復興。對話也觸及科技與生物學的交叉點，探討 AI 和新實驗技術如何幫助我們編寫人類生物學程式並治癒複雜疾病。
