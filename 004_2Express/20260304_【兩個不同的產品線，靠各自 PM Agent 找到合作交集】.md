【 兩個不同的產品線，靠各自 PM Agent 找到合作交集 】
ㅤ
為什麼很多跨部門會議都是浪費時間？
ㅤ
因為問題是即興的，回答是靠記憶的，紀錄是不完整的。
ㅤ
我換了一種方式：讓 Agent 閱讀雙方的角色 Repo，理解兩邊的產品背景和決策邏輯，然後生成一份結構化訪談問卷，按「現況→痛點→合作價值→可行性→優先度」層層遞進。
ㅤ
這有點像是一份跨角色的知識轉移協議。
ㅤ
ㅤ
▋ 起點
ㅤ
我手上有電動車數據管理平台，對方有零售廣告看板系統案子。兩個完全不同的產品，但在零售場域有一個交集——廣告帶動了需求，貨卻沒到店。
ㅤ
我想跟零售端的 PM 提一個物流數據合作的提案。
ㅤ
傳統做法大家都知道：約個時間、口頭問、靠記憶整理。

我這次想試點不一樣的。
ㅤ
ㅤ
▋ 實驗一：把角色寫成 Repo
ㅤ
我幫 VMS 和 SmartSignage 各建了一個 PM Repo，裡面有幾個關鍵檔案：
ㅤ
`.ai/identity.yaml`：這個 PM 是誰、管什麼、用什麼標準做決策
`.ai/principles.md`：決策邏輯和優先序
`00_context/`：產品背景、數據架構、客戶情境
`05_product/`：正在進行的 OPP 和 Backlog
ㅤ
這些檔案加在一起，就是一個「角色產生器」。
ㅤ
把它們餵給任何一個 LLM，它就能用那個 PM 的視角、語言和優先序來回答問題。角色從此存在於 Repo，而不只在某個人的腦子裡。
ㅤ
我把自己的決策原則和產品上下文全部寫進去之後，Agent 回答「這個合作對電動車數據管理系統的優先序是什麼」時，邏輯跟我自己想的幾乎一樣。那個瞬間蠻驚訝的。
ㅤ
ㅤ
▋ 實驗二：結構化訪談取代臨時開會
ㅤ
有了兩個角色之後，問題是怎麼讓他們「討論」。
ㅤ
開會的問題我們都清楚：對話是線性的、容易跑偏、很難事後追溯。
ㅤ
我選的方式是結構化訪談：
ㅤ
1. Agent（VMS PM）先讀完雙方的 Repo，理解提案背景
2. 設計一份 16 題的訪談問卷，按「現況→痛點→合作價值→可行性→優先度」分層
3. 人（扮演 SmartSignage PM）帶入自己的 Repo 上下文逐題回答
4. 回答直接寫進同一份文件，成為永久記錄
ㅤ
讓我驚訝的是回答的品質。
ㅤ
「目前有沒有辦法掌握商品是否已到店？」
ㅤ
如果在會議上問，回答大概就是「沒有」。但帶著 Repo 上下文回答時，答案變成：「完全沒有。我們的數據鏈路是 AI Camera / CMS → MongoDB → Databricks → 報表，這條路上完全沒有物流資訊。廣告排程以促銷檔期為基準，不會動態參考到貨狀態。」
ㅤ
這個精確度，在傳統會議裡不容易出現。因為沒有人會在開會前花時間把自己的產品架構重新翻一遍。
ㅤ
ㅤ
▋ 實驗三：訪談文件本身就是最有價值的產出
ㅤ
訪談結束後，我用 PptxGenJS 把核心洞察轉成了 10 頁的提案簡報。
ㅤ
但真正有價值的，是那份訪談文件。
ㅤ
它記錄了廣告平台端的真實痛點（用「結構性黑洞」描述庫存可見度的缺口）、電車數據最有價值的應用場景（歸因報告）、PoC 的具體設計（20–50 家門市 × 3–6 個月歷史數據），還有四個提案必須先解的問題（ID 對應、SKU 層級、商業歸屬、契約責任）。
ㅤ
如果只是開了個會，這些洞察幾乎不可能被完整保留。現在它們都在 Repo 裡，任何後續參與的人或 Agent，都能讀到完整的討論脈絡。
ㅤ
三個月後有新的工程師加入，他不用找人問「當初為什麼選歸因報告」。答案就在那裡，連推理過程都保留著。
ㅤ
ㅤ
▋ 我對這件事的看法：Context 是一筆好的投資
ㅤ
回頭看整個流程：
ㅤ
定義角色（Repo）→ 設計討論協議（訪談）→ 執行討論（帶上下文問答）→ 沉澱產出（文件 + 簡報）
ㅤ
這套方法最值得投入的地方在於：你花時間維護的 Context，會隨著 AI 能力的提升而越來越有價值。
ㅤ
今天的模型能讀你的 Repo 回答問題，明年的模型可能能自己判斷「哪些地方對不上、需要問」，然後主動生成訪談問卷。再之後，也許工程主管、法務、客戶各自有自己的 Repo，Agent 能設計多方訪談協議，一次性拿到所有關鍵視角。
ㅤ
Repo 還有 git history，角色的演進可追溯。三個月後回來看這份訪談，能知道當時的 PM 處於什麼狀態的 Backlog，做了什麼判斷。
ㅤ
這些可能性，都建立在同一個前提上：你的 Context 有被好好維護。
ㅤ
AI 會越來越強，但它能做的事情，永遠受限於你餵給它的上下文。把角色的思考方式、決策原則、產品脈絡結構化地寫下來，是一筆會增值的投資。
ㅤ
ㅤ
▋ 收尾
ㅤ
PM 的未來，可能在於學會指揮。
ㅤ
當每個角色都有自己的 Repo，PM 不再需要親自跑每一場對齊會議。你設計協議、定義角色，然後讓 Agent 去執行知識交換。
ㅤ
從親自演奏每個樂器，變成站在指揮台上，讓整個樂團各司其職。
ㅤ
你不需要比每個人都懂。你只需要讓對的知識，在對的時間，出現在對的地方。
