【7 週攻克 SAA-C03 安全領域 (30%)：專為「非資安背景」的 6 人開發團隊打造 - 精簡版】

▋ 為什麼開發者需要資安知識

開發外部產品時，我們常被提醒要注意資安。

這份大綱整合「開發安全系統」的實務知識，

同時對標 AWS SAA-C03 認證中佔比 30% 的 Domain 1。

原則很簡單：實務導向，避免過深理論，始終以開發者視角學習。

這是從開發者角度出發，兼顧實戰與認證的整合學習計畫。

---



▋ 四階段學習路徑

整個學習路徑拆分為四個階段，逐步深入。

第一階段兩週，專注資安基礎與 AWS 身份管理。

第二階段兩週，重點是雲端架構安全與網路防護。

第三階段兩週，深入 API 與後端安全。

第四階段一週，收尾前端安全並整合自動化工具。



以下是精簡版，就不特別標註是哪一階段

---

▋ 資安核心概念

學習資安的第一步，是建立「安全思考方式」。

核心在於了解攻擊者會從哪些角度攻擊系統。

我們必須先懂 CIA 三要素：Confidentiality (機密性) 確保資料不被未授權存取，

Integrity (完整性) 確保資料不被竄改，

Availability (可用性) 確保系統隨時可用。

接著是 Zero Trust 原則，

核心精神是「不相信任何人，

每次存取都要驗證」，搭配「最小權限原則」。

我們還需要認識常見攻擊類型：SQL Injection 在輸入框塞惡意指令，XSS 在網頁植入惡意程式碼，CSRF 偽造使用者操作，以及 Token 被偷。

最後要學會簡化版的威脅模型：畫出系統架構圖，標出「哪些地方有資料」、「誰可以存取」，找出「如果被攻擊會怎樣」的弱點。

實務練習是在系統架構圖上標出 5 個可能的攻擊點。

---



▋ AWS 責任共擔模型

很多人誤以為上了雲，AWS 就會搞定所有安全。

AWS Shared Responsibility Model 要澄清的就是這點。

AWS 負責「雲端本身」的安全 (Security OF the Cloud)，

包括實體機房、硬體設備、網路基礎設施，以及底層虛擬化平台。

而你負責「雲端上」的安全 (Security IN the Cloud)：自己設定 IAM 權限，

負責資料加密 (at rest / in transit)，

管理作業系統與應用程式更新，

設定網路 (Security Group, NACL) 和應用程式層安全 (API 驗證、Input Validation)。

責任分界依服務模式而定：IaaS (如 EC2) 你負責作業系統以上，PaaS (如 RDS) AWS 管理更多，SaaS (如 S3) AWS 管理平台你管理權限。

最經典的例子：S3 bucket 設定錯誤導致資料外洩，這是你的責任。

了解「哪些安全是 AWS 負責，哪些是你負責」，是避免上雲迷思的關鍵。

---



▋ AWS 身份與存取管理 (IAM)

IAM 是雲端安全的基礎，核心是控制「誰可以做什麼」，

直接對應 SAA-C03 Domain 1 的 1.1 和 1.2。

首先要懂 IAM 核心概念：User (個人帳號)，

Group (一群人)，Role (臨時身份，如讓 EC2 讀 S3)，Policy (權限規則)。

IAM Policy 結構包含：Effect (Allow/Deny)，Action (如 s3:GetObject)，Resource (如特定 S3 bucket)，Condition (什麼條件下生效)。

我們必須遵循 IAM Best Practices：Root 帳號只用來開帳號平常不用，所有人開啟 MFA (多因素驗證)，盡量用 Role 而非 Access Key 避免金鑰外洩，定期檢查權限移除不需要的。

實務練習是建立一個 IAM Role，讓 EC2 可以讀取特定 S3 bucket 但不能刪除。

---



▋ IAM Access Analyzer

IAM Access Analyzer 是 AWS 免費工具，

專門找出「權限給太多」的問題。

它自動分析 IAM Policy，找出過寬權限，尤其是「可被外部存取」的資源。

常見發現包括：S3 bucket 被設為 Public，IAM Role 允許任何帳號存取，Lambda 可被外部呼叫，KMS Key 可被其他帳號使用。

只需在 IAM Console 開啟 Access Analyzer，定期檢查 Findings 並修正高風險設定。

建議每季檢查一次，或在新增重要資源時立即檢查。

實務練習是開啟 Access Analyzer，檢查有沒有「可被外部存取」的資源。

---



▋ AWS Secrets 與金鑰管理

密碼、API Key、資料庫連線字串等機密資訊必須安全存放，

對應 SAA-C03 Domain 1 的 1.1。

AWS Secrets Manager 用來存放資料庫密碼、API Key，

最大特色是支援自動輪換密碼，應用程式在啟動時才讀取機密。

AWS Systems Manager Parameter Store 也用來存放設定參數，

比 Secrets Manager 便宜但沒有自動輪換功能，適合非關鍵設定值。

AWS KMS (Key Management Service) 用來管理「加密金鑰」，支援 Customer Managed Keys (CMK) 和自動輪換，S3、RDS、EBS 等服務都可用 KMS 加密。

實務最高原則：絕對不要把密碼寫在程式碼裡，絕對不要把 Access Key 放在 Git 裡，一律使用環境變數搭配 Secrets Manager。

一般資料用 SSE-S3，機密資料用 CMK。

實務練習是把資料庫連線字串存到 Secrets Manager，並讓應用程式去讀取。

---



▋ AWS 網路安全架構

網路架構安全的核心是設計「防火牆」，

控制流量進出，對應 SAA-C03 Domain 1 的 1.2。

一切從 VPC (Virtual Private Cloud) 開始，

這是你在 AWS 裡的私人網路區域。

VPC 內部切分 Subnet：Public Subnet 可連網際網路放 Web Server，

Private Subnet 不能直接連網際網路放資料庫。

Security Group (安全群組) 是第一層虛擬防火牆，

控制「進出」EC2 Instance 的流量，是 Stateful 的，

允許進來的流量其回應可自動出去，預設全部拒絕。

Network ACL 是 Subnet 層級的防火牆，

是 Stateless 的，進出流量規則要分別設定。

Private Subnet 的 EC2 要存取 S3 又不希望流量繞經網際網路，

就用 Private Endpoint (VPC Endpoint)。

Private Subnet 的機器需要連外部網路，

就用 NAT Gateway，允許內部主動連出去但外部不能主動連進來。

安全的網路分層架構：Public Subnet 放 Load Balancer、Bastion Host，

Private Subnet 放應用程式伺服器，Data Subnet 放資料庫。

實務練習是畫出系統的 VPC 架構圖，

標示哪些元件在 Public 或 Private Subnet。

▋ 多帳號與環境隔離

了解公司為何分多個 AWS 帳號，是理解環境隔離的第一步。

分帳號的原因：Dev 環境出問題不會影響 Prod，不同團隊資源完全隔離，帳單分開易追蹤成本，大幅降低「誤刪正式環境資源」的風險。

常見帳號架構：Sandbox (工程師實驗用)，Dev (開發環境)，Staging (測試環境)，Production (正式環境，最嚴格權限)。

通常透過 AWS Organizations 統一管理多個帳號，集中付費、統一權限政策。

環境隔離的好處：開發者在 Dev 有較大權限可快速測試，到了 Prod 權限變嚴格需要審核，即使 Dev 被入侵駭客也無法碰到正式環境。

作為開發者，你只需知道：你在哪個環境工作，為什麼某些操作在 Prod 會被擋，需要 Prod 權限時該找誰申請。

實務練習是畫一張圖，標示公司有哪幾個 AWS 環境及你在每個環境的權限差異。

▋ AWS 資料加密與保護

我們必須確保資料在「存放時」和「傳輸時」都是安全的，

對應 SAA-C03 Domain 1 的 1.3。

加密的基本概念：Encryption at rest (靜態加密) 指資料存在硬碟上要加密，

Encryption in transit (傳輸加密) 指資料在網路上傳要加密如 HTTPS。

在 S3 安全設定上：用 Bucket Policy 控制誰可以存取 bucket，

務必開啟 Block Public Access 預設關閉公開存取，

開啟 Versioning (版本控管) 保留檔案歷史版本防止誤刪，

設定 S3 Encryption 可選 SSE-S3 (AWS 管理) 或 SSE-KMS (你管理金鑰)。

在 RDS 安全設定上：必須啟用 encryption at rest，自動備份也要加密，最好使用 IAM Database Authentication 用 IAM Role 登入資料庫，一定要把資料庫放在 Private Subnet。

資料分類與保護：Public (公開資料如產品目錄)，Internal (公司內部資料如員工名單)，Confidential (機密資料如客戶個資)，Restricted (最高機密如核心商業機密)。

有時還需要資料遮蔽與匿名化：Masking (遮蔽如信用卡號顯示後四碼)，Pseudonymization (假名化如測試環境用假名)，Tokenization (符記化用亂數取代敏感資料)。

最後要有基本法規認知，知道 GDPR (歐盟個資法) 和台灣個資法。

實務練習是設定 S3 bucket 的加密與存取控制，確保只有授權的 IAM Role 可以讀取。

▋ 備份與復原基礎

備份與復原的目的，是確保資料不會因為誤刪、故障或攻擊而永久消失。

這裡有兩個基本概念：RPO 與 RTO。

RPO (Recovery Point Objective) 指可以接受遺失多少資料，例如 RPO = 1 小時代表最多遺失 1 小時的資料。

RTO (Recovery Time Objective) 指系統必須在多久內復原，例如 RTO = 4 小時代表 4 小時內要讓系統恢復運作。

開發者需要知道系統的 RPO/RTO 目標。

RDS 提供自動備份功能：預設每天自動備份，保留期限可設定 1-35 天建議至少 7 天，可以還原到「某個時間點」。

S3 則依靠版本控管 (Versioning)，保留檔案所有歷史版本，如果檔案被誤刪或竄改都可以還原。

高可用性 (High Availability) 依賴跨 AZ 部署：RDS Multi-AZ 能在主資料庫故障時自動切換到備援，S3 則是自動跨 AZ 資料會自動備份到多個資料中心。

實務練習是檢查 RDS 是否開啟自動備份，以及 S3 bucket 是否開啟版本控管。

▋ 安全 API 開發實務

開發 API 時，最大的挑戰是保護它不被未授權存取或攻擊，

對應 SAA-C03 Domain 1 的 1.2。

首先是身份驗證 (Authentication)：常用 OAuth 2.0 作為授權流程標準，

使用 JWT (JSON Web Token) 作為 token 格式，

JWT 結構包含 Header、Payload 和 Signature，

必須設定 Token 過期機制如 15 分鐘，

搭配 Refresh Token 換取新的 Access Token。

在 API Gateway 層要善用安全機制：Throttling (節流) 限制請求次數防止 DDoS，

API Key 用於簡單存取控制，WAF (Web Application Firewall) 擋掉惡意請求，

設定 CORS 控制哪些網站可以呼叫 API。

Input Validation (輸入驗證) 至關重要，

為了防止 SQL Injection 和 XSS，

做法是檢查資料格式、長度、特殊字元，使用「白名單」通常比「黑名單」更安全。

也要設定安全的 Headers：Content-Security-Policy (CSP) 防止 XSS，

X-Frame-Options 防止 Clickjacking，

Strict-Transport-Security (HSTS) 強制使用 HTTPS。

在處理密碼時要區分加密 (Encryption) 和雜湊 (Hash)：加密是可以解密回來的，

雜湊是不可逆的專門用來存密碼，絕對不要用 MD5 或 SHA-1，

應該使用 bcrypt、PBKDF2 或 Argon2。

最後是安全日誌 (Security Log)：要記錄登入失敗、權限拒絕、異常操作，但絕不能記錄密碼、信用卡號或完整的 token。

實務練習是撰寫一個受 JWT 保護的 API，並加上輸入驗證與錯誤處理。

▋ AWS 應用層安全服務

善用 AWS 提供的安全工具，能大幅強化應用程式的防護能力。

AWS WAF (Web Application Firewall) 用於保護 Web 應用程式免受常見攻擊如 SQL Injection 和 XSS，可以使用 AWS Managed Rules (預設規則集) 或自訂規則。

AWS Shield 提供 DDoS 防護：Shield Standard 是免費的提供基本自動防護，Shield Advanced 則是付費的進階防護。

Amazon Cognito 用來處理使用者註冊、登入和權限管理，包含 User Pools (管理使用者帳號) 和 Identity Pools (給前端臨時 AWS 憑證)，能整合 Google、Facebook 等外部登入。

AWS Certificate Manager (ACM) 專門用來管理 SSL/TLS 憑證，最大的好處是憑證到期會自動換新，通常搭配 CloudFront、ALB 或 API Gateway 使用。

實務練習是在 API Gateway 前面加上 WAF，並設定基本防護規則。

---



▋ 前端安全基礎

前端安全的核心，在於保護使用者的瀏覽器環境避免資料外洩。

在使用 Vue.js 或 React 時有常見安全問題：XSS 防護絕對不要使用 v-html 或 dangerouslySetInnerHTML，框架預設的資料綁定通常會自動跳脫特殊字元是安全的，表單提交時應帶上 CSRF Token 來驗證。

Token 的管理尤其重要：不要存在 localStorage 因為它很容易被 XSS 腳本偷走，建議的方式是使用 HttpOnly Cookie 或存在 Memory 中，設計 Token 刷新機制在快到期時自動換新。

最後必須強制使用 HTTPS，透過設定 HSTS (Strict-Transport-Security) header 可以防止中間人攻擊。

實務練習是檢查前端專案的 Token 存放位置，並改為更安全的方式。

▋ 安全自動化工具

將安全檢查整合進開發流程 (CI/CD)，才能真正實現「自動發現問題」。

我們可以在 CI/CD Pipeline 中加入安全檢查步驟：Secret Scanning 檢查有沒有人把密碼 commit 進去，

SAST (靜態分析) 檢查程式碼本身有沒有安全漏洞，

Dependency Check 檢查相依套件有沒有已知漏洞，

Container Image Scanning 檢查 Docker Image 有沒有漏洞。

市面上有很多常用工具：GitHub Secret Scanning (免費)，SonarQube 或 SonarCloud，Snyk 或 Dependabot，以及 Trivy。

實務上就是在 GitHub Actions 或 Azure DevOps 加上掃描步驟，

設定當發現高風險漏洞時自動阻擋部署，並定期產生安全報告。

實務練習是在 CI/CD pipeline 中加上 Secret Scanning 與 Dependency Check。

▋ 監控與偵測基礎

了解「系統被攻擊時會發生什麼」以及如何快速發現問題，是安全的最後一道防線。

CloudTrail 是操作記錄，記錄了所有 AWS API 操作也就是「誰在什麼時候做了什麼」，這在被入侵時至關重要可以用來追查「駭客做了什麼」，建議全區開啟並將記錄存到 S3。

GuardDuty 則是威脅偵測，透過 AI 分析 CloudTrail、VPC Flow Logs 自動偵測異常行為，例如偵測到有來自非常規地區的登入。

CloudWatch Logs 與 Alarm 用來收集應用程式 log 並設定告警，常見的告警如 API 錯誤率突然升高、CPU 使用率爆表，開發者需要會看 log 知道怎麼查錯誤訊息。

有三個基本告警是你需要關心的：Root 帳號被使用，S3 bucket 被設為 Public，IAM Policy 被大幅修改。

開發者需要知道這些服務存在並保持開啟，遇到安全告警時知道去哪裡看 log。

實務練習是確認 AWS 帳號有開啟 CloudTrail 和 GuardDuty，並找到在哪裡查看 log。
