【 系統動力學告訴我：Coding Agent 消除的不是瓶頸，而是把瓶頸推到你身上 】

Kent Beck 說：「AI 讓我 90% 的技能價值歸零，但剩下 10% 的槓桿放大了 1000 倍。」

DORA 2025 報告說：「AI 是放大器，放大你組織既有的強項與弱點。」

我把這兩個觀點疊在一起，用系統動力學的角度拆解 Coding Agent 時代可能出現的新瓶頸——然後發現最先遇到的，不是技術問題，是 PM 的問題定義。

所以我最近很積極在測試用 AI 輔助 PM 工作來加速。以下是我目前正在實驗的方案。

▋ 瓶頸正在轉移

用系統動力學來看現在的軟體交付：

Agent 變強 → 產出 code/PR 更快 → 更多任務交給 Agent。

這個增強迴路已經是現實。GitHub Copilot coding agent 在背景完成任務開 PR，Codex 在雲端平行處理多個任務，Gemini CLI 用 ReAct loop 不停迭代。

但任何增強迴路都會撞上限制因子。我觀察到至少五個平衡迴路正在形成：

B1：需求定義不足 → 模糊需求被更快實作 → 返工增加
B2：Review 壓力上升 → 待審 PR 堆積 → 合併變慢
B3：CI / 測試 / 資安壓力 → 驗證系統跟不上產出速度
B4：架構整合壓力 → 局部最優侵蝕整體邊界
B5：風險與不信任 → 組織加更多人工控制 → Agent 效益被吃掉

我自己排的第一名瓶頸：需求定義與問題切分。

因為這決定了 Agent 是高效做對，還是高效做錯。

一個很具體的症狀：task 很多，真正 merge 的少。Agent 很忙，但 roadmap 感覺更亂。PR 很多，但需求重開也很多。

DORA 的核心訊號講得很直白——AI 的回報取決於底層組織系統。如果你的需求定義混亂，AI 只會更快把混亂擴大。

▋ 那 10% 的槓桿到底是什麼

Kent Beck 說的那 10%，我認為對 PM 來說就是「定義問題」的能力。

Andy Grove 有個管理槓桿率公式：經理人產出 = 管理槓桿率 × 管理活動。AI 讓每個管理活動更快，但方向錯了，你只是更有效率地浪費時間。

我自己觀察到三個常見陷阱：
　產出膨脹——做更多，但沒有做更好
　方向錯誤——快速往錯的地方前進
　複製平庸——AI 放大的是你餵給它的框架品質

所以真正的槓桿點在「設計正確的框架讓 Agent 做對的事」。

▋ 我的實驗：把 PM 判斷力編碼成 Git Repo

我同時管三條產品線，兼職小團隊的 PM。面對的現實很具體：

在個別產品細節與整體 portfolio 策略間頻繁跳轉，切換成本很高。
資源有限，三個專案都說急，但我沒有一個明確的判斷框架。
過往的取捨散落在會議記錄裡，兩個月後沒人記得為什麼當初這樣決定。

與其讓 Agent 幫我「做更多事」，我決定先把「什麼值得做」這件事系統化。

做法是把 PM 的判斷力編碼成一個 Git Repo，讓任何 AI Agent 都能在我設計的框架內執行。

目前分三層在做：

第一層，角色與邊界。PM 總管看 Portfolio 全局，子 PM 看單一產品。總管只讀取狀態、做跨產品線判斷，不越級干預。這其實對應到 Agent 設計裡一個我很認同的概念——Harness 是品味的具體實現。同一個模型，不同的 Harness 產出天差地別。PM 的 Harness 就是你怎麼定義角色邊界、決策權限、協作介面。

第二層，決策原則編碼。我寫了五條可執行的決策框架，每條對應一個真實的兩難情境。舉兩個例子：「全局槓桿優先」——先做 A，因為 B 也會受惠；「好策略有明確取捨」——三個專案都重要？全做等於全做不好。把好策略的三層結構（診斷、方針、協調行動）寫進 Repo，它就變成 Agent 可以執行的指令，不只是簡報上的口號。

第三層，技能化工作流程。把重複性任務模組化——定期報告、跨專案分析、優先順序仲裁。每個技能有觸發條件、執行流程、品質檢查清單。Agent 負責資料彙整與流程執行，關鍵決策還是我自己確認。

▋ 取捨與目前的效益

架構上，PM 總管與各子 PM 各自獨立 repo，用 AGENTS.md 作為單一真實源，橋接同步到 Claude Code、GitHub Copilot、Gemini CLI。不被單一工具綁定，但指令定義統一。

核心取捨：讓 Agent 在我設計的框架內執行，結構化降低認知負擔，同時保留未預期任務依 principles 處理的彈性。

實際跑起來：
　週報從 2 小時降到 30 分鐘
　決策可追溯——為什麼當時選 A 不選 B，三個月後還能查到
　跨產品線共用組件識別，減少重複開發

老實說，前提條件很重要：產品線要有共通技術棧、你要願意花時間把模糊的 PM 工作結構化、組織文化要重視決策脈絡。這些不成立的話，效果會打折。

▋ 回到那張系統圖

Agent 越強，越不該問「能不能寫更多 code」，而要問：

我們定義問題的速度跟得上嗎？
有判斷權的人 review 得過來嗎？
架構邊界會不會被局部最優侵蝕？

我選擇先處理第一個。這是一個正在進行中的實驗，還不是完成式。

Coding Agent 消除的不是瓶頸，而是把瓶頸從「寫 code」推到「定義、review、驗證、整合、治理」。推到的第一站，正好是 PM 的地盤。這是機會——前提是你願意把自己的判斷力系統化。
