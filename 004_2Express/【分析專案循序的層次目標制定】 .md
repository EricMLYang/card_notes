【分析專案循序的層次目標制定】 

—— 定錯目標會導致大量的白功，絕對要避免

在數據產生價值的專案中，好高騖遠絕對是大地雷。 



早期我有一個公司內部市場分析案就發生這狀況。 



User (公司的市場分析單位) 期望可以預測產品價格。 



最後發現結果不符合預期導致最後的不愉快。 



這裡總結三個檢討與三個建議。



如果你是數據工程師或剛開始轉做數據分析的PM。



希望這篇文章能幫你避免一些專案初期常見的錯誤。



以下針對專案失敗的三點檢討：

1. 知識與評估經驗不足

2. 過於熱情嘗試新技術

3. 溝通認知落差未即時解決



為避免這些問題，提供三個建議：

1. 找出最真實的目標

2. 以生產線的角度看待數據產品

3. 利用 DIKW 金字塔逐步推出可用功能

\
\
 ▋知識與評估的經驗不足 

我原專長是深度學習，對產業分析本來就比較不足。 



初期嘗試分析建模後發現價格的影響因素太多。 



要做出太精準的預測根本就難以達成。 



更何況當時 User 期望可以預測未來 6 個月的價格。 



事後也沒踩煞車的情況下。 



讓團隊成員儘管非常努力了還是沒有對應的成果。 



整個專案到後期讓團隊非常沮喪。



\
 ▋對技術和難題太過熱情

年輕工程師往往有很多嘗試新技術的熱情。 



而資深工程師已經經歷很多維護噩夢。



常常會對於新技術的選擇非常保守。 



而當時此案開案時。 



因為是我們團隊不熟的領域。 



因此我們找了經濟學教授來進行預測建模。 



大家初期對於預測建模的技術非常有熱忱。 



花很多時間討論理論和手法。 



導致沒有即時對高標準目標踩煞車。



\
 ▋太慢溝通認知的不同

User 期望價格預測是能直接參考的預測結果。 



而我們預計用模型當作推估的工具。 



這兩者其實差異非常大。 



例如，因應月頻率的數據，數據很少，我們用多模型策略因應。



我們可能建構多個模型，每個有不同特徵。



可能是總經、庫存、多組產品價格、價格本身。



每個模型代表該特徵隱含的推估資訊，搭配領域知識解讀。



但 User 還是希望得到一個比較明確的結果。



我們其實專案走到一半就發現這樣的問題，但沒有及早溝通。





 ——\
以上經驗讓我對於分析專案的目標制定，會特別注意以下事項 

\
▋ 挖掘最真實的目標\
專案起始前到規劃階段，我會盡量去釐清 User 真正期望。



例如上述專案，我現在應該會在一開始就持續釐清：

．誰實際要看預測數據?

．什麼情境下要看？

．看完之後通常會進行什麼動作或決定？

．以上持續追問 Why，通常要問好幾次



這樣追問下，很有可能會把專案目標。



精準預測未來 3 個月價格，變成，下個月價格降低的機率多高。

\
\
▋ 用生產線的角度來看待產品

還是菜鳥時，數據分析就像是在 Jupyter Notebook 上塗鴉。



做各種不同嘗試跟產生單次結果。



但這樣做的最大壞處是。



結果很難系統化，很難系統化就表示你無法讓事情有成果。



整個分析案就會變成類似單次報告的產出。



現在我會提醒自己，要把數據當成生產線產品。



Pipeline 是你的加工產線。



進行分析時就同步思考，產品要怎麼從半成品到最終成品。



而你存放產品的地方就是資料庫。



\
\
▋ DIKW金字塔層次\
上述的半成品、成品的比喻。



其實是對比 DIKW 金字塔層次，組成如下：

．數據：原始的事實或觀察結果。如：銷售數字。

．資訊：經處理和組織的數據，具有意義。如：「本月 MoM -5% 」。 

．知識：有經驗、理解的運用知識。如：旺季 MoM -5% 很罕見 ，一定發生什麼事。

．智慧：深刻的理解和判斷，能回答「為什麼」並做出決策。如：旺季 MoM -5% 要儘速確認是否為同業競價消庫存，是的話我們要跟上。



而這層次的重點是，就算真的要預測價格。



我們也要及早在資訊、知識層次發布一些功能讓 User 使用。



而非一股腦想把價格預測做到盡善盡美。

——



雖然我還是很感謝當初該專案讓我學到很多東西。



但事後回想，如果能儘早把專案往有共識的地方進行。



也許團隊可以做的更有成就感。
