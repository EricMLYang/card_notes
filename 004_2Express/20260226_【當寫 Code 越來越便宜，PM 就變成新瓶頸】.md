當寫 Code 越來越便宜，PM 就變成新瓶頸

—

▋ PM 最花時間的是什麼？

對我來說，不是寫 PRD
是「想清楚要做什麼」

想完還要去對齊、被挑戰、再回來想
加上我這個 PM 還得兼著寫 code 應付報告
時間常常被切得很碎

—
▋大家都在 AI Coding

最近公司同仁都在追 Coding Agent
開發速度確實提升了

但我開始想：
code 寫得再快，需求沒想清楚
是不是只是在高速製造技術債？

所以我試了一件事——
用 Coding Agent 建了一個 PM Agent

—

▋ 為什麼先做 PM Agent？

我的觀察是：
Coding 速度提升之後，瓶頸會往上游移動

誰來定義「值得做的需求」？
誰來確保需求夠完整？
不會讓工程師做到一半才發現方向錯了？

這才是 PM 的核心工作：
產出有價值的需求

—

▋ 這個 PM Agent 在做什麼？

本質上就是一個 Coding Repo + Coding Agent
（GitHub Copilot、Claude Code、Codex CLI 混用）

整個流程：

Inbox → Outcomes → OPP+Experiments → Backlog → Product Brief
（訊號）→（期望結果）→（機會+驗證）→（待辦）→（移交文件）

分三階段，把零散訊號逐步轉成 PRD 前文件：

1. 收斂訊號
日常觀察、回饋、競品情報 → 可追蹤的問題

2. 反芻質疑
策略思考、腦力激盪、挑戰前提

3. 整理移交
洞察整理成工程前文件，降低 PRD 成本

—

▋ 兩個「反芻」節點

這個流程不是一路往前衝的
我在兩個地方設計了停下來重問的機制：

1.策略質疑：我們現在的假設還成立嗎？
2️. 機會發散：有沒有更高價值、但被忽略的需求？

Agent 只會順著流程一直走
不會停下來檢查方向對不對
所以這個機制對我來說很必要

—

▋ 還在摸索的部分

・反芻節點要固定週期跑，還是有事件才觸發？
・Prototype 優先給內部對齊，還是對外展示？
・未來跟 Daily 和 Sprint 節奏怎麼配合？

—

▋ 小結

出發點很單純：
我想讓自己在「想清楚要做什麼」這件事上
做得更快、更完整一點

目前跑了幾天，感覺有幫助
但還不成熟

如果手上幾個專案都順利建好
下一步想讓不同專案的 Agent 互相看對方內容
給交叉意見

等穩定之後，也許可以往其他角色擴展
——架構師、QA、維運

畢竟 Coding 速度提升後
這些環節可能才是下一個瓶頸

先把這個整理下來
之後再補更多實作結果
