最近有點 Local Agent 的快速發展，想關注集中 Context 並建立角色來幫忙做(就是一個有代表角色且充滿 Context 的 Repo.，例如專案 PM、個人知識管理、考試協助、架構師...等)，因為想建立的想法越來越多，因此索性建立一個 Repo，RoleFoundry 是一個用來快速鍛造高品質角色 Repo 的框架與示範專案，目標是把角色的思考方式、決策原則與協作介面標準化，讓 Agent 能穩定進入角色、可重複協作並持續迭代。


# 建立角色產生器，讓 Agent 討論

今天做了一個有趣的實驗。

起點是想將電動車數據管理分析案子經驗，轉給零售相關方案的 PM 做介紹，看看有無機會接觸零售物流，跟 RMN 串成一個更有上下游資訊的分析中心， 是零售媒體網路的數據平台，兩個完全不同的產品，但有一個共同的痛點——廣告帶動需求，但貨沒到店。我想跟零售端的 PM 提一個物流數據分析合作的提案，但在開會之前，我需要先理解對方的世界。

傳統做法：約個時間，口頭問，然後靠記憶整理。

我今天嘗試：

---

## 一、把角色寫成 Repo

廣告管理 和 電動車管理 各自有一個 PM Repo，裡面有幾個關鍵檔案：

- `.ai/identity.yaml`：這個 PM 是誰、管什麼、用什麼標準做決策
- `.ai/principles.md`：決策邏輯和優先序
- `00_context/`：產品背景、數據架構、客戶情境
- `05_product/`：正在進行的 OPP 和 Backlog

這些檔案加在一起，就是一個「角色產生器」。

當你把這些檔案餵給任何一個 LLM，它就能「變成」那個 PM——不是完美複製，而是能用那個角色的視角、語言和優先序來回答問題。角色不再存在於某個人的腦子裡，它存在於 Repo。

---

## 二、用訪談作為跨 Agent 的溝通協議

確定了兩個角色之後，下一個問題是：怎麼讓他們「討論」？

開個會不是好方法。對話是線性的、容易跑偏、而且很難事後追溯。

我選擇的協議是**結構化訪談**：

1. Agent（VMS PM）先閱讀雙方的 Repo，理解提案背景
2. 設計一份 16 題的訪談問卷，按「現況→痛點→合作價值→可行性→優先度」分層
3. 人（扮演 SmartSignage PM）閱讀問卷，帶入自己的 Repo 上下文逐題回答
4. 回答直接寫進同一份文件，成為永久記錄

這個過程讓我驚訝的地方在於：當對方「帶著 Repo 上下文」回答問題，答案的品質遠高於一般會議記錄。

「目前有沒有辦法掌握商品是否已到店？」

回答不是「沒有」，而是：「直接說：完全沒有。我們的數據鏈路是 `AI Camera / CMS → MongoDB → Databricks → 報表`，這條路上完全沒有物流資訊。廣告排程以促銷檔期為基準，不會動態參考到貨狀態。」

這個層次的回答，在傳統會議裡很難出現——因為沒有足夠的時間讓對方去查自己的 Repo 來組織答案。

---

## 三、產出不只是簡報，是兩個角色對話的記錄

訪談結束後，我用 PptxGenJS 把訪談的核心洞察轉成了 10 頁的提案簡報。

但其實更有價值的東西，是那份訪談文件本身。

它記錄了：
- SmartSignage 端的真實痛點（用「結構性黑洞」描述庫存可見度的缺口）
- VMS 數據對他們最有價值的應用場景（歸因報告，不是即時排程）
- PoC 的具體設計（20–50 家門市 × 3–6 個月歷史數據）
- 四個提案要成功必須先解的問題（ID 對應、SKU 層級、商業歸屬、契約責任）

這些洞察，如果只是開了個會，幾乎不可能被完整保留。現在它們都在 `_collab/SmartSignage_logistics/interview_retail_pm.md` 裡，任何一個後續參與這個合作的人或 Agent，都能讀到完整的討論脈絡。

---

## 四、這個模式的本質

回頭看整個流程：

```
定義角色（Repo）→ 設計討論協議（訪談）→ 執行討論（帶上下文問答）→ 沉澱產出（文件 + 簡報）
```

它其實在解決一個老問題：**跨角色的知識轉移**。

傳統上，兩個 PM 要對齊，靠的是信任、關係、和共同語言。這些都很脆弱——換人了就重來，會議紀錄不夠精確，很多隱性知識根本沒有被說出來。

當角色被「寫進 Repo」，知識對齊就變成了一個技術問題：讓 Agent 讀兩個 Repo，然後用結構化的方式觸發知識交換。

這不是在取代人的判斷。SmartSignage PM 回答問題時，靠的是他對自己產品的真實理解——這是 Repo 裡沒有的部分。但 Repo 提供了足夠的上下文，讓他能快速定位「哪些答案最重要」，而不是在回答問題前先花三十分鐘 recap 背景。

---

## 延伸思考

這個模式還可以往哪裡走？

**多角色同時討論**：如果工程主管、法務、客戶各自有自己的 Repo，是不是可以設計一個「多方訪談協議」，一次性拿到所有關鍵視角？

**Agent 自己提問**：這次是人設計訪談問卷。但如果 Agent 閱讀完兩個 Repo 後，能自己判斷「哪些地方對不上、需要問」，然後自動生成問卷——提案準備的循環就可以大幅壓縮。

**角色版本控制**：Repo 是有 git history 的，意味著角色的演進也是可追溯的。三個月後再回來看這個訪談，可以知道當時的 SmartSignage PM 處於什麼狀態的 Backlog，做了什麼判斷。

這些都不是科幻。它們是把現有工具——git、markdown、LLM、結構化問卷——組合在一起之後自然浮現的可能性。

#### 標題 1 鉤子：【 開會前，我先讓兩個 AI 替我們聊完了 】

**A-1**
我要跟一個完全不同產品線的 PM 提合作案。電動車數據管理 × 零售廣告看板，兩個宇宙的交集在哪裡？我完全沒概念。傳統做法是約個會、口頭問、靠記憶整理。但這次我換了一個方式——我把兩個 PM 的角色寫成了 Git Repo，讓 Agent 先替我們跑了一輪 16 題的結構化訪談。結果？Agent 問出了我在任何會議桌上都不會想到要問的問題。以下是我怎麼做到的，以及你明天就能複製的完整流程。

**A-2**
上週五我面對一個棘手的狀況：需要在週一之前搞懂零售廣告看板的 PM 在想什麼，然後準備一份跨產品線的合作提案。問題是，我對他的產品幾乎一無所知。我沒有約會議，而是花了一個下午做了一件事——把我自己和對方的 PM 角色，各寫成一個有 identity.yaml、principles.md 和產品上下文的 Repo。然後讓 Agent 讀完兩邊的 Repo，自動設計訪談問卷。這份問卷挖出的東西，比我過去三年任何一場跨部門會議都深。

**A-3**
你有沒有過這種經驗？開完跨部門對齊會議，走出會議室的那一刻就知道——真正重要的問題，一個都沒被問到。不是沒時間，是沒人有足夠的上下文去問出對的問題。我今天做了一個實驗，用一個你可能沒想過的方式解決這個問題：把 PM 的決策邏輯、產品背景、和優先序全部寫進 Git Repo，然後讓 AI 用結構化訪談的方式，替我問出那些「對的問題」。

#### 標題 3 鉤子：【 傳統 PM 對齊靠關係，我用 Git Repo 讓 Agent 三十分鐘搞定 】

**B-1**
兩個產品線的 PM 要對齊，傳統上你需要什麼？一段信任關係、幾次午餐、一個共同的老闆、和至少三場「先聊聊看」的會議。我這次什麼都沒有。電動車數據平台和零售廣告看板，兩個 BU，兩套語言，兩種優先序。但我發現，當你把角色的思考方式寫成檔案、把決策原則寫成 markdown，對齊就不再需要關係——它變成了一個技術問題。以下是我用 Git Repo + 結構化訪談，在一個下午完成的事。

**B-2**
如果我告訴你，我上週做了一份跨產品線的合作提案，從「完全不理解對方產品」到「產出 10 頁洞察簡報」只花了一個下午——你可能覺得我在吹牛。但關鍵不是我多厲害。關鍵是我把兩個 PM 的角色各寫成了一個 Repo，裡面有身份定義、決策原則、產品上下文。然後讓 Agent 讀完這些檔案後，用 16 道結構化問題進行了一場「代理訪談」。那份訪談紀錄裡的洞察深度，讓我自己都嚇了一跳。

**B-3**
每次跨部門合作，最貴的成本不是開發，是對齊。你花了兩週搞懂對方在做什麼，又花了兩週讓對方搞懂你要什麼，然後發現雙方理解的「合作」根本不是同一件事。我決定用一個工程師的方式來解這個問題：把角色寫成代碼。不是寫程式，是把 PM 的身份、原則、和產品上下文結構化成 Git Repo 裡的檔案。然後，讓 Agent 基於這些檔案，執行一場品質遠超人工會議的結構化訪談。

#### 標題 5 鉤子：【 16 道結構化問題，挖出傳統會議永遠問不到的答案 】

**C-1**

**C-2**

### 問題：跨部門合作，最貴的成本是「對齊」

我手上有一個電動車數據管理平台（VMS），對方有一個零售廣告看板系統（SmartSignage）。兩個完全不同的產品，但有一個共同的痛點——廣告帶動了需求，但貨沒到店。

我想跟零售端的 PM 提一個物流數據分析合作的提案。

但在開會之前，我需要先理解對方的世界。

傳統做法：約個時間，口頭問，然後靠記憶整理。

問題是——每次跨部門合作，最貴的成本不是開發，是對齊。你花了兩週搞懂對方在做什麼，又花了兩週讓對方搞懂你要什麼，然後發現雙方理解的「合作」根本不是同一件事。

會議紀錄不夠精確，很多隱性知識根本沒有被說出來，換人了就得重來。

---

### 機制：問題出在「隱性知識沒有被結構化」

為什麼傳統會議很難產出高品質的對齊結果？

三個根本原因：

**問題是即興的。** 大部分跨部門會議，沒有人事先設計過「該問什麼」。問題隨著對話流動，想到哪問到哪，重要的盲區被跳過，不重要的細節被展開。

**回答是靠記憶的。** 被問到「你們的數據鏈路長什麼樣」時，PM 靠的是腦中的印象，而不是產品文件。回答的精確度取決於他今天的狀態，而不是事實。

**紀錄是不完整的。** 會議結束後，會議紀錄只留下結論，過程中的推理鏈和邊界條件全部消失。三個月後回來看，沒有人記得當時為什麼做了那個判斷。

本質上，這是一個**隱性知識沒有被結構化**的問題。每個 PM 腦中都有一套決策模型——我管什麼產品、我用什麼標準做判斷、我的優先序是什麼——但這些東西從來沒有被寫下來。

所以每次對齊，都得從頭建立共同語言。

---

### 解法一：把角色寫成 Repo

我的做法是：把 PM 的角色定義，寫成一個 Git Repo。

VMS 和 SmartSignage 各自有一個 PM Repo，裡面有幾個關鍵檔案：

- `.ai/identity.yaml`：這個 PM 是誰、管什麼、用什麼標準做決策
- `.ai/principles.md`：決策邏輯和優先序
- `00_context/`：產品背景、數據架構、客戶情境
- `05_product/`：正在進行的 OPP 和 Backlog

這些檔案加在一起，就是一個「角色產生器」。

當你把這些檔案餵給任何一個 LLM，它就能「變成」那個 PM——不是完美複製，而是能用那個角色的視角、語言和優先序來回答問題。

角色不再存在於某個人的腦子裡，它存在於 Repo。

這跟 Multi-Repo Agent 架構的概念一樣：把職責邊界分離，減少 Context 汙染，讓 Agent 之間透過明確的 contract 溝通。每個 Repo 就是一個 Agent 的「身份證」。

**武力展示**：我把自己（VMS PM）的決策原則、產品上下文、和進行中的 OPP 全部寫進了 Repo。當 Agent 讀完這些檔案後，它能用我的語言回答「這個合作對 VMS 的優先序是什麼」——而且回答的邏輯跟我自己想的幾乎一樣。

---

### 解法二：用結構化訪談作為跨 Agent 的溝通協議

確定了兩個角色之後，下一個問題是：怎麼讓他們「討論」？

開個會不是好方法。對話是線性的、容易跑偏、而且很難事後追溯。

我選擇的協議是**結構化訪談**：

1. Agent（VMS PM）先閱讀雙方的 Repo，理解提案背景
2. 設計一份 16 題的訪談問卷，按「現況→痛點→合作價值→可行性→優先度」分層
3. 人（扮演 SmartSignage PM）閱讀問卷，帶入自己的 Repo 上下文逐題回答
4. 回答直接寫進同一份文件，成為永久記錄

關鍵不在問了什麼問題，而在回答的人有多少上下文。

**武力展示**：「目前有沒有辦法掌握商品是否已到店？」

傳統會議裡，回答大概是「沒有」。

但當回答的人帶著整個 Repo 的上下文時，答案變成：「直接說：完全沒有。我們的數據鏈路是 `AI Camera / CMS → MongoDB → Databricks → 報表`，這條路上完全沒有物流資訊。廣告排程以促銷檔期為基準，不會動態參考到貨狀態。」

這個層次的精確度，在傳統會議裡幾乎不可能出現——因為沒有足夠的時間讓對方去查自己的產品文件來組織答案。

### 解法三：產出不只是簡報，是可追溯的知識資產

訪談結束後，我用 PptxGenJS 把訪談的核心洞察轉成了 10 頁的提案簡報。

但其實更有價值的東西，是那份訪談文件本身。

它記錄了：
- SmartSignage 端的真實痛點（用「結構性黑洞」描述庫存可見度的缺口）
- VMS 數據對他們最有價值的應用場景（歸因報告，不是即時排程）
- PoC 的具體設計（20–50 家門市 × 3–6 個月歷史數據）
- 四個提案要成功必須先解的問題（ID 對應、SKU 層級、商業歸屬、契約責任）

這些洞察，如果只是開了個會，幾乎不可能被完整保留。

現在它們都在 `_collab/SmartSignage_logistics/interview_retail_pm.md` 裡，任何一個後續參與這個合作的人或 Agent，都能讀到完整的討論脈絡。

**武力展示**：三個月後，如果有新的工程師加入這個合作案，他不需要找人問「當初為什麼選歸因報告而不是即時排程」。答案就在那份訪談文件裡，連推理過程都保留著。

### 這個模式的本質

回頭看整個流程：

```
定義角色（Repo）→ 設計討論協議（訪談）→ 執行討論（帶上下文問答）→ 沉澱產出（文件 + 簡報）
```

它其實在解決一個老問題：**跨角色的知識轉移**。

傳統上，兩個 PM 要對齊，靠的是信任、關係、和共同語言。這些都很脆弱——換人了就重來，會議紀錄不夠精確，很多隱性知識根本沒有被說出來。

當角色被「寫進 Repo」，知識對齊就變成了一個技術問題：讓 Agent 讀兩個 Repo，然後用結構化的方式觸發知識交換。

這不是在取代人的判斷。SmartSignage PM 回答問題時，靠的是他對自己產品的真實理解——這是 Repo 裡沒有的部分。但 Repo 提供了足夠的上下文，讓他能快速定位「哪些答案最重要」，而不是在回答問題前先花三十分鐘 recap 背景。

### 延伸：這個模式還能往哪裡走？

**多角色同時討論**：如果工程主管、法務、客戶各自有自己的 Repo，是不是可以設計一個「多方訪談協議」，一次性拿到所有關鍵視角？

**Agent 自己提問**：這次是人設計訪談問卷。但如果 Agent 閱讀完兩個 Repo 後，能自己判斷「哪些地方對不上、需要問」，然後自動生成問卷——提案準備的循環就可以大幅壓縮。

**角色版本控制**：Repo 是有 git history 的，意味著角色的演進也是可追溯的。三個月後再回來看這個訪談，可以知道當時的 SmartSignage PM 處於什麼狀態的 Backlog，做了什麼判斷。

這些都不是科幻。它們是把現有工具——git、markdown、LLM、結構化問卷——組合在一起之後自然浮現的可能性。

### 收尾選項

**收尾版本 A：Prototype as SPEC**
角色 Repo 不是文件，它是一份可執行的 spec。就像 Prototype as SPEC 的概念——當你的原型足夠精確，它本身就成為規格書。你的角色 Repo 也是：當它足夠完整，任何 Agent 讀完都能進入那個角色，不需要再問任何人。角色不再依附於人，它依附於 Repo。而 Repo 是可以被 fork、被迭代、被版本控制的。

**收尾版本 B：從演奏者到指揮家**
PM 的未來不是學會更多工具，而是學會指揮。當每個角色都有自己的 Repo，PM 不再需要親自跑每一場對齊會議——他設計協議、定義角色、然後讓 Agent 去執行知識交換。從親自演奏每個樂器，變成站在指揮台上，讓整個樂團各司其職。你不需要比每個人都懂，你只需要讓對的知識在對的時間出現在對的地方。

**收尾版本 C：非同步協作的未來**
我們習慣了「同步對齊」——同一時間、同一房間、同一段對話。但當角色被結構化、訪談被文件化、產出被版本控制，對齊就不再需要同步。你描述方向，Agent 平行處理，人判斷品質。這才是 AI 時代協作的真正形態：不是用 AI 取代人開會，而是讓「開會」這件事本身變得不再必要。
