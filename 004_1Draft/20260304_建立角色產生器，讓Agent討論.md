## 本 Repo 的目的
RoleFoundry 是一個用來快速鍛造高品質角色 Repo 的框架與示範專案，目標是把角色的思考方式、決策原則與協作介面標準化，讓 Agent 能穩定進入角色、可重複協作並持續迭代。
# 建立角色產生器，讓 Agent 討論

今天做了一個有趣的實驗。

起點是一個 PM 的實際需求：VMS 是電動車隊的遙測系統，SmartSignage 是零售媒體網路的數據平台，兩個完全不同的產品，但有一個共同的痛點——廣告帶動需求，但貨沒到店。我想跟零售端的 PM 提一個物流數據分析合作的提案，但在開會之前，我需要先理解對方的世界。

傳統做法：約個時間，口頭問，然後靠記憶整理。

這次的做法不一樣。

---

## 一、把角色寫成 Repo

SmartSignage 和 VMS 各自有一個 PM Repo，裡面有幾個關鍵檔案：

- `.ai/identity.yaml`：這個 PM 是誰、管什麼、用什麼標準做決策
- `.ai/principles.md`：決策邏輯和優先序
- `00_context/`：產品背景、數據架構、客戶情境
- `05_product/`：正在進行的 OPP 和 Backlog

這些檔案加在一起，就是一個「角色產生器」。

當你把這些檔案餵給任何一個 LLM，它就能「變成」那個 PM——不是完美複製，而是能用那個角色的視角、語言和優先序來回答問題。角色不再存在於某個人的腦子裡，它存在於 Repo。

---

## 二、用訪談作為跨 Agent 的溝通協議

確定了兩個角色之後，下一個問題是：怎麼讓他們「討論」？

開個會不是好方法。對話是線性的、容易跑偏、而且很難事後追溯。

我選擇的協議是**結構化訪談**：

1. Agent（VMS PM）先閱讀雙方的 Repo，理解提案背景
2. 設計一份 16 題的訪談問卷，按「現況→痛點→合作價值→可行性→優先度」分層
3. 人（扮演 SmartSignage PM）閱讀問卷，帶入自己的 Repo 上下文逐題回答
4. 回答直接寫進同一份文件，成為永久記錄

這個過程讓我驚訝的地方在於：當對方「帶著 Repo 上下文」回答問題，答案的品質遠高於一般會議記錄。

「目前有沒有辦法掌握商品是否已到店？」

回答不是「沒有」，而是：「直接說：完全沒有。我們的數據鏈路是 `AI Camera / CMS → MongoDB → Databricks → 報表`，這條路上完全沒有物流資訊。廣告排程以促銷檔期為基準，不會動態參考到貨狀態。」

這個層次的回答，在傳統會議裡很難出現——因為沒有足夠的時間讓對方去查自己的 Repo 來組織答案。

---

## 三、產出不只是簡報，是兩個角色對話的記錄

訪談結束後，我用 PptxGenJS 把訪談的核心洞察轉成了 10 頁的提案簡報。

但其實更有價值的東西，是那份訪談文件本身。

它記錄了：
- SmartSignage 端的真實痛點（用「結構性黑洞」描述庫存可見度的缺口）
- VMS 數據對他們最有價值的應用場景（歸因報告，不是即時排程）
- PoC 的具體設計（20–50 家門市 × 3–6 個月歷史數據）
- 四個提案要成功必須先解的問題（ID 對應、SKU 層級、商業歸屬、契約責任）

這些洞察，如果只是開了個會，幾乎不可能被完整保留。現在它們都在 `_collab/SmartSignage_logistics/interview_retail_pm.md` 裡，任何一個後續參與這個合作的人或 Agent，都能讀到完整的討論脈絡。

## 四、這個模式的本質

回頭看整個流程：

```
定義角色（Repo）→ 設計討論協議（訪談）→ 執行討論（帶上下文問答）→ 沉澱產出（文件 + 簡報）
```

它其實在解決一個老問題：**跨角色的知識轉移**。

傳統上，兩個 PM 要對齊，靠的是信任、關係、和共同語言。這些都很脆弱——換人了就重來，會議紀錄不夠精確，很多隱性知識根本沒有被說出來。

當角色被「寫進 Repo」，知識對齊就變成了一個技術問題：讓 Agent 讀兩個 Repo，然後用結構化的方式觸發知識交換。

這不是在取代人的判斷。SmartSignage PM 回答問題時，靠的是他對自己產品的真實理解——這是 Repo 裡沒有的部分。但 Repo 提供了足夠的上下文，讓他能快速定位「哪些答案最重要」，而不是在回答問題前先花三十分鐘 recap 背景。

## 延伸思考

這個模式還可以往哪裡走？

**多角色同時討論**：如果工程主管、法務、客戶各自有自己的 Repo，是不是可以設計一個「多方訪談協議」，一次性拿到所有關鍵視角？

**Agent 自己提問**：這次是人設計訪談問卷。但如果 Agent 閱讀完兩個 Repo 後，能自己判斷「哪些地方對不上、需要問」，然後自動生成問卷——提案準備的循環就可以大幅壓縮。

**角色版本控制**：Repo 是有 git history 的，意味著角色的演進也是可追溯的。三個月後再回來看這個訪談，可以知道當時的 SmartSignage PM 處於什麼狀態的 Backlog，做了什麼判斷。

這些都不是科幻。它們是把現有工具——git、markdown、LLM、結構化問卷——組合在一起之後自然浮現的可能性。
