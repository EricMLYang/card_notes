# 用 AI Agent 建立產品探索管線的實作紀錄

---

## 起點：一份 Context 文件

MI 2.0 是一個內部系統改版專案——把十年前的 Drupal + Tableau + SQL Server 架構，遷移到 Vue.js + ASP.NET Core + PostgreSQL + Databricks，同時從一個事業群擴展到三個。

專案啟動時，我手上只有一份 `init_context.md`，裡面記錄了專案背景、利害關係人、技術現狀、以及我對 PM 方法論的整理（四大風險框架、FLOP、Cost of Delay、Pretotyping）。

問題是：這些方法論要怎麼落地？一個人的 PM，面對跨事業群的系統遷移，怎麼確保每個決策都有依據、每個機會都被驗證、每次會議都有結構？

答案是建一個 PM Agent——不是取代 PM 的判斷，而是把 PM 的判斷力放大。

---

## 設計原則：Agent 做量，人做判斷

整個 Agent 的定位很明確：**Product Discovery 助手，專注在產品開發的最上游**。

產出邊界畫在 Discovery Brief——也就是工程團隊接手寫 PRD 之前的所有東西。信號捕獲、機會識別、驗證實驗設計、Backlog 排序，這些是 Agent 的範圍。至於「賭哪個方案」、「砍什麼留什麼」，這些判斷由人來做。

這個分工不是客氣，是務實。AI 能源源不絕產出選項，但不能為結果負責。PM 的價值在於：從 Agent 給你的十個方案裡，選一個去賭，然後扛住後果。

---

## 建構過程：從空 Repo 到可運作的管線

整個建構過程大約花了一個下午，經歷了幾個階段：

**第一步：定義結構**

參考了另一個較成熟的專案 Repo（SmartSignagePM）的模式，但刻意保持最簡——只建立真正需要的資料夾和檔案。核心結構是：

- `context/`：5 份文件構成專案的活快照（產品背景、儀表板、決策記錄、Story Map、方法論底本）
- `product/`：產品工作流的所有產出（信號收件匣、Opportunity Solution Tree、驗證實驗、Backlog、Discovery Brief、溝通彙整）
- `.github/prompts/` 和 `.github/skills/`：Agent 的能力定義

**第二步：建立工作流**

五個階段的管線：

```
信號捕獲 → Triage & OST → 驗證實驗 → Backlog 排序 → Discovery Brief
```

每個階段都有對應的 Prompt 或 Skill。關鍵設計是：**每個 Opportunity 必須至少有一個驗證實驗，通過才能進 Backlog**。這不是理想主義，而是因為 MI 2.0 是內部專案——使用者不到 20 人且可以直接接觸，做驗證的成本極低，沒理由跳過。

**第三步：餵入真實信號，執行 Triage**

把實際的 8 個痛點信號（Tableau 遷移風險、CSV 人工瓶頸、爬蟲維護成本、集團雲連線未知……）餵進系統，讓 Agent 執行 Triage。

結果是：8 個信號收斂為 6 個 Opportunity，其中 3 個相關信號被合併成一個複合型機會（爬蟲後處理 + Pipeline 遷移 + 數據進雲路徑）。每個 Opportunity 都配上了驗證實驗，包含假說、方法、成功標準。

這個過程最有價值的不是產出本身，而是**被迫把模糊的擔憂結構化**。「爬蟲很麻煩」變成了「EXP-003：1 條完整爬蟲數據路徑可在 2 天內跑通 POC」。有了明確的假說，工程團隊就知道要驗證什麼、花多少時間、成功長什麼樣。

**第四步：產出溝通文件**

最後一步是產出工程團隊的討論彙整——一份可以直接在瀏覽器開啟的 HTML 頁面。裡面包含技術棧對照、Blocker 警示、實驗優先序、候選方案比較、待決策事項。

這份文件的價值不在於漂亮，而在於**會議開始前，所有人已經看到結構化的議題**，不用花前 20 分鐘在「我們今天要討論什麼」上打轉。

## 幾個實作中的體會

**Context 是 Agent 的燃料，不是裝飾**

Agent 的產出品質完全取決於 Context 的品質。5 份 context 文件不是一次寫完的，而是隨著每次操作逐步豐富。儀表板在 Triage 後自動更新風險追蹤、實驗進度、Now/Next/Later 優先序。這些不是文件，是活的狀態機。

**內部專案的驗證優勢被嚴重低估**

使用者就坐在隔壁，走廊測試的成本趨近於零。但多數內部專案從不做驗證——因為「反正是內部的，做了再說」。結果就是做了三個月才發現方向錯了。強制每個 Opportunity 綁定驗證實驗，是這套系統最重要的護欄。

**Agent 最大的價值是逼你結構化思考**

不是 Agent 多聰明，而是它要求你把輸入結構化。你不能丟一句「爬蟲很麻煩」給它，你得說清楚：誰受影響、影響什麼 Outcome、假說是什麼、怎麼驗證。這個「被迫清晰」的過程，本身就是 PM 最該做的事。

## 目前狀態與下一步

一個下午建完的系統，現在的狀態是：

- 8 個信號已 Triage，6 個 Opportunity 已登錄
- 6 個驗證實驗已設計，2 個是 Blocker 等級（集團雲部署、爬蟲數據路徑）
- 第一份工程團隊溝通彙整已產出
- 任何人 Clone Repo 後可直接用 `ask-context` Skill 查詢專案資訊

下一步是執行那兩個 Blocker 實驗。如果集團雲部署不了，其他都是空談——這也是 Triage 最直接的價值：**不是告訴你什麼最重要，而是告訴你什麼不通的話其他都不用做**。

PM 的工作不是產出文件，是降低不確定性。Agent 只是讓這件事變得更快、更結構化、更難逃避。

**B-1（非主流成功的秘密故事）**
專案啟動時，我列出了八個「令人不安但說不清楚」的風險：Tableau 要遷到哪裡？爬蟲維護成本？集團雲到底連不連得上？這些東西在我腦子裡轉了兩週，每個都覺得重要，但沒有一個有明確的下一步。直到我把它們全部丟進 AI Agent 做 Triage——八個擔憂收斂成六個 Opportunity，六個 Opportunity 裡只有兩個是 Blocker。如果那兩個不通，剩下的全部可以不用做。這是我第一次體驗到「結構化的殘忍」：

**B-2（我幫你做完了最難的事）**
你有沒有過這種經驗——腦袋裡同時跑著八個擔憂，每個都覺得該處理，但就是排不出優先序？我在 MI 2.0 專案啟動時就是這個狀態。Tableau 遷移、CSV 人工流程、爬蟲維護、集團雲連線——每個都像是在喊「先處理我」。我做了一件事：建一個 AI Agent，把這些模糊的擔憂強制轉成有結構的驗證實驗。結果八個變六個，六個裡只有兩個是真正的 Blocker。以下是這個壓縮過程的具體步驟：

**B-3（我幫你付了錢所以你不用付了）**
我花了兩週在腦中排列組合那八個風險，排不出優先序。然後我花了一個下午讓 AI 幫我做 Triage，它用了十分鐘就告訴我：你只有兩個問題真正重要，其他都是附帶的。不是 AI 比我聰明，是它逼我把每個擔憂寫清楚——寫清楚之後，答案其實很明顯。以下是這個過程教我的三件事：

#### 標題 3 鉤子（「爬蟲很麻煩 vs. EXP-003：2 天內跑通 POC」）

**C-1（非主流成功的秘密故事）**
「爬蟲很麻煩。」這是我在專案會議上說了至少三次的話。每次說完，大家點點頭，然後繼續討論下一個議題。直到我把這句話丟進 AI Agent，它回我一個問題：「麻煩是指什麼？維護時間？失敗率？資料缺口？」我被迫回答，然後它幫我寫出：「EXP-003：1 條完整爬蟲數據路徑可在 2 天內跑通 POC，成功標準是端到端資料完整率 > 95%。」同一個問題，從「感覺麻煩」變成「2 天內可驗證」。這就是 AI 逼出來的清晰度：

**C-2（我幫你做完了最難的事）**
PM 最常犯的錯是用形容詞描述問題——「很複雜」「很麻煩」「風險很高」。這些話在會議上沒有人會反對，但也沒有人知道下一步該做什麼。我在建 AI Agent 管線時發現：它不接受形容詞。你說「爬蟲很麻煩」，它會要求你定義「麻煩」。你說「風險很高」，它會要求你給出假說和驗證方法。這個「被迫清晰」的過程，是我從這整個實作中學到最值得分享的一件事：

**C-3（我幫你付了錢所以你不用付了）**
我花了兩年跟工程團隊開會，學到一件事：工程師最討厭的不是困難的需求，是模糊的需求。「爬蟲很麻煩」這種話，他們聽了不知道該做什麼。但如果你說「EXP-003：1 條完整爬蟲數據路徑，2 天 POC，成功標準端到端完整率 > 95%」——他們馬上知道要幹嘛。AI Agent 幫我做的最重要的事，就是把我腦中的模糊翻譯成工程師能接手的規格。以下是完整的翻譯過程：

PM 最常犯的錯，是用形容詞描述問題——「很複雜」「很麻煩」「風險很高」。

這些話在會議上沒有人會反對，但也沒有人知道下一步該做什麼。

我在建 AI Agent 產品探索管線時發現一件事：AI 不接受形容詞。你說「爬蟲很麻煩」，它會要求你定義「麻煩」。你說「風險很高」，它會要求你給出假說和驗證方法。

這個「被迫清晰」的過程，是我從整個實作中學到最值得分享的一件事。

以下是一個人的 PM，如何在一個下午用 AI Agent 建出一條產品探索管線的完整紀錄。

---

### 起點：一份 Context 文件，和八個說不清楚的擔憂

MI 2.0 是一個內部系統改版專案——把十年前的 Drupal + Tableau + SQL Server 架構，遷移到 Vue.js + ASP.NET Core + PostgreSQL + Databricks，同時從一個事業群擴展到三個。

專案啟動時，我手上只有一份 `init_context.md`，裡面記錄了專案背景、利害關係人、技術現狀、以及我對 PM 方法論的整理（四大風險框架、FLOP、Cost of Delay、Pretotyping）。

問題是：這些方法論要怎麼落地？

一個人的 PM，面對跨事業群的系統遷移，沒有第二雙眼睛幫你檢查決策，沒有人幫你追蹤哪個風險最致命。腦袋裡有八個擔憂在打轉——Tableau 遷移風險、CSV 人工瓶頸、爬蟲維護成本、集團雲連線未知——每個都覺得重要，但沒有一個有明確的下一步。

答案是建一個 PM Agent。不是取代 PM 的判斷，而是把 PM 的判斷力放大。

---

### 設計原則：Agent 做量，人做判斷

整個 Agent 的定位很明確：**Product Discovery 助手，專注在產品開發的最上游**。

產出邊界畫在 Discovery Brief——也就是工程團隊接手寫 PRD 之前的所有東西。信號捕獲、機會識別、驗證實驗設計、Backlog 排序，這些是 Agent 的範圍。至於「賭哪個方案」、「砍什麼留什麼」，這些判斷由人來做。

這個分工不是客氣，是務實。

AI 能源源不絕產出選項，但不能為結果負責。就像一張卡片裡寫的：AI 降低了執行門檻，但提高了判斷門檻。Agent 可以在十分鐘內幫你生出六個 Opportunity 和對應的驗證實驗，但「該不該賭這個方向」這件事，只有人能決定。

PM 的價值在於：從 Agent 給你的十個方案裡，選一個去賭，然後扛住後果。

---

### 建構過程：從空 Repo 到可運作的管線

整個建構過程花了一個下午，四個步驟。

**第一步：定義結構**

參考了另一個較成熟的專案 Repo 的模式，但刻意保持最簡——只建立真正需要的資料夾和檔案。核心結構是：

- `context/`：5 份文件構成專案的活快照（產品背景、儀表板、決策記錄、Story Map、方法論底本）
- `product/`：產品工作流的所有產出（信號收件匣、Opportunity Solution Tree、驗證實驗、Backlog、Discovery Brief、溝通彙整）
- `.github/prompts/` 和 `.github/skills/`：Agent 的能力定義

這裡有一個關鍵設計：context 文件不是一次寫完的裝飾品，而是隨著每次操作逐步豐富的活狀態機。儀表板在 Triage 後自動更新風險追蹤、實驗進度、Now/Next/Later 優先序。

Context 是 Agent 的燃料，不是裝飾。Agent 的產出品質完全取決於你餵給它的 context 品質——這跟「垃圾進、垃圾出」是同一件事，只是在 AI 時代更直接。

**第二步：建立工作流**

五個階段的管線：

```
信號捕獲 → Triage & OST → 驗證實驗 → Backlog 排序 → Discovery Brief
```

每個階段都有對應的 Prompt 或 Skill。關鍵設計是：**每個 Opportunity 必須至少有一個驗證實驗，通過才能進 Backlog**。

這不是理想主義。MI 2.0 是內部專案，使用者不到 20 人且可以直接接觸，做驗證的成本極低。走廊測試的成本趨近於零——使用者就坐在隔壁。但多數內部專案從不做驗證，因為「反正是內部的，做了再說」。結果就是做了三個月才發現方向錯了。

強制每個 Opportunity 綁定驗證實驗，是這套系統最重要的護欄。

**第三步：餵入真實信號，執行 Triage**

把實際的 8 個痛點信號餵進系統，讓 Agent 執行 Triage。

結果：8 個信號收斂為 6 個 Opportunity，其中 3 個相關信號被合併成一個複合型機會（爬蟲後處理 + Pipeline 遷移 + 數據進雲路徑）。每個 Opportunity 都配上了驗證實驗，包含假說、方法、成功標準。

這個過程最有價值的不是產出本身，而是**被迫把模糊的擔憂結構化**。

「爬蟲很麻煩」變成了「EXP-003：1 條完整爬蟲數據路徑可在 2 天內跑通 POC」。有了明確的假說，工程團隊就知道要驗證什麼、花多少時間、成功長什麼樣。

不是 Agent 多聰明。而是它要求你把輸入結構化——你不能丟一句「爬蟲很麻煩」給它，你得說清楚：誰受影響、影響什麼 Outcome、假說是什麼、怎麼驗證。這個「被迫清晰」的過程，本身就是 PM 最該做的事。

**第四步：產出溝通文件**

最後一步是產出工程團隊的討論彙整——一份可以直接在瀏覽器開啟的 HTML 頁面。裡面包含技術棧對照、Blocker 警示、實驗優先序、候選方案比較、待決策事項。

這份文件的價值不在於漂亮，而在於**會議開始前，所有人已經看到結構化的議題**，不用花前 20 分鐘在「我們今天要討論什麼」上打轉。

---

### 一個下午之後的狀態

- 8 個信號已 Triage，6 個 Opportunity 已登錄
- 6 個驗證實驗已設計，2 個是 Blocker 等級（集團雲部署、爬蟲數據路徑）
- 第一份工程團隊溝通彙整已產出
- 任何人 Clone Repo 後可直接用 `ask-context` Skill 查詢專案資訊

下一步是執行那兩個 Blocker 實驗。如果集團雲部署不了，其他都是空談。

這也是 Triage 最直接的價值：**不是告訴你什麼最重要，而是告訴你什麼不通的話其他都不用做**。好策略的第一步永遠是診斷——搞清楚真正的瓶頸在哪，然後把所有資源集中到那一點。

---

### 收尾

PM 的工作不是產出文件，是降低不確定性。

Agent 不會讓你變成更好的 PM。它只是讓「不做 PM 該做的事」變得更難逃避——你不能用形容詞敷衍它，你不能跳過驗證直接進開發，你不能把八個模糊的擔憂同時丟在腦子裡假裝自己在思考。

定義問題的能力，比找到答案更稀缺。Agent 幫你找答案。但問什麼問題，還是你的事。
