# 當 Agent 越強，PM 該先解決什麼瓶頸？我用 Git Repo 把定義問題編碼成系統

> 日期：2026-03-07





## 起點：系統動力學告訴我，Agent 越強，瓶頸在哪裡

用系統動力學來看 Coding Agent 時代的軟體交付，有一個清楚的結論：

當 Agent 讓「產生變更」的邊際成本快速下降，瓶頸不再是寫 code，而是依序轉移到——需求定義、判斷權、驗證、架構整合、治理。

這不是抽象猜測。GitHub Copilot coding agent 已經能在背景完成任務、開 PR、接受 review 繼續迭代；Codex 在雲端平行處理多個任務；DORA 2025 也直說：AI 是放大器，放大組織既有的強項與弱點。

畫成系統圖，長這樣：

```
[Agent 變強] → [產出 code/PR 更快] → [更多任務交給 Agent]（R1 增強環）
   ↓
   但會撞上 5 個平衡迴路
   ├─→ (B1) 需求定義不足 → 返工增加
   ├─→ (B2) Review 壓力上升 → 合併變慢
   ├─→ (B3) CI/測試/資安壓力 → 等待變慢
   ├─→ (B4) 架構整合壓力 → 技術債增加
   └─→ (B5) 風險與不信任 → 治理變重
```

**第一名瓶頸：需求定義與問題切分。** 因為這決定 Agent 是高效做對，還是高效做錯。模糊需求被更快實作，錯誤方向的產出增加，返工增加——PM 被迫花更多時間重定義問題，真正可推進的流量反而下降。

Kent Beck 說：「AI 讓我 90% 的技能價值歸零，但剩下 10% 的槓桿放大了 1000 倍。」

那 10% 是什麼？就是「定義問題」的能力。

---

## 所以我決定：先解決定義問題的瓶頸

我同時管理三條產品線（MI2.0 影像平台、VMS 影像管理系統、SmartSignage 智慧電子看板），兼職小團隊的 PM。面對的現實是：

- 視角切換困難：在個別產品細節與整體 portfolio 策略間頻繁跳
- 優先順序拍腦袋：資源有限，三個專案都說急，但沒有判斷框架
- 決策脈絡遺失：過往的取捨散落在會議記錄裡，兩個月後沒人記得為什麼這樣決定

與其讓 Agent 幫我「做更多事」，我選擇先把「什麼值得做」這個定義問題系統化。

做法：**把 PM 的判斷力編碼成一個 Git Repo，讓任何 AI Agent 都能在我設計的框架內執行。**

---

## 思考框架：三層設計

### 第一層：角色與邊界

PM 總管（Portfolio 視角）和子 PM（單一產品視角）明確分工。總管只讀取狀態、做跨產品線判斷，不越級干預子 PM 的在地決策。

這對應到 Agent 設計的核心概念——Harness 是品味的具體實現。同一個模型，不同的 Harness 產出天差地別。PM 的 Harness 就是：你怎麼定義角色邊界、決策權限、協作介面。

### 第二層：決策原則編碼

系統內建五條可執行的決策框架：

- **全局槓桿優先**：先做 A，因為 B 也會受惠
- **抽離即複利**：共同組件投資能跨專案回收，效益比 > 1.5 就做
- **子 PM 在地判斷優先**：不越級干預，除非影響其他產品線
- **短期 × 長期並重**：若短期扛得住，就投資長期效益
- **好策略有明確取捨**：三個專案都重要？全做等於全做不好

好策略的三層結構——診斷、方針、協調行動——寫成 Git Repo 後，就不只是 PPT 上的口號。它變成 Agent 可以執行的指令、可以追溯的決策記錄、可以迭代的系統。

### 第三層：技能化工作流程

將 PM 的重複性任務模組化為「技能」：定期報告彙整、跨專案策略分析、優先順序仲裁、利害關係人對齊。每個技能有觸發條件、執行流程、品質檢查清單。

不用每次想「報告怎麼寫」，直接執行技能。AI Agent 負責資料彙整與流程執行，關鍵決策仍由人類確認。

---

## 目前的做法與取捨

**架構選擇**：PM 總管與各子 PM 各自獨立 repo。各產品線獨立演進，但保留 portfolio 視角的彙整點。用 AGENTS.md 作為單一真實源，橋接同步到 Claude Code、GitHub Copilot、Gemini CLI 等工具。

**關鍵取捨**：
- 結構 vs 彈性 → 結構化降低認知負擔，但保留「未匹配任務依 principles 處理」的彈性
- 自動化 vs 人工判斷 → Agent 做資料彙整，人做策略判斷——不是讓 Agent 做決策，是讓 Agent 在你設計的框架內執行

**實際效益**：
- 週報從 2 小時降到 30 分鐘
- 決策可追溯——為什麼當時選 A 不選 B，三個月後還能查到
- 跨產品線共用組件識別，減少重複開發

---

## 反思：這個思路的前提與侷限

這個做法的前提假設：
1. 產品線有共通技術棧，共用元件才有槓桿價值
2. 你願意花時間把「模糊的 PM 工作」結構化
3. 組織文化重視決策脈絡，不只看結果

尚未解決的問題：即時狀態同步（目前週度同步）、利害關係人報告客製化、團隊協作學習曲線。

---

## 回到系統動力學的結論

Agent 越強，越不該問「能不能寫更多 code」，而要問：

- 我們定義問題的速度跟得上嗎？
- 有判斷權的人 review 得過來嗎？
- 架構邊界會不會被局部最優侵蝕？

我選擇先處理第一個問題。把「什麼值得做」的判斷力編碼成系統，讓 Agent 在框架內自主運作。

PM 的護城河不是做更多，而是知道「什麼值得存在」。你的 Harness 反映你的價值觀——在 AI 時代，這就是最稀缺的資產。

▍瓶頸正在轉移，而且轉到 PM 身上

用系統動力學來看 Coding Agent 時代的軟體交付：

Agent 變強 → 產出 code/PR 更快 → 更多任務交給 Agent。這是增強迴路，現在已經是現實——GitHub Copilot coding agent 在背景完成任務開 PR，Codex 在雲端平行處理多個任務，Gemini CLI 用 ReAct loop 不停迭代。

但任何增強迴路都會撞上限制因子。

我看到至少五個平衡迴路正在形成：

B1：需求定義不足 → 模糊需求被更快實作 → 返工增加
B2：Review 壓力上升 → 待審 PR 堆積 → 合併變慢
B3：CI / 測試 / 資安壓力 → 驗證系統跟不上 → 等待變慢
B4：架構整合壓力 → 局部最優侵蝕邊界 → 技術債增加
B5：風險與不信任 → 組織加更多人工控制 → Agent 效益被吃掉

排名第一的瓶頸：需求定義與問題切分。

因為這決定 Agent 是高效做對，還是高效做錯。你會看到一個典型症狀：task 很多，但真正 merge 的少；Agent 很忙，但 roadmap 感覺更亂；PR 很多，但需求重開也很多。

DORA 的核心訊號就是：AI 的回報取決於底層組織系統，不是工具本身。如果你的需求定義混亂，AI 只會更快把混亂擴大。

—

▍那 10% 的槓桿是什麼？

Kent Beck 說的那 10%，我認為對 PM 來說就是「定義問題」的能力。

不是「能不能用 AI 工具」的問題。是「你能不能把判斷力變成 Agent 可執行的指令」的問題。

▍我的實驗：把 PM 判斷力編碼成 Git Repo
