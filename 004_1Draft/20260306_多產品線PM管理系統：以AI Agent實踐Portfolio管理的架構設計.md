# 多產品線 PM 管理系統：以 AI Agent 實踐 Portfolio 管理的架構設計

> 本文描述一個真實運作的 PM 管理 repo 架構，用於協調多個產品線的策略、進度與資源分配。
> 日期：2026-03-06

---

## 一、系統定位與設計目標

### 1.1 核心問題

當一位 PM 需要同時管理多條產品線（如：MI2.0 影像平台、VMS 影像管理系統、SmartSignage 智慧電子看板）時，面臨以下挑戰：

- **視角切換困難**：在個別產品細節與整體 portfolio 策略間頻繁切換
- **資源衝突仲裁**：多產品線同時需要資源，需要明確的優先順序框架
- **重複工作浪費**：各產品線獨立開發類似功能，缺乏共用組件識別機制
- **報告成本高**：定期向上層彙整多產品線狀態，耗費大量時間在資料整理
- **決策脈絡遺失**：過往的取捨判斷與理由散落在會議記錄中，難以追溯

### 1.2 設計目標

設計一個「PM 總管」系統，透過：

1. **明確的角色分工**：PM 總管（Portfolio 視角）vs 子 PM（單一產品視角）
2. **結構化的決策框架**：可執行的優先順序評估流程，而非拍腦袋
3. **技能化的工作流程**：將定期報告、策略分析、優先順序仲裁等任務模組化
4. **可追溯的決策記錄**：重要判斷的理由、脈絡、trade-off 完整留存
5. **AI Agent 輔助執行**：將 PM 總管的視角與決策框架編碼為 AI Agent 可執行的指令

## 二、架構設計原則

### 2.1 角色層次設計

```
PM 總管（pm-head）
├─ 視角：俯瞰多產品線，看策略協同、資源配置、長期效益
├─ 決策權：跨產品線優先順序、資源分配、共同組件抽離
└─ 尊重邊界：子 PM 的在地判斷、domain-expert 的領域知識、tech-lead 的工程實現

子 PM（各自獨立 repo）
├─ MI_PM（MI2.0 影像平台）
├─ VMS_PM（VMS 影像管理系統）
└─ SmartSignagePM（智慧電子看板）
```

**設計考量**：
- 不越級干預：PM 總管只讀取子 PM 的狀態，不直接修改其內容
- 明確能力邊界：哪些決策由 PM 總管主導、哪些只給建議、哪些尊重他人專業

### 2.2 核心決策原則

系統內建五條決策框架（記錄於 `.ai/principles.md`）：

| 原則 | 核心邏輯 | 真實應用場景 |
|------|---------|------------|
| **全局槓桿優先** | 先做 A，因為 B 也會受惠 | MI_PM 和 VMS_PM 都需要數據分析模組，抽共用比各做各的划算 |
| **抽離即複利** | 共同組件投資能跨專案回收 | 某功能抽離多花 2 週，但下個專案省 3 週，效益比 > 1.5 就做 |
| **子 PM 在地判斷優先** | 不越級干預，只在全局衝突時介入 | 尊重子 PM 的優先順序排法，除非影響其他產品線 |
| **短期 × 長期並重** | 不只看急迫性，也看團隊能力投資 | 趕需求 vs 建自動化流程，若短期扛得住就投資長期效益 |
| **好策略壞策略** | 好策略有明確取捨，壞策略什麼都做 | 三個專案都重要？資源有限就明確排序，全做等於全做不好 |

**設計考量**：
- 可執行：每條原則都對應到真實兩難情境的判斷流程
- 可追溯：決策時引用原則，日後可回顧當時為何這樣判斷

### 2.3 技能化工作流程

系統將 PM 總管的常見任務模組化為「技能」（Skills），每個技能包含：
- 觸發條件：何時該啟動這個技能
- 執行流程：step-by-step 的工作步驟
- 品質檢查清單：確保產出符合標準
- 範例與反例：what good/bad looks like

**核心技能清單**：

| 技能名稱 | 觸發時機 | 產出 | 模式 |
|---------|---------|------|------|
| 定期報告彙整 | 週報/月報/季報 | 跨產品線狀態報告（含策略觀點）| hybrid |
| 跨專案策略分析 | 發現資源/效益/衝突 | 共同組件識別、衝突分析 | hybrid |
| 優先順序仲裁 | 多產品線搶資源 | 基於全局槓桿的優先順序建議 | interactive |
| 利害關係人對齊 | 需要向上報告 | 利害關係人分析與對齊方案 | interactive |
| 子 Repo Git 同步 | 檢查各產品線更新 | 批量同步各子 PM 的 git 狀態 | hybrid |
| PPTX 簡報流程 | 製作策略簡報 | 結構化的簡報檔產出與 QA | autonomous |
| 技能擴充 | 發現重複性任務 | 將新任務模組化為技能 | hybrid |
| Copilot 橋接同步 | 系統指令更新 | 確保 AI Agent 指令一致性 | hybrid |

**設計考量**：
- 減少認知負擔：不用每次想「報告怎麼寫」，直接執行技能
- 持續改進：技能可以迭代優化，累積最佳實踐

## 三、目錄結構與資訊流

### 3.1 核心目錄設計

```
PM_Head/
├── 0_portfolio/                 # Portfolio 層級資訊
│   ├── product_matrix.md        # 產品矩陣（狀態、階段、關鍵指標）
│   ├── resource_allocation.md   # 資源分配現況
│   ├── cross_project_conflicts.md # 跨專案衝突與依賴
│   ├── common_components.md     # 共同組件識別
│   ├── strategy_roadmap.md      # 策略路線圖
│   └── stakeholder_map.md       # 利害關係人地圖
│
├── 5_management/                # PM 產出
│   ├── weekly/                  # 週報
│   ├── monthly/                 # 月報
│   ├── quarterly/               # 季報
│   ├── decision_records/        # 決策記錄
│   └── stakeholder_briefs/      # 利害關係人簡報
│
├── projects/                    # 子 PM Repo（各自獨立 git）
│   ├── MI_PM/                   # MI2.0 子 PM
│   ├── VMS_PM/                  # VMS 子 PM
│   └── SmartSignagePM/          # SmartSignage 子 PM
│
├── .ai/                         # AI Agent 設定
│   ├── identity.yaml            # 角色身份卡
│   ├── principles.md            # 決策原則
│   ├── knowledge/               # 術語表、規範
│   ├── interfaces/              # 跨 Repo 協作介面
│   └── memory/                  # 決策記錄
│
├── .github/skills/              # 技能定義
│   ├── _index.yaml              # 技能索引
│   ├── portfolio-report/        # 定期報告技能
│   ├── cross-project-strategy/  # 跨專案分析技能
│   └── priority-arbitrate/      # 優先順序仲裁技能
│
└── AGENTS.md                    # AI Agent 核心指令（單一真實源）
```

### 3.2 資訊流設計

```
[子 PM Repo] → 定期同步 → [0_portfolio/] → 策略分析 → [5_management/]
     ↓                          ↓                           ↓
  狀態、需求              product_matrix.md            週報/月報
  決策記錄              resource_allocation.md         決策記錄
                       cross_project_conflicts.md      利害關係人簡報
```

**同步頻率**：
- 每週一：執行 `sync-portfolio` 更新 portfolio 狀態
- 每週五：執行 `portfolio-report` 產生週報
- 每月末：產生月報
- 每季末：產生季報 + 跨專案策略分析

## 四、AI Agent 整合設計

### 4.1 多 AI 工具橋接架構

系統需要支援多種 AI 工具（Claude Code, GitHub Copilot, Gemini CLI, Codex CLI），設計策略：

```
AGENTS.md（單一真實源）
    ↓ 橋接同步
    ├→ .github/copilot-instructions.md（GitHub Copilot）
    ├→ CLAUDE.md（Claude Code）
    └→ GEMINI.md（Gemini CLI）
```

**設計考量**：
- 單一真實源：AGENTS.md 是權威定義，其他檔案是橋接摘要
- 自動同步：透過 `copilot-sync` 技能維持一致性
- 工具差異：各工具有不同讀取方式，橋接檔案格式需適配

### 4.2 角色身份卡設計

系統使用 `.ai/identity.yaml` 定義 PM 總管的：
- 角色定位（一句話描述）
- 視角與溝通風格
- 核心能力（primary vs secondary）
- 能力邊界（owns vs advises vs defers_to）
- 品質標準（self-check questions）

**設計考量**：
- 明確性：AI Agent 需要知道「我是誰」「我能做什麼」「我不該做什麼」
- 一致性：所有任務執行前先載入身份卡，確保視角一致

### 4.3 技能觸發機制

系統設計三種觸發模式：

| 模式 | 何時觸發 | 範例 |
|------|---------|------|
| **定期觸發** | 週度/月度固定時間 | 每週一同步 portfolio、每週五寫週報 |
| **事件觸發** | 特定事件發生時 | 子 PM 提出資源衝突 → 優先順序仲裁 |
| **人類觸發** | 使用者口語化指令 | 「現在各產品線狀況如何？」→ quick-status |

**設計考量**：
- 降低使用門檻：不用記指令，用自然語言描述需求即可
- 智能匹配：系統根據 `.github/skills/_index.yaml` 的 trigger 條件自動匹配技能

## 五、實踐價值與設計啟示

### 5.1 解決的核心問題

| 問題 | 傳統做法 | 本系統做法 | 價值 |
|------|---------|-----------|------|
| 多產品線狀態彙整 | 手動複製貼上各專案資料 | `portfolio-report` 技能自動彙整 | 週報時間從 2hr → 30min |
| 優先順序拍腦袋 | 憑感覺或先來先做 | `priority-arbitrate` 基於決策框架 | 可解釋、可追溯 |
| 重複造輪子 | 各專案各做各的 | `cross-project-strategy` 識別共用 | 減少 30% 重複開發 |
| 決策脈絡遺失 | 散落在會議記錄 | `.ai/memory/decisions.md` 結構化記錄 | 可查詢、可學習 |
| AI Agent 不知道該怎麼幫 | 每次都要重新解釋 | AGENTS.md 編碼角色與框架 | 一次定義持續受益 |

### 5.2 關鍵設計取捨

**取捨 1：結構 vs 彈性**
- 決策：採用結構化的技能與框架，但保留「未匹配任務依 principles 處理」的彈性
- 理由：結構化降低認知負擔，彈性確保不會遇到新狀況就卡住

**取捨 2：自動化 vs 人工判斷**
- 決策：AI Agent 負責資料彙整與流程執行，關鍵決策仍由人類確認
- 理由：自動化提升效率，人工判斷保留策略思考的價值

**取捨 3：單一 Repo vs 多 Repo**
- 決策：PM 總管與各子 PM 各自獨立 repo，透過 `projects/` 目錄管理
- 理由：各產品線獨立演進，但保留 portfolio 視角的彙整點

**取捨 4：工具鎖定 vs 工具中立**
- 決策：支援多種 AI 工具（Claude, Copilot, Gemini），但指令定義統一
- 理由：不被單一工具綁定，但避免各工具指令不一致的混亂

### 5.3 可遷移的設計模式

本系統的以下設計模式可應用於其他 PM 場景：

1. **角色身份卡**：用 YAML 定義角色的視角、能力、邊界 → 適用於任何需要 AI Agent 協作的角色
2. **技能化工作流程**：將重複性任務模組化為技能 → 適用於任何有 SOP 的工作
3. **決策框架編碼**：將判斷邏輯寫成可執行的框架 → 適用於需要一致性決策的場景
4. **單一真實源 + 橋接**：一份核心指令，多工具橋接 → 適用於多 AI 工具並存環境
5. **Portfolio 層與執行層分離**：總管彙整 vs 子 PM 執行 → 適用於管理多專案的情境

## 六、反思與侷限

### 6.1 系統的前提假設

本系統設計基於以下假設，若不成立則效果打折：

1. **產品線有共通技術棧**：本案中為 Databricks + 領域應用，共用元件才有槓桿價值
2. **子 PM 狀態可結構化讀取**：需要各子 PM repo 有基本的狀態文件結構
3. **決策需要可追溯性**：組織文化重視決策脈絡，而非只看結果
4. **AI Agent 可信任度**：使用者願意讓 AI Agent 執行資料彙整等任務

### 6.2 尚未解決的問題

1. **跨 Repo 依賴版本管理**：子 PM 之間有依賴時，版本同步機制尚不完善
2. **即時狀態同步**：目前週度同步，無法捕捉子 PM 的即時變化
3. **利害關係人多樣性**：不同利害關係人需要不同層次的資訊，報告客製化成本高
4. **團隊協作學習曲線**：需要團隊理解並遵循 repo 結構與技能觸發邏輯

### 6.3 未來可能的演進方向

1. **事件驅動同步**：子 PM 有重大更新時主動通知 PM 總管
2. **智能報告客製化**：根據利害關係人角色自動調整報告詳細度
3. **跨專案影響力模擬**：調整一個專案的資源，預測對其他專案的影響
4. **團隊能力地圖**：記錄各專案累積的技能與知識，作為資源分配參考

## 七、總結

本 PM_Head repo 是一個將「多產品線 Portfolio 管理」編碼為 AI Agent 可執行系統的實踐案例。其核心價值在於：

1. **明確的角色定位**：PM 總管 vs 子 PM 的能力邊界與協作介面
2. **可執行的決策框架**：不是口號，而是有真實兩難情境的判斷邏輯
3. **技能化的工作流程**：將定期報告、策略分析等任務模組化，降低認知負擔
4. **可追溯的決策記錄**：重要判斷的理由、脈絡、trade-off 完整留存
5. **AI Agent 友善設計**：透過 AGENTS.md 與身份卡，讓 AI 理解「我是誰、該做什麼」

**適用場景**：
- 一位 PM 管理多條產品線，需要在 portfolio 視角與執行細節間切換
- 產品線間有共通技術棧或潛在共用組件
- 組織重視決策的可追溯性與策略一致性
- 團隊願意投資結構化的 PM 工作流程

**不適用場景**：
- 產品線完全獨立，無共用元件或協同機會
- 組織文化不重視流程結構化
- PM 更偏好完全自由的即興決策風格
