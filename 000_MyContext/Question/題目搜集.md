# <題目>

▋主線：垂直領域新創，怎麼把資料流做成決策流
- 為什麼很多新創收了很多數據，最後還是做不出真正有用的 AI 產品
- 垂直領域產品最該先定義的，不是模型，而是事件、語意與驗證
- 什麼資料值得長期蒐集，什麼資料只是看起來很多但沒有產品價值
- 能持續做出好商業判斷的產品，背後都不是單一模型，而是穩定的資料循環
- 真正有潛力的 AI-native product，從一開始就不是功能堆疊，而是資料與決策架構設計

▋Business Data Science / Agentic Analytics
- AI Agent 做數據分析 (我高度興趣)
- dashboard、報表、分析流程，怎麼長成可重用的 decision system
- 分析 workflow 怎麼 productize，而不是每次都重做一次
- 什麼樣的分析適合做成 Agent，什麼樣的分析一定要保留人工
- 因果推論 / 情境模擬 / 評估系統，怎麼進入商業分析流程
- Palantir Ontology / semantic layer / metrics layer 在真實產品裡的價值

▋Databricks / Data Platform
- Databricks Cluster 當你按開始運算，底層在忙什麼？
- Databricks CICD / Asset Bundle / workflow 上線經驗
- Databricks Data Engineer Professional 認證考上心得
- Genie / Unity Catalog / Metrics / Agent Bricks 怎麼把分析方法 productize
- Spark 底層
- Databricks 在垂直領域 data product 的角色

▋AWS / 治理 / 可靠性
- AWS 證照考試拆解（若寫，應轉成上線責任與最小必備治理能力）
- Shared Responsibility Model 對資料產品團隊真正意味著什麼
- 哪些 IAM / secrets / audit / guardrail 是分析系統上線前不能欠的
- AI workflow 上線時，資安與可靠性怎麼變成產品問題

▋團隊流程 / 軟體部門小主管
- AI Agent Coding & 團隊開發效率極大化 (我高度興趣)
- 哪些規格、測試、review 流程在 Agent 時代反而更重要
- **[不指責人的事故檢討 (blameless postmortem)](https://www.facebook.com/explainthis.io/posts/809970068657283)**

▋特定領域延伸場景
- 土木 / 建築 / 營造顧問公司如何把資料、流程與 AI 接起來
- 智慧建築 / 車隊 / IoT / 數位看板，哪些場域最適合先長出 data product
- 「紙上作業」和「實際施工」是兩回事：真實場域如何讓資料與產品架構變難
